# Thread Safety Analysis: Lazy Evaluation Hierarchy

## 1. Send/Sync Properties of Each Lazy Type

### RcLazy (`Lazy<'a, A, RcLazyConfig>`)

- **Internal structure**: `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>` (line 178 of `lazy.rs`).
- **Send**: No. `Rc` is `!Send`, so `RcLazy` is `!Send` regardless of `A`.
- **Sync**: No. `Rc` is `!Sync`, and `LazyCell` is `!Sync`.
- **Correctness**: Sound. Single-threaded memoized values should not cross thread boundaries.

### ArcLazy (`Lazy<'a, A, ArcLazyConfig>`)

- **Internal structure**: `Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>` (line 292 of `lazy.rs`).
- **Send**: Yes, when `A: Send + Sync`. `Arc<T>` is `Send` when `T: Send + Sync`. `LazyLock<A>` is `Send` when `A: Send`, and `Sync` when `A: Send + Sync`. The closure is `Box<dyn FnOnce() -> A + Send>`, which is `Send`.
- **Sync**: Yes, when `A: Send + Sync`. `Arc<T>` is `Sync` when `T: Send + Sync`, and `LazyLock` provides internal synchronization.
- **Correctness**: Sound. The `ArcLazy::new` constructor (line 622) requires `f: impl FnOnce() -> A + Send + 'a`, ensuring the closure itself is `Send`. The auto-derived `Send`/`Sync` bounds from `Arc<LazyLock<...>>` are correct.

### Thunk (`Thunk<'a, A>`)

- **Internal structure**: `Box<dyn FnOnce() -> A + 'a>` (line 90 of `thunk.rs`).
- **Send**: Only if the captured closure is `Send`. Since `dyn FnOnce() -> A + 'a` does not include `Send`, `Thunk` is `!Send` by default. If someone constructs a `Thunk` with a `Send` closure, the auto-trait derivation still prevents it from being `Send` because the trait object erases the `Send` bound.
- **Sync**: No. `Box<dyn FnOnce()>` is `!Sync` because it contains a mutable closure (consumed on call).
- **Correctness**: Sound. The doc comment on line 62 of `thunk.rs` correctly states "Not `Send`".

### Trampoline (`Trampoline<A>`)

- **Internal structure**: `Free<ThunkBrand, A>` which uses `Thunk<'static, ...>` internally plus `CatList` of continuations with `Box<dyn Any>` type-erased binds.
- **Send**: Not `Send`. Even though `A: 'static`, the internal `Thunk` captures closures as `Box<dyn FnOnce()>` (without `Send`), and `CatList` uses `Rc` internally.
- **Sync**: Not `Sync` for the same reasons.
- **Correctness**: Sound. The doc states this correctly. The `'static` bound exists for `Box<dyn Any>` type erasure, not for thread safety.

### TryLazy (`TryLazy<'a, A, E, Config>`)

- Wraps `Config::TryLazy<'a, A, E>`. For `RcLazyConfig`, this is `Rc<LazyCell<Result<A, E>, ...>>` (not Send). For `ArcLazyConfig`, this is `Arc<LazyLock<Result<A, E>, ...>>` (Send/Sync when A + E are Send + Sync).
- **Correctness**: Sound. Follows the same pattern as `Lazy`.

### TryThunk (`TryThunk<'a, A, E>`)

- Wraps `Thunk<'a, Result<A, E>>` (line 70 of `try_thunk.rs`).
- **Send/Sync**: Inherits from `Thunk`, so `!Send` and `!Sync`.
- **Correctness**: Sound.

### TryTrampoline (`TryTrampoline<A, E>`)

- Wraps `Trampoline<Result<A, E>>` (line 40 of `try_trampoline.rs`).
- **Send/Sync**: Inherits from `Trampoline`, so `!Send` and `!Sync`.
- **Correctness**: Sound.

**Summary table:**

| Type | Send | Sync | Correct |
|------|------|------|---------|
| RcLazy | No | No | Yes |
| ArcLazy | Yes (when A: Send+Sync) | Yes (when A: Send+Sync) | Yes |
| Thunk | No | No | Yes |
| Trampoline | No | No | Yes |
| TryLazy (Rc) | No | No | Yes |
| TryLazy (Arc) | Yes (when A+E: Send+Sync) | Yes (when A+E: Send+Sync) | Yes |
| TryThunk | No | No | Yes |
| TryTrampoline | No | No | Yes |

## 2. ArcLazy Thread Safety: OnceLock/LazyLock Soundness

`ArcLazy` uses `Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>` (line 292).

`LazyLock` is a standard library type (stabilized in Rust 1.80) that provides thread-safe lazy initialization. Its key properties:

- **Initialization**: `LazyLock::force()` uses an internal `Once` primitive. If two threads call `force()` concurrently, exactly one runs the initializer; the other blocks until initialization completes.
- **Poisoning**: If the initializer panics, `LazyLock` is poisoned. Subsequent calls to `force()` will panic. This matches the documented behavior in the `Lazy` type's doc comment (lines 406-412 of `lazy.rs`).
- **Memory ordering**: `LazyLock` uses the same synchronization primitives as `std::sync::Once`, providing proper happens-before relationships.

The `arc_lazy_fix` function (line 788) uses `Arc<OnceLock<ArcLazy<'a, A>>>` to break the self-referential cycle. The `OnceLock` is populated via `cell.set(result.clone()).ok()` on line 801, which happens before returning the `ArcLazy` to the caller. The closure inside the `ArcLazy` reads from the `OnceLock` via `cell_ref.get().unwrap()`. Since `OnceLock::set` and `OnceLock::get` are properly synchronized, concurrent evaluation is safe, though the `unwrap()` on line 798 could panic if the cell is somehow not set (which the comment asserts cannot happen, and the logic confirms).

**Verdict**: The `LazyLock`-based approach is sound. Standard library synchronization guarantees are correctly leveraged.

## 3. Thunk: Send Properties with Rc vs Arc Captures

`Thunk<'a, A>` wraps `Box<dyn FnOnce() -> A + 'a>` (line 90 of `thunk.rs`). The trait object does not include `Send`, so:

- A `Thunk` capturing `Rc<T>` is `!Send`, which is correct.
- A `Thunk` capturing `Arc<T>` is still `!Send`, because the `dyn FnOnce()` trait object erases the `Send` marker. The type system is conservative here.

This is a deliberate design decision. There is no `SendThunk` variant. If users need a `Send` deferred computation, they should use:
- `ArcLazy` for memoized computation.
- A raw `Box<dyn FnOnce() -> A + Send>` for one-shot computation.

The `memoize_arc` method on `Thunk` (line 290 of `thunk.rs`) works around this by eagerly evaluating the thunk and wrapping the result in `ArcLazy`. This is the correct pattern: since `Thunk` cannot be `Send`, we must evaluate it before crossing thread boundaries.

**Verdict**: Correct and conservative. No unsoundness.

## 4. Trampoline: The `'static` Bound and Send

`Trampoline<A>` requires `A: 'static` (line 73 of `trampoline.rs`). This exists because `Free<ThunkBrand, A>` uses `Box<dyn Any>` for type erasure, and `Any` requires `'static`.

However, the `'static` bound does NOT make `Trampoline` `Send`. The internal `Thunk` closures are `Box<dyn FnOnce()>` (without `Send`), and `CatList` uses `Rc` internally. So `Trampoline` is `!Send` and `!Sync`.

The `memoize_arc` method on `Trampoline` (line 293) handles the cross-thread case correctly: it eagerly evaluates the trampoline (consuming it) and wraps the result in `ArcLazy`, requiring `A: Send + Sync`.

**Is `'static` always correct?** Yes, it is necessary for the `Box<dyn Any>` type erasure in `Free`. It does not introduce unsoundness; it merely restricts `Trampoline` to owned, non-borrowed data. This is documented in the type's doc comment and in the comparison table (line 57-63 of `thunk.rs`).

**Verdict**: Correct. The `'static` bound is necessary for the implementation strategy and does not imply thread safety.

## 5. Rc/Arc Parameterization Consistency

The library uses the `LazyConfig` trait (line 65 of `lazy.rs`) to parameterize the Rc/Arc split:

- `RcLazyConfig`: `Rc<LazyCell<...>>`, thunks are `dyn FnOnce() -> A + 'a`.
- `ArcLazyConfig`: `Arc<LazyLock<...>>`, thunks are `dyn FnOnce() -> A + Send + 'a`.

This pattern is applied consistently across:
- `Lazy<'a, A, Config>` (line 424 of `lazy.rs`).
- `TryLazy<'a, A, E, Config>` (line 44 of `try_lazy.rs`).

The pattern is NOT applied to `Thunk`, `Trampoline`, `TryThunk`, or `TryTrampoline`. These types are always `!Send`:

- `Thunk` uses `Box<dyn FnOnce()>` without Send. There is no `SendThunk` or `Thunk<Config>` variant.
- `Trampoline` uses `Free<ThunkBrand, A>` which internally uses `Thunk` and `Rc`-based `CatList`.
- `TryThunk` wraps `Thunk`.
- `TryTrampoline` wraps `Trampoline`.

**Consistency assessment**: The Rc/Arc split is consistent for the memoized types (`Lazy`, `TryLazy`) through `LazyConfig`. The non-memoized types (`Thunk`, `Trampoline` and their `Try` variants) do not have Arc variants, which is a design choice rather than an inconsistency. The rationale is clear: `Thunk` and `Trampoline` are consumed on evaluation (not shared), so reference counting is not needed.

**Potential gap**: There is no `SendThunk` type. Users who want a `Send` deferred, non-memoized computation must use `Box<dyn FnOnce() -> A + Send>` directly. This is arguably fine since `Thunk`'s value proposition is HKT compatibility, and a `Send` variant would require a separate brand and separate trait impls.

## 6. SendDeferrable: Necessity and Value

`SendDeferrable<'a>` (in `send_deferrable.rs`) mirrors `Deferrable<'a>` but requires `f: impl FnOnce() -> Self + Send + Sync + 'a` (line 42).

**Implementations found**:
- `Lazy<'a, A, ArcLazyConfig>` implements `SendDeferrable` (line 847 of `lazy.rs`).
- `TryLazy<'a, A, E, ArcLazyConfig>` implements `SendDeferrable` (line 621 of `try_lazy.rs`).

**Is it needed?** The alternative would be `Deferrable + Send` bounds. However, this does not work because:
1. `Deferrable::defer` takes `f: impl FnOnce() -> Self + 'a`, which does not require `Send`.
2. Adding `Send` to the type (e.g., `D: Deferrable + Send`) constrains the result to be `Send`, but does not constrain the closure argument to be `Send`.
3. `SendDeferrable` constrains the closure itself to be `Send + Sync`, which is necessary for constructing `ArcLazy` values where the initializer must be sendable.

**Verdict**: `SendDeferrable` is necessary. It cannot be replaced by `Deferrable + Send` because the `Send` requirement applies to the closure parameter, not just the result type. This is analogous to the `SendCloneableFn` / `CloneableFn` split.

## 7. Concurrent Evaluation of ArcLazy

When two threads call `evaluate()` on the same `ArcLazy` simultaneously:

1. Both threads hold `Arc` clones pointing to the same `LazyLock<A, Box<dyn FnOnce() -> A + Send>>`.
2. Both call `LazyLock::force()`, which internally uses an `Once` synchronization primitive.
3. Exactly one thread "wins" and runs the initializer closure. The other thread blocks on the `Once` barrier.
4. After initialization completes, both threads receive `&A` references into the same allocation.
5. Subsequent calls to `evaluate()` from any thread return the cached value without synchronization overhead (just an atomic load to check initialization status).

**Edge case: panic during initialization**. If the winning thread's initializer panics, the `LazyLock` is poisoned. All waiting threads (and future callers) will panic with the same poison error. This is documented in the `Lazy` type's doc comment (lines 406-412).

**Memory safety**: No data races are possible. `LazyLock` provides the necessary happens-before relationship between the initializer's write and subsequent reads. The `Arc` ensures the allocation lives as long as any reference exists.

**Verdict**: Fully sound under concurrent access.

## 8. Pointer Brands and Lazy Type Variants

The relationship between pointer brands and lazy types:

| Pointer Brand | Lazy Config | Lazy Type | Function Brand |
|--------------|-------------|-----------|----------------|
| `RcBrand` | `RcLazyConfig` | `RcLazy` / `RcTryLazy` | `RcFnBrand = FnBrand<RcBrand>` |
| `ArcBrand` | `ArcLazyConfig` | `ArcLazy` / `ArcTryLazy` | `ArcFnBrand = FnBrand<ArcBrand>` |

The `LazyBrand<Config>` (line 92 of `brands.rs`) parameterizes `Lazy` at the HKT level, enabling generic programming over both variants.

The pointer hierarchy (`Pointer` -> `RefCountedPointer` -> `SendRefCountedPointer`) maps onto the lazy hierarchy as follows:
- `RcBrand` implements `Pointer` and `RefCountedPointer` but not `SendRefCountedPointer`.
- `ArcBrand` implements all three, including `SendRefCountedPointer` (line 172 of `arc_ptr.rs`).

The `FnBrand<P>` type (line 84 of `brands.rs`) is parameterized over a `RefCountedPointer`, allowing functions to be wrapped in either `Rc<dyn Fn>` or `Arc<dyn Fn>`. This is orthogonal to `LazyConfig` but follows the same Rc/Arc duality.

**Consistency**: The pointer brand hierarchy and lazy config hierarchy are separate but parallel abstractions. `LazyConfig` controls the lazy cell implementation; pointer brands control the reference-counting wrapper for shared function objects. Both use the same Rc/Arc split, but they do not directly reference each other. This is appropriate since `Lazy` does not wrap functions in `Rc`/`Arc`; it wraps the lazy cell itself.

## 9. Missing Send/Sync Implementations

**No manual `unsafe impl Send` or `unsafe impl Sync`** exists in any of the lazy types. All Send/Sync derivation is automatic, based on the fields of each struct.

**Potentially missing impls**:

1. **No `RefFunctor` for `LazyBrand<ArcLazyConfig>`**: The doc comment on `ArcLazy::ref_map` (lines 660-662 of `lazy.rs`) explains this: the `RefFunctor` trait does not require `Send` on the mapping function, but `ArcLazy::new` requires it. An inherent method is provided instead. This is not a bug but a documented limitation of the trait design.

2. **No `Deferrable` for `ArcLazy`**: `ArcLazy` implements `SendDeferrable` instead of `Deferrable`, because `Deferrable::defer` does not require `Send` on the closure. This means `ArcLazy` cannot be used generically where `Deferrable` is expected. This is correct behavior but may limit ergonomics.

3. **No `Send` bound on `Thunk`**: As discussed, there is no mechanism to create a `Send` thunk. The `memoize_arc` escape hatch exists for crossing thread boundaries.

**Verdict**: No missing impls that would constitute unsoundness. The limitations are deliberate and documented.

## 10. Unsafe Code and Data Race Potential

**There is zero `unsafe` code in any of the lazy evaluation types.** The grep across all files in `fp-library/src/types/` found no `unsafe` blocks in `lazy.rs`, `thunk.rs`, `trampoline.rs`, `try_lazy.rs`, `try_thunk.rs`, or `try_trampoline.rs`.

All thread safety is delegated to standard library types:
- `Arc<LazyLock<...>>` for `ArcLazy` / `ArcTryLazy`.
- `Arc<OnceLock<...>>` for `arc_lazy_fix`.
- `Rc<LazyCell<...>>` for `RcLazy` / `RcTryLazy`.
- `Rc<OnceCell<...>>` for `rc_lazy_fix`.

Since there is no `unsafe` code and no manual `Send`/`Sync` implementations, data races are impossible unless the standard library types themselves have bugs (which is outside the scope of this analysis).

The only `unsafe`-adjacent concern is the use of `Box<dyn Any>` in the `Free` monad (used by `Trampoline`), which requires `'static`. This uses `downcast` operations that can fail at runtime (returning `Err`) but cannot cause memory unsafety.

**Verdict**: No unsafe code, no data race potential. The implementation is sound.

## Summary of Findings

1. **All Send/Sync derivations are correct.** The library relies entirely on auto-trait derivation from field types, with no manual `unsafe impl`. `RcLazy` is correctly `!Send`, `ArcLazy` is correctly `Send + Sync` when `A: Send + Sync`.

2. **ArcLazy's LazyLock-based approach is sound.** `LazyLock` provides proper synchronization for concurrent evaluation, and the `Arc` wrapper provides correct shared ownership semantics.

3. **Thunk is correctly `!Send`.** The trait object erasure (`Box<dyn FnOnce()>` without `Send`) prevents sending thunks across threads regardless of what they capture. The `memoize_arc` escape hatch is the correct pattern.

4. **Trampoline's `'static` bound is necessary and correct.** It enables `Box<dyn Any>` type erasure but does not grant thread safety. Trampoline is `!Send` due to internal `Rc` and `Thunk` usage.

5. **Rc/Arc parameterization is consistent for memoized types** via `LazyConfig`. Non-memoized types (`Thunk`, `Trampoline`) are always single-threaded, which is a reasonable design choice.

6. **`SendDeferrable` is necessary.** It cannot be replaced by `Deferrable + Send` because the `Send` constraint must apply to the closure argument, not the result type.

7. **Concurrent evaluation of ArcLazy** is fully safe, with proper happens-before semantics from `LazyLock`.

8. **No unsafe code exists** in any lazy type. Thread safety is entirely delegated to `std::sync` primitives.

**Overall assessment**: The thread safety design of the lazy evaluation hierarchy is sound, conservative, and well-documented. There are no unsoundness issues, no missing `Send`/`Sync` impls that would be needed for safety, and no data race potential.
