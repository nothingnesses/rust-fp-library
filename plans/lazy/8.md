# Type Class Law Compliance Analysis for Lazy Types

## Scope

This analysis examines whether the type class instances for `Thunk`, `Lazy`, and `Trampoline` satisfy their required algebraic laws. It covers Functor, Monad, Applicative, RefFunctor, Deferrable, Semigroup, and Monoid instances, and identifies gaps in property-based test coverage.

---

## 1. Functor Laws for Thunk

The Functor laws are:
- **Identity**: `map(id, fa) = fa`
- **Composition**: `map(f . g, fa) = map(f, map(g, fa))`

### Implementation (`ThunkBrand::map`)

```rust
fn map<'a, A: 'a, B: 'a>(
    func: impl Fn(A) -> B + 'a,
    fa: Thunk<'a, A>,
) -> Thunk<'a, B> {
    fa.map(func)
}
```

Which delegates to the inherent `Thunk::map`:

```rust
pub fn map<B: 'a>(self, f: impl FnOnce(A) -> B + 'a) -> Thunk<'a, B> {
    Thunk::new(move || f((self.0)()))
}
```

### Identity Law Analysis

For `map(id, thunk)`, the result is `Thunk::new(move || id(thunk.evaluate()))`, which simplifies to `Thunk::new(move || thunk.evaluate())`. This produces the same value as `thunk.evaluate()` when forced.

**Verdict**: Satisfied (extensionally). The resulting thunk produces the same value as the original. It is not the same object in memory (a new closure is always allocated), but this is expected for a lazy, non-memoized type. Extensional equality (equality of produced values) is the appropriate notion here.

### Composition Law Analysis

For `map(compose(f, g), thunk)`, the result evaluates as `f(g(thunk.evaluate()))`. For `map(f, map(g, thunk))`, the result evaluates as `f((map(g, thunk)).evaluate())` = `f(g(thunk.evaluate()))`.

**Verdict**: Satisfied. Both sides produce identical values for any pure function `f` and `g`.

### Side-Effect Caveat

Since `Thunk` is not memoized, each evaluation re-executes the computation. If the inner thunk performs side effects, `map(id, thunk)` re-wraps and will re-execute, just like the original. Side effects do not break the law because both sides execute the same number of times when forced the same number of times. The law holds under extensional equality of the evaluated result.

## 2. Monad Laws for Thunk

The Monad laws are:
- **Left identity**: `bind(pure(a), f) = f(a)`
- **Right identity**: `bind(m, pure) = m`
- **Associativity**: `bind(bind(m, f), g) = bind(m, |x| bind(f(x), g))`

### Implementation

`Semimonad::bind` delegates to `Thunk::bind`:

```rust
pub fn bind<B: 'a>(self, f: impl FnOnce(A) -> Thunk<'a, B> + 'a) -> Thunk<'a, B> {
    Thunk::new(move || {
        let a = (self.0)();
        let thunk_b = f(a);
        (thunk_b.0)()
    })
}
```

### Left Identity: `bind(pure(a), f) = f(a)`

`bind(pure(a), f)` evaluates as: evaluate `pure(a)` to get `a`, then evaluate `f(a)`.
`f(a)` directly.

When forced, both produce `f(a).evaluate()`. However, `bind(pure(a), f)` wraps the entire sequence in a new `Thunk`, adding one layer of indirection. The evaluated result is the same.

**Verdict**: Satisfied extensionally. Both produce the same value when evaluated.

### Right Identity: `bind(m, pure) = m`

`bind(m, pure)` evaluates as: evaluate `m` to get `a`, then evaluate `pure(a)` to get `a`.
`m` evaluates as: evaluate `m` to get `a`.

**Verdict**: Satisfied extensionally. Same evaluated result.

### Associativity: `bind(bind(m, f), g) = bind(m, |x| bind(f(x), g))`

Left side: Force `m` to get `a`, force `f(a)` to get `b`, force `g(b)` to get `c`.
Right side: Force `m` to get `a`, force `f(a)` to get `b`, force `g(b)` to get `c`.

Both execute the same sequence of computations in the same order.

**Verdict**: Satisfied extensionally.

### Important Note on Evaluation Semantics

The `Semimonad` trait requires `func: impl Fn(A) -> ... + 'a` (note: `Fn`, not `FnOnce`). However, `Thunk::bind` inherently takes `FnOnce`. The HKT-level `Semimonad` implementation calls the inherent `bind` which takes `FnOnce`. This works because the `Fn` closure passed to `Semimonad::bind` is used exactly once in the `Thunk::new` wrapper. The `Fn` bound is more restrictive than needed, but it does not break laws.

## 3. Applicative Laws for Thunk

The Applicative laws are:
- **Identity**: `apply(pure(id), v) = v`
- **Homomorphism**: `apply(pure(f), pure(x)) = pure(f(x))`
- **Interchange**: `apply(u, pure(y)) = apply(pure(|f| f(y)), u)`
- **Composition**: (follows from the above in a lawful Monad)

### Implementation

`Semiapplicative::apply` for `ThunkBrand`:

```rust
fn apply<'a, FnBrand: 'a + CloneableFn, A: 'a + Clone, B: 'a>(
    ff: Thunk<'a, CloneableFn::Of<'a, A, B>>,
    fa: Thunk<'a, A>,
) -> Thunk<'a, B> {
    ff.bind(move |f| fa.map(move |a| f(a)))
}
```

Since `apply` is derived from `bind` and `map`, and those satisfy their laws, the Applicative laws hold by the standard proof that any lawful Monad yields a lawful Applicative when `apply` is defined as `apply ff fa = bind ff (\f -> map f fa)`.

**Verdict**: Satisfied, as a consequence of lawful Monad. This is the standard derivation.

## 4. RefFunctor Laws for Lazy

### What Laws Should RefFunctor Satisfy?

The `RefFunctor` trait documentation in `classes/ref_functor.rs` does **not** explicitly state any laws. However, as a functor-like abstraction, it should reasonably satisfy adapted functor laws:

- **Identity**: `ref_map(|x| x.clone(), fa)` should produce a `Lazy` whose evaluated value equals the original's evaluated value.
- **Composition**: `ref_map(|x| f(g(x)), fa)` should produce the same evaluated value as `ref_map(f, ref_map(g, fa))`.

Note that `ref_map` takes `FnOnce(&A) -> B`, so the identity function is `|x: &A| x.clone()` rather than a pure identity.

### Implementation

```rust
pub fn ref_map<B: 'a>(self, f: impl FnOnce(&A) -> B + 'a) -> Lazy<'a, B, RcLazyConfig> {
    let fa = self.clone();
    let init: Box<dyn FnOnce() -> B + 'a> = Box::new(move || f(fa.evaluate()));
    Lazy(RcLazyConfig::lazy_new(init))
}
```

### Identity Law Analysis

`ref_map(|x: &A| x.clone(), lazy)` creates a new `Lazy` that, when forced, evaluates `lazy` and clones the result. The evaluated value equals the original's evaluated value (assuming `Clone` is lawful, i.e., produces an equal value).

**Verdict**: Satisfied for `Clone` types that maintain equality under cloning.

### Composition Law Analysis

`ref_map(|x| f(&g(x)), lazy)` evaluates `lazy` once and applies both transformations.
`ref_map(f, ref_map(g, lazy))` evaluates `lazy` once (memoized), then `g` is applied to produce an intermediate `Lazy`, which is then evaluated and `f` applied.

Both produce the same final value. However, there is a subtlety: the intermediate `Lazy` from `ref_map(g, lazy)` is itself memoized. Composition allocates an extra `Lazy` cell, but the final computed value is identical.

**Verdict**: Satisfied extensionally.

### Observation: Missing Law Documentation

The `RefFunctor` trait lacks documented laws. This is a documentation gap. Functor-like abstractions should declare their laws explicitly, even if they follow the same pattern as `Functor`.

**Recommendation**: Add law documentation to `RefFunctor`:
```rust
/// ### Laws
///
/// `RefFunctor` instances should satisfy:
/// * Identity: `ref_map(|x| x.clone(), fa)` produces a value equal to `fa`.
/// * Composition: `ref_map(|x| f(&g(x)), fa)` equals `ref_map(f, ref_map(g, fa))`.
```

## 5. Deferrable Laws

### What Laws Should Deferrable Satisfy?

The `Deferrable` trait (`classes/deferrable.rs`) has **no documented laws**. This is a significant documentation gap.

By analogy with Haskell/PureScript, a reasonable law for `defer` is:

- **Transparency**: `defer(|| x)` should be observationally equivalent to `x` (the deferred computation produces the same result as eager construction).

Formally:
- `defer(|| pure(a)).evaluate() = pure(a).evaluate() = a`
- `defer(|| m).evaluate() = m.evaluate()` for any computation `m`.

### Thunk Implementation

```rust
fn defer(f: impl FnOnce() -> Self + 'a) -> Self {
    Thunk::new(move || f().evaluate())
}
```

When evaluated: `defer(|| m).evaluate()` = `(|| m)().evaluate()` = `m.evaluate()`.

**Verdict**: Satisfied. `defer` is transparent for `Thunk`.

### Lazy (RcLazy) Implementation

```rust
fn defer(f: impl FnOnce() -> Self + 'a) -> Self {
    RcLazy::new(move || f().evaluate().clone())
}
```

When evaluated: `defer(|| m).evaluate()` yields `m.evaluate().clone()`. For `Clone` types where cloning preserves equality, this equals `m.evaluate()`.

**Verdict**: Satisfied for types with lawful `Clone`.

### Trampoline Implementation

```rust
fn defer(f: impl FnOnce() -> Self + 'static) -> Self {
    Trampoline::defer(f)
}
```

Which delegates to:
```rust
pub fn defer(f: impl FnOnce() -> Trampoline<A> + 'static) -> Self {
    Trampoline(Free::wrap(Thunk::new(move || f().0)))
}
```

When evaluated, `defer(|| m).evaluate()` runs the deferred construction and then evaluates the result, yielding the same value as `m.evaluate()`.

**Verdict**: Satisfied.

### Recommendation

Document the transparency law for `Deferrable`:
```rust
/// ### Laws
///
/// * Transparency: `defer(|| x).evaluate()` produces the same result as `x.evaluate()`.
```

## 6. Memoization and Law Compliance in Lazy

### Does Memoization Affect Law Compliance?

Memoization changes the *operational* behavior but not the *denotational* behavior, assuming:
1. The computation is pure (no observable side effects).
2. The result type has a well-behaved `Clone` implementation.

For pure computations, memoization is invisible: `lazy.evaluate()` returns the same value regardless of whether it was previously forced. This is consistent with referential transparency.

### Side Effects and Memoization

If the initializer performs side effects (e.g., incrementing a counter, I/O), memoization does affect behavior:

```rust
let counter = Rc::new(RefCell::new(0));
let c = counter.clone();
let lazy = RcLazy::new(move || { *c.borrow_mut() += 1; 42 });
lazy.evaluate(); // counter = 1
lazy.evaluate(); // counter = 1 (memoized, not re-executed)
```

Versus a non-memoized `Thunk`:

```rust
let thunk = Thunk::new(move || { *c.borrow_mut() += 1; 42 });
thunk.evaluate(); // counter = 1
// (Cannot evaluate again; FnOnce consumed)
```

This difference is by design: `Lazy` guarantees at-most-once execution. The functor laws still hold because:
- `RefFunctor::ref_map` creates a new `Lazy` that evaluates the source once (via memoization), then applies the function once.
- The law `ref_map(id_clone, lazy)` produces a value equal to the original.

**Verdict**: Memoization does not break functor laws for pure computations. For effectful computations, the "at most once" semantics is a feature, not a bug, but users must understand that `Lazy` is not referentially transparent when side effects are present.

## 7. Lazy Evaluation and Referential Transparency

### Thunk

`Thunk` wraps `Box<dyn FnOnce() -> A>`. Since `FnOnce` is consumed on evaluation, a `Thunk` can only be evaluated once. This is a linear use pattern. Referential transparency holds if the wrapped function is pure.

If the function captures mutable state, referential transparency is lost:
```rust
let cell = RefCell::new(0);
let thunk = Thunk::new(|| { *cell.borrow_mut() += 1; *cell.borrow() });
// thunk.evaluate() returns 1; but there's no way to call it again.
```

Since `FnOnce` prevents re-evaluation, the referential transparency concern is mitigated.

### Lazy

`Lazy` supports multiple evaluations via shared memoization (`Rc<LazyCell<...>>`). For pure computations, `lazy.evaluate()` always returns the same reference. This is referentially transparent.

For impure computations, the first call executes the side effect and caches the result. Subsequent calls return the cached value without re-executing. This is consistent with call-by-need semantics (as in Haskell), where effects during evaluation are considered "benign" if the result is deterministic.

### Trampoline

`Trampoline` is built on `Free<ThunkBrand, A>`. Each evaluation re-runs the computation (no memoization). Referential transparency holds for pure computations but is not enforced by the type system.

### Summary

| Type | Referential Transparency (Pure) | Referential Transparency (Impure) |
|------|------|------|
| `Thunk` | Yes | N/A (single-use) |
| `Lazy` | Yes | Partial (at-most-once execution) |
| `Trampoline` | Yes | No (re-executes each time) |

#[cfg(test)]
mod law_tests {

    #[quickcheck]
    fn functor_identity(x: i32) -> bool {

        map::<ThunkBrand, _, _>(compose(f, g), pure::<ThunkBrand, _>(x)).evaluate()

            bind::<ThunkBrand, _, _>(pure::<ThunkBrand, _>(a), &f),
            &g,

            == bind::<ThunkBrand, _, _>(pure::<ThunkBrand, _>(a), |x| {
                bind::<ThunkBrand, _, _>(f(x), &g)

    }

## 9. Semigroup/Monoid Laws for Thunk

### Semigroup Implementation

```rust
impl<'a, A: Semigroup + 'a> Semigroup for Thunk<'a, A> {
    fn append(a: Self, b: Self) -> Self {
        Thunk::new(move || Semigroup::append(a.evaluate(), b.evaluate()))
    }
}
```

**Associativity**: `append(append(t1, t2), t3)` evaluates as `append(append(a, b), c)` on the inner values. `append(t1, append(t2, t3))` evaluates as `append(a, append(b, c))`. By the inner type's Semigroup associativity, these are equal.

**Verdict**: Satisfied, assuming the inner type `A` has a lawful `Semigroup`.

### Monoid Implementation

```rust
impl<'a, A: Monoid + 'a> Monoid for Thunk<'a, A> {
    fn empty() -> Self {
        Thunk::new(|| Monoid::empty())
    }
}
```

**Left identity**: `append(empty(), t).evaluate()` = `append(A::empty(), t.evaluate())` = `t.evaluate()` by `A`'s Monoid law.
**Right identity**: `append(t, empty()).evaluate()` = `append(t.evaluate(), A::empty())` = `t.evaluate()` by `A`'s Monoid law.

**Verdict**: Satisfied, assuming the inner type `A` has a lawful `Monoid`.

## 10. Interaction Between Deferrable and Monad

### The Key Question

Does `defer` interact correctly with `bind`? Specifically, does the following hold?

```
defer(|| m).bind(f) ~~= m.bind(f)
```

Where `~~=` denotes extensional equality (same evaluated result).

### Thunk

`defer(|| m)` = `Thunk::new(move || m.evaluate())`.
`defer(|| m).bind(f)` = `Thunk::new(move || { let a = m.evaluate(); f(a).evaluate() })`.
`m.bind(f)` = `Thunk::new(move || { let a = m.evaluate(); f(a).evaluate() })`.

These are structurally identical.

**Verdict**: `defer` and `bind` compose correctly for `Thunk`.

### Lazy

`Lazy` implements `Deferrable` but not `Semimonad`/`Monad`. It only implements `RefFunctor`. So the `defer` + `bind` interaction does not arise for `Lazy`.

However, `defer` + `ref_map` should be examined:

`defer(|| lazy).ref_map(f)`: First, `defer` creates a new `Lazy` that evaluates `lazy` on first access (with `.clone()`). Then `ref_map` creates another `Lazy` that applies `f` to the result.

`lazy.ref_map(f)`: Creates a `Lazy` that evaluates `lazy` and applies `f`.

Both produce the same evaluated value.

**Verdict**: `defer` composes correctly with `ref_map` for `Lazy`.

### Trampoline

`Trampoline::defer(|| m)` wraps the construction in a `Free::wrap(Thunk::new(|| ...))`, adding a trampolining step.

`Trampoline::defer(|| m).bind(f)` evaluates `m` via the trampoline loop, then applies `f`.
`m.bind(f)` evaluates `m`, then applies `f`.

Both produce the same result; `defer` merely adds a level of indirection that is resolved during the trampoline evaluation loop.

**Verdict**: `defer` and `bind` compose correctly for `Trampoline`.
