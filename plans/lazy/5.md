# Lazy Type: Deep Analysis of Memoized Lazy Evaluation

## 1. Memoization Mechanism: How Is the Cached Value Stored?

### Architecture Overview

`Lazy` uses a two-layer architecture:

1. **Outer pointer** (`Rc` or `Arc`): Provides shared ownership so clones share the same cache.
2. **Inner lazy cell** (`LazyCell` or `LazyLock`): Provides interior mutability for one-shot initialization.

The concrete representation, extracted from `LazyConfig` implementations:

```
RcLazyConfig::Lazy<'a, A> = Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>
ArcLazyConfig::Lazy<'a, A> = Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>
```

(See `/home/jessea/Documents/projects/rust-fp-lib/fp-library/src/types/lazy.rs`, lines 157 and 271.)

### Soundness Assessment

The design is sound for the following reasons:

- **No `unsafe` code.** The entire implementation delegates to `std::cell::LazyCell` (stabilized in Rust 1.80) and `std::sync::LazyLock`. All interior mutability is encapsulated within these standard library types, which have been formally verified for soundness.
- **`Rc` prevents `Send`/`Sync`.** `RcLazy` cannot cross thread boundaries because `Rc` is `!Send + !Sync`, which is correct since `LazyCell` is also `!Sync`.
- **`Arc` + `LazyLock` provides thread safety.** `ArcLazy` uses `LazyLock` which uses internal synchronization (a blocking mutex or similar) to ensure exactly-once initialization even under concurrent access.
- **The `FnOnce` closure is consumed** by `LazyCell::force`/`LazyLock::force`, preventing double-execution. The Box is dropped after initialization.

### Potential Concern: Panic During Initialization

If the initializer panics, behavior depends on the underlying cell:
- `LazyCell`: The cell is poisoned; subsequent accesses will re-run the initializer (if it was not consumed) or panic.
- `LazyLock`: The lock is poisoned; subsequent accesses will panic.

The library does not document this behavior. The `TryLazy` variant partially addresses this by catching panics via `catch_unwind`, but `Lazy` itself provides no protection.

**Recommendation:** Document panic behavior explicitly in `Lazy`'s type-level docs. Consider whether a panic-safe variant (wrapping the initializer in `catch_unwind`) would be valuable as a method on `Lazy` itself, not just `TryLazy`.

## 2. Interior Mutability Approach

### Design Via the `LazyConfig` Trait

Rather than using raw `Cell`/`RefCell`/`UnsafeCell`, the library abstracts the memoization strategy through the `LazyConfig` trait (lines 44-148):

```rust
pub trait LazyConfig: 'static {
    type Lazy<'a, A: 'a>: Clone;
    type TryLazy<'a, A: 'a, E: 'a>: Clone;
    type Thunk<'a, A: 'a>: ?Sized;
    type TryThunk<'a, A: 'a, E: 'a>: ?Sized;
    fn lazy_new<'a, A: 'a>(f: Box<Self::Thunk<'a, A>>) -> Self::Lazy<'a, A>;
    fn evaluate<'a, 'b, A: 'a>(lazy: &'b Self::Lazy<'a, A>) -> &'b A;
    // ... try_ variants omitted
}
```

This is a well-executed typeclass-style abstraction. The key insight is that `LazyConfig` bundles four decisions: pointer type, cell type, thunk type, and evaluation method. The two provided configs are:

| Config | Pointer | Cell | Thunk Bound | Thread Safety |
|--------|---------|------|-------------|---------------|
| `RcLazyConfig` | `Rc` | `LazyCell` | `dyn FnOnce() -> A + 'a` | `!Send`, `!Sync` |
| `ArcLazyConfig` | `Arc` | `LazyLock` | `dyn FnOnce() -> A + Send + 'a` | `Send + Sync` |

### Comparison with Alternative Approaches

- **`OnceCell`**: The standard library's `OnceCell` requires the value to be provided externally (`set`), not computed from a closure. `LazyCell` is the closure-taking counterpart and is the correct choice here.
- **`RefCell`**: Would allow mutation but lacks the one-shot semantics. Using `RefCell<Option<A>>` would work but requires runtime borrow checking on every access, not just initialization.
- **`UnsafeCell`**: Would be lower-level and require manual synchronization. The library correctly avoids this.

**Assessment:** Using `LazyCell`/`LazyLock` is the idiomatic Rust approach as of 1.80+. The abstraction through `LazyConfig` is clean and extensible.

## 3. Why Only "Partial" HKT Traits (RefFunctor)? Why Not Full Functor/Monad?

### The Core Problem: `evaluate` Returns `&A`, Not `A`

`Lazy::evaluate` has signature `fn evaluate(&self) -> &A` (line 455). This is fundamentally different from `Thunk::evaluate` which returns `A` by value (consuming `self`).

The `Functor` trait requires:

```rust
fn map<'a, A: 'a, B: 'a>(func: impl Fn(A) -> B + 'a, fa: F<A>) -> F<B>;
```

This signature demands `F<A>` be consumed and a new `F<B>` produced. But `Lazy<A>` cannot provide `A` by value without either:
1. Cloning the cached value (requiring `A: Clone`).
2. Consuming the `Lazy`, which would invalidate all shared clones.

### Why `RefFunctor` Instead

`RefFunctor` (defined in `/home/jessea/Documents/projects/rust-fp-lib/fp-library/src/classes/ref_functor.rs`) has signature:

```rust
fn ref_map<'a, A: 'a, B: 'a>(
    func: impl FnOnce(&A) -> B + 'a,
    fa: Lazy<'a, A>,
) -> Lazy<'a, B>;
```

The function receives `&A` rather than `A`. This matches `Lazy`'s semantics perfectly: force the value, pass a reference to the mapping function, and wrap the result in a new `Lazy`. The implementation (lines 524-531):

```rust
pub fn ref_map<B: 'a>(self, f: impl FnOnce(&A) -> B + 'a) -> Lazy<'a, B, RcLazyConfig> {
    let fa = self.clone();
    let init: Box<dyn FnOnce() -> B + 'a> = Box::new(move || f(fa.evaluate()));
    Lazy(RcLazyConfig::lazy_new(init))
}
```

Note: `self.clone()` is cheap (just an `Rc::clone`, i.e., incrementing a reference count).

### Why `Functor` Cannot Be Lawfully Implemented

Even if we added `A: Clone` and cloned the value out, the resulting `Functor` instance would violate the identity law. Consider:

```rust
// Hypothetical Functor impl
fn map(f: impl Fn(A) -> B, fa: Lazy<A>) -> Lazy<B> {
    Lazy::new(move || f(fa.evaluate().clone()))
}
```

The identity law says `map(id, fa) == fa`. But `map(id, fa)` creates a *new* `Lazy` that, when forced, forces `fa` and clones. The result is a separate memoization cell, not structurally equal to `fa`. While the *values* would be equal (given `PartialEq`), the memoization identity (same underlying `Rc`) would be lost. In a strict interpretation, this is a law violation because forcing the mapped version also forces the original, creating an observable side-effect dependency.

### PureScript's Approach: A Fundamental Difference

PureScript's `Data.Lazy` (from `/home/jessea/Documents/projects/purescript-lazy/src/Data/Lazy.purs`) implements the full Functor/Applicative/Monad/Comonad hierarchy:

```purescript
instance functorLazy :: Functor Lazy where
  map f l = defer \_ -> f (force l)

instance bindLazy :: Bind Lazy where
  bind l f = defer \_ -> force $ f (force l)
```

This works because:
1. PureScript's `Lazy` is a runtime-level concept; `defer` creates a new memoized thunk every time, and `force` evaluates it.
2. The garbage collector handles the lifecycle of the thunk and its captured references.
3. There is no ownership/borrowing distinction; `force` can return the value freely.

In Rust, creating `Lazy::new(move || f(fa.evaluate().clone()))` requires `Clone`, creates a new allocation, and creates an observable coupling between the new and old `Lazy` values.

## 4. Semantic Difference Between Lazy and Thunk

### Beyond Memoization: Three Key Differences

| Aspect | `Thunk<'a, A>` | `Lazy<'a, A, Config>` |
|--------|----------------|----------------------|
| **Memoization** | No; re-executes on every `.evaluate()` | Yes; computes at most once |
| **Evaluation semantics** | Consumes `self` (`fn evaluate(self) -> A`) | Borrows `self` (`fn evaluate(&self) -> &A`) |
| **Cloning** | Not cloneable (wraps `Box<dyn FnOnce>`) | Cloneable; clones share the same cache |

The second difference is arguably more fundamental than the first. `Thunk` is a *linear* computation: you can only run it once because `evaluate` takes ownership. `Lazy` is a *shared* computation: you can evaluate it any number of times from any number of clones.

This ownership distinction is why `Thunk` can implement full `Functor`/`Monad` (it yields `A` by value, consuming the thunk) while `Lazy` can only implement `RefFunctor` (it yields `&A`).

### Conversion Between the Two

The library provides bidirectional conversions:

- `From<Thunk<'a, A>> for Lazy<'a, A, RcLazyConfig>` (line 535): Wraps thunk evaluation in a lazy cell.
- `From<Lazy<'a, A, Config>> for Thunk<'a, A>` (line 248 of thunk.rs): Evaluates and clones (`A: Clone` required).

The `Lazy -> Thunk` direction requires `Clone` because `Thunk::evaluate` must return `A` by value, but `Lazy::evaluate` only provides `&A`.

## 5. Thread Safety

### Current State

The library provides clean separation:

- `RcLazy` (`Lazy<'a, A, RcLazyConfig>`): `!Send`, `!Sync`. Uses `Rc<LazyCell<...>>`.
- `ArcLazy` (`Lazy<'a, A, ArcLazyConfig>`): `Send + Sync` (when `A: Send + Sync`). Uses `Arc<LazyLock<...>>`.

This is correct. `LazyLock` handles the synchronization for concurrent first-access, and `Arc` handles the shared ownership across threads.

### Thread-Safety Test

The test `test_arc_memo_thread_safety` (lines 800-830) spawns 10 threads that all call `.evaluate()` on clones of the same `ArcLazy`, then asserts the counter was incremented exactly once. This properly validates the exactly-once guarantee under concurrent access.

### Missing: `RefFunctor` for `ArcLazyConfig`

Currently, `RefFunctor` is only implemented for `LazyBrand<RcLazyConfig>` (line 705). There is no `RefFunctor` implementation for `LazyBrand<ArcLazyConfig>`. This means `ArcLazy` cannot be ref-mapped through the type class. The inherent method `ref_map` is also only defined on the `Lazy<'a, A, RcLazyConfig>` impl block (line 462).

**Recommendation:** Add `RefFunctor for LazyBrand<ArcLazyConfig>` and an inherent `ref_map` method for `ArcLazy`. The implementation would be analogous, requiring `A: Send + Sync` and `B: Send + Sync` bounds, and using `ArcLazyConfig::lazy_new`.

## 6. Comparison with `std::lazy::Lazy`, `once_cell::Lazy`, and PureScript's Data.Lazy

### vs `std::cell::LazyCell` / `std::sync::LazyLock`

The library's `Lazy` is built directly on top of these types. The key additions are:

1. **Shared ownership via `Rc`/`Arc`.** Bare `LazyCell` is owned by a single location. `Lazy` wraps it in `Rc`/`Arc` to enable clone-based sharing.
2. **`LazyConfig` trait.** Provides a uniform interface over both single-threaded and thread-safe variants.
3. **HKT integration.** `LazyBrand<Config>` enables generic programming via `RefFunctor` and the `Kind` system.
4. **Conversions from `Thunk` and `Trampoline`.** Provides ergonomic interop with the other evaluation types.

### vs `once_cell::Lazy`

`once_cell::Lazy` is essentially the predecessor to `std::cell::LazyCell`. Since the library targets Rust 1.80+, using the standard library types directly is the correct modern choice. No dependency on `once_cell` is needed.

### vs PureScript's `Data.Lazy`

PureScript's `Data.Lazy` implements a much richer type class hierarchy:

| Type Class | PureScript `Lazy` | Rust `Lazy` |
|------------|-------------------|-------------|
| Functor | Yes | No (RefFunctor only) |
| Apply/Applicative | Yes | No |
| Bind/Monad | Yes | No |
| Foldable | Yes | No |
| Traversable | Yes | No |
| Extend/Comonad | Yes | No |
| Semigroup/Monoid | Yes | No |
| Eq/Ord/Show | Yes | No |

The gap is enormous. PureScript's garbage collection and lack of ownership semantics allow `force` to return `a` (not `&a`), making standard `Functor`/`Monad` trivially implementable.

**Key PureScript patterns that cannot directly translate:**

- `instance bindLazy :: Bind Lazy where bind l f = defer \_ -> force $ f (force l)` requires being able to pass the forced value to `f` by value; in Rust this requires `Clone`.
- `instance comonadLazy :: Comonad Lazy where extract = force` requires `force` to return `A`, not `&A`.
- `instance traversableLazy :: Traversable Lazy where traverse f l = defer <<< const <$> f (force l)` requires `force` to yield the value for mapping, plus the ability to wrap the result back in `Lazy` via `defer <<< const`.

## 7. Memory Behavior: Does Memoization Interact Well with Rust's Ownership?

### Cache Lifetime

The cached value lives as long as the last `Rc`/`Arc` clone. This is correct; the cache is shared, and when all references are dropped, both the cached value and the (already consumed) thunk are freed.

### Memory Leak Potential

Because `Lazy` uses `Rc`, reference cycles are possible:

```rust
// Hypothetical cycle
let lazy: RcLazy<RcLazy<i32>> = RcLazy::new(|| {
    // If this captured an Rc to the outer Lazy, we'd have a cycle
});
```

This is a general `Rc` concern, not specific to `Lazy`. The `ArcLazy` variant has the same theoretical risk but is less likely in practice.

### Closure Capture and the Thunk Box

The initializer closure is `Box<dyn FnOnce() -> A + 'a>`. After initialization, the `LazyCell` internally drops the closure and stores only the computed value. This means any captured references are released after initialization, which is good for memory.

However, there is a subtlety: if the `Lazy` is never forced, the closure lives for the entire lifetime of the `Lazy` (and all its clones). This can unintentionally keep large data structures alive. This is a well-known issue with lazy evaluation in general, but worth documenting.

### The `'a` Lifetime and Cache Persistence

The lifetime `'a` on `Lazy<'a, A, Config>` constrains how long the closure can borrow. The cached value type `A` must also satisfy `A: 'a`. This means:

```rust
fn example<'a>(data: &'a [i32]) -> RcLazy<'a, i32> {
    RcLazy::new(move || data.iter().sum())
}
```

The `Lazy` borrows `data` via the closure. The cached value (an `i32`) outlives `'a` trivially. But the `Lazy` itself cannot outlive `'a` because the closure captures `&'a [i32]`. This interaction is correct and well-typed.

## 8. Clone/Copy Semantics

### Clone: Shared Cache

`Clone` is manually implemented (line 407):

```rust
impl<'a, A, Config: LazyConfig> Clone for Lazy<'a, A, Config> where A: 'a {
    fn clone(&self) -> Self {
        Self(self.0.clone())
    }
}
```

For `RcLazyConfig`, `self.0` is `Rc<LazyCell<...>>`, so `.clone()` increments the reference count. For `ArcLazyConfig`, it increments the atomic reference count. In both cases, the clone shares the same underlying `LazyCell`/`LazyLock`.

This means:
- Forcing one clone forces all clones (they share the same cell).
- The value is computed at most once across all clones.
- Dropping one clone does not affect others.

This is precisely the semantic of PureScript's `Lazy` cloning.

### No Copy

`Lazy` is not `Copy` because `Rc` and `Arc` are not `Copy`. This is correct; reference counting requires explicit clone calls to maintain the count invariant.

### No `A: Clone` Requirement for `Lazy::clone`

Importantly, `Lazy<'a, A, Config>: Clone` does NOT require `A: Clone`. Only the pointer (`Rc`/`Arc`) is cloned, not the value. This is a significant advantage over an alternative design where each clone would get its own memoization cell.

## 9. The `'a` Lifetime Parameter: Does It Cause Issues with Memoization?

### How `'a` Interacts with the Design

The lifetime `'a` appears in two places:
1. The `Lazy` struct itself: `Lazy<'a, A, Config>`.
2. The closure type: `dyn FnOnce() -> A + 'a`.

The `'a` enables borrowing in the initializer:

```rust
let local = vec![1, 2, 3];
let lazy = RcLazy::new(|| local.iter().sum::<i32>()); // borrows local
```

### Lifetime Constraint on `LazyConfig`

`LazyConfig: 'static` (line 44). This means the config type itself has no borrowed data, which is correct since `RcLazyConfig` and `ArcLazyConfig` are unit structs.

### Potential Issue: `Deferrable` Requires `A: Clone`

The `Deferrable` implementation for `Lazy` (line 637) requires `A: Clone`:

```rust
impl<'a, A> Deferrable<'a> for Lazy<'a, A, RcLazyConfig> where A: Clone + 'a {
    fn defer(f: impl FnOnce() -> Self + 'a) -> Self {
        RcLazy::new(move || f().evaluate().clone())
    }
}
```

The `.clone()` is needed because `f()` produces a `Lazy<A>` whose `evaluate()` returns `&A`, and we need to store `A` in the new cell. This is an unavoidable consequence of the reference-returning design.

### `'a` and HKT: The Kind Signature

```rust
impl_kind! {
    impl<Config: LazyConfig> for LazyBrand<Config> {
        type Of<'a, A: 'a>: 'a = Lazy<'a, A, Config>;
    }
}
```

This registers `LazyBrand` in the HKT system with signature `type Of<'a, A: 'a>: 'a`, matching the standard `Kind_cdc7cd43dac7585f` trait. The lifetime `'a` flows correctly through the kind machinery.

### Issue: `'static` Operations Are Awkward

For `ArcLazy`, the closure must be `Send`. For `Trampoline` interop, `A: 'static` is required. When combining `Lazy` with `Trampoline`:

```rust
impl<'a, A> From<Trampoline<A>> for Lazy<'a, A, RcLazyConfig> where A: Send {
    fn from(task: Trampoline<A>) -> Self {
        Self::new(move || task.evaluate())
    }
}
```

This works because `Trampoline<A>` is `'static`, which satisfies any `'a` in `Lazy<'a, ...>`. But the reverse direction (`Lazy -> Trampoline`) in trampoline.rs (line 440) requires `Lazy<'static, A, Config>`, restricting the `Lazy` to non-borrowing initializers.

## 10. Should Lazy Support Monad? What Would That Look Like?

### The Fundamental Tension

A `Monad` implementation for `Lazy` would require `bind`:

```rust
fn bind<'a, A: 'a, B: 'a>(
    ma: Lazy<'a, A>,
    f: impl Fn(A) -> Lazy<'a, B> + 'a,
) -> Lazy<'a, B>
```

The signature demands `f` receives `A` by value. But `Lazy::evaluate` returns `&A`. There are three possible approaches:

### Approach 1: Require `A: Clone` (Most Practical)

```rust
impl Semimonad for LazyBrand<RcLazyConfig> {
    fn bind<'a, A: Clone + 'a, B: 'a>(
        ma: Lazy<'a, A>,
        f: impl Fn(A) -> Lazy<'a, B> + 'a,
    ) -> Lazy<'a, B> {
        RcLazy::new(move || f(ma.evaluate().clone()).evaluate().clone())
    }
}
```

Problems:
- Adds `Clone` bound not present in the `Semimonad` trait definition.
- Requires `B: Clone` too (since the inner `evaluate()` returns `&B`).
- These bounds would need to be added to the trait itself or handled via a separate trait.

### Approach 2: RefMonad (New Trait)

Analogous to `RefFunctor`, create `RefSemimonad`:

```rust
trait RefSemimonad: RefFunctor {
    fn ref_bind<'a, A: 'a, B: 'a>(
        ma: Lazy<'a, A>,
        f: impl Fn(&A) -> Lazy<'a, B> + 'a,
    ) -> Lazy<'a, B>;
}
```

Here `f` receives `&A`, matching `Lazy`'s semantics. The implementation:

```rust
fn ref_bind<'a, A: 'a, B: 'a>(
    ma: Lazy<'a, A>,
    f: impl FnOnce(&A) -> Lazy<'a, B> + 'a,
) -> Lazy<'a, B> {
    let fa = ma.clone();
    RcLazy::new(move || f(fa.evaluate()).evaluate().clone())
}
```

This still requires `B: Clone` for the same reason. And it creates a new type class hierarchy parallel to the main one.

### Approach 3: Follow PureScript (With Clone)

PureScript's `bind` for `Lazy` is: `bind l f = defer \_ -> force $ f (force l)`.

The Rust translation would be:

```rust
fn bind(ma: Lazy<A>, f: impl Fn(A) -> Lazy<B>) -> Lazy<B>
where A: Clone, B: Clone
{
    RcLazy::new(move || {
        let a = ma.evaluate().clone();
        let lb = f(a);
        lb.evaluate().clone()
    })
}
```

### Recommendation: Add `RefBind` as an Inherent Method, Not a Type Class

Rather than creating a full `RefMonad` hierarchy, add an inherent method:

```rust
impl<'a, A> Lazy<'a, A, RcLazyConfig> where A: 'a {
    pub fn ref_bind<B: 'a>(
        self,
        f: impl FnOnce(&A) -> Lazy<'a, B, RcLazyConfig> + 'a,
    ) -> Lazy<'a, B, RcLazyConfig> {
        let fa = self.clone();
        RcLazy::new(move || {
            let inner = f(fa.evaluate());
            inner.evaluate().clone()
        })
    }
}
```

Wait: this still requires `B: Clone` for the inner unwrap. An alternative that avoids the clone:

```rust
pub fn ref_bind<B: 'a>(
    self,
    f: impl FnOnce(&A) -> Lazy<'a, B, RcLazyConfig> + 'a,
) -> Lazy<'a, B, RcLazyConfig> {
    // Store the intermediate Lazy<B> once computed
    let intermediate: RcLazy<Lazy<'a, B, RcLazyConfig>> = {
        let fa = self.clone();
        RcLazy::new(move || f(fa.evaluate()))
    };
    // Flatten: Lazy<Lazy<B>> -> Lazy<B>
    // But this still needs to unwrap the inner Lazy...
}
```

The fundamental issue remains: flattening `Lazy<Lazy<B>>` to `Lazy<B>` requires either cloning the inner value or breaking the shared cache abstraction.

### Verdict

Full `Monad` support for `Lazy` is not practical in Rust without pervasive `Clone` bounds. The `RefFunctor` approach is the right one. If monadic chaining is needed, the recommended pattern is to convert to `Thunk` (which has full Monad support), perform the monadic operations, and convert back to `Lazy` for memoization:

```rust
let lazy: RcLazy<i32> = RcLazy::new(|| 21);
let thunk = Thunk::from(lazy); // requires Clone
let result = thunk.bind(|x| Thunk::pure(x * 2));
let memoized = RcLazy::from(result);
```

This pattern is already supported by the existing `From` conversions.

## Additional Observations

### Missing Trait Implementations

Compared to PureScript's `Data.Lazy`, the following could be added with `Clone` bounds:

1. **`Semigroup`/`Monoid` for `Lazy`**: `append(a, b) = Lazy::new(|| Semigroup::append(a.evaluate().clone(), b.evaluate().clone()))`. This mirrors PureScript's `instance semigroupLazy` and `Thunk`'s existing impl.
2. **`Eq`/`Ord` for `Lazy`**: By forcing both sides and comparing. Already done in PureScript.
3. **`Foldable` for `LazyBrand`**: `fold_right(f, init, fa) = f(fa.evaluate().clone(), init)`. Mirrors PureScript's `foldableLazy`.

These are low-hanging fruit and would bring `Lazy` closer to parity with both PureScript's `Data.Lazy` and the library's own `Thunk` type.

### The `Evaluable` Gap

`Evaluable` is defined for `ThunkBrand` but not for `LazyBrand`. This is because `Evaluable` requires `Functor` (it extends `Functor`), which `Lazy` cannot implement. However, a `RefEvaluable` variant could be useful:

```rust
trait RefEvaluable: RefFunctor {
    fn ref_evaluate<'a, A: 'a>(fa: Lazy<'a, A>) -> &A;
}
```

This would complete the `Ref*` hierarchy.

### The `LazyConfig` Extensibility

The `LazyConfig` trait is open for extension. A user could provide:
- A `parking_lot`-based config for faster locking.
- A `tokio`-based async config using `tokio::sync::OnceCell`.
- A no-op config that always recomputes (making `Lazy` behave like `Thunk`).

This extensibility is well-designed but currently undocumented.

### `Deferrable` vs `pure`: Semantic Overlap

Both `Deferrable::defer` and `Lazy::pure` create a `Lazy` from a value. The difference:
- `pure(a)`: Wraps `a` directly (no outer computation).
- `defer(f)`: Runs `f` to get a `Lazy<A>`, then flattens (join-like operation).

`Deferrable::defer` for `Lazy` is essentially `join` in the `Monad` sense, but implemented as a standalone operation because full `Monad` is not available.

## Summary of Recommendations

1. **Document panic behavior** in `Lazy` type docs, noting the interaction with `LazyCell`/`LazyLock` poisoning.
2. **Add `RefFunctor` for `LazyBrand<ArcLazyConfig>`** and a corresponding `ref_map` inherent method on `ArcLazy`.
3. **Add `Semigroup`/`Monoid` implementations** for `Lazy` (with `A: Clone` bounds), mirroring the existing `Thunk` implementations.
4. **Add `Foldable`-like functionality** via a `RefFoldable` trait or inherent methods.
5. **Document the `LazyConfig` extensibility** and provide guidance for custom configs.
6. **Consider adding `Eq`/`Ord`/`Debug`** implementations for `Lazy`, delegating to the cached value.
7. **Do not attempt full `Functor`/`Monad`** for `Lazy`; the `RefFunctor` approach is correct for Rust's ownership model. Instead, document the `Lazy -> Thunk -> ... -> Lazy` roundtrip pattern for monadic operations.
8. **Consider a `ref_bind` inherent method** on `RcLazy` and `ArcLazy` for convenience, accepting the `Clone` requirement on the result type.
