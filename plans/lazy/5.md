# Analysis 5: Try* Variant Pattern and Error Handling Design

## Overview

The library provides three fallible computation types that mirror the base lazy evaluation hierarchy:

| Base Type | Try* Variant | Wraps |
|-----------|-------------|-------|
| `Thunk<'a, A>` | `TryThunk<'a, A, E>` | `Thunk<'a, Result<A, E>>` |
| `Trampoline<A>` | `TryTrampoline<A, E>` | `Trampoline<Result<A, E>>` |
| `Lazy<'a, A, Config>` | `TryLazy<'a, A, E, Config>` | `Config::TryLazy<'a, A, E>` |

Each Try* type is a newtype wrapper around the base type applied to `Result<A, E>`, with ergonomic combinators for error handling (short-circuiting `bind`, `map_err`, `catch`, etc.).

---

## 1. Is the Try* Pattern the Right Approach?

### Current design: separate newtype wrappers

The Try* types are structurally simple. `TryThunk<'a, A, E>` is literally `TryThunk(Thunk<'a, Result<A, E>>)` (line 68-71 of `try_thunk.rs`), and `TryTrampoline<A, E>` is `TryTrampoline(Trampoline<Result<A, E>>)` (line 38-41 of `try_trampoline.rs`). This is the "specialized newtype" approach.

### Alternative 1: Generic over Result (no wrapper)

Users could simply write `Thunk<'a, Result<A, E>>` directly. This eliminates the wrapper types but sacrifices ergonomics; users would need to manually handle `Result` in every `bind` call, with no short-circuiting semantics.

**Verdict:** The Try* types provide substantial ergonomic value. Writing `try_thunk.bind(|x| ...)` with automatic error propagation is vastly more ergonomic than `thunk.bind(|result| match result { Ok(a) => ..., Err(e) => Thunk::pure(Err(e)) })`.

### Alternative 2: EitherT monad transformer

A monad transformer `EitherT<F, E, A>` would unify all three Try* types into a single generic type. This is the standard approach in Haskell/PureScript. However, implementing monad transformers in Rust faces severe ergonomic barriers:

- The HKT encoding via Brand types already requires verbose type annotations. Adding a transformer layer would compound this significantly.
- Rust's lack of first-class HKT means each transformer instantiation would need its own Brand type anyway.
- The library's use of `Box<dyn FnOnce>` in `Thunk` and `Free` in `Trampoline` means the underlying mechanics differ enough that a unified transformer would either lose features or require complex conditional behavior.

**Verdict:** Monad transformers are theoretically cleaner but practically infeasible in Rust's type system without significant abstraction cost. The current approach is pragmatic.

### Alternative 3: Single type with error parameter

Something like `Computation<'a, A, E = Infallible, Stack = NoStack, Memo = NoMemo>` that unifies all six types. This would be overparameterized and confuse users.

**Verdict:** The current six-type hierarchy (three base, three Try*) is reasonable for Rust. The types have genuinely different characteristics (lifetime requirements, stack safety, memoization), and making the error channel explicit through separate types is clear and discoverable.

### Assessment

The Try* pattern is **a reasonable design for Rust**. It duplicates some code but provides clear ergonomics and discoverability. The main risk is maintenance burden from duplication, which is analyzed below.

## 2. Code Duplication Between Base Types and Try* Variants

### Quantitative comparison

| File | Lines | Public methods (inherent) |
|------|-------|--------------------------|
| `thunk.rs` | 1,083 | 7 (`new`, `pure`, `defer`, `bind`, `map`, `evaluate`, `memoize`, `memoize_arc`) |
| `try_thunk.rs` | 2,285 | 15 (`new`, `pure`, `ok`, `err`, `defer`, `bind`, `map`, `map_err`, `catch`, `evaluate`, `lift2`, `then`, `memoize`, `memoize_arc`, `catch_unwind`) |
| `trampoline.rs` | 922 | 11 (`pure`, `new`, `defer`, `bind`, `map`, `evaluate`, `memoize`, `memoize_arc`, `lift2`, `then`, `tail_rec_m`, `arc_tail_rec_m`) |
| `try_trampoline.rs` | 1,195 | 14 (`ok`, `err`, `new`, `defer`, `map`, `map_err`, `bind`, `catch`, `lift2`, `then`, `tail_rec_m`, `arc_tail_rec_m`, `evaluate`, `memoize`, `memoize_arc`, `catch_unwind`) |
| `lazy.rs` | 2,168 | 8 (`evaluate`, `new`, `pure`, `ref_map` x2 for Rc/Arc) |
| `try_lazy.rs` | 976 | 10 (`evaluate`, `new`, `ok`, `err`, `map`, `map_err` x2 for Rc/Arc, `catch_unwind`) |

`TryThunk` is more than double the size of `Thunk` (2,285 vs 1,083 lines). The excess comes from:

1. **Error-specific methods** (`ok`, `err`, `map_err`, `catch`, `catch_unwind`): ~150 lines of unique functionality.
2. **Duplicated HKT impls**: TryThunk has full Functor/Pointed/Lift/Semiapplicative/Semimonad/MonadRec/Foldable implementations for `TryThunkErrAppliedBrand<E>` (~600 lines), plus Functor/Pointed/Lift/Semiapplicative/Semimonad/MonadRec/Foldable for `TryThunkOkAppliedBrand<A>` (~another 600 lines), plus Bifunctor/Bifoldable for `TryThunkBrand` (~200 lines). The base `Thunk` has ~400 lines of HKT impls.
3. **From impls**: TryThunk has 6 `From` impls; Thunk has 2. The extra ones handle cross-type conversions with TryLazy, TryTrampoline, and Result.
4. **Duplicated tests**: Similar test patterns but with `Result` wrapping.

### Duplication that could be reduced

The inherent methods (`bind`, `map`, `evaluate`, `defer`, `lift2`, `then`, `memoize`, `memoize_arc`) follow a mechanical pattern: unwrap the inner type, apply the base type's operation (with Result handling for bind/map), rewrap. This pattern is consistent across all three Try* types. A macro could generate these, though the documentation annotations would make macro generation complex.

The HKT impls for `TryThunkErrAppliedBrand<E>` are nearly identical to those for `ThunkBrand`, differing only in the `Result` wrapping/unwrapping in method bodies. This is the most significant duplication.

### Verdict

Approximately **40-50%** of Try* code is structurally duplicated from base types with mechanical `Result` wrapping. A proc macro like `derive_try_variant!` could generate the newtype wrapper, inherent methods, and HKT impls from a base type definition. Whether this is worth the added complexity depends on how often these types change. Given the library is maturing, the duplication is manageable but represents a maintenance risk.

## 3. Consistency of Error Handling Across Try* Types

### Method signatures

All three Try* types provide a consistent core API:

| Method | TryThunk | TryTrampoline | TryLazy |
|--------|----------|---------------|---------|
| `ok(a)` | Yes | Yes | Yes (Rc/Arc) |
| `err(e)` | Yes | Yes | Yes (Rc/Arc) |
| `new(f)` | `FnOnce() -> Result<A,E> + 'a` | `FnOnce() -> Result<A,E> + 'static` | `FnOnce() -> Result<A,E> + 'a` |
| `map(f)` | `FnOnce(A) -> B + 'a` | `FnOnce(A) -> B + 'static` | `FnOnce(&A) -> B + 'a` (note: takes `&A`) |
| `map_err(f)` | `FnOnce(E) -> E2 + 'a` | `FnOnce(E) -> E2 + 'static` | `FnOnce(&E) -> E2 + 'a` (note: takes `&E`) |
| `bind` | Yes | Yes | **No** |
| `catch` | Yes | Yes | **No** |
| `evaluate` | `self -> Result<A,E>` | `self -> Result<A,E>` | `&self -> Result<&A, &E>` |
| `lift2` | Yes | Yes | **No** |
| `then` | Yes | Yes | **No** |
| `memoize` | Yes | Yes | N/A |
| `catch_unwind` | Yes (`E=String`) | Yes (`E=String`) | Yes (`E=String`, RcLazyConfig only) |

### Subtle differences

1. **TryLazy's `map` takes `&A`, not `A`**. This is a fundamental difference. Because `TryLazy` memoizes and returns references, its `map` and `map_err` receive references to the cached value. This requires `E: Clone` for `map` and `A: Clone` for `map_err`, because the new `TryLazy` cell must own its result independently. Lines 200-206 (`try_lazy.rs`): `map` requires `E: Clone + 'a`; lines 231-237: `map_err` requires `A: Clone + 'a`.

2. **TryLazy lacks `bind`, `catch`, `lift2`, `then`**. This is consistent with the base `Lazy` type, which also lacks monadic operations (it only has `RefFunctor`). Since `Lazy` returns `&A`, chaining computations would require cloning at every step, which contradicts the zero-cost philosophy.

3. **Lifetime bounds differ**: TryThunk uses `'a`, TryTrampoline uses `'static`, TryLazy uses `'a`. This mirrors the base types and is correct.

4. **`pure` is deprecated on TryThunk** (line 99-123) in favor of `ok`, but there is no `pure` on TryTrampoline or TryLazy at all. This is a minor inconsistency in history, not in current API.

5. **`catch_unwind` is only on RcLazyConfig for TryLazy** (lines 398-427 of `try_lazy.rs`). There is no `catch_unwind` for `ArcTryLazy`. This is a gap; `ArcLazyConfig` users who want panic safety must construct the closure externally.

### Verdict

The error handling is **largely consistent**, with differences that are justified by the underlying type semantics (reference vs. owned evaluation, lifetime constraints). The main inconsistency is the missing `catch_unwind` for `ArcTryLazy`, which should be added for completeness.

## 4. HKT Integration

### TryThunk: Full HKT participation

`TryThunk` has three Brand types:

- `TryThunkBrand` (Bifunctor, Bifoldable): polymorphic over both `E` and `A`.
- `TryThunkErrAppliedBrand<E>` (Functor, Pointed, Lift, Semiapplicative, Semimonad, MonadRec, Foldable): fixes `E`, varies `A`.
- `TryThunkOkAppliedBrand<A>` (Functor, Pointed, Lift, Semiapplicative, Semimonad, MonadRec, Foldable): fixes `A`, varies `E`.

This is thorough and mirrors the `Result`/`ResultBrand` pattern. The dual-functor approach allows both `map` (over `Ok`) and `map_err` (over `Err`) to be expressed via HKT.

### TryTrampoline: No HKT participation

`TryTrampoline` has **no Brand type** and **no typeclass implementations**. There is no `TryTrampolineBrand` anywhere in the codebase. This mirrors `Trampoline`, which also has no Brand/HKT support because its `'static` requirement and `Free` monad internals make HKT integration impractical.

This is a significant gap for users who want generic programming over fallible stack-safe computations, but it is a deliberate design choice inherited from the base `Trampoline` type.

### TryLazy: Partial HKT participation

`TryLazy` has `TryLazyBrand<E, Config>` with `impl_kind!` (line 609-614 of `try_lazy.rs`), making it a proper HKT type. However, it only participates in:

- `Deferrable` / `SendDeferrable`
- No Functor, Pointed, Semimonad, or Foldable implementations.

This mirrors the base `Lazy`, which only has `RefFunctor`, `Deferrable`, `Foldable`, and `Semigroup`/`Monoid`. The lack of standard `Functor` is because `evaluate` returns `&A`, not `A`.

### Verdict

HKT integration is **correct and consistent with the base types**. TryThunk has the richest HKT support, TryLazy has minimal support, and TryTrampoline has none. The gap in TryTrampoline is inherited from Trampoline and would require significant refactoring to address.

## 5. TryThunk's `lift2`, `then`, `memoize` Additions

### `lift2` (lines 332-338 of `try_thunk.rs`)

```rust
pub fn lift2<B: 'a, C: 'a>(
    self,
    other: TryThunk<'a, B, E>,
    f: impl FnOnce(A, B) -> C + 'a,
) -> TryThunk<'a, C, E> {
    self.bind(move |a| other.map(move |b| f(a, b)))
}
```

This is well-designed: it follows the standard applicative `lift2` pattern, implementing it via `bind` + `map` for short-circuiting semantics. The implementation is identical across all three types (TryThunk, TryTrampoline, Trampoline), which is correct.

**Note:** The function `f` takes owned values `(A, B) -> C`, which is appropriate since `TryThunk::evaluate` consumes `self`. For `TryLazy`, a `lift2` would need to take `(&A, &B) -> C` due to reference semantics, which is why it is absent.

### `then` (lines 368-373 of `try_thunk.rs`)

```rust
pub fn then<B: 'a>(
    self,
    other: TryThunk<'a, B, E>,
) -> TryThunk<'a, B, E> {
    self.bind(move |_| other)
}
```

Clean implementation as `bind` with a discarding closure. Consistent with PureScript's `*>` / Haskell's `>>`. Short-circuits on error from `self`, which is correct.

### `memoize` (lines 392-394 of `try_thunk.rs`)

```rust
pub fn memoize(self) -> RcTryLazy<'a, A, E> {
    RcTryLazy::from(self)
}
```

Delegates to the `From` impl. Clean and correct.

### `memoize_arc` (lines 413-419 of `try_thunk.rs`)

```rust
pub fn memoize_arc(self) -> ArcTryLazy<'a, A, E>
where
    A: Send + Sync + 'a,
    E: Send + Sync + 'a, {
    let result = self.evaluate();
    ArcTryLazy::new(move || result)
}
```

**Design concern:** This eagerly evaluates the thunk to produce the `Result`, then wraps it in `ArcTryLazy`. The eager evaluation is necessary because `TryThunk`'s inner closure is not `Send` (it's `Box<dyn FnOnce() -> Result<A, E> + 'a>`). This is consistent with `Thunk::memoize_arc` and `Trampoline::memoize_arc`, but the name `memoize` is slightly misleading since the computation has already happened. The documentation correctly states this: "The thunk is evaluated eagerly because its inner closure is not `Send`."

### Verdict

All additions are **well-designed**, consistent with their counterparts on the base types and on `TryTrampoline`. The eager evaluation in `memoize_arc` is a necessary consequence of thread-safety requirements and is properly documented.

## 6. Error Type Constraints

### Current constraints

The error type `E` in Try* types has minimal constraints:

- `TryThunk<'a, A, E>`: requires `E: 'a` only.
- `TryTrampoline<A, E>`: requires `E: 'static` only.
- `TryLazy<'a, A, E, Config>`: requires `E: 'a` only.

Additional constraints appear only where needed:

- `catch_unwind`: constrains `E = String` (all three types).
- `TryLazy::map`: requires `E: Clone` because a new cell is created.
- `TryLazy::map_err`: requires `A: Clone` for the same reason.
- `From<TryLazy> for TryThunk`: requires `A: Clone, E: Clone`.

### Assessment

The constraints are **appropriately minimal**. The library does not require `E: std::error::Error` or `E: Display`, which is correct; these would be unnecessarily restrictive for a generic library. The `Clone` requirements on `TryLazy`'s `map`/`map_err` are a consequence of memoization semantics and are well-documented.

The `catch_unwind` methods constraining to `E = String` is pragmatic but somewhat limiting. An alternative would be `E: From<String>` or a generic `catch_unwind_with(f, handler: impl FnOnce(Box<dyn Any>) -> E)`, but the current design is simpler and covers the common case.

### Verdict

Error type constraints are **well-calibrated**: loose enough for generic use, tight enough where semantics require it.

## 7. Conversion Ergonomics

### From implementations inventory

**TryThunk converts from:**
- `Thunk<'a, A>` (wraps in `Ok`)
- `Lazy<'a, A, Config>` (clones cached value, wraps in `Ok`)
- `TryLazy<'a, A, E, Config>` (clones cached result)
- `TryTrampoline<A, E>` (evaluates, returns result)
- `Result<A, E>` (wraps in pure)

**TryTrampoline converts from:**
- `Trampoline<A>` (wraps in `Ok`)
- `Lazy<'static, A, Config>` (clones cached value, wraps in `Ok`)
- `TryLazy<'static, A, E, Config>` (clones cached result)
- `TryThunk<'static, A, E>` (evaluates thunk)
- `Result<A, E>` (wraps in pure)

**TryLazy converts from:**
- `TryThunk<'a, A, E>` (wraps evaluation in lazy cell)
- `TryTrampoline<A, E>` (wraps evaluation in lazy cell)
- `Lazy<'a, A, RcLazyConfig>` (clones cached value, wraps in `Ok`)
- `Lazy<'a, A, ArcLazyConfig>` (clones cached value, wraps in `Ok`)
- `Result<A, E>` (wraps in lazy cell, both Rc and Arc)

|-----------|----------|---------------|-------------|--------------|

| TryTrampoline | Yes (line 583) | - | Yes (line 284) | **No** |

### Gaps

1. **No `From<TryThunk> for ArcTryLazy`** or **`From<TryTrampoline> for ArcTryLazy`**: Users must go through `memoize_arc()` instead, which eagerly evaluates. This is intentional (non-Send closures), but worth noting.
2. **No direct `From<Thunk> for TryTrampoline`**: Requires two hops. This is a minor gap.
3. **No `From<Trampoline> for TryThunk`**: Also requires two hops.

### Verdict

Conversions are **mostly complete** for practical use. The gaps involving Arc types are forced by thread-safety constraints. The two-hop paths between non-corresponding base/Try types (Thunk to TryTrampoline) are a minor inconvenience but not a significant problem in practice.

## 8. Could Monad Transformers Eliminate Try* Types?

### Theoretical perspective

An `EitherT<F, E, A>` transformer would generalize over the base functor `F`:

```rust
struct EitherT<F: Brand, E, A>(Kind<F, Result<A, E>>);
```

This would unify TryThunk, TryTrampoline (if it had a brand), and TryLazy into a single type. The `bind`, `map`, `catch`, etc., would be implemented generically in terms of `F`'s monad operations.

### Practical barriers in this codebase

1. **Trampoline has no Brand**: `Trampoline` does not participate in the HKT hierarchy, so `EitherT<TrampolineBrand, E, A>` is impossible without adding a brand to Trampoline first.

2. **Lazy is not a Monad**: `Lazy` only supports `RefFunctor`, not `Functor` or `Semimonad`. An `EitherT` over `Lazy` would need special handling since `evaluate` returns `&A`.

3. **Lifetime mismatches**: `Thunk` uses `'a`, `Trampoline` uses `'static`. A unified `EitherT` would need to be generic over lifetimes in ways that Rust's type system does not easily support.

4. **Performance**: The current newtype wrappers are zero-cost (the `TryThunk(Thunk<...>)` wrapper compiles away). A generic transformer would likely also be zero-cost if using associated types, but the type inference burden would be much higher.

5. **Documentation ergonomics**: Users would see `EitherT<ThunkBrand, MyError, i32>` instead of `TryThunk<'a, i32, MyError>`. The latter is clearer.

### Verdict

Monad transformers **cannot practically replace the Try* types** in this codebase. The prerequisites (Brand for Trampoline, Monad for Lazy) do not exist and would require fundamental design changes. Even if they did exist, the ergonomic cost in Rust would be high. The separate Try* types are the right call.

## 9. From Implementations: Completeness and Consistency

### Pattern consistency

All `From` implementations follow consistent patterns:

- **Infallible to fallible**: Wrap in `Ok`. Example: `From<Thunk<'a, A>> for TryThunk<'a, A, E>` (line 537-539) does `TryThunk(eval.map(Ok))`.
- **Memoized to non-memoized**: Clone the cached value. Example: `From<Lazy<'a, A, Config>> for TryThunk<'a, A, E>` (line 483-485) does `TryThunk::new(move || Ok(memo.evaluate().clone()))`.
- **Non-memoized to memoized**: Wrap evaluation in lazy cell. Example: `From<TryThunk<'a, A, E>> for TryLazy<'a, A, E, RcLazyConfig>` (line 260-262) does `Self::new(move || eval.evaluate())`.
- **Result to any**: Wrap in pure/thunk. Always consistent.

### Potential issue: `From<TryLazy> for TryThunk` cloning semantics

Line 515-517 of `try_thunk.rs`:
```rust
fn from(memo: TryLazy<'a, A, E, Config>) -> Self {
    TryThunk::new(move || memo.evaluate().cloned().map_err(Clone::clone))
}
```

This calls `.cloned()` on `Result<&A, &E>` and then `.map_err(Clone::clone)`. The `.cloned()` method on `Result<&A, &E>` only clones the `Ok` variant's inner value and copies the reference for `Err`. The subsequent `.map_err(Clone::clone)` then clones the error. This is correct but slightly redundant; `memo.evaluate().map(|a| a.clone()).map_err(|e| e.clone())` would be equivalent and arguably clearer.

### Consistency check: `From<TryLazy> for TryTrampoline`

Line 617-618 of `try_trampoline.rs`:
```rust
fn from(memo: TryLazy<'static, A, E, Config>) -> Self {
    TryTrampoline(Trampoline::pure(memo.evaluate().cloned().map_err(Clone::clone)))
}
```

Uses the same `.cloned().map_err(Clone::clone)` pattern. **However**, this eagerly evaluates the `TryLazy` during `From`, which means the conversion itself is not lazy. In contrast, `From<TryThunk> for TryTrampoline` wraps in `TryTrampoline::new(move || thunk.evaluate())`, which is lazy. The eager evaluation for `TryLazy` is acceptable (the value is already cached), but the semantics differ subtly from other `From` impls.

### Verdict

The `From` implementations are **complete for practical use and internally consistent** in their patterns. Minor style differences in the cloning approach exist but do not affect correctness.

## Summary of Recommendations

1. **Keep the Try* pattern.** It is pragmatic for Rust and provides genuine ergonomic value. Monad transformers are not viable in this codebase.

2. **Consider a code-generation macro** for the mechanical parts of Try* types (newtype wrapper, inherent methods, basic HKT impls). This would reduce the ~40-50% structural duplication.

3. **Add missing `catch_unwind` for ArcTryLazy.** This is the most notable API inconsistency.

4. **Add tests for TryThunk's `lift2`, `then`, `memoize`, and `memoize_arc`.** These were recently added but lack test coverage.

5. **Add tests for TryLazy's `map` and `map_err`** on both Rc and Arc variants.

6. **Add property tests for Bifunctor/Bifoldable laws** on TryThunk to complement the existing unit tests.

7. **Consider adding `From<Thunk> for TryTrampoline`** and `From<Trampoline> for TryThunk`** as convenience conversions, removing the need for two-hop conversions through intermediate types.
