# Deep Analysis: Thunk Type Implementation

## 1. Internal Representation

### The Enum-less Approach

Unlike many lazy evaluation types that use an enum (e.g., `Pure(A) | Deferred(Box<dyn FnOnce() -> A>)`), `Thunk` uses a single-variant newtype wrapper around a boxed closure:

```rust
pub struct Thunk<'a, A>(Box<dyn FnOnce() -> A + 'a>);  // line 88-91
```

The `pure` constructor (line 137-141) wraps an already-computed value inside a closure:

```rust
pub fn pure(a: A) -> Self where A: 'a {
    Thunk::new(move || a)
}
```

**Analysis**: This design is sound and arguably preferable to an enum for several reasons:

1. **Uniform code paths.** Every `Thunk` is evaluated the same way: call the closure. There is no branch on `Pure` vs `Deferred`. This simplifies `evaluate()`, `map()`, and `bind()` to single-expression functions.

2. **Smaller struct size.** A `Box<dyn FnOnce()>` is two words (pointer + vtable). An enum `Pure(A) | Deferred(Box<dyn FnOnce()>)` would be `max(size_of::<A>(), 2 words) + discriminant`, which is at least 3 words for pointer-sized A, and potentially larger for big A.

3. **No `Pure` variant means no temptation to inspect.** Since `Thunk` is always opaque, there is no way to accidentally "peek" without consuming it.

**Trade-off**: The `pure` constructor pays for a heap allocation (`Box::new`) even for immediate values. For types like `i32`, this is wasteful; the value is moved into a closure that is immediately boxed. An enum with a `Pure(A)` variant would avoid this allocation. However, the cost is one small allocation per `pure` call, which is acceptable given Thunk's intended use as glue code rather than a hot-path data structure.

**Alternative considered**: A `MaybeUninit`-based approach where `Pure` stores the value inline would eliminate the allocation but would require unsafe code and complicate the API. The current design is safe, simple, and correct.

## 2. `evaluate()` Implementation

```rust
pub fn evaluate(self) -> A {   // line 248
    (self.0)()                 // line 249
}
```

**Correctness**: This is trivially correct. The `self` parameter is taken by value, which moves the `Box<dyn FnOnce()>` out of the struct and calls it. The `FnOnce` trait guarantees the closure is called exactly once.

**Performance**: The call goes through a vtable dispatch (dynamic dispatch on `dyn FnOnce`). This is one indirect call. For a single evaluation this is negligible. The real cost concern is in chains (see section 7).

**Edge cases**:
- **Panic in closure**: If the closure panics, the panic propagates normally. The `Box` is dropped during unwinding. No resources leak.
- **Double evaluation**: Impossible by construction; `evaluate` takes `self` by value, so the `Thunk` is consumed.

## 3. Functor/Applicative/Monad Law Compliance

### Functor Laws

The `map` implementation (line 225-230):

```rust
pub fn map<B: 'a>(self, f: impl FnOnce(A) -> B + 'a) -> Thunk<'a, B> {
    Thunk::new(move || f((self.0)()))
}
```

The HKT-level `Functor::map` (line 412-417) delegates to the inherent method:

```rust
fn map<'a, A: 'a, B: 'a>(func: impl Fn(A) -> B + 'a, fa: ...) -> ... {
    fa.map(func)
}
```

**Identity law**: `map(identity, thunk)` creates `Thunk::new(|| identity(thunk.0()))`, which is observationally equivalent to `thunk` since `identity` is a no-op. The result produces the same value. Satisfied.

**Composition law**: `map(f . g, thunk)` creates `Thunk::new(|| (f . g)(thunk.0()))`, which equals `Thunk::new(|| f(g(thunk.0())))`. Meanwhile `map(f, map(g, thunk))` creates `Thunk::new(|| f(Thunk::new(|| g(thunk.0())).0()))` which also reduces to `Thunk::new(|| f(g(thunk.0())))`. Satisfied.

**Note on `Fn` vs `FnOnce`**: The inherent `map` takes `FnOnce`, but the HKT `Functor::map` takes `Fn`. This discrepancy exists because HKT-level traits must work for types like `Vec` where `map` calls `f` multiple times. For `Thunk`, the `Fn` bound is unnecessarily strict but not incorrect; any `Fn` is also `FnOnce`. Users who want `FnOnce` flexibility should use the inherent method.

### Monad Laws

The inherent `bind` (line 194-203):

```rust
pub fn bind<B: 'a>(self, f: impl FnOnce(A) -> Thunk<'a, B> + 'a) -> Thunk<'a, B> {
    Thunk::new(move || {
        let a = (self.0)();
        let thunk_b = f(a);
        (thunk_b.0)()
    })
}
```

**Left identity**: `pure(a).bind(f)` creates a closure that calls `pure(a).0()` to get `a`, then calls `f(a)`, then evaluates the result. This is observationally `f(a).evaluate()`, which matches `f(a)` wrapped in a `Thunk`. Satisfied.

**Right identity**: `thunk.bind(pure)` creates a closure that evaluates `thunk` to get `a`, then calls `Thunk::pure(a)`, then evaluates that to get `a`. Observationally equivalent to `thunk`. Satisfied.

**Associativity**: `thunk.bind(f).bind(g)` and `thunk.bind(|a| f(a).bind(g))` both ultimately produce the same value because closures compose associatively. Satisfied.

The QuickCheck property tests (lines 998-1043) verify these laws empirically across random inputs.

### Applicative Laws

The `Semiapplicative::apply` (line 531-541) and `Lift::lift2` (line 483-493) are both implemented via `bind` and `map`. Since Thunk's monad is lawful, the derived applicative is automatically lawful (this is the standard construction of applicative from monad).

**Potential concern**: The applicative `apply` for Thunk is sequential (evaluates ff before fa). This is correct but means it cannot be used for parallel execution. Since Thunk is `!Send`, this is not a practical concern.

## 4. `tail_rec_m` Implementation: Stack Safety

```rust
fn tail_rec_m<'a, A: 'a, B: 'a>(
    f: impl Fn(A) -> Thunk<'a, Step<A, B>> + Clone + 'a,
    a: A,
) -> Thunk<'a, B> {
    Thunk::new(move || {          // line 616
        let mut current = a;      // line 617
        loop {                    // line 618
            match f(current).evaluate() {  // line 619
                Step::Loop(next) => current = next,  // line 620
                Step::Done(res) => break res,        // line 621
            }
        }
    })
}
```

**Stack safety**: This implementation is genuinely stack-safe. The `loop` replaces recursion with iteration. Each call to `f(current)` returns a `Thunk<Step<A, B>>`, which is immediately evaluated. The key insight is that `f` returns a `Thunk` wrapping a `Step`, not a `Thunk` that chains into more `Thunk`s via `bind`. So each iteration is:
1. Call `f(current)` (one stack frame for `f`).
2. Call `.evaluate()` (one vtable dispatch).
3. Match on the result.

No stack accumulation occurs across iterations.

**Subtle point**: If `f` internally uses `bind` chains (e.g., `f(x) = thunk1.bind(|_| thunk2.bind(|_| ...))`), those inner bind chains are NOT stack-safe. But the `tail_rec_m` loop itself is safe. This is documented in the `MonadRec` trait (line 48-49): "Thunk's bind chains are NOT stack-safe."

**Memory behavior**: Each iteration allocates a new `Thunk` (via `f(current)`) and immediately evaluates and drops it. This is O(1) live heap memory per iteration, with O(n) total allocations. The allocator pressure is real but unavoidable given the `Box<dyn FnOnce>` design.

**No bugs detected.** The implementation is correct and truly stack-safe for the loop itself.

## 5. Thunk vs TryThunk: Duplication Analysis

`TryThunk` (line 68-71) wraps a `Thunk<'a, Result<A, E>>`:

```rust
pub struct TryThunk<'a, A, E>(Thunk<'a, Result<A, E>>);
```

### Code Duplication Inventory

**Duplicated patterns:**
- `new`, `pure`/`ok`, `defer`, `evaluate`, `map`, `bind`, `memoize`, `memoize_arc`: all follow the pattern of delegating to the inner `Thunk` and wrapping/unwrapping `Result`.
- `Functor`, `Pointed`, `Lift`, `Semiapplicative`, `Semimonad`, `MonadRec` for `TryThunkErrAppliedBrand<E>`: structurally identical to Thunk's implementations but with `Result` handling.
- `Semigroup`, `Monoid`: similar but with `Result` matching.
- `Foldable`: switches from direct evaluation to `match` on `Ok`/`Err`.

**Unique to TryThunk:**
- `err()` constructor.
- `map_err()`: transforms the error channel.
- `catch()`: error recovery.
- `lift2()`, `then()`: convenience combinators (these could be derived from bind+map for Thunk too, but are explicitly provided).
- `catch_unwind()`: panic capture.
- `Bifunctor`, `Bifoldable` for `TryThunkBrand`: these have no analogue in Thunk.
- `TryThunkOkAppliedBrand`: functor over the error channel.
- `From<Result<A, E>>`, `From<TryTrampoline>`, `From<TryLazy>`: conversion ecosystem.

### Could They Be Unified?

**Option A: Generic over Result.** One could imagine `Thunk<'a, A>` as the base and TryThunk as a type alias `type TryThunk<'a, A, E> = Thunk<'a, Result<A, E>>` with extension traits. However, this would lose the newtype's ability to implement separate HKT brands and type class instances. The `TryThunkErrAppliedBrand<E>` needs its own `Functor` that maps over `A` in `Result<A, E>`, not over `Result<A, E>` itself.

**Option B: Macro-generated implementations.** The boilerplate in the trait implementations could be reduced with macros, but given the documentation requirements (each method has extensive doc comments), the savings would be minimal.

**Conclusion**: The duplication is justified. `TryThunk` is not merely `Thunk<Result<A,E>>`; it provides a fundamentally different HKT structure (bifunctor, error-channel functor) and error-handling combinators. The newtype pattern is the right choice here.

## 6. Memory Behavior

### Allocation Profile

- `Thunk::new(f)`: one `Box` allocation for the closure. Size depends on captured environment.
- `Thunk::pure(a)`: one `Box` allocation for a `move || a` closure. For small `A`, this is approximately 16-24 bytes (closure size + vtable overhead).
- `map(f, thunk)`: creates a new `Box` closure that captures both `f` and the original `thunk`. The original `thunk`'s `Box` is moved into the new closure, not cloned. So total live allocations grow linearly with chain depth.
- `bind(thunk, f)`: same as `map` in terms of allocation.

### Retained Reference Chains

A chain like `thunk.map(f1).map(f2).map(f3)` creates a nesting of closures:
```
Box(|| f3(Box(|| f2(Box(|| f1(Box(|| value)()))()))()))
```

All four `Box` allocations are live simultaneously until `evaluate()` is called. This is O(n) memory for n operations in a chain. Each `evaluate` step deallocates the outermost `Box` and the closure's captured data.

**No leaks**: Each `Box<dyn FnOnce>` is consumed exactly once by `evaluate()`. The `FnOnce` call moves the closure out of the box, after which the box is freed. Drop ordering is correct: the innermost thunk is evaluated first, its box is freed, then the outer closure continues.

### Comparison with a Reified Chain

If `Thunk` used an enum like `Pure(A) | Map(Box<dyn FnOnce(?) -> A>, Thunk<?, ?>)`, it could potentially fuse operations or enable tail-call optimization. However, the existential type problem (`?` above) makes this impractical without unsafe code or `dyn Any` downcasting.

## 7. The `bind` Chain Stack Overflow Issue

### The Problem

Each `bind` in a chain adds a stack frame when evaluated:

```rust
// thunk.bind(f) creates:
Thunk::new(move || {
    let a = (self.0)();     // evaluate inner thunk (recurse into its closure)
    let thunk_b = f(a);     // call f
    (thunk_b.0)()           // evaluate result
})
```

For a chain of N binds, evaluation requires O(N) stack frames because each outer closure calls the inner closure. At around 10,000-100,000 binds (depending on closure size and stack limit), this will overflow.

### Documentation

The issue is documented in several places:
- The struct-level doc comment (line 56-63): trade-off table mentions "Partial (tail_rec_m only)" for stack safety.
- The inherent `bind` method doc (line 167-169): "Each bind adds to the call stack. For deep recursion, use Trampoline instead."
- The `MonadRec` trait doc (line 47-49): explicitly states bind chains are not stack-safe.

**Assessment**: The documentation is adequate but could be more prominent. A user who reads only the struct-level docs might miss the bind-specific warning. Consider adding a "WARNING" section at the struct level.

### Mitigations

1. **`tail_rec_m`**: The primary mitigation. Users should restructure recursive computations to use `tail_rec_m` instead of `bind` chains.
2. **`Trampoline`**: For truly deep recursion, users should switch to `Trampoline` which is fully stack-safe. The `From<Trampoline>` conversion (line 329-343) makes this interoperable.
3. **No automatic trampolining**: Unlike some FP libraries (e.g., Cats' `Eval` in Scala), Thunk does not automatically trampoline bind chains. This is a deliberate design choice: automatic trampolining would require either an enum-based representation (sacrificing the simple closure model) or heap-allocated continuation chains (sacrificing performance).

### Missing Mitigation: `stacker` or Manual Trampolining

The library could optionally use the `stacker` crate to grow the stack on demand, but this would add a dependency and platform-specific behavior. It is reasonable to leave this as the user's responsibility.

## 8. Lifetime Handling

### `'a` Parameter Usage

The lifetime `'a` on `Thunk<'a, A>` constrains the closure:

```rust
pub struct Thunk<'a, A>(Box<dyn FnOnce() -> A + 'a>);
```

This means the closure can borrow data with lifetime `'a`. The `A: 'a` bound on the `impl` block (line 98) ensures the output type also lives at least as long as `'a`.

**Correctness**: The `'a` parameter is used consistently:
- `new(f: impl FnOnce() -> A + 'a)`: the closure must live at least `'a`.
- `pure(a: A) where A: 'a`: the value must live at least `'a`.
- `map`, `bind`: the transformation function must be `+ 'a`.
- `defer(f: impl FnOnce() -> Thunk<'a, A> + 'a)`: both the outer and inner thunks share the same lifetime.

**HKT integration**: The `impl_kind!` (line 346-350) maps `ThunkBrand::Of<'a, A> = Thunk<'a, A>`, which correctly threads the lifetime through the Kind system. The `A: 'a` bound on the Kind trait ensures type safety.

**Borrowing test** (line 900-905) confirms that references to stack values work correctly:

```rust
let x = 42;
let thunk = Thunk::new(|| &x);
assert_eq!(thunk.evaluate(), &42);
```

**Potential issue**: The `'a` lifetime cannot be `'static` if the closure borrows local data. This is correct behavior, but it means `Thunk<'a, A>` values cannot be stored in long-lived data structures unless `'a = 'static`. This is documented in the trade-off table.

### `From<Trampoline>` Lifetime Restriction

```rust
impl<A: 'static> From<Trampoline<A>> for Thunk<'static, A> { ... }
```

This correctly restricts the conversion to `'static` lifetime, matching `Trampoline`'s requirement.

## 9. Clone/Drop Semantics

### Clone

`Thunk` does **not** implement `Clone`. This is correct because `Box<dyn FnOnce()>` cannot be cloned; `FnOnce` closures may capture non-cloneable state, and calling a clone would violate the "once" guarantee.

**Implication for HKT traits**: The `Semiapplicative::apply` signature requires `A: Clone` (for the value inside the thunk). This is because `apply` is defined generically and types like `Vec` may need to clone values. For `Thunk`, the `A: Clone` bound is not strictly necessary (since each value is used exactly once), but it is required by the trait signature. This is a minor over-constraint.

### Drop

`Thunk` derives `Drop` implicitly through `Box`. When a `Thunk` is dropped without being evaluated:
- The `Box<dyn FnOnce()>` is dropped.
- The closure's captured environment is dropped.
- Any values owned by the closure are freed.

This is correct. No resources leak when a `Thunk` is dropped unevaluated.

**Side effects**: If the closure was intended to perform side effects (e.g., writing to a file), dropping without evaluating silently discards those effects. This is inherent to lazy evaluation and is not a bug.

## 10. TryThunk Enhancement Analysis

### `lift2` (line 332-338)

```rust
pub fn lift2<B: 'a, C: 'a>(
    self, other: TryThunk<'a, B, E>, f: impl FnOnce(A, B) -> C + 'a,
) -> TryThunk<'a, C, E> {
    self.bind(move |a| other.map(move |b| f(a, b)))
}
```

**Correctness**: This is the standard applicative lift2 derived from bind+map. It short-circuits on the first error (from `self.bind`). If `self` fails, `other` is never evaluated.

**Strictness note**: The `f` is applied only after both `self` and `other` succeed. The implementation is correct.

**Comparison with HKT `Lift::lift2`** (line 736-746 in try_thunk.rs): The HKT version has the same implementation. The inherent method provides `FnOnce` flexibility while the HKT version requires `Fn`.

### `then` (line 368-373)

```rust
pub fn then<B: 'a>(self, other: TryThunk<'a, B, E>) -> TryThunk<'a, B, E> {
    self.bind(move |_| other)
}
```

**Correctness**: Standard monadic sequencing (`>>` in Haskell). Evaluates `self` for its effect, discards the result, returns `other`. Short-circuits on error.

**Potential issue**: None. The `_` pattern correctly discards the success value.

### `memoize` (line 392-394)

```rust
pub fn memoize(self) -> RcTryLazy<'a, A, E> {
    RcTryLazy::from(self)
}
```

**Correctness**: Delegates to the `From<TryThunk>` implementation on `RcTryLazy`. This should create a lazy value that evaluates the thunk on first access and caches the `Result`.

### `memoize_arc` (line 413-419)

```rust
pub fn memoize_arc(self) -> ArcTryLazy<'a, A, E>
where A: Send + Sync + 'a, E: Send + Sync + 'a {
    let result = self.evaluate();
    ArcTryLazy::new(move || result)
}
```

**Design note**: This eagerly evaluates the thunk because `Thunk`'s inner closure is not `Send`. The eager evaluation is necessary and correctly documented. The resulting `ArcTryLazy` wraps an already-computed value.

**Potential concern**: If the evaluation panics, the panic occurs at `memoize_arc` call site rather than at lazy access time. This is different from `memoize()` which defers evaluation. The behavioral asymmetry is documented (line 396-399).

### `From<Result<A, E>>` (line 547-563)

```rust
fn from(result: Result<A, E>) -> Self {
    TryThunk(Thunk::pure(result))
}
```

**Correctness**: Wraps a pre-computed result in a thunk. Equivalent to `TryThunk::new(move || result)`. The `Thunk::pure` path is used, which is correct.

### `From<Thunk<'a, A>>` (line 525-539)

```rust
fn from(eval: Thunk<'a, A>) -> Self {
    TryThunk(eval.map(Ok))
}
```

**Correctness**: Maps the infallible thunk's output into `Ok(a)`. The error type `E` is unconstrained, which is correct since no error is produced.

### `From<TryTrampoline>` (line 567-586)

```rust
fn from(tramp: TryTrampoline<A, E>) -> Self {
    TryThunk::new(move || tramp.evaluate())
}
```

**Correctness**: Defers trampoline evaluation to thunk evaluation time. Requires `'static` (line 567), which matches `TryTrampoline`'s requirements.

### `From<TryLazy>` (line 494-518)

```rust
fn from(memo: TryLazy<'a, A, E, Config>) -> Self {
    TryThunk::new(move || memo.evaluate().cloned().map_err(Clone::clone))
}
```

**Note**: This requires both `A: Clone` and `E: Clone` because `TryLazy::evaluate()` returns references. The double clone (`.cloned()` for the Ok value, `.map_err(Clone::clone)` for the Err value) is necessary. This is a correct if slightly expensive conversion.

### `catch_unwind` (line 453-457)

```rust
pub fn catch_unwind(f: impl FnOnce() -> A + UnwindSafe + 'a) -> Self {
    TryThunk::new(move || {
        std::panic::catch_unwind(f).map_err(crate::utils::panic_payload_to_string)
    })
}
```

**Correctness**: Requires `E = String` (constrained by the `impl` block at line 426). The `UnwindSafe` bound is correctly required. Panics are converted to `Err(String)` using a shared utility.

**Limitation**: Only works with `E = String`. If users need a different error type, they must `map_err` afterward.

## Summary of Findings

| Area | Assessment |
|------|-----------|
| Internal representation | Sound. Single-closure design is simpler than enum; the allocation overhead of `pure` is acceptable. |
| `evaluate()` | Trivially correct. Cannot be called twice (moved self). |
| Functor/Monad laws | Satisfied. Verified by QuickCheck property tests. |
| `tail_rec_m` | Truly stack-safe. No bugs. O(1) stack, O(n) allocations. |
| Thunk vs TryThunk duplication | Justified by distinct HKT brands and bifunctor structure. |
| Memory behavior | O(n) live memory for n-deep chains. No leaks. |
| bind chain stack overflow | Documented in three places. Could be more prominent. |
| Lifetime handling | Correct throughout. `'a` is threaded consistently. |
| Clone/Drop | Correct. Non-cloneable by design. Drop is safe. |
| TryThunk enhancements | All implementations are correct. `memoize_arc` eagerly evaluates by necessity. |

### Recommendations

1. **Documentation**: Add a top-level "Limitations" callout about bind chain depth directly in the struct doc (it is mentioned in the table but easy to miss).
2. **`pure` optimization**: Consider adding a `#[inline]` hint to `evaluate()` and `pure()` to help the optimizer eliminate the closure indirection for trivial thunks.
3. **`Fn` vs `FnOnce` in HKT**: The mismatch between inherent methods (`FnOnce`) and HKT traits (`Fn`) is unavoidable but should be documented more explicitly so users know to prefer inherent methods when possible.
4. **TryThunk `Semigroup::append`**: The current implementation evaluates both thunks even if the first one fails (line 1061). This differs from `bind`-based short-circuiting. The behavior matches `Result`'s Semigroup (both sides are evaluated), but the doc comment at line 303-305 says "Short-circuits on error" for `lift2`. The `Semigroup` does not short-circuit, which is consistent with its definition but potentially surprising. Worth documenting.
