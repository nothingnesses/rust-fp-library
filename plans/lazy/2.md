# HKT Integration Analysis: Lazy Evaluation Types

## 1. Overview of the Lazy Hierarchy

The library provides six lazy evaluation types organized into infallible and fallible tiers:

| Type | Memoized | Stack-safe | Lifetime | HKT Brand |
|------|----------|------------|----------|-----------|
| `Thunk<'a, A>` | No | Partial | `'a` | `ThunkBrand` |
| `Lazy<'a, A, Config>` | Yes | N/A | `'a` | `LazyBrand<Config>` |
| `Trampoline<A>` | No | Yes | `'static` | None |
| `TryThunk<'a, A, E>` | No | Partial | `'a` | `TryThunkErrAppliedBrand<E>`, `TryThunkOkAppliedBrand<A>`, `TryThunkBrand` |
| `TryLazy<'a, A, E, Config>` | Yes | N/A | `'a` | `TryLazyBrand<E, Config>` |
| `TryTrampoline<A, E>` | No | Yes | `'static` | None |

## 2. Brand and Kind Integration

### 2.1 ThunkBrand

**Definition** (`brands.rs:194`):
```rust
pub struct ThunkBrand;
```

**Kind implementation** (`thunk.rs:269`):
```rust
impl_kind! {
    for ThunkBrand {
        type Of<'a, A: 'a>: 'a = Thunk<'a, A>;
    }
}
```

**Assessment**: Correct and complete. The `Kind` signature `type Of<'a, A: 'a>: 'a` properly maps to `Thunk<'a, A>`. The lifetime parameter `'a` threads through correctly, allowing `ThunkBrand` to participate fully in the HKT system. The output bound `: 'a` ensures the result type outlives `'a`, which is satisfied since `Thunk<'a, A>` contains `Box<dyn FnOnce() -> A + 'a>`.

### 2.2 LazyBrand

**Definition** (`brands.rs:92`):
```rust
pub struct LazyBrand<Config>(PhantomData<Config>);
```

**Kind implementation** (`lazy.rs:630`):
```rust
impl_kind! {
    impl<Config: LazyConfig> for LazyBrand<Config> {
        type Of<'a, A: 'a>: 'a = Lazy<'a, A, Config>;
    }
}
```

**Assessment**: Correct. The `Config` parameter is correctly hoisted to the brand level (not part of the `Kind::Of` signature), making `LazyBrand<Config>` a proper `* -> *` type constructor after partial application. This allows `LazyBrand<RcLazyConfig>` and `LazyBrand<ArcLazyConfig>` to be used as distinct brands with the same Kind signature.

**Concern**: `LazyConfig` has a `'static` bound (`pub trait LazyConfig: 'static`). This is appropriate because `LazyConfig` is a strategy/configuration type (not data), and the `'static` bound does not restrict the lifetime of the values stored inside `Lazy`.

### 2.3 TryThunkBrand (Bifunctor)

**Definition** (`brands.rs:203`):
```rust
pub struct TryThunkBrand;
```

**Kind implementation** (`try_thunk.rs:879`):
```rust
impl_kind! {
    for TryThunkBrand {
        type Of<'a, E: 'a, A: 'a>: 'a = TryThunk<'a, A, E>;
    }
}
```

**Assessment**: Correct. Uses the two-parameter Kind signature `type Of<'a, E: 'a, A: 'a>: 'a` for bifunctor support. Note the parameter ordering: `Of<'a, E, A>` maps to `TryThunk<'a, A, E>` (error first in the Kind, matching the Haskell/PureScript `Either e a` convention where the "varying" parameter is last).

### 2.4 TryThunkErrAppliedBrand and TryThunkOkAppliedBrand

**Kind implementations** (`try_thunk.rs:408, 940`):
```rust
impl_kind! {
    impl<E: 'static> for TryThunkErrAppliedBrand<E> {
        type Of<'a, A: 'a>: 'a = TryThunk<'a, A, E>;
    }
}

impl_kind! {
    impl<A: 'static> for TryThunkOkAppliedBrand<A> {
        type Of<'a, E: 'a>: 'a = TryThunk<'a, A, E>;
    }
}
```

**Assessment**: Both are correct. They partially apply the bifunctor to create proper `* -> *` type constructors. The `'static` bounds on the fixed type parameters (`E: 'static`, `A: 'static`) are notable; they ensure the partially-applied brand can be used in contexts where the brand itself must be `'static`, but this does impose a restriction on the fixed parameter. This is acceptable for most use cases but prevents using short-lived references as the fixed error or success type in HKT-generic code.

### 2.5 TryLazyBrand

**Kind implementation** (`try_lazy.rs:357`):
```rust
impl_kind! {
    impl<E: 'static, Config: LazyConfig> for TryLazyBrand<E, Config> {
        type Of<'a, A: 'a>: 'a = TryLazy<'a, A, E, Config>;
    }
}
```

**Assessment**: Correct. Both `E` and `Config` are hoisted to the brand level. The `E: 'static` bound mirrors `TryThunkErrAppliedBrand`'s approach. The resulting brand is a proper `* -> *` constructor over the success type.

### 2.6 Trampoline and TryTrampoline: No HKT Integration

Neither `Trampoline` nor `TryTrampoline` has a brand or Kind implementation.

**Why**: `Trampoline<A>` requires `A: 'static + Send` because it is built on `Free<ThunkBrand, A>`, which uses `Box<dyn Any>` for type erasure. The `Any` trait requires `'static`. The library's Kind system uses `type Of<'a, A: 'a>: 'a` with lifetime polymorphism, which is fundamentally incompatible with `'static`-only types.

**Correctness of this decision**: This is the right call. Attempting to implement `Kind` for a hypothetical `TrampolineBrand` would require either:
1. `type Of<'a, A: 'a>: 'a = Trampoline<A>` which would fail because `Trampoline<A>` requires `A: 'static + Send`, but the `Kind` trait only guarantees `A: 'a`.
2. Specializing to `'a = 'static` only, which would break the uniform interface.

## 3. Type Class Instance Analysis

### 3.1 Thunk (ThunkBrand)

| Type Class | Implemented | Correct | Notes |
|------------|-------------|---------|-------|
| `Functor` | Yes | Yes | Delegates to `Thunk::map`. |
| `Pointed` | Yes | Yes | Wraps in `Thunk::pure`. |
| `Lift` | Yes | Yes | Uses `bind`+`map` composition. |
| `ApplyFirst` | Yes | Yes | Blanket default. |
| `ApplySecond` | Yes | Yes | Blanket default. |
| `Semiapplicative` | Yes | Yes | `ff.bind(|f| fa.map(f))` pattern. |
| `Semimonad` | Yes | Yes | Delegates to `Thunk::bind`. |
| `Applicative` | Yes (blanket) | Yes | Via `Pointed + Semiapplicative + ApplyFirst + ApplySecond`. |
| `Monad` | Yes (blanket) | Yes | Via `Applicative + Semimonad`. |
| `MonadRec` | Yes | Yes | Imperative loop inside `Thunk::new`. |
| `Evaluable` | Yes | Yes | Delegates to `Thunk::evaluate`. |
| `Foldable` | Yes | Yes | Single-element fold. |
| `Semigroup` | Yes (on type) | Yes | Combines by evaluating both and appending. |
| `Monoid` | Yes (on type) | Yes | Creates thunk returning `Monoid::empty()`. |
| `Deferrable` | Yes (on type) | Yes | Delegates to `Thunk::defer`. |
| `Traversable` | No | Correct omission | `FnOnce` cannot be cloned; documented. |

**Law compliance**: The `Thunk` monad laws hold because evaluation is deterministic. `pure(a).bind(f)` produces `f(a)` since `pure` wraps the value in a closure that returns it, and `bind` evaluates the closure then applies `f`. Associativity holds because `bind` composes closures linearly.

**Missing instances that could exist**:
- **`FunctorWithIndex`**: Could be implemented with `()` as the index type (single-element container), but this is of marginal utility.

### 3.2 Lazy (LazyBrand)

| Type Class | Implemented | Correct | Notes |
|------------|-------------|---------|-------|
| `RefFunctor` | Yes (RcLazyConfig only) | Yes | Maps by reference over cached value. |
| `Deferrable` | Yes (RcLazyConfig only) | Yes | Flattens `Lazy<Lazy<A>>` to `Lazy<A>`. |
| `SendDeferrable` | Yes (ArcLazyConfig only) | Yes | Thread-safe variant. |
| `Functor` | **No** | See analysis below | |
| `Pointed` | **No** | See analysis below | |
| `Semimonad` | **No** | See analysis below | |

**Why Lazy cannot implement standard Functor**: `Lazy::evaluate` returns `&A` (a reference), not `A` (an owned value). The standard `Functor` trait signature is `fn map<A, B>(f: impl Fn(A) -> B, fa: F<A>) -> F<B>`, which requires producing an owned `B` from an owned `A`. Since `Lazy` only provides `&A`, it cannot satisfy this signature without a `Clone` bound. The `RefFunctor` trait addresses this by using `impl FnOnce(&A) -> B` instead.

**Could Lazy implement Functor with a Clone bound?** In principle, yes: `map(f, lazy) = Lazy::new(|| f(lazy.evaluate().clone()))`. However, this would require `A: Clone` in the Functor implementation, which the trait does not allow. The library correctly introduces `RefFunctor` as a weaker abstraction.

**Missing RefFunctor for ArcLazyConfig**: Only `LazyBrand<RcLazyConfig>` implements `RefFunctor`. There is no `RefFunctor` implementation for `LazyBrand<ArcLazyConfig>`. This appears to be an oversight, since `ArcLazy` has the same `evaluate -> &A` semantics and `ref_map` would work identically (requiring `Send` bounds on the closure).

### 3.3 Trampoline (No Brand)

`Trampoline` provides inherent methods (`pure`, `new`, `bind`, `map`, `defer`, `lift2`, `then`, `tail_rec_m`, `evaluate`) that mirror the type class interfaces without implementing any traits. This is by design due to the `'static` constraint.

**Implements `Deferrable<'static>`**: Yes (`trampoline.rs:460`). This is a type-level instance, not a brand-level instance, using `Deferrable<'static>` directly on `Trampoline<A>`.

**Could Trampoline have a restricted Brand?** Theoretically, one could define a `TrampolineBrand` with `type Of<'a, A: 'a + 'static + Send>: 'a = Trampoline<A>`, but this would require the Kind trait to support additional bounds on `A`, which the current `Kind` system does not. The current approach (inherent methods + `Deferrable` on the type) is the pragmatic choice.

### 3.4 TryThunk

**TryThunkErrAppliedBrand<E> (Functor over success)**:

| Type Class | Implemented | Correct | Notes |
|------------|-------------|---------|-------|
| `Functor` | Yes | Yes | Maps success value. |
| `Pointed` | Yes | Yes | `TryThunk::pure(a)` wraps in `Ok`. |
| `Lift` | Yes | Yes | `bind`+`map` composition. |
| `ApplyFirst` | Yes | Yes | Blanket. |
| `ApplySecond` | Yes | Yes | Blanket. |
| `Semiapplicative` | Yes | Yes | `bind`+`map`. |
| `Semimonad` | Yes | Yes | Short-circuits on `Err`. |
| `MonadRec` | Yes | Yes | Loop with error short-circuit. |
| `Foldable` | Yes | Yes | Folds over `Ok`, ignores `Err`. |
| `Evaluable` | **No** | Debatable | See below. |

**Missing `Evaluable` for `TryThunkErrAppliedBrand<E>`**: The `Evaluable` trait requires `fn evaluate<A>(fa: F<A>) -> A`, meaning it must produce a bare `A`. For `TryThunk<A, E>`, evaluation yields `Result<A, E>`, not `A`. Therefore `Evaluable` cannot be implemented unless `E` has some fallback (e.g., `panic`). The omission is correct.

**TryThunkOkAppliedBrand<A> (Functor over error)**:

| Type Class | Implemented | Correct | Notes |
|------------|-------------|---------|-------|
| `Functor` | Yes | Yes | Maps error value via `map_err`. |
| `Pointed` | Yes | Yes | `TryThunk::err(e)` wraps in `Err`. |
| `Lift` | Yes | Yes | Combines two errors. |
| `ApplyFirst` | Yes | Yes | Blanket. |
| `ApplySecond` | Yes | Yes | Blanket. |
| `Semiapplicative` | Yes | Yes | Applies function in `Err` to value in `Err`. |
| `Semimonad` | Yes | Yes | Short-circuits on `Ok`. |
| `Foldable` | Yes | Yes | Folds over `Err`, ignores `Ok`. |
| `MonadRec` | **No** | See below | |

**Missing `MonadRec` for `TryThunkOkAppliedBrand<A>`**: This is a notable omission. The error-side monad could support `tail_rec_m` with the same loop structure as the success-side version, but short-circuiting on `Ok` instead of `Err`. The implementation would be:
```rust
fn tail_rec_m<'a, E: 'a, E2: 'a>(
    f: impl Fn(E) -> TryThunk<'a, A, Step<E, E2>> + Clone + 'a,
    e: E,
) -> TryThunk<'a, A, E2> {
    TryThunk::new(move || {
        let mut current = e;
        loop {
            match f(current).evaluate() {
                Err(Step::Loop(next)) => current = next,
                Err(Step::Done(res)) => break Err(res),
                Ok(a) => break Ok(a),
            }
        }
    })
}
```

**TryThunkBrand (Bifunctor)**:

| Type Class | Implemented | Correct | Notes |
|------------|-------------|---------|-------|
| `Bifunctor` | Yes | Yes | Maps both `Ok` and `Err`. |
| `Bifoldable` | **No** | Missing | Could fold over both sides. |
| `Bitraversable` | **No** | Missing | Could traverse both sides. |

### 3.5 TryLazy (TryLazyBrand)

| Type Class | Implemented | Correct | Notes |
|------------|-------------|---------|-------|
| `Deferrable` | Yes (RcLazyConfig) | Yes | Flattens nested `TryLazy`. |
| `SendDeferrable` | Yes (ArcLazyConfig) | Yes | Thread-safe variant. |
| `Functor` | **No** | Correct omission | Same `&A` issue as `Lazy`. |
| `RefFunctor` | **No** | Missing | Could return `&Result<A, E>` or `Result<&A, &E>`. |

**Missing `RefFunctor` for `TryLazyBrand`**: `TryLazy::evaluate` returns `Result<&A, &E>`, which does not directly match the `RefFunctor` signature (which expects `&A`). A `RefFunctor` for the success value would require extracting `&A` from `Result<&A, &E>`, which fails on `Err`. This omission is therefore correct; `TryLazy` fundamentally cannot provide a total `ref_map` operation.

### 3.6 TryTrampoline (No Brand)

`TryTrampoline` wraps `Trampoline<Result<A, E>>` and provides inherent methods. Like `Trampoline`, it cannot participate in the HKT system due to the `'static` requirement.

**Implements `Deferrable<'static>`**: Yes (`try_trampoline.rs:341`).

## 4. Lifetime Handling Assessment

### 4.1 The `'a` Parameter in Thunk and Lazy

The `Thunk<'a, A>` and `Lazy<'a, A, Config>` types carry a lifetime `'a` that represents the lifetime of captured references. The Kind mapping `type Of<'a, A: 'a>: 'a` correctly threads this lifetime:

- `A: 'a` ensures the inner type outlives the captured references.
- The output bound `: 'a` ensures the constructed type itself outlives `'a`.
- `Thunk<'a, A>` stores `Box<dyn FnOnce() -> A + 'a>`, which correctly requires `'a`.

This allows writing generic code like:
```rust
fn delayed_double<'a>(x: &'a i32) -> Thunk<'a, i32> {
    Thunk::new(move || *x * 2)
}
```

### 4.2 The `'static` Wall for Trampoline

The `'static` requirement on `Trampoline` creates a hard boundary in the type hierarchy:
- `From<Trampoline<A>> for Lazy<'a, A, RcLazyConfig>` works because `Trampoline<A>` is `'static` and therefore satisfies any `'a`.
- `From<Lazy<'static, A, Config>> for Trampoline<A>` requires `'static` on the `Lazy` lifetime, restricting the conversion.

This asymmetry is correct. `Trampoline` cannot hold borrowed data because `Free` uses `Box<dyn Any>`, which requires `'static`. The conversions correctly encode this constraint in their bounds.

### 4.3 The `Send` Requirement on Trampoline

`Trampoline<A>` requires `A: 'static + Send`. The `Send` requirement comes from the internal `Free` monad's use of type-erased continuations. This means `Trampoline` cannot be used with `Rc`-wrapped values or other `!Send` types. For single-threaded stack-safe recursion with `!Send` types, the user must fall back to `Thunk` with `MonadRec::tail_rec_m`, which provides partial stack safety (the loop itself is stack-safe, but the step function may not be).

## 5. Conversion Graph

The library provides a comprehensive `From` conversion network:

```
Thunk<'a, A> -----> Lazy<'a, A, RcLazyConfig>
Trampoline<A> ----> Lazy<'a, A, RcLazyConfig>
Lazy<'a, A, _> ---> Thunk<'a, A>                  (requires A: Clone)

TryThunk<'a, A, E> ----> TryLazy<'a, A, E, RcLazyConfig>
TryTrampoline<A, E> ---> TryLazy<'a, A, E, RcLazyConfig>  (requires A,E: Send)
TryLazy<'a, A, E, _> --> TryThunk<'a, A, E>               (requires A,E: Clone)

Thunk<'a, A> -----------> TryThunk<'a, A, E>
Lazy<'a, A, _> ----------> TryThunk<'a, A, E>     (requires A: Clone)
TryLazy<'a, A, E, _> ----> TryThunk<'a, A, E>     (requires A,E: Clone)

Trampoline<A> -----------> TryTrampoline<A, E>
Lazy<'static, A, _> -----> TryTrampoline<A, E>    (requires A: Clone+Send)
TryLazy<'static, A, E, _> -> TryTrampoline<A, E>  (requires A,E: Clone+Send)

Lazy<'a, A, RcLazyConfig> -> TryLazy<'a, A, E, RcLazyConfig>
Lazy<'a, A, ArcLazyConfig> -> TryLazy<'a, A, E, ArcLazyConfig>
Lazy<'static, A, _> -> Trampoline<A>              (requires A: Clone+Send)
```

**Assessment**: The conversion graph is well-designed. Information-losing conversions (memoized to non-memoized) require `Clone` because the memoized type returns `&A`, so the non-memoized type must clone to produce `A`. The `'static` restrictions on conversions involving `Trampoline`/`TryTrampoline` are correctly applied.

## 6. Suggestions for Improvement

### 6.1 Add `RefFunctor` for `LazyBrand<ArcLazyConfig>` (Missing Implementation)

Only `LazyBrand<RcLazyConfig>` has a `RefFunctor` instance. `LazyBrand<ArcLazyConfig>` should also have one. The implementation would be nearly identical, needing `Send + Sync` bounds on the closure:

```rust
impl RefFunctor for LazyBrand<ArcLazyConfig> {
    fn ref_map<'a, A: 'a, B: 'a>(
        f: impl FnOnce(&A) -> B + Send + 'a,
        fa: Lazy<'a, A, ArcLazyConfig>,
    ) -> Lazy<'a, B, ArcLazyConfig> {
        ArcLazy::new(move || f(fa.evaluate()))
    }
}
```

Note: This may require adjusting the `RefFunctor` trait's closure bound to support `Send`, or introducing a `SendRefFunctor` trait. The current `RefFunctor` uses `impl FnOnce(&A) -> B + 'a` without `Send`, so the trait itself may need to be examined for whether a parallel variant is appropriate.

### 6.2 Add `MonadRec` for `TryThunkOkAppliedBrand<A>`

For symmetry and completeness, implement tail-recursive monadic computation over the error channel. This would allow stack-safe recursion in error-handling pipelines where the error type evolves through iterations.

### 6.3 Add `Bifoldable` and `Bitraversable` for `TryThunkBrand`

`TryThunkBrand` implements `Bifunctor` but not `Bifoldable` or `Bitraversable`. Since `TryThunk` is a single-element container (either `Ok` or `Err`), these would be straightforward:

```rust
impl Bifoldable for TryThunkBrand {
    fn bifold_right<'a, A: 'a, B: 'a, C: 'a>(
        f: impl Fn(A, C) -> C + 'a,
        g: impl Fn(B, C) -> C + 'a,
        initial: C,
        fab: TryThunk<'a, B, A>,
    ) -> C {
        match fab.evaluate() {
            Ok(b) => g(b, initial),
            Err(a) => f(a, initial),
        }
    }
}
```

### 6.4 Consider a Unified `Evaluable` Trait Variant for Fallible Types

Currently, `Evaluable` requires `fn evaluate(fa) -> A`, which cannot accommodate fallible types. A `TryEvaluable` trait with signature `fn try_evaluate(fa) -> Result<A, E>` would allow `TryThunkErrAppliedBrand` and potentially `TryLazyBrand` to participate in generic evaluation pipelines.

### 6.5 Document the `Send` Requirement on Trampoline More Prominently

The doc comment on `Trampoline` mentions `A: 'static + Send` at the type level, but the "When to Use" comparison table (`thunk.rs:53-58`) does not mention the `Send` requirement. Users choosing between `Thunk` and `Trampoline` should be aware that `Trampoline` excludes `!Send` types (e.g., `Rc`-wrapped closures).

### 6.6 Evaluate Whether `TryLazy` Should Have Brand-Level Type Class Instances

Currently, `TryLazyBrand<E, Config>` has a Kind implementation but no type class instances beyond `Deferrable` (which is on the type, not the brand). If the library intends to support `ref_map` for `TryLazy` (mapping over the success reference), a custom trait like `TryRefFunctor` with signature `fn try_ref_map(f: impl FnOnce(&A) -> B, fa: TryLazy<A, E>) -> TryLazy<B, E>` could be introduced. However, this adds complexity for a narrow use case; the current approach of converting to `TryThunk` for functional operations is reasonable.

### 6.7 Consider `Semigroup`/`Monoid` for `Lazy`

`Thunk` and `TryThunk` both implement `Semigroup` and `Monoid` (when the inner type does). `Lazy` does not, which is an asymmetry. A `Semigroup` for `Lazy` could be:

```rust
impl<'a, A: Semigroup + Clone + 'a> Semigroup for Lazy<'a, A, RcLazyConfig> {
    fn append(a: Self, b: Self) -> Self {
        RcLazy::new(move || Semigroup::append(a.evaluate().clone(), b.evaluate().clone()))
    }
}
```

The `Clone` requirement comes from `evaluate()` returning `&A`. This is a trade-off worth documenting even if the implementation is deferred.

## 7. Law Compliance Summary

### Thunk Monad Laws

1. **Left identity**: `pure(a).bind(f)` = `Thunk::new(|| a).bind(f)` = `Thunk::new(|| { let a = a; f(a).evaluate() })` = `f(a)`. Holds.
2. **Right identity**: `thunk.bind(pure)` = `Thunk::new(|| { let a = thunk.evaluate(); Thunk::pure(a).evaluate() })` = `Thunk::new(|| thunk.evaluate())`. Equivalent to `thunk`. Holds.
3. **Associativity**: Holds because closures compose linearly.

### Thunk Functor Laws

1. **Identity**: `map(id, thunk)` = `thunk.map(id)` = `Thunk::new(|| id(thunk.evaluate()))` = `Thunk::new(|| thunk.evaluate())`. Equivalent to `thunk`. Holds.
2. **Composition**: `map(g . f, thunk)` = `map(g, map(f, thunk))`. Holds because closure composition is associative.

### TryThunk Monad Laws (Error-Applied)

The `TryThunkErrAppliedBrand<E>` monad laws hold with the same reasoning as `Thunk`, with the additional property that `Err` short-circuits (analogous to `Option`'s `None` or `Result`'s `Err`).

### TryThunk Monad Laws (Ok-Applied)

The `TryThunkOkAppliedBrand<A>` monad operates over the `Err` channel. Laws hold symmetrically, with `Ok` short-circuiting.

**Potential law violation concern**: In `TryThunkOkAppliedBrand<A>`'s `Semiapplicative::apply` (`try_thunk.rs:1111`), when both sides are `Ok`, the implementation returns the first `Ok(a)`. This means `apply(Ok(f), Ok(x))` returns `Ok(a)` from the first argument rather than applying `f` to `x`. This is correct for the error-channel semantics: `apply` operates on the `Err` values, and `Ok` values represent "no error" that short-circuits. The implementation is consistent with `Semimonad::bind` which also short-circuits on `Ok`.

### Lazy RefFunctor Laws

`RefFunctor` should satisfy:
1. **Identity**: `ref_map(|x| *x, lazy)` should be equivalent to `lazy` (modulo creating a new `Lazy` wrapper). Holds for `Clone` types.
2. **Composition**: `ref_map(g, ref_map(f, lazy))` = `ref_map(|x| g(&f(x)), lazy)`. Holds.

Note: `RefFunctor` is weaker than `Functor` because the identity law only holds up to re-evaluation and re-wrapping, not structural equality.

## 8. Architectural Observations

### 8.1 The Thunk/Trampoline Split is Well-Motivated

The division between `Thunk` (HKT-compatible, lifetime-polymorphic, not stack-safe) and `Trampoline` (stack-safe, `'static`-only, no HKT) reflects a real tension in Rust's type system. The `Box<dyn Any>` required for type erasure in `Free` forces `'static`, and the library's HKT system requires lifetime polymorphism. These are fundamentally incompatible, so two separate types are the correct solution.

### 8.2 The Config Pattern for Lazy is Elegant

Using `LazyConfig` as a type-level strategy pattern to abstract over `Rc<LazyCell>` vs `Arc<LazyLock>` is a clean design. It avoids code duplication while keeping the brand system clean (one `LazyBrand<Config>` per configuration). The `LazyConfig` trait also bundles the fallible variant (`TryLazy`), which is well-organized.

### 8.3 The Deferrable Trait is Type-Level, Not Brand-Level

`Deferrable` is implemented directly on types (`Thunk<'a, A>`, `Lazy<'a, A, RcLazyConfig>`, etc.) rather than on brands. This is appropriate because `Deferrable` requires `Self: Sized` and operates on concrete types. It cannot be meaningfully abstracted at the brand level because the trait's `defer` method returns `Self`, which is the concrete type.

### 8.4 The Evaluable Trait Enables the Free Monad

`ThunkBrand` implements `Evaluable`, which is used by `Free<ThunkBrand, A>::evaluate()` to run the trampoline loop. This is the critical link between `Thunk` (the base functor) and `Trampoline` (the Free monad over `Thunk`). The Evaluable instance extracts the value from `Thunk` at each step of the trampoline, making the loop work.
