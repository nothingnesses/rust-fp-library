# Lazy Evaluation Hierarchy: Comparison with PureScript Originals

## 1. Deferrable vs PureScript's Control.Lazy Class

### PureScript Original (Control/Lazy.purs)

PureScript's `Lazy` class (lines 10-11) is minimal:

```purescript
class Lazy l where
  defer :: (Unit -> l) -> l
```

It also provides a `fix` combinator (lines 22-25):

```purescript
fix :: forall l. Lazy l => (l -> l) -> l
fix f = go
  where go = defer \_ -> f go
```

And two instances: one for functions (`a -> b`) and one for `Unit`.

### Rust's Deferrable (classes/deferrable.rs)

The Rust `Deferrable` trait (line 54) closely mirrors PureScript:

```rust
pub trait Deferrable<'a> {
    fn defer(f: impl FnOnce() -> Self + 'a) -> Self where Self: Sized;
}
```

**Faithfulness assessment:**

- The core `defer` operation is a direct translation. PureScript uses `Unit -> l`; Rust uses `FnOnce() -> Self`, which is the idiomatic equivalent.
- The lifetime parameter `'a` is an addition forced by Rust's ownership model, absent in PureScript where all values live on the GC heap.
- The `fix` function is deliberately omitted from the trait. The documentation at line 26 explains why: lazy self-reference requires shared ownership (`Rc`/`Arc`) and interior mutability, which are specific to `Lazy`, not to all `Deferrable` types. `Thunk` is consumed on evaluation, making self-referential construction impossible. This is a well-reasoned divergence.
- PureScript's instance for functions (`Lazy (a -> b)`) has no Rust equivalent, which makes sense since Rust functions are not lazy by default.

**Verdict:** Faithful translation with appropriate Rust adaptations. The omission of `fix` from the trait is well-motivated and documented.

## 2. Rust Lazy vs PureScript Data.Lazy

### PureScript's Data.Lazy

PureScript's `Lazy` (Data/Lazy.purs) is a foreign type with two core operations:

```purescript
foreign import data Lazy :: Type -> Type
foreign import defer :: forall a. (Unit -> a) -> Lazy a
foreign import force :: forall a. Lazy a -> a
```

Key properties:
- Single type, no configuration parameters.
- `force` returns owned values (PureScript has no borrowing).
- Memoized: computed at most once.
- Extensive typeclass instances (lines 41-143): Semiring, Ring, CommutativeRing, EuclideanRing, Eq, Eq1, Ord, Ord1, Bounded, Semigroup, Monoid, HeytingAlgebra, BooleanAlgebra, Functor, FunctorWithIndex, Foldable, FoldableWithIndex, Foldable1, Traversable, TraversableWithIndex, Traversable1, Invariant, Apply, Applicative, Bind, Monad, Extend, Comonad, Show, and the `Lazy` class itself.

### Rust's Lazy (types/lazy.rs)

Rust's `Lazy<'a, A, Config>` (line 424) is parameterized by:
- A lifetime `'a`.
- A `LazyConfig` trait that selects the pointer and synchronization strategy.
- Two concrete configurations: `RcLazyConfig` (single-threaded) and `ArcLazyConfig` (thread-safe).

Core operations:
- `Lazy::new(f)` constructs a lazy value.
- `Lazy::evaluate(&self) -> &A` forces computation and returns a **reference**.

**What was gained:**
1. Thread safety is a first-class concern. `ArcLazy` provides `Send + Sync` lazy values with zero API changes beyond the config parameter. PureScript has no equivalent.
2. The `LazyConfig` trait (line 65) is open for extension, allowing third-party lazy cell implementations.
3. Explicit lifetime support means `Lazy` can close over borrowed data.
4. The `Rc`/`Arc` shared ownership is explicit, making the sharing semantics visible.
5. `rc_lazy_fix` and `arc_lazy_fix` (lines 735, 788) provide the `fix` combinator as concrete functions, correctly scoped to the types that can support them.

**What was lost:**
1. `evaluate` returns `&A`, not `A`. This is fundamental: Rust's memoization stores the value inside the `Rc<LazyCell<...>>`, so you get a borrow. PureScript's `force` returns an owned value because the GC handles lifetimes. This means Rust code must clone or work with references, adding friction.
2. Full `Functor`/`Monad` support is missing. `Lazy` only implements `RefFunctor`, not `Functor`. The reason: `Functor::map` expects `fn(A) -> B`, but `Lazy::evaluate` returns `&A`, not `A`. The library created `RefFunctor` (line 62 of ref_functor.rs) specifically for this, where `ref_map` receives `fn(&A) -> B`. This is honest but limits composability with generic code expecting `Functor`.
3. No `Apply`, `Applicative`, `Bind`, or `Monad` instances. In PureScript, `bind l f = defer \_ -> force $ f (force l)`. In Rust, `evaluate` returns `&A`, and `f` would need to accept `&A` and return `Lazy<B>`, which does not match the standard `Monad` trait signature `fn(A) -> M<B>`.
4. No `Traversable` instance.
5. No `Extend` or `Comonad` instances (these traits do not exist in the Rust library at all).
6. No `Eq`, `Ord`, `Show` (Debug) instances that force evaluation. Rust's `Lazy` has no `Debug` impl that shows the value.

## 3. Why Did Rust Split into Three Types?

PureScript's `Lazy` is a single type with full typeclass support. Rust splits into:

| Type | Memoized? | Stack-safe? | Lifetimes | HKT traits |
|------|-----------|-------------|-----------|------------|
| `Lazy<'a, A, Config>` | Yes | N/A | `'a` | `RefFunctor`, `Foldable`, `Semigroup`, `Monoid` |
| `Thunk<'a, A>` | No | No | `'a` | `Functor`, `Monad`, `Foldable`, `Semiapplicative`, `MonadRec` |
| `Trampoline<A>` | No | Yes | `'static` | None (inherent methods only) |

The split is motivated by three fundamental Rust constraints:

**Constraint 1: Memoization vs Functor.** A memoized value stores its result behind a shared pointer (`Rc`/`Arc`). Once stored, you can only get `&A`. A proper `Functor::map` needs to consume `A` or at least move it. PureScript sidesteps this because `force` returns owned values (GC handles sharing). Rust cannot have both memoization and standard `Functor` on the same type.

**Constraint 2: Stack safety vs lifetime polymorphism.** The `Free` monad that powers `Trampoline` uses `Box<dyn Any>` for type erasure in its CatList of continuations. `dyn Any` requires `'static`. Therefore, stack-safe monadic chaining (`O(1)` bind) is incompatible with lifetime polymorphism. You cannot have a `Trampoline<&'a str>`.

**Constraint 3: HKT compatibility vs type erasure.** The `Kind` trait requires `type Of<'a, A: 'a>: 'a`, meaning the type must work for any lifetime. `Trampoline` requires `'static`, so it cannot implement `Kind` and therefore cannot participate in the HKT typeclass hierarchy.

PureScript avoids all three constraints because:
- It has a GC, so `force` returns owned values.
- It uses `unsafeCoerce` for type erasure without lifetime concerns.
- It has native higher-kinded types.

The three-way split is the most honest solution given Rust's constraints. Each type occupies a different point in the design space, and conversions between them (`From` impls, `memoize()`, `memoize_arc()`) let users move between points as needed.

## 4. Missing PureScript Typeclass Instances

Comparing PureScript `Data.Lazy`'s instances against the Rust implementations:

### Present in Rust (for `Lazy`)
- Semigroup (lines 921, 956 of lazy.rs)
- Monoid (lines 993, 1019)
- Foldable (line 1043)
- RefFunctor (line 880, Rust-specific replacement for Functor)
- Deferrable / Control.Lazy (lines 812, 847)

### Missing from Rust `Lazy`, present in PureScript
- **Functor** (replaced by RefFunctor due to `&A` return)
- **Apply / Applicative** (cannot implement; would need owned values)
- **Bind / Monad** (cannot implement; same reason)
- **Extend / Comonad** (these traits do not exist in the Rust library)
- **Traversable** (would require cloning the lazy value and wrapping in an applicative)
- **FunctorWithIndex, FoldableWithIndex, TraversableWithIndex** (WithIndex variants not implemented)
- **Foldable1, Traversable1** (non-empty fold/traverse not in the library)
- **Invariant** (trait does not exist in the library)
- **Eq, Ord, Eq1, Ord1** (could potentially be implemented by forcing evaluation)
- **Bounded** (could be implemented)
- **Semiring, Ring, CommutativeRing, EuclideanRing** (arithmetic instances not attempted)
- **HeytingAlgebra, BooleanAlgebra** (not in the library)
- **Show** (Rust's `Debug` is not implemented for `Lazy` in a way that shows the value)

### Present in Rust `Thunk` (compensating for Lazy's gaps)
- Functor (line 384 of thunk.rs)
- Pointed (line 420)
- Lift (line 450)
- Semiapplicative (line 499)
- Semimonad (line 544)
- MonadRec (line 580)
- Foldable (present in thunk.rs)
- Evaluable (enables Free monad integration)

The strategy is clear: `Thunk` carries the monad tower that `Lazy` cannot. Users who need both memoization and monadic composition must compose `Thunk` computations and then `.memoize()` the result.

## 5. Rust-Specific Additions Without PureScript Equivalents

### Thunk

PureScript has no direct equivalent of `Thunk`. The closest analog is a zero-argument function `Unit -> a`, which PureScript's `Lazy` class already abstracts over. In PureScript, `\_ -> expr` is lightweight; in Rust, `Box<dyn FnOnce() -> A>` has allocation overhead. `Thunk` wraps this pattern and adds the full monad tower, making it a first-class citizen.

**Well-motivated?** Yes. Rust needs an explicit non-memoized deferred computation type because:
1. Closures are not first-class values in the way PureScript thunks are.
2. It serves as the HKT-compatible computation type that `Lazy` and `Trampoline` cannot be.
3. It is the `Evaluable` functor that powers the `Free` monad.

### Trampoline

PureScript has `Control.Monad.Trampoline` in purescript-free, defined as `type Trampoline = Free Lazy`. The Rust version is essentially `Free<ThunkBrand, A>` with a newtype wrapper. The key difference is that PureScript's `Trampoline` is a type alias and inherits all `Free` typeclass instances, while Rust's is a standalone struct with inherent methods only (no HKT traits).

**Well-motivated?** Yes. Stack safety is more critical in Rust than PureScript because:
1. Rust has a fixed-size stack (typically 8MB).
2. PureScript compiles to JavaScript, which has tail-call optimization in strict mode and generally deeper stacks.
3. The `Free` monad with CatList provides guaranteed O(1) bind, which is essential for practical use.

### Try* Variants (TryThunk, TryLazy, TryTrampoline)

PureScript has no direct equivalents. Error handling in PureScript typically uses `Either` or `ExceptT`, and `Lazy (Either e a)` is trivial to construct. The Rust `Try*` types are newtypes over the base types wrapping `Result`:
- `TryThunk<'a, A, E>` wraps `Thunk<'a, Result<A, E>>`
- `TryLazy<'a, A, E, Config>` wraps `Config::TryLazy<'a, A, E>`
- `TryTrampoline<A, E>` wraps `Trampoline<Result<A, E>>`

**Well-motivated?** Partially. The ergonomic combinators (`map`, `map_err`, `bind`, `and_then`) reduce boilerplate compared to manually threading `Result` through the base types. The `Bifunctor` and `Bifoldable` instances for `TryThunk` are a nice touch. However, in a language with `?` operator sugar, some might argue that `Thunk<'a, Result<A, E>>` with the `?` operator is sufficient. The `Try*` types add API surface without adding new capability.

The `TryTrampoline::catch_unwind` (mentioned in recent commits) is genuinely Rust-specific, allowing recovery from panics in trampolined computations.

### Free Monad

The `Free<F, A>` implementation (types/free.rs) uses the "Reflection without Remorse" technique with `CatList` for O(1) bind. PureScript's `Free` is more general (supports `resume`, `foldFree`, `hoistFree` for arbitrary natural transformations), while Rust's is focused on execution via `Evaluable`. The Rust version explicitly acknowledges this narrower scope in its documentation (lines 13-42 of free.rs).

### LazyConfig Trait

The `LazyConfig` trait (line 65 of lazy.rs) is entirely Rust-specific. PureScript has no need to parameterize over synchronization primitives. This is a well-designed abstraction that cleanly separates the `Rc` vs `Arc` concern from the lazy evaluation logic.

## 6. The `fix` Function

### PureScript

```purescript
fix :: forall l. Lazy l => (l -> l) -> l
fix f = go
  where go = defer \_ -> f go
```

This is generic over any `Lazy` instance and enables tying recursive knots.

### Rust

`fix` is not on the `Deferrable` trait. Instead, it is provided as two concrete functions:
- `rc_lazy_fix` (line 735 of lazy.rs): for `RcLazy`
- `arc_lazy_fix` (line 788): for `ArcLazy`

The implementation uses `Rc<OnceCell<RcLazy<A>>>` (or `Arc<OnceLock<...>>`) to break the self-referential cycle. The lazy value is allocated with a closure that reads from the cell, and the cell is populated immediately after construction.

**Should `fix` be on the trait?** No, the current design is correct. In PureScript, `fix` works because `go` is a lazy thunk that can reference itself through PureScript's lazy evaluation. In Rust, self-reference requires shared ownership and interior mutability, which is specific to `Lazy` (with its `Rc`/`Arc` wrapper). `Thunk` is linear (consumed on evaluation), so a `Thunk`-based `fix` would need an infinite chain of allocations. `Trampoline` could theoretically support a `fix` via its `defer` method, but the `'static` constraint and lack of memoization make it impractical for the knot-tying pattern.

The concrete functions are the right choice. They require `A: Clone` because the `OnceCell` approach means the value must be clonable to share between the cell and the lazy value.

## 7. Lifetime Constraints vs PureScript's Implicit Laziness

PureScript is a lazy-by-default language (at the JS FFI boundary) with garbage collection. Every value implicitly supports:
- Arbitrary sharing (no lifetime tracking).
- Deferred evaluation (thunks are invisible).
- Memoization (thunks evaluate once).

Rust requires explicit management of all three:

| Concern | PureScript | Rust |
|---------|-----------|------|
| Sharing | GC (implicit) | `Rc`/`Arc` (explicit) |
| Deferral | Thunks (implicit) | `Box<dyn FnOnce()>` (explicit, allocated) |
| Memoization | JS runtime | `LazyCell`/`LazyLock` (explicit) |
| Lifetimes | None | `'a` parameter on `Thunk`, `Lazy`; `'static` on `Trampoline` |

The `'static` requirement on `Trampoline` is the most impactful divergence. In PureScript, you can trampoline over borrowed data freely. In Rust, any data in a `Trampoline` computation must be owned. This forces cloning or `Arc`-wrapping data that would otherwise be borrowed.

The `'a` parameter on `Thunk` and `Lazy` is more flexible, allowing borrowed data, but it means these types cannot outlive their borrows. PureScript values have no such constraint.

## 8. PureScript Patterns That Are Impossible or Awkward in Rust

### Lazy Monadic Composition

In PureScript:
```purescript
result = do
  x <- defer \_ -> expensive1
  y <- defer \_ -> expensive2 x
  pure (x + y)
```

This produces a `Lazy` value with full monadic composition and memoization. In Rust, `Lazy` has no `Monad` instance. The equivalent requires composing `Thunk` values and then memoizing:

```rust
let result = Thunk::new(|| expensive1())
    .bind(|x| Thunk::new(move || expensive2(x)).map(move |y| x + y))
    .memoize();
```

This works but loses intermediate memoization; only the final result is cached.

### Comonadic Extraction

PureScript's `Lazy` is a `Comonad`:
```purescript
extract :: Lazy a -> a
extract = force

extend :: (Lazy a -> b) -> Lazy a -> Lazy b
extend f x = defer \_ -> f x
```

Rust has no `Comonad` trait at all. `Lazy::evaluate` returns `&A`, not `A`, so even if a `Comonad` trait existed, `Lazy` could not implement `extract :: Lazy a -> a` without cloning.

### Traversing Lazy Values

PureScript: `traverse f l = defer <<< const <$> f (force l)`. This wraps a lazy value in an applicative effect. Rust's `Lazy` has no `Traversable` instance, and implementing one would require the applicative functor to produce a `Lazy` containing a reference (or a clone), which is awkward.

### Transparent Lifting of Algebraic Structures

PureScript trivially lifts `Semiring`, `Ring`, etc. through `Lazy`:
```purescript
instance semiringLazy :: Semiring a => Semiring (Lazy a) where
  add a b = defer \_ -> force a + force b
```

The Rust library provides `Semigroup` and `Monoid` for `Lazy` but stops there. Extending to `Semiring`-like structures is possible in principle (the library would need those traits first) but the `&A` return type from `evaluate` means every algebraic operation requires cloning, adding overhead that PureScript avoids.

### Self-Referential Data Structures

PureScript's `fix` enables patterns like:
```purescript
ones = fix \xs -> Cons 1 xs
```

Rust's `rc_lazy_fix` can achieve similar patterns, but the `Clone` bound on `A` and the explicit `Rc<OnceCell>` machinery make it significantly more verbose. Additionally, since Rust `Lazy` is not a proper `Functor`, you cannot `map` over such recursive structures generically.

## 9. Improvements the Rust Version Makes Over PureScript

1. **Thread safety as a first-class concern.** `ArcLazy` and `ArcLazyConfig` provide thread-safe lazy values with the same API, something PureScript does not address (its runtime is single-threaded). The `memoize_arc()` methods on `Thunk` and `Trampoline` bridge single-threaded computation to thread-safe caching.

2. **Stack safety is explicit and guaranteed.** `Trampoline` provides a hard guarantee against stack overflow with O(1) bind. PureScript's `Trampoline` (from purescript-free) provides the same, but in Rust this is more critical due to fixed stack sizes.

3. **The LazyConfig trait.** This is an elegant abstraction that PureScript has no need for but that solves a real Rust problem: parameterizing over `Rc` vs `Arc` without duplicating the entire API. The trait is open for third-party implementations.

4. **Explicit conversion graph.** The `From` implementations between `Thunk`, `Lazy`, and `Trampoline` form a well-documented conversion graph. Users can start with any type and convert to another as needs change. PureScript has no need for this because `Lazy` does everything.

5. **MonadRec on Thunk.** The `MonadRec` implementation for `ThunkBrand` (line 580 of thunk.rs) provides stack-safe tail recursion within the HKT framework. PureScript's `MonadRec` is typically implemented for `Free`-based types; having it on `Thunk` gives users a lightweight option.

6. **Try* variants with ergonomic error handling.** While PureScript relies on `ExceptT` or `Either`, the Rust `Try*` types provide purpose-built combinators for fallible lazy computations, including `catch_unwind` for panic recovery.

7. **Zero-cost abstractions in the core path.** `Thunk`'s `map` and `bind` use `impl FnOnce`, avoiding any heap allocation for the composition itself (beyond the initial `Box<dyn FnOnce>` for the thunk). PureScript's lazy operations always go through the runtime's thunk mechanism.

## 10. Overall Assessment

### Is This a Good Adaptation?

Yes. The Rust lazy evaluation hierarchy is a thoughtful adaptation that honestly confronts the fundamental differences between PureScript and Rust rather than trying to paper over them.

**Strengths:**

- The three-type split (`Lazy`/`Thunk`/`Trampoline`) is well-motivated by genuine language constraints, not over-engineering. Each type fills a distinct niche that the others cannot serve.
- The `Deferrable` trait is a faithful translation of PureScript's `Lazy` class, with the `fix` omission properly justified.
- The `RefFunctor` trait is an honest acknowledgment that memoized lazy values cannot satisfy standard `Functor`. Rather than fighting the borrow checker or introducing unsafe code, the library introduces a new abstraction.
- Thread safety integration is first-class and well-designed.
- The conversion graph between types is comprehensive and well-documented.

**Weaknesses:**

- The `Lazy` type loses most of PureScript's typeclass richness. Only `RefFunctor`, `Foldable`, `Semigroup`, and `Monoid` survive. This means `Lazy` cannot participate in generic monadic or applicative code, which significantly limits its utility in a typeclass-heavy FP library.
- The `Trampoline` type stands entirely outside the HKT system. It has `map`, `bind`, `pure`, and `lift2` as inherent methods, but these cannot be used generically. Code that is polymorphic over `Functor` or `Monad` cannot accept `Trampoline`.
- The `Try*` types add significant API surface. `TryThunk`, `TryLazy`, and `TryTrampoline` each duplicate much of their base type's API with `Result` wrapping. In a language with `?` operator sugar, the value proposition is debatable.
- The `Clone` requirement on `Lazy`'s `Deferrable` and `Foldable` impls (forced by `evaluate` returning `&A`) adds friction compared to PureScript where cloning is invisible.

**Fundamental Mismatches:**

There are no fundamental *mismatches* in the sense of incorrectly translated semantics. The adaptation correctly identifies where PureScript's design relies on GC and implicit laziness, and it provides appropriate Rust alternatives. The main tension is between memoization and the standard `Functor` interface, which is an inherent property of Rust's ownership model, not a design flaw.

The one area that could be improved is bridging `Lazy` back into the typeclass hierarchy more aggressively. For example, if `Lazy::evaluate` returned `A` (consuming the cache, or requiring `Clone`), it could implement standard `Functor`. The `RefFunctor` approach is safer but creates a parallel typeclass universe that does not compose with the rest of the library. An alternative design might offer both a reference-returning `get()` and a consuming `evaluate()` to give users the choice.

### Summary

The Rust lazy evaluation hierarchy is a **good adaptation** of PureScript's `Data.Lazy` ecosystem. It loses typeclass generality (inevitably, given Rust's ownership model) but gains thread safety, stack safety guarantees, and explicit control over sharing. The three-type split is the correct response to Rust's constraints, and the conversion graph makes movement between types practical. The main gap is the inability of `Lazy` to participate in generic monadic code, which limits the composability that makes PureScript's `Lazy` so powerful.
