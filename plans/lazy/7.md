# Agent 7: Fallible Variants and Error Handling Analysis

## 1. Inventory of Fallible Variants

The library provides a complete set of `Try*` counterparts for all three lazy types:

| Infallible | Fallible | Location |
|---|---|---|
| `Thunk<'a, A>` | `TryThunk<'a, A, E>` | `fp-library/src/types/try_thunk.rs` |
| `Trampoline<A>` | `TryTrampoline<A, E>` | `fp-library/src/types/try_trampoline.rs` |
| `Lazy<'a, A, Config>` | `TryLazy<'a, A, E, Config>` | `fp-library/src/types/try_lazy.rs` |

Each fallible type follows the same structural pattern as its infallible counterpart but wraps a `Result<A, E>` in the output channel.

### Internal Representations

- **`TryThunk<'a, A, E>`**: Wraps `Box<dyn FnOnce() -> Result<A, E> + 'a>`. This is a direct `Result`-producing closure, not a wrapper around `Thunk`.
- **`TryTrampoline<A, E>`**: Wraps `Trampoline<Result<A, E>>`. This is a newtype wrapper that delegates to the infallible `Trampoline`.
- **`TryLazy<'a, A, E, Config>`**: Wraps `Config::TryLazy<'a, A, E>`, which resolves to `Rc<LazyCell<Result<A, E>, ...>>` or `Arc<LazyLock<Result<A, E>, ...>>`. The `LazyConfig` trait has dedicated associated types (`TryLazy`, `TryThunk`) specifically for fallible memoization.

**Observation**: The internal representation strategy is inconsistent. `TryTrampoline` delegates to `Trampoline<Result<A, E>>` (a wrapper), while `TryThunk` uses a standalone `Box<dyn FnOnce() -> Result<A, E>>` (a parallel implementation). `TryLazy` uses dedicated associated types baked into the `LazyConfig` trait. This means fallibility was integrated at different levels of abstraction for each type.

## 2. How Error Handling Integrates with Lazy Evaluation

### Construction Patterns

All three fallible types provide consistent construction APIs:

```
TryThunk::new(|| Ok(42))        / TryThunk::ok(42)        / TryThunk::err(e)
TryTrampoline::new(|| Ok(42))   / TryTrampoline::ok(42)   / TryTrampoline::err(e)
TryLazy::new(|| Ok(42))         // no ok()/err() shortcuts
```

**Gap**: `TryLazy` lacks `ok()` and `err()` convenience constructors. While this is arguably less important for a memoized type (you would rarely memoize a pure value), it breaks the symmetry with the other two types.

### Evaluation Patterns

| Type | Evaluate Signature | Returns |
|---|---|---|
| `Thunk` | `evaluate(self) -> A` | Owned value |
| `TryThunk` | `evaluate(self) -> Result<A, E>` | Owned result |
| `Trampoline` | `evaluate(self) -> A` | Owned value |
| `TryTrampoline` | `evaluate(self) -> Result<A, E>` | Owned result |
| `Lazy` | `evaluate(&self) -> &A` | Borrowed reference |
| `TryLazy` | `evaluate(&self) -> Result<&A, &E>` | Borrowed references |

The `Lazy`/`TryLazy` evaluate by reference (they are memoized; you only borrow the cached value), while the others consume self. The fallible `TryLazy::evaluate` returns `Result<&A, &E>`, not `&Result<A, E>`, which is more ergonomic because it lets callers use the `?` operator on the result.

### Monadic Operations

All fallible types implement `bind` to short-circuit on error:

```rust
// TryThunk::bind (line 196)
TryThunk::new(move || match (self.0)() {
    Ok(a) => (f(a).0)(),
    Err(e) => Err(e),
})

// TryTrampoline::bind (line 214)
TryTrampoline(self.0.bind(|result| match result {
    Ok(a) => f(a).0,
    Err(e) => Trampoline::pure(Err(e)),
}))
```

Both patterns correctly propagate errors without executing the continuation. The implementations are idiomatic.

### Error Recovery

`TryThunk` and `TryTrampoline` both provide a `catch` combinator:

```rust
// TryThunk (line 265)
pub fn catch(self, f: impl FnOnce(E) -> TryThunk<'a, A, E> + 'a) -> Self

// TryTrampoline (line 236)
pub fn catch(self, f: impl FnOnce(E) -> TryTrampoline<A, E> + 'a) -> Self
```

**Gap**: `TryLazy` does not have a `catch` combinator. This makes sense for a memoized type (you cannot "retry" after caching an error), but the library could provide a `recover` or `or_else` that creates a *new* `TryLazy` with fallback logic before evaluation occurs.

### Panic Safety

`TryLazy` uniquely provides `catch_unwind` (line 272 of `try_lazy.rs`), which wraps `std::panic::catch_unwind` and converts panics to `Err(String)`:

```rust
pub fn catch_unwind(f: impl FnOnce() -> A + std::panic::UnwindSafe + 'a) -> Self
```

**Observation**: This is only available on `TryLazy` (specifically the `RcLazyConfig` variant with `E = String`). `TryThunk` and `TryTrampoline` do not offer this. While memoizing a panic result is arguably the primary use case, `TryThunk` could also benefit from a `catch_unwind` constructor.

## 3. Type Class Instances: Fallible vs. Infallible

### Thunk vs. TryThunk

| Type Class | `ThunkBrand` | `TryThunkErrAppliedBrand<E>` | `TryThunkOkAppliedBrand<A>` | `TryThunkBrand` |
|---|---|---|---|---|
| `Functor` | Yes | Yes | Yes (over error) | N/A (Bifunctor) |
| `Pointed` | Yes | Yes | Yes (wraps as Err) | N/A |
| `Semiapplicative` | Yes | Yes | Yes | N/A |
| `Semimonad` | Yes | Yes | N/A | N/A |
| `MonadRec` | Yes | Yes | N/A | N/A |
| `Foldable` | Yes | Yes | N/A | N/A |
| `Semigroup` | Yes | Yes (for A: Semigroup) | N/A | N/A |
| `Monoid` | Yes | Yes (for A: Monoid) | N/A | N/A |
| `Bifunctor` | N/A | N/A | N/A | Yes |
| `Lift` | Yes | Yes | Yes | N/A |
| `ApplyFirst` | Yes | Yes | Yes | N/A |
| `ApplySecond` | Yes | Yes | Yes | N/A |
| `Evaluable` | Yes | **No** | N/A | N/A |
| `Deferrable` | Yes | Yes (via TryThunk directly) | N/A | N/A |

`TryThunk` has nearly complete type class parity with `Thunk`. The key difference is that `TryThunkErrAppliedBrand<E>` does not implement `Evaluable`. This makes sense because `Evaluable::evaluate` returns `A`, not `Result<A, E>`, so the type class is fundamentally incompatible with fallible evaluation. Additionally, `TryThunk` gains `Bifunctor` and `TryThunkOkAppliedBrand<A>` for mapping over the error type, which `Thunk` does not need.

### Trampoline vs. TryTrampoline

| Capability | `Trampoline` | `TryTrampoline` |
|---|---|---|
| HKT Brand | None (due to `'static`) | None |
| `bind` | Yes (inherent) | Yes (inherent) |
| `map` | Yes (inherent) | Yes (inherent) |
| `map_err` | N/A | Yes (inherent) |
| `defer` | Yes | Yes |
| `lift2` | Yes | No |
| `then` | Yes | No |
| `tail_rec_m` | Yes | **No** |
| `arc_tail_rec_m` | Yes | **No** |
| `catch` | N/A | Yes |
| `Deferrable` | Yes | Yes |

**Significant gap**: `TryTrampoline` lacks `tail_rec_m`, `lift2`, and `then`. The absence of `tail_rec_m` is notable because the primary purpose of `Trampoline` is stack-safe recursion, and fallible computations that recurse deeply are a common use case (e.g., recursive descent parsers, tree transformations). Without `tail_rec_m`, users who need stack-safe fallible recursion must manually wrap `Trampoline<Result<Step<S, A>, E>>` themselves, losing the ergonomic `TryTrampoline` combinators.

### Lazy vs. TryLazy

| Capability | `Lazy` | `TryLazy` |
|---|---|---|
| HKT Brand | `LazyBrand<Config>` | `TryLazyBrand<E, Config>` |
| `RefFunctor` | Yes (`RcLazyConfig` only) | **No** |
| `Deferrable` | Yes (Rc) | Yes (Rc) |
| `SendDeferrable` | Yes (Arc) | Yes (Arc) |
| `ref_map` | Yes (inherent, Rc) | **No** |

**Gap**: `TryLazy` does not implement `RefFunctor`. A `try_ref_map` that maps `Result<&A, &E>` to something useful would be valuable. The mapping would need to handle the error case, so the signature would differ from `RefFunctor` (it would need to be a separate trait or inherent method).

## 4. Design Consistency Assessment

### Naming

The naming is consistent: `Try` prefix uniformly marks fallible variants. The library avoids the `Fallible` prefix mentioned in the task description; `Try` is used throughout. This is a good choice, as it matches Rust's `TryFrom`, `TryInto` conventions.

### Structural Inconsistency

As noted in section 1, the internal architecture differs:

1. **`TryThunk`**: Standalone type with `Box<dyn FnOnce() -> Result<A, E>>`. Fully independent from `Thunk`.
2. **`TryTrampoline`**: Newtype wrapper around `Trampoline<Result<A, E>>`. Delegates to the infallible type.
3. **`TryLazy`**: Uses `LazyConfig::TryLazy` associated type. Fallibility is baked into the config trait itself.

This inconsistency has practical consequences:

- `TryThunk`'s code is a near-complete copy of `Thunk`'s code with `Result` threaded through. This maximizes duplication but gives full control.
- `TryTrampoline`'s delegation strategy is elegant and minimizes duplication. Each method delegates to `Trampoline`'s equivalent with a `Result` wrapper.
- `TryLazy`'s integration into `LazyConfig` means the config trait is more complex (4 associated types instead of 2), but avoids a separate wrapper layer.

### Conversion Web

The library provides a comprehensive set of `From` implementations. Here is the conversion graph:

```
Thunk  ----> TryThunk    (From<Thunk> for TryThunk)
Lazy   ----> TryThunk    (From<Lazy> for TryThunk)
TryLazy --> TryThunk     (From<TryLazy> for TryThunk)

Thunk  ----> Lazy        (From<Thunk> for Lazy)
Trampoline -> Lazy       (From<Trampoline> for Lazy)

Lazy   ----> TryLazy     (From<Lazy> for TryLazy, both Rc and Arc)
TryThunk --> TryLazy     (From<TryThunk> for TryLazy)
TryTrampoline -> TryLazy (From<TryTrampoline> for TryLazy)

Trampoline --> TryTrampoline (From<Trampoline> for TryTrampoline)
Lazy -------> TryTrampoline  (From<Lazy> for TryTrampoline)
TryLazy ----> TryTrampoline  (From<TryLazy> for TryTrampoline)
```

**Missing conversions**:
- No `From<TryThunk> for TryTrampoline`. This means you cannot easily upgrade a fallible thunk to a stack-safe fallible computation.
- No `From<Thunk> for TryTrampoline` (would need to go `Thunk -> TryThunk -> TryLazy -> TryTrampoline`, or directly).

## 5. Error Type Parameterization

All three fallible types are generic over `E`:

```rust
TryThunk<'a, A, E>
TryTrampoline<A, E>
TryLazy<'a, A, E, Config>
```

This is maximally flexible. Users can use any error type: `String`, custom enums, `Box<dyn Error>`, `anyhow::Error`, etc.

### HKT Branding and Error Types

For `TryThunk`, the error type must be fixed to form a `Functor` over the success type:

```rust
TryThunkErrAppliedBrand<E>   // Functor over A (error fixed)
TryThunkOkAppliedBrand<A>    // Functor over E (success fixed)
TryThunkBrand                // Bifunctor over (E, A)
```

For `TryLazy`, the brand requires both `E` and `Config`:

```rust
TryLazyBrand<E, Config>      // Functor over A
```

`TryTrampoline` has no HKT brand at all (same limitation as `Trampoline`, due to `'static`).

### The `'static` Constraint on E

`TryTrampoline` requires `A: 'static + Send` and `E: 'static + Send`. The `'static` constraint on `E` means you cannot use borrowed error types (e.g., `&str`) with `TryTrampoline`, while `TryThunk` can. This mirrors the same split between `Thunk` (supports borrows) and `Trampoline` (requires `'static`).

For `TryThunk` and `TryLazy`, the error type only needs `E: 'a`, which is flexible enough for most use cases.

## 6. Interaction with `Result` and the `?` Operator

### Direct `?` Operator Usage

None of the fallible types directly implement `Try` (the unstable trait behind `?`). Users must call `.evaluate()` first to get a `Result`, then use `?`:

```rust
let result = TryThunk::new(|| Ok(42)).evaluate()?;
```

This is a reasonable design choice. Implementing `Try` would require the nightly toolchain and would conflate evaluation with error propagation. The two-step pattern (evaluate, then propagate) is explicit and predictable.

### Composing with `?` Inside Closures

The `TryThunk::new` and `TryTrampoline::new` constructors accept `FnOnce() -> Result<A, E>`, so the `?` operator works naturally inside the closure:

```rust
let thunk = TryThunk::new(|| {
    let x = some_fallible_op()?;
    let y = another_fallible_op(x)?;
    Ok(y * 2)
});
```

This is ergonomic and idiomatic Rust.

### From<Result> Missing

There is no `From<Result<A, E>>` for any of the `Try*` types. While trivial to construct via `TryThunk::new(|| result)`, a `From` impl would improve ergonomics for converting already-computed results into the deferred context:

```rust
// Currently required:
TryThunk::new(|| already_computed_result)

// Could be:
TryThunk::from(already_computed_result)
```

## 7. Should There Be a `FallibleDeferrable` Trait?

Currently, `Deferrable` is implemented for all six types (both infallible and fallible). The trait signature is:

```rust
pub trait Deferrable<'a> {
    fn defer(f: impl FnOnce() -> Self + 'a) -> Self where Self: Sized;
}
```

This works for fallible types because `Self` already includes the error type. There is no separate `FallibleDeferrable` or `TryDeferrable` trait.

### Analysis

A dedicated `FallibleDeferrable` trait could provide:

```rust
pub trait FallibleDeferrable<'a, E> {
    fn try_defer(f: impl FnOnce() -> Result<Self, E> + 'a) -> Self where Self: Sized;
}
```

This would allow constructing a deferred fallible value from a closure that itself might fail during construction (as opposed to the current `Deferrable::defer` where the closure infallibly produces a `TryThunk` that then may fail on evaluation).

**Verdict**: The current design is adequate. The `Deferrable` trait already handles the fallible case because `TryThunk<'a, A, E>` is the `Self` type, and the closure `FnOnce() -> TryThunk<'a, A, E>` can internally fail. A separate `FallibleDeferrable` would be a leaky abstraction, forcing the failure concern into the trait interface when it belongs in the constructor. The existing pattern of `TryThunk::new(|| some_result?)` is more idiomatic.

## 8. Comparison with Other FP Libraries

### Haskell

Haskell does not typically have "fallible lazy" types because laziness is pervasive and error handling is done through monad transformers (`ExceptT`). The equivalent of `TryThunk` would be `ExceptT e IO a` or simply `IO (Either e a)`.

### PureScript

PureScript uses `ExceptT` as a monad transformer to add error handling to any base monad:

```purescript
type FallibleAff a = ExceptT Error Aff a
```

This is more general than having separate `Try*` types, but requires transformer boilerplate.

### Scala (Cats Effect / ZIO)

- **Cats Effect `IO[A]`**: Error handling is built into the `IO` type itself. `IO` is always fallible. There is no separate `TryIO`.
- **ZIO `ZIO[R, E, A]`**: Error type `E` is a first-class parameter. `ZIO` unifies infallible and fallible computation by allowing `E = Nothing`.

### This Library's Approach

The library's approach of parallel `Try*` types is more explicit than Haskell/PureScript's transformer approach but less unified than ZIO's. It trades generality for simplicity: users choose `Thunk` or `TryThunk` based on whether they need error handling, without needing to understand monad transformers.

## 9. Code Duplication Between Fallible and Infallible Variants

### TryThunk vs. Thunk: High Duplication

`TryThunk` reimplements nearly every method from `Thunk` with `Result` threaded through:

- `new`, `pure`, `defer`, `bind`, `map`, `evaluate` all have near-identical structure.
- All type class impls (`Functor`, `Pointed`, `Semiapplicative`, `Semimonad`, `MonadRec`, `Foldable`, `Lift`, `Semigroup`, `Monoid`) are duplicated with `Result` handling added.

The `try_thunk.rs` file is substantially larger than `thunk.rs` because it also includes `Bifunctor`, `TryThunkOkAppliedBrand`, and all the associated impls. In total, there are approximately 1000+ lines of type class implementations in `try_thunk.rs` that closely mirror `thunk.rs`.

### TryTrampoline vs. Trampoline: Low Duplication

`TryTrampoline` delegates to `Trampoline<Result<A, E>>`, so the implementations are thin wrappers:

```rust
pub fn map<B>(self, func: impl FnOnce(A) -> B) -> TryTrampoline<B, E> {
    TryTrampoline(self.0.map(|result| result.map(func)))
}
```

This is the ideal pattern for minimizing duplication. Each method is 1-3 lines.

### TryLazy vs. Lazy: Medium Duplication

`TryLazy` shares the `LazyConfig` infrastructure with `Lazy`, so the config-level code is shared. However, the `TryLazy` type itself has separate `new`, `evaluate`, `Clone`, `Deferrable`, and `SendDeferrable` impls.

### Duplication Risk

The primary risk is that changes to the infallible types (e.g., adding a new combinator to `Thunk`) require manually mirroring the change in `TryThunk`. The `TryTrampoline` pattern mitigates this naturally, but `TryThunk` does not.

## 10. Could a Monad Transformer Approach Eliminate Separate Fallible Types?

### The Transformer Approach

A generic `ExceptT` monad transformer could unify all fallible types:

```rust
struct ExceptT<E, M, A>(M::Of<Result<A, E>>);
// where M is a monad brand
```

Then:
- `ExceptT<E, ThunkBrand, A>` replaces `TryThunk<A, E>`
- `ExceptT<E, LazyBrand<Config>, A>` replaces `TryLazy<A, E, Config>`

### Why This Is Difficult in the Current Design

1. **Lifetime polymorphism**: `ExceptT` would need to work with `Kind` and the library's HKT machinery. The `Apply!` macro would need to handle nested type application: `Apply!(M::Of<'a, Result<A, E>>)`.

2. **Trampoline has no brand**: `Trampoline` cannot participate in HKT traits due to its `'static` requirement, so `ExceptT<E, TrampolineBrand, A>` is not possible with the current architecture.

3. **Lazy returns references**: `Lazy::evaluate` returns `&A`, not `A`. A generic `ExceptT` with `evaluate` returning the inner monad's result type would get `&Result<A, E>`, not `Result<&A, &E>`. The current `TryLazy` explicitly provides the more ergonomic `Result<&A, &E>` via `.as_ref()`. A transformer would lose this.

4. **Config parameterization**: `TryLazy`'s integration with `LazyConfig` (which has `TryLazy` and `TryThunk` associated types) means fallibility is part of the config abstraction itself. This cannot be factored through a generic transformer without redesigning the config trait.

### Verdict

A monad transformer approach is theoretically cleaner but practically difficult given the library's constraints:

- The `'static` requirement on `Trampoline` prevents HKT integration.
- Reference-returning `Lazy::evaluate` does not compose with a generic transformer.
- The `LazyConfig` trait already bakes fallibility into its associated types.

The current parallel-types approach is the pragmatic choice. The main improvement opportunity is to adopt `TryTrampoline`'s delegation pattern for `TryThunk` (i.e., make `TryThunk` a newtype over `Thunk<'a, Result<A, E>>`) to reduce code duplication.

## Summary of Recommendations

### High Priority

1. **Add `tail_rec_m` to `TryTrampoline`**: The absence of stack-safe tail recursion on the fallible variant undermines the primary value proposition of `Trampoline`. Implementation would delegate to `Trampoline::tail_rec_m` with `Result<Step<S, A>, E>` unwrapping.

2. **Add `lift2` and `then` to `TryTrampoline`**: These are straightforward delegations that complete the API surface.

3. **Add `From<TryThunk> for TryTrampoline`**: Complete the conversion web so users can upgrade fallible thunks to stack-safe fallible computations.

### Medium Priority

4. **Refactor `TryThunk` as a newtype over `Thunk<'a, Result<A, E>>`**: This would follow the `TryTrampoline` pattern, dramatically reducing code duplication in type class implementations. The inherent methods (`ok`, `err`, `map_err`, `catch`) would remain as thin wrappers.

5. **Add `catch_unwind` to `TryThunk`**: Panic catching is useful beyond memoized contexts. The current implementation only lives on `TryLazy`.

6. **Add `ok()` / `err()` convenience constructors to `TryLazy`**: For symmetry with `TryThunk` and `TryTrampoline`.

### Low Priority

7. **Add `From<Result<A, E>>` impls**: For ergonomic conversion from already-computed results.

8. **Consider `try_ref_map` on `TryLazy`**: An inherent method like `try_ref_map(self, f: impl FnOnce(&A) -> Result<B, E2>) -> TryLazy<B, E2>` would let users map over memoized fallible values.

9. **Explore unifying `LazyConfig::TryLazy` with a wrapper**: Instead of dedicated associated types on `LazyConfig`, `TryLazy` could be `Lazy<'a, Result<A, E>, Config>` with ergonomic wrappers. This would simplify the config trait from 4 associated types to 2, at the cost of storing `Result` inside the lazy cell (which is what the current implementation does anyway).
