# Lazy Evaluation Hierarchy: Architecture and Design Coherence Analysis

## 1. The Three-Type Split: Thunk / Trampoline / Lazy

### Is It Well-Motivated?

Yes. The three-way split addresses three genuinely distinct concerns in an eagerly evaluated language, and each concern imposes constraints that are mutually exclusive in Rust's type system:

| Concern | Type | Key Constraint |
|---------|------|---------------|
| Deferred computation with lifetime polymorphism | `Thunk<'a, A>` | Supports `'a`, HKT-compatible |
| Stack-safe deferred computation | `Trampoline<A>` | Requires `'static` (uses `Box<dyn Any>`) |
| Memoized computation | `Lazy<'a, A, Config>` | Returns `&A` not `A`; requires `Clone` for sharing |

The README (lines 188-193) articulates this clearly. In Haskell or PureScript, the runtime handles all three concerns transparently via thunks with built-in memoization and an RTS that manages stack. Rust has no such runtime, so the user must choose explicitly.

The split is forced by fundamental Rust constraints:

1. **Thunk vs Trampoline**: `Free<F, A>` requires `Box<dyn Any>` for type erasure in its `CatList` of continuations (`free.rs`, line 93), which requires `'static`. This makes `Trampoline` unable to support arbitrary lifetimes. Meanwhile, `Thunk` wraps `Box<dyn FnOnce() -> A + 'a>` (line 90 of `thunk.rs`), preserving lifetime polymorphism but building a call-stack-consuming chain of closures on `bind`.

2. **Computation vs Caching**: `Lazy` returns `&A` from `evaluate` (line 485 of `lazy.rs`), which is the correct API for memoization (you get a reference to the cached value). This makes it fundamentally incompatible with `Functor` (which requires returning owned `B`, not `&B`). `Thunk::evaluate` consumes `self` and returns `A` (line 248 of `thunk.rs`), which is the right API for a computation that runs once and yields ownership.

### Verdict

The split is architecturally sound and well-motivated by Rust's ownership model. No reasonable unification is possible without sacrificing correctness or ergonomics.

## 2. Relationships Between Types

### Conversion Web

The types form a rich conversion graph via `From` implementations:

```
Thunk <---> Trampoline    (From impls, thunk.rs:329, trampoline.rs:512)
Thunk  ---> Lazy           (From impl, lazy.rs:565)
Lazy   ---> Thunk          (From impl, thunk.rs:303; requires Clone)
Lazy   ---> Trampoline     (From impl, trampoline.rs:490; requires Clone)
Trampoline ---> Lazy       (From impl, lazy.rs:583)
```

The same pattern repeats for `Try*` variants (e.g., `TryThunk <-> TryTrampoline`, `TryThunk -> TryLazy`, etc.). This is thorough and consistent.

### Memoization Bridge

Each computation type provides `.memoize()` and `.memoize_arc()` methods:

- `Thunk::memoize()` -> `Lazy<'a, A, RcLazyConfig>` (line 269 of `thunk.rs`)
- `Thunk::memoize_arc()` -> `Lazy<'a, A, ArcLazyConfig>` (line 290; evaluates eagerly because `Thunk` is `!Send`)
- `Trampoline::memoize()` -> `Lazy<'static, A, RcLazyConfig>` (line 272 of `trampoline.rs`)
- `Trampoline::memoize_arc()` -> `Lazy<'static, A, ArcLazyConfig>` (line 293; also evaluates eagerly)

The eager evaluation for `memoize_arc` is a necessary consequence: `Thunk` and `Trampoline` contain `!Send` closures (they use `Box<dyn FnOnce>` without `Send`), so the only way to produce an `ArcLazy` (which requires `Send`) is to evaluate the computation first and wrap the result. This is clearly documented.

### Consistency Assessment

The relationships are clear and consistent. Every computation type can be promoted to any other, with appropriate bounds (`Clone` for extracting from `Lazy`, `'static` for entering `Trampoline`). The directionality makes sense: computation types flow into memoization types naturally, while the reverse requires cloning.

One minor inconsistency: `Thunk` has bidirectional `From` impls with both `Lazy` and `Trampoline`, but `Lazy` to `Trampoline` only goes through `Lazy -> Trampoline` (requiring `Clone`). There is no `Trampoline -> Lazy` via the `From` trait on `Trampoline`; instead it goes through `From<Trampoline> for Lazy`. This is correct but the direction might confuse newcomers.

## 3. Duplication and Unification Opportunities

### The `LazyConfig` Trait: A Successful Unification

The `LazyConfig` trait (lines 65-170 of `lazy.rs`) is a well-designed abstraction that unifies `RcLazy` and `ArcLazy` into a single generic type `Lazy<'a, A, Config>`. This avoids duplicating the entire `Lazy` implementation for the two pointer strategies. The associated types (`Lazy`, `TryLazy`, `Thunk`, `TryThunk`) bundle together the pointer, cell, and closure types for each strategy.

This design is extensible: third-party code could implement `LazyConfig` for `parking_lot`-based locks or async-aware cells without modifying the library.

### Duplicated Inherent Methods Across Thunk / Trampoline / TryThunk / TryTrampoline

Each type independently implements:
- `new`, `pure`/`ok`, `err` (where applicable)
- `map`, `bind`, `defer`
- `evaluate`
- `lift2`, `then`
- `memoize`, `memoize_arc`
- `tail_rec_m`, `arc_tail_rec_m` (Trampoline and TryTrampoline)
- `catch` (TryThunk and TryTrampoline)
- `catch_unwind` (all three Try* types)

The implementations follow identical patterns. For example, `TryThunk::bind` (line 203-211 of `try_thunk.rs`) and `TryTrampoline::bind` (line 212-220 of `try_trampoline.rs`) are structurally identical, differing only in the underlying type (`Thunk` vs `Trampoline`):

```rust
// TryThunk::bind
TryThunk(self.0.bind(|result| match result {
    Ok(a) => f(a).0,
    Err(e) => Thunk::pure(Err(e)),
}))

// TryTrampoline::bind
TryTrampoline(self.0.bind(|result| match result {
    Ok(a) => f(a).0,
    Err(e) => Trampoline::pure(Err(e)),
}))
```

Similarly, `TryThunk::tail_rec_m` (lines 884-895 of `try_thunk.rs`) mirrors the loop-based pattern exactly.

### Could This Be Unified?

In principle, a generic `TryComputation<Inner>` wrapper could eliminate this duplication. However, this would require an abstraction over "computation type" that captures `new`, `pure`, `bind`, `map`, and `evaluate` in a trait. The problem is:

1. `Thunk` and `Trampoline` have different lifetime constraints (`'a` vs `'static`).
2. `Thunk::evaluate` consumes `self`, while `Lazy::evaluate` borrows `&self`.
3. The HKT system already provides this abstraction for `Thunk` (via `Functor`, `Semimonad`) but cannot for `Trampoline`.

A macro-based approach (generating the `Try*` wrapper from the base type) would be more practical than a trait-based one. The current duplication is manageable (six types total) but would become a maintenance burden if more variants were added.

### Verdict on Duplication

The duplication is real but tolerable. The `Try*` types are thin wrappers (each stores a single inner computation of `Result<A, E>`) with predictable, mechanical implementations. The `LazyConfig` trait demonstrates that unification is pursued where it makes architectural sense. The remaining duplication is a pragmatic choice given Rust's type system constraints.

## 4. The Try* Variant Pattern

### Design

Each base type has a fallible counterpart:
- `Thunk<'a, A>` -> `TryThunk<'a, A, E>`
- `Trampoline<A>` -> `TryTrampoline<A, E>`
- `Lazy<'a, A, Config>` -> `TryLazy<'a, A, E, Config>`

Each `Try*` type is a newtype wrapper around the base type applied to `Result<A, E>`:
- `TryThunk<'a, A, E>` wraps `Thunk<'a, Result<A, E>>` (line 68-71 of `try_thunk.rs`)
- `TryTrampoline<A, E>` wraps `Trampoline<Result<A, E>>` (line 38-41 of `try_trampoline.rs`)
- `TryLazy<'a, A, E, Config>` wraps `Config::TryLazy<'a, A, E>` (line 44-47 of `try_lazy.rs`)

### Does It Scale?

The pattern is systematic and predictable, which is good. However, it doubles the surface area of the API. If a fourth base type were added (e.g., a `Stream` or `IO`), the combinatorial explosion would produce eight types. Currently, with three base types and three `Try*` variants, the total is six, which is manageable.

### Is It the Right Approach?

The alternative would be to not have `Try*` types and instead ask users to work with `Thunk<'a, Result<A, E>>` directly. The `Try*` types provide:

1. **Short-circuiting bind**: `TryThunk::bind` only calls `f` on `Ok`, propagating `Err` automatically. Without the wrapper, users would need to match on `Result` manually in every `bind` call.
2. **Consistent error-handling API**: `catch`, `map_err`, `catch_unwind` are available as methods.
3. **Type clarity**: `TryTrampoline<i32, String>` communicates intent more clearly than `Trampoline<Result<i32, String>>`.

The wrappers earn their keep primarily through the short-circuiting `bind` semantics. This is the same reason `Result` exists as a distinct type rather than just being a tagged union.

### HKT Integration for Try* Types

`TryThunk` has full HKT support via three brands (line 62-67 of `try_thunk.rs`):
- `TryThunkBrand` (bifunctor)
- `TryThunkErrAppliedBrand<E>` (functor/monad over Ok)
- `TryThunkOkAppliedBrand<A>` (functor over Err)

`TryTrampoline` has no brands and no HKT integration, mirroring `Trampoline`'s exclusion. `TryLazy` has a brand (`TryLazyBrand<E, Config>`) but limited trait implementations.

This is consistent with the base types' HKT participation.

### `catch_unwind` Across All Try* Types

All three `Try*` types provide `catch_unwind` for panic recovery (specialized to `E = String`). The implementations are nearly identical, each delegating to `std::panic::catch_unwind` and a shared `panic_payload_to_string` utility. This is a practical feature that integrates well with the fallible computation model.

### Verdict

The `Try*` pattern is the right approach. It provides meaningful ergonomic benefits (short-circuiting bind, error handling combinators) without excessive complexity. The pattern scales adequately for the current type count. A macro could reduce boilerplate if more base types are added.

## 5. Integration with the Broader HKT/Typeclass System

### Thunk: Full Citizen

`Thunk` (via `ThunkBrand`) implements the complete typeclass tower:
- `Functor` (line 384 of `thunk.rs`)
- `Pointed` (line 420)
- `Lift` (line 450)
- `Semiapplicative` (line 499)
- `Semimonad` (line 544)
- `MonadRec` (line 580)
- `Evaluable` (line 628)
- `Foldable` (line 661)
- `Deferrable` (line 356)

This makes `Thunk` the primary way to work with deferred computation in generic (HKT-polymorphic) code. The `Evaluable` trait (line 41 of `evaluable.rs`) is particularly important: it is the mechanism by which `Free::evaluate` runs effects, and only `ThunkBrand` implements it.

### Trampoline: Deliberately Excluded

`Trampoline` cannot participate in HKT because `Free` requires `'static` types, and the `Kind` trait requires lifetime polymorphism (`type Of<'a, A: 'a>: 'a`). The comment in `brands.rs` (line 192-193) makes this explicit: "Note: This is for `Thunk<'a, A>`, NOT for `Trampoline<A>`. `Trampoline` cannot implement HKT traits due to its `'static` requirement."

This means `Trampoline` lives outside the typeclass hierarchy entirely, with only inherent methods and `Deferrable` (line 530 of `trampoline.rs`). This is the correct design: forcing `'static` into the `Kind` system would break the lifetime polymorphism that other types depend on.

### Lazy: Partial Citizen via RefFunctor

`Lazy` cannot implement `Functor` because `Functor::map` requires producing `F<B>` from `F<A>` and `A -> B`, but `Lazy::evaluate` returns `&A`, not `A`. The library solves this with `RefFunctor` (line 62 of `ref_functor.rs`), which maps `(&A) -> B` instead of `A -> B`.

Only `LazyBrand<RcLazyConfig>` implements `RefFunctor` (line 880 of `lazy.rs`). `LazyBrand<ArcLazyConfig>` does not get a blanket `RefFunctor` impl because the trait does not require `Send` on the mapping function, but `ArcLazy::new` requires `Send`. The `ArcLazy` variant instead provides an inherent `ref_map` method with the additional `Send` bound (lines 677-685 of `lazy.rs`). This is well-documented in the code comment at line 660-663.

`Lazy` also implements `Foldable` (line 1043 of `lazy.rs`), `Semigroup`, `Monoid`, and `Deferrable`/`SendDeferrable`. It does not implement `Semimonad` or `Applicative`, which makes sense: you cannot meaningfully chain `Lazy` computations with `bind` since `evaluate` returns a reference.

### The Evaluable Bottleneck

The `Evaluable` trait (line 41 of `evaluable.rs`) is documented as: "Currently only `ThunkBrand` implements this trait. `Lazy` cannot implement it because `evaluate` returns `&A` (a reference), not an owned `A`. `Trampoline` does not have a brand and therefore cannot participate in HKT traits."

This is accurate but creates a design constraint: `Free<F, A>` can only be instantiated with `F = ThunkBrand` in practice, which means `Trampoline` is the only practical consumer of `Free`. If other effect types wanted to use `Free`, they would need to implement `Evaluable`, which requires returning owned values from a functor context.

## 6. Trait Assignment Assessment

### Correct Assignments

- **Thunk**: `Functor`, `Monad`, `MonadRec`, `Evaluable`, `Foldable`, `Deferrable`. All correct. `Thunk` is the canonical "computation" type and should have the full stack.
- **Thunk does NOT implement Traversable**: Correctly excluded (line 74-78 of `thunk.rs`). `Traversable` requires `Clone`, which `Box<dyn FnOnce>` cannot provide.
- **Lazy**: `RefFunctor`, `Foldable`, `Semigroup`, `Monoid`, `Deferrable`. All correct. These are the traits that make sense for a memoized, shared-reference type.
- **Lazy does NOT implement Functor or Monad**: Correct. The `&A` return type from `evaluate` makes these traits semantically inappropriate.
- **Trampoline**: `Deferrable` only. Correct, given the `'static` constraint.
- **TryThunk**: `Functor`, `Monad`, `MonadRec`, `Foldable`, `Bifunctor`, `Bifoldable`, `Deferrable`. Correct and comprehensive. The bifunctor support (via `TryThunkBrand`) is a nice touch for mapping over both `Ok` and `Err`.

### Potentially Missing

- **Trampoline could implement Semigroup/Monoid**: If `A: Semigroup`, then `Trampoline<A>` could combine two trampolines by evaluating both and combining results. This is not currently implemented. It would be consistent with `Lazy`'s `Semigroup` impl.
- **TryLazy lacks RefFunctor**: `TryLazy` has inherent `map` and `map_err` methods (lines 200-238 of `try_lazy.rs`) but no `RefFunctor` trait impl. This is understandable since `RefFunctor` does not model the error channel, but a `TryRefFunctor` or similar could be considered.

## 7. Architectural Gaps and Missed Opportunities

### 7.1. No Send-Safe Thunk

`Thunk` wraps `Box<dyn FnOnce() -> A + 'a>`, which is `!Send`. There is no `SendThunk` analogue. For thread-safe deferred computation, users must use `ArcLazy` (memoized) or construct their own wrappers. A `SendThunk<'a, A>` wrapping `Box<dyn FnOnce() -> A + Send + 'a>` would fill this gap and could implement `Functor` with `Send` bounds, similar to how `ArcLazy` adds `Send` to `Lazy`.

This is not necessarily a deficiency; the library's approach of providing `ArcLazy` for thread-safe scenarios may be sufficient. But it means there is no non-memoized, `Send`-capable computation type.

### 7.2. Trampoline Cannot Be Used in Generic Code

Because `Trampoline` has no brand, it cannot be used with generic functions like `map::<Brand, _, _>` or `bind::<Brand, _, _>`. Users must use inherent methods exclusively. This is a known limitation documented in the codebase.

If a use case arises where stack-safe computation needs to participate in generic typeclass code, the only option is `Thunk` with `MonadRec::tail_rec_m`. This is adequate for most cases but means that deep `bind` chains in generic code (using `Thunk`) are not stack-safe.

### 7.3. The `Deferrable` Trait Could Be More Useful

`Deferrable` is a minimal trait with a single method (`defer`). It is implemented by `Thunk`, `Trampoline`, `Lazy` (Rc variant), `TryThunk`, and `TryTrampoline`. However, it does not require any relationship with `Evaluable` or `Functor`, so it cannot be used as a generic "lazy type" abstraction.

The PureScript `Lazy` typeclass includes `defer` and `fix`. In this library, `fix` is intentionally excluded from the trait (documented at lines 26-37 of `deferrable.rs`) because it requires shared ownership (`Rc`/`Arc`), which only `Lazy` has. The concrete `rc_lazy_fix` and `arc_lazy_fix` functions fill this role. This is a reasonable Rust-specific adaptation.

### 7.4. Missing `From` Impls for ArcLazy

`From<Thunk>` and `From<Trampoline>` are implemented for `Lazy<RcLazyConfig>` (lines 565 and 583 of `lazy.rs`), but there are no corresponding `From` impls for `Lazy<ArcLazyConfig>`. This is because `Thunk` and `Trampoline` contain `!Send` closures, so direct conversion to `ArcLazy` would require eager evaluation. The `memoize_arc()` methods handle this case explicitly. This is correct, but a `From` impl that eagerly evaluates could be provided for ergonomic parity, with the eager evaluation documented clearly.

### 7.5. TryLazy Has Limited HKT Integration

`TryLazy` has a brand (`TryLazyBrand<E, Config>`) declared in `brands.rs` (line 199), and `impl_kind!` is presumably applied somewhere, but it implements fewer traits than `Lazy`. Notably, it lacks `RefFunctor` and `Foldable`. Given that `TryLazy::map` exists as an inherent method (line 200 of `try_lazy.rs`), a `RefFunctor` impl for the error-applied brand would be natural.

### 7.6. No Unified "Evaluate" Concept

The three base types use different signatures for `evaluate`:
- `Thunk::evaluate(self) -> A` (consumes self)
- `Trampoline::evaluate(self) -> A` (consumes self)
- `Lazy::evaluate(&self) -> &A` (borrows self)

This is correct given their semantics, but it means there is no trait that abstracts over "types you can force." The `Evaluable` typeclass exists but only works at the HKT level and only for `ThunkBrand`. A simpler non-HKT trait like:

```rust
trait Force {
    type Output;
    fn force(self) -> Self::Output;
}
```

...could unify `Thunk::evaluate` and `Trampoline::evaluate` (both `self -> A`). However, `Lazy` would not fit because its output is `&A`, not `A`. This is a fundamental tension rather than a fixable gap.

### 7.7. Thunk's bind Is Not Stack-Safe

The documentation is clear about this (line 168-169 of `thunk.rs`), and `MonadRec` provides a stack-safe alternative via `tail_rec_m` (which uses a loop, lines 616-625). But this means that generic monadic code using `bind::<ThunkBrand, _, _>` can overflow the stack for deep chains. Users must know to use `tail_rec_m` instead.

This is an inherent limitation, not a bug. The alternative (making `Thunk` stack-safe by default) would require `Thunk` to use `Free` internally, which would require `'static`, defeating its purpose. The design correctly trades stack safety for lifetime flexibility.

## 8. Summary

### Strengths

1. **Clear separation of concerns**: Each type addresses a specific need (lifetime-flexible computation, stack-safe computation, memoization) with well-documented trade-offs.
2. **Comprehensive conversions**: The `From` impl web allows smooth transitions between types as requirements change.
3. **LazyConfig abstraction**: Elegant unification of `Rc` and `Arc` variants without code duplication.
4. **Correct HKT integration**: Types participate in the typeclass hierarchy exactly where their semantics allow, no more, no less.
5. **Try* variants**: Well-motivated by the short-circuiting `bind` semantics, with consistent API surface.
6. **Thread safety story**: The `Rc`/`Arc` split is handled uniformly via `LazyConfig`, and the `memoize_arc` methods correctly handle the `!Send` boundary by eager evaluation.

### Weaknesses

1. **Mechanical duplication**: The `Try*` types duplicate methods from their base types with predictable `Result`-wrapping logic. A macro or code generation step could reduce this.
2. **No non-memoized `Send` computation type**: There is no `SendThunk` for thread-safe deferred computation without memoization.
3. **TryLazy has limited typeclass integration**: Despite having a brand, it implements fewer traits than might be expected.
4. **No unified force/evaluate abstraction**: The different `evaluate` signatures prevent a single trait from abstracting over all lazy types.

### Overall Assessment

The lazy evaluation hierarchy is architecturally sound. The three-type split is well-motivated by Rust's ownership and lifetime constraints, and the types integrate correctly with the broader HKT system. The `Try*` variant pattern is the right approach, providing meaningful ergonomic value despite the duplication cost. The primary improvement opportunities are mechanical (reducing duplication via macros) rather than architectural.
