# Lazy Hierarchy: Architecture and Design Analysis

## 1. Overview of the Hierarchy

The library provides three primary lazy evaluation types, plus three fallible counterparts:

| Type | File | Memoized | Stack-safe | HKT traits | Lifetime | Cloneable |
|------|------|----------|------------|------------|----------|-----------|
| `Thunk<'a, A>` | `types/thunk.rs` | No | Partial (`tail_rec_m`) | Yes (full) | `'a` | No |
| `Trampoline<A>` | `types/trampoline.rs` | No | Yes (unlimited) | No | `'static` | No |
| `Lazy<'a, A, Config>` | `types/lazy.rs` | Yes | N/A | Partial (`RefFunctor`) | `'a` | Yes |
| `TryThunk<'a, A, E>` | `types/try_thunk.rs` | No | Partial | Yes (full) | `'a` | No |
| `TryTrampoline<A, E>` | `types/try_trampoline.rs` | No | Yes | No | `'static` | No |
| `TryLazy<'a, A, E, Config>` | `types/try_lazy.rs` | Yes | N/A | Partial | `'a` | Yes |

The types are organized along two orthogonal axes:

1. **Infallible vs. Fallible** (Thunk/Trampoline/Lazy vs. TryThunk/TryTrampoline/TryLazy).
2. **Evaluation semantics** (non-memoized computation vs. stack-safe recursion vs. memoized caching).

Additionally, the `Deferrable` trait and `SendDeferrable` trait provide a uniform construction interface across all types.

---

## 2. Design Coherence and Complementarity

### 2.1 Clear responsibility separation

The three infallible types occupy distinct, well-motivated niches:

- **Thunk** is the "glue code" type: lightweight deferred computation with full HKT integration. Its `Box<dyn FnOnce() -> A + 'a>` representation is minimal. It supports `Functor`, `Semimonad`, `Semiapplicative`, `Foldable`, `MonadRec`, and `Evaluable`. This makes it the workhorse for generic programming over deferred values.

- **Trampoline** is the "heavy-duty recursion" type: built on `Free<ThunkBrand, A>` with a `CatList`-based bind stack for O(1) bind and unlimited stack safety. It sacrifices HKT compatibility (due to its `'static + Send` requirement) and cloneability in exchange for guaranteed safety at any recursion depth.

- **Lazy** is the "caching" type: computes at most once, shares the result across clones. It leverages `Rc<LazyCell>` or `Arc<LazyLock>` from the standard library and returns `&A` from `evaluate`, making it fundamentally different from the other two (which consume themselves to produce `A`).

This delineation is well-chosen. The three types do not overlap in their core capabilities; rather, they form a spectrum:

```
Thunk (lightweight, composable, no memoization, partial stack safety)
  |
  v  [wrap in Free]
Trampoline (heavy, stack-safe, 'static only, no HKT)
  |
  v  [wrap in Lazy]
Lazy (memoized, cloneable, returns &A, no monadic composition)
```

### 2.2 Conversion graph

The types form a well-connected conversion graph via `From` impls:

```
Thunk ---From---> Lazy (RcLazy)
Trampoline ---From---> Lazy (RcLazy)
Lazy ---From---> Thunk (requires Clone)
Lazy ---From---> Trampoline (requires Clone + 'static)
TryThunk ---From---> TryLazy
TryTrampoline ---From---> TryLazy
Lazy ---From---> TryLazy (wraps in Ok)
Trampoline ---From---> TryTrampoline (wraps in Ok)
TryLazy ---From---> TryTrampoline (requires Clone)
```

This is comprehensive. Users can freely convert between types as their requirements change. One notable observation: the Lazy-to-Thunk and Lazy-to-Trampoline conversions require `Clone` because `Lazy::evaluate` returns `&A`, so the conversion must clone the value to produce an owned `A`. This is a necessary consequence of `Lazy`'s reference-returning design.

### 2.3 The `Deferrable` unification

All six types implement `Deferrable<'a>` (with `Trampoline` and `TryTrampoline` limited to `'static`):

```rust
pub trait Deferrable<'a> {
    fn defer(f: impl FnOnce() -> Self + 'a) -> Self where Self: Sized;
}
```

This directly mirrors PureScript's `Control.Lazy` class:

```purescript
class Lazy l where
  defer :: (Unit -> l) -> l
```

The key difference is that PureScript's `defer` takes `Unit -> l` (a thunk) while Rust's takes `FnOnce() -> Self + 'a`. The Rust version is strictly more expressive because the closure can capture state with a bounded lifetime, whereas PureScript's `Unit -> l` has no lifetime constraint (GC handles everything).

For memoized types (`Lazy`, `TryLazy`), `defer` flattens the nested structure: instead of producing `Lazy<Lazy<A>>`, it evaluates the inner lazy and re-wraps:

```rust
// Lazy<'a, A, RcLazyConfig>::defer
fn defer(f: impl FnOnce() -> Self + 'a) -> Self {
    RcLazy::new(move || f().evaluate().clone())
}
```

This requires `A: Clone`, which is the cost of the memoization approach (since `evaluate` returns `&A`).

## 3. Detailed Type Analysis

### 3.1 Thunk

**Representation:**

```rust
pub struct Thunk<'a, A>(Box<dyn FnOnce() -> A + 'a>);
```

**Strengths:**

- Full HKT support via `ThunkBrand`. Implements `Functor`, `Pointed`, `Semiapplicative`, `Semimonad`, `MonadRec`, `Foldable`, `Evaluable`, `Lift`, `ApplyFirst`, `ApplySecond`, `Semigroup`, `Monoid`.
- Lifetime polymorphism allows borrowing from the environment.
- Serves as the functor parameter for `Free<ThunkBrand, A>`, which underpins `Trampoline`.
- `MonadRec` provides a stack-safe `tail_rec_m` that runs an imperative loop internally.

**Design decisions:**

- `evaluate(self) -> A` consumes the thunk. This is correct for `FnOnce` semantics but means a thunk cannot be evaluated twice. This is intentional, as the doc comment states: "each call to `evaluate` re-executes the computation." This is slightly misleading since you cannot actually call evaluate twice on the same value (it moves `self`). The "re-execution" applies when you create the same thunk again, not when you evaluate it again.

- The `bind` implementation nests closures:

```rust
pub fn bind<B: 'a>(self, f: impl FnOnce(A) -> Thunk<'a, B> + 'a) -> Thunk<'a, B> {
    Thunk::new(move || {
        let a = (self.0)();
        let thunk_b = f(a);
        (thunk_b.0)()
    })
}
```

Each `bind` adds a stack frame when evaluated. For deep chains, this can overflow. This is correctly documented and is why `Trampoline` exists.

**Observation on Traversable:**

The documentation correctly explains why `Traversable` cannot be implemented: `FnOnce` closures cannot be cloned, and `Traversable` requires reconstructing the structure, which for a single-element container would require running the closure to get the value, wrapping it in the applicative, and then re-wrapping in `Thunk`. The issue is that `traverse` needs to produce `F (Thunk A)`, but reconstructing a `Thunk` from a value means the original deferred computation is lost. The explanation in the doc comment focuses on `Clone` bounds, which is a slightly indirect way to express this.

### 3.2 Trampoline

**Representation:**

```rust
pub struct Trampoline<A: 'static>(Free<ThunkBrand, A>);
```

**Strengths:**

- Guaranteed stack safety via the `Free` monad's `CatList`-based continuation queue.
- O(1) `bind` operations (no left-associated chain degradation).
- `defer` is the critical combinator for stack-safe recursion, wrapping `Free::wrap(Thunk::new(...))`.
- Both `tail_rec_m` (for `Clone` closures) and `arc_tail_rec_m` (for non-`Clone` closures) are provided.

**Design decisions:**

- `A: 'static + Send` is required. The `'static` constraint comes from `Free` using `Box<dyn Any>` for type erasure. The `Send` constraint is baked into the `impl` block. This means `Trampoline` cannot hold borrowed data, which is a fundamental trade-off documented clearly.

- No HKT brand exists for `Trampoline`. The doc comment on `ThunkBrand` explicitly notes: "This is for `Thunk<'a, A>`, NOT for `Trampoline<A>`. `Trampoline` cannot implement HKT traits due to its `'static` requirement." This is correct; the `Kind` trait requires `type Of<'a, A: 'a>: 'a`, but `Trampoline` cannot vary over lifetimes.

- `map` is implemented via `bind`:

```rust
pub fn map<B: 'static + Send>(self, f: impl FnOnce(A) -> B + 'static) -> Trampoline<B> {
    self.bind(move |a| Trampoline::pure(f(a)))
}
```

This is correct but slightly less efficient than a direct `Free::map` would be, since it adds an extra continuation to the CatList. For a trampoline whose primary purpose is deep recursion, this overhead is negligible.

### 3.3 Lazy

**Representation:**

```rust
pub struct Lazy<'a, A, Config: LazyConfig = RcLazyConfig>(Config::Lazy<'a, A>) where A: 'a;
```

Where `Config::Lazy<'a, A>` is either `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>` or `Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>`.

**Strengths:**

- Leverages Rust 1.80's `LazyCell` and `LazyLock` for initialization-once semantics, avoiding hand-rolled interior mutability.
- The `LazyConfig` trait elegantly parameterizes over the threading model (`Rc`/`LazyCell` vs. `Arc`/`LazyLock`).
- `evaluate(&self) -> &A` is non-consuming, enabling repeated access to the cached value.
- `Clone` shares the cache, not the computation. All clones see the same value.
- `ref_map` creates a new `Lazy` that lazily maps over the cached reference.

**HKT integration:**

`Lazy` implements `RefFunctor` rather than `Functor`:

```rust
impl RefFunctor for LazyBrand<RcLazyConfig> {
    fn ref_map<'a, A: 'a, B: 'a>(
        f: impl FnOnce(&A) -> B + 'a,
        fa: Lazy<'a, A, RcLazyConfig>,
    ) -> Lazy<'a, B, RcLazyConfig> {
        fa.ref_map(f)
    }
}
```

This is a good design choice. Since `evaluate` returns `&A`, a standard `Functor::map` taking `Fn(A) -> B` would require cloning the value out, which is wasteful when you have a reference. `RefFunctor` with `Fn(&A) -> B` is the natural fit.

However, `RefFunctor` is only implemented for `LazyBrand<RcLazyConfig>`, not for `LazyBrand<ArcLazyConfig>`. This is a gap; the `ArcLazy` variant could also support `RefFunctor` with the appropriate `Send` bounds.

**The `LazyConfig` trait:**

```rust
pub trait LazyConfig: 'static {
    type Lazy<'a, A: 'a>: Clone;
    type TryLazy<'a, A: 'a, E: 'a>: Clone;
    type Thunk<'a, A: 'a>: ?Sized;
    type TryThunk<'a, A: 'a, E: 'a>: ?Sized;
    fn lazy_new<'a, A: 'a>(f: Box<Self::Thunk<'a, A>>) -> Self::Lazy<'a, A>;
    fn try_lazy_new<'a, A: 'a, E: 'a>(f: Box<Self::TryThunk<'a, A, E>>) -> Self::TryLazy<'a, A, E>;
    fn evaluate<'a, 'b, A: 'a>(lazy: &'b Self::Lazy<'a, A>) -> &'b A;
    fn try_evaluate<'a, 'b, A: 'a, E: 'a>(lazy: &'b Self::TryLazy<'a, A, E>) -> Result<&'b A, &'b E>;
}
```

This is well-designed. It bundles the infallible and fallible lazy cell types together, which is pragmatic since `TryLazy` needs the same config mechanism. The `'static` bound on the trait itself ensures configs are compile-time choices, not runtime values.

One minor concern: `LazyConfig` contains both `Lazy` and `TryLazy` associated types. This means you cannot have a config that only supports one but not the other. In practice this is fine since both `RcLazyConfig` and `ArcLazyConfig` support both, but it reduces extensibility if someone wanted to define a custom config.

### 3.4 Fallible Variants

The fallible types (`TryThunk`, `TryTrampoline`, `TryLazy`) mirror their infallible counterparts with `Result<A, E>` semantics.

**TryThunk** is essentially `Thunk<'a, Result<A, E>>` with ergonomic wrappers (`ok`, `err`, `bind` that short-circuits on error, `map_err`, `catch`). It has full HKT support via `TryThunkBrand` (bifunctor), `TryThunkErrAppliedBrand<E>` (functor over Ok), and `TryThunkOkAppliedBrand<A>` (functor over Err).

**TryTrampoline** is `Trampoline<Result<A, E>>` with the same ergonomic wrappers. It has no HKT brand (inherits `Trampoline`'s limitation).

**TryLazy** is `Lazy` parameterized with `Config::TryLazy<'a, A, E>` which wraps `LazyCell<Result<A, E>, ...>`. It caches either the success value or the error. It has a `TryLazyBrand<E, Config>` for HKT (functor over the success type with fixed error type).

The fallible hierarchy is cleanly parallel to the infallible one. The `TryLazy::catch_unwind` method is a nice addition that bridges Rust panics into the fallible lazy system.

## 4. Comparison with PureScript Originals

### 4.1 Control.Lazy (PureScript) vs. Deferrable (Rust)

**PureScript:**

```purescript
class Lazy l where
  defer :: (Unit -> l) -> l

fix :: forall l. Lazy l => (l -> l) -> l
fix f = go where go = defer \_ -> f go
```

**Rust:**

```rust
pub trait Deferrable<'a> {
    fn defer(f: impl FnOnce() -> Self + 'a) -> Self where Self: Sized;
}
```

**What was gained:**
- Lifetime parameterization (`'a`) allows the thunk to borrow from the environment, which PureScript cannot express (it relies on GC).
- `impl FnOnce() -> Self + 'a` is more ergonomic than PureScript's `Unit -> l` convention.
- The naming (`Deferrable` instead of `Lazy`) avoids confusion with `Data.Lazy` (the type).

**What was lost:**
- `fix` is absent. PureScript's `fix` computes a fixed point by leveraging laziness; the equivalent in Rust would be tricky due to ownership semantics but could potentially work with `Lazy`:
  ```rust
  fn fix<'a, A: Clone + 'a>(f: impl Fn(&A) -> A + 'a) -> Lazy<'a, A, RcLazyConfig>
  ```
  This is a notable omission, since `fix` is the primary combinator that makes `Control.Lazy` useful in practice (for tying recursive knots).

- PureScript has a `Lazy` instance for functions (`defer f = \x -> f unit x`), which Rust cannot express since `Deferrable` requires `Sized` and closures have opaque types.

### 4.2 Data.Lazy (PureScript) vs. Lazy (Rust)

**PureScript's Data.Lazy** implements a rich set of type classes:

| PureScript | Rust equivalent | Implemented? |
|------------|-----------------|--------------|
| Functor | Functor | No (uses RefFunctor instead) |
| Apply | Semiapplicative | No |
| Applicative | Pointed + Semiapplicative | No |
| Bind | Semimonad | No |
| Monad | Monad | No |
| Foldable | Foldable | No |
| Traversable | Traversable | No |
| Extend | (not in library) | No |
| Comonad | (not in library) | No |
| Semigroup | Semigroup | No |
| Monoid | Monoid | No |
| Eq | PartialEq | No |
| Ord | PartialOrd | No |
| Show | Display | No |
| Lazy (Control.Lazy) | Deferrable | Yes |

**What was gained:**
- Thread-safe variant (`ArcLazy`) has no PureScript equivalent since PureScript is single-threaded.
- The `LazyConfig` parameterization elegantly handles the Rc/Arc distinction at the type level.
- The `TryLazy` fallible variant has no direct PureScript equivalent (PureScript would use `Lazy (Either e a)` without special support).

**What was lost:**
- The vast majority of type class instances. PureScript's `Data.Lazy` is a full Monad, Comonad, Traversable, etc. Rust's `Lazy` only implements `RefFunctor` and `Deferrable`. This is the largest gap.

The root cause is fundamental: PureScript's `force :: Lazy a -> a` produces an owned value (the runtime handles sharing), while Rust's `evaluate(&self) -> &A` produces a reference. This means:

1. Standard `Functor` (`map :: (a -> b) -> f a -> f b`) would need to consume the value, but `Lazy` only gives references.
2. `Applicative`/`Monad` instances would need to produce new `Lazy` values from the results, requiring `Clone` to extract the cached value.
3. `Foldable` could be implemented trivially (since you have `&A`).
4. `Traversable` could be implemented if `evaluate` gave an owned value.

The design choice to return `&A` from `evaluate` is well-motivated (avoids unnecessary cloning) but comes at the cost of type class richness. This is a core trade-off that could be explored further (see suggestions below).

## 5. Observations and Suggestions

### 5.1 Missing `fix` combinator

PureScript's `Control.Lazy` exists primarily to support `fix`:

```purescript
fix :: forall l. Lazy l => (l -> l) -> l
fix f = go where go = defer \_ -> f go
```

This is used to tie recursive knots in lazy data structures. The Rust `Deferrable` trait has no equivalent. A possible implementation for `RcLazy`:

```rust
pub fn fix<'a, A: Clone + 'a>(f: impl Fn(RcLazy<'a, A>) -> RcLazy<'a, A> + 'a) -> RcLazy<'a, A> {
    // Requires careful handling of the self-reference
    // This may need Rc-based indirection
}
```

The challenge in Rust is creating a self-referential lazy value without unsafe code. This may require an `Rc<RefCell<Option<...>>>` approach or a dedicated `LazyFix` type. Worth investigating, as `fix` is genuinely useful for defining recursive data structures.

### 5.2 RefFunctor not implemented for ArcLazy

`RefFunctor` is only implemented for `LazyBrand<RcLazyConfig>`. The `ArcLazy` variant should also support this, with the mapping closure required to be `Send`:

```rust
impl RefFunctor for LazyBrand<ArcLazyConfig> {
    fn ref_map<'a, A: 'a, B: 'a>(
        f: impl FnOnce(&A) -> B + 'a,
        fa: Lazy<'a, A, ArcLazyConfig>,
    ) -> Lazy<'a, B, ArcLazyConfig> {
        // Requires f: Send, which the current trait signature doesn't support
    }
}
```

The issue is that `RefFunctor::ref_map` does not require `Send` on the closure, but `ArcLazy::new` requires `Send`. This is a genuine API limitation. A possible solution is a `SendRefFunctor` trait or adding a `Send`-bounded `ref_map` method directly on `ArcLazy`.

### 5.3 Foldable and additional type classes for Lazy

Even though `Lazy` returns `&A`, several type classes could be implemented:

- **Foldable**: Straightforward. `fold_right(f, init, lazy) = f(lazy.evaluate().clone(), init)` (with `A: Clone`).
- **Semigroup**: `append(a, b) = Lazy::new(move || Semigroup::append(a.evaluate().clone(), b.evaluate().clone()))`.
- **Monoid**: `empty() = Lazy::new(|| Monoid::empty())`.
- **Eq/PartialEq**: `lazy1 == lazy2` iff `*lazy1.evaluate() == *lazy2.evaluate()`.
- **Ord/PartialOrd**: Similar delegation to the inner value.
- **Display/Debug**: Delegate to the inner value.

These are all implemented for PureScript's `Data.Lazy` and would be useful additions.

For `Functor`/`Monad`, the situation is more nuanced. You could implement them with a `Clone` bound:

```rust
impl Functor for LazyBrand<RcLazyConfig> {
    fn map<'a, A: 'a + Clone, B: 'a>(
        f: impl Fn(A) -> B + 'a,
        fa: Lazy<'a, A, RcLazyConfig>,
    ) -> Lazy<'a, B, RcLazyConfig> {
        RcLazy::new(move || f(fa.evaluate().clone()))
    }
}
```

But the `Clone` bound is a problem: the `Functor` trait does not require `Clone` on `A`. This would require a separate bounded Functor, which is not idiomatic.

### 5.4 Thunk documentation clarity

The `Thunk` doc comment says "each call to `evaluate` re-executes the computation," but `evaluate` takes `self` by value, so you cannot call it twice on the same instance. This should be clarified to say something like: "Thunk does not cache results. If you need the same computation's result more than once, wrap it in `Lazy`."

### 5.5 Evaluable trait connection

The `Evaluable` trait (`classes/evaluable.rs`) is implemented only for `ThunkBrand`:

```rust
pub trait Evaluable: Functor {
    fn evaluate<'a, A: 'a>(fa: Apply!(...)::Of<'a, A>) -> A;
}
```

This trait is the bridge between `Free` and `Trampoline`: `Free::evaluate` requires `F: Evaluable` to run the effects. Since `Thunk` is the only `Evaluable` functor, `Trampoline` is effectively `Free<ThunkBrand, A>`. If additional `Evaluable` functors were added (e.g., an `IO` brand), they could power different kinds of stack-safe monads. This is a clean extension point that the current design supports well.

### 5.6 Consider a unified `evaluate` naming convention

Currently, evaluation is triggered differently across types:

| Type | Method | Signature |
|------|--------|-----------|
| `Thunk` | `evaluate(self) -> A` | Consuming |
| `Trampoline` | `evaluate(self) -> A` | Consuming |
| `Lazy` | `evaluate(&self) -> &A` | Borrowing |
| `TryThunk` | `evaluate(self) -> Result<A, E>` | Consuming |
| `TryTrampoline` | `evaluate(self) -> Result<A, E>` | Consuming |
| `TryLazy` | `evaluate(&self) -> Result<&A, &E>` | Borrowing |

The naming is consistent (`evaluate` everywhere), which is good. The signature differences (consuming vs. borrowing) accurately reflect the semantic differences between memoized and non-memoized types.

### 5.7 The SendDeferrable asymmetry

`SendDeferrable` is a separate trait from `Deferrable`, requiring `Send + Sync` on the closure:

```rust
pub trait SendDeferrable<'a> {
    fn send_defer(f: impl FnOnce() -> Self + Send + Sync + 'a) -> Self where Self: Sized;
}
```

Only `ArcLazy` and `ArcTryLazy` implement this. This is pragmatic but creates a two-class system. An alternative would be to parameterize `Deferrable` itself over thread-safety bounds, but Rust's trait system makes this difficult without higher-kinded traits on bounds.

### 5.8 TryLazy's Deferrable requires Clone

```rust
impl<'a, A, E> Deferrable<'a> for TryLazy<'a, A, E, RcLazyConfig>
where
    A: Clone + 'a,
    E: Clone + 'a,
{
    fn defer(f: impl FnOnce() -> Self + 'a) -> Self {
        Self::new(move || f().evaluate().cloned().map_err(Clone::clone))
    }
}
```

Both `A: Clone` and `E: Clone` are required because `evaluate` returns `Result<&A, &E>` and the deferred version needs to produce owned values for the new `TryLazy`. This is unavoidable given the design, but worth noting that `Deferrable` for `TryLazy` is less general than for `TryThunk` (which has no `Clone` requirement).

## 6. Summary Assessment

### Strengths

1. **Clear separation of concerns.** Each type serves a distinct purpose with minimal overlap. The choice between Thunk, Trampoline, and Lazy is driven by clear criteria (HKT compatibility, stack safety, memoization).

2. **Comprehensive conversion graph.** `From` implementations allow fluid movement between types as requirements change.

3. **Consistent API surface.** The `evaluate` method, `pure` constructor, `bind`/`map` combinators, and `defer` factory are present across all types with consistent naming.

4. **Sound threading model.** The `LazyConfig` trait cleanly separates single-threaded and thread-safe variants without code duplication.

5. **Faithful PureScript translation.** The `Deferrable` trait correctly captures the essence of PureScript's `Control.Lazy`, and the evaluation semantics of `Lazy` match `Data.Lazy`'s memoization guarantee.

6. **Complete fallible counterparts.** The TryThunk/TryTrampoline/TryLazy types provide ergonomic error handling over the same deferred computation patterns.

### Gaps

1. **Missing `fix` combinator.** The primary use case for `Control.Lazy` in PureScript is absent.

2. **Limited type class coverage for `Lazy`.** PureScript's `Data.Lazy` is a full Monad/Comonad/Traversable. Rust's `Lazy` only has `RefFunctor` and `Deferrable`. At minimum, `Foldable`, `Semigroup`, `Monoid`, `PartialEq`, `PartialOrd`, `Debug`, and `Display` could be added.

3. **No `RefFunctor` for `ArcLazy`.** The thread-safe variant lacks the mapping capability.

4. **No Comonad.** PureScript's `Data.Lazy` is a Comonad with `extract = force`. The Rust equivalent would be `extract(&self) -> &A` for `Lazy`, or a `RefComonad` trait. This is a natural fit for `Lazy` and would be useful for lazy stream-like patterns.

5. **No `Extend`.** PureScript's `extend f x = defer \_ -> f x` is a simple but useful combinator for `Lazy`. Again, this would need a reference-based variant in Rust.
