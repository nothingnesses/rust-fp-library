# Analysis 6: HKT/Brand System Integration with Lazy Types and Typeclass Law Compliance

## 1. Brand Definitions for Lazy Types

### Overview

The lazy type family defines the following brands in `fp-library/src/brands.rs`:

| Brand | Line | Type Constructor | Parameters |
|-------|------|-----------------|------------|
| `ThunkBrand` | 194 | `Thunk<'a, A>` | None |
| `LazyBrand<Config>` | 92 | `Lazy<'a, A, Config>` | `Config: LazyConfig` |
| `TryThunkBrand` | 203 | `TryThunk<'a, A, E>` (bifunctor) | None |
| `TryThunkErrAppliedBrand<E>` | 207 | `TryThunk<'a, A, E>` (functor over `Ok`) | Error type `E` |
| `TryThunkOkAppliedBrand<A>` | 211 | `TryThunk<'a, A, E>` (functor over `Err`) | Success type `A` |
| `TryLazyBrand<E, Config>` | 199 | `TryLazy<'a, A, E, Config>` | Error type `E`, `Config: LazyConfig` |

### Assessment

**Correct and well-structured.** The brands correctly follow the library's conventions:

- `ThunkBrand` is a simple zero-sized marker with no parameters, matching the single type parameter of `Thunk<'a, A>`.
- `LazyBrand<Config>` parameterizes over the memoization strategy, allowing `LazyBrand<RcLazyConfig>` and `LazyBrand<ArcLazyConfig>` to serve as distinct brands. This is a clean design that avoids duplicating brand definitions.
- `TryThunkBrand` plus its two partially-applied variants (`TryThunkErrAppliedBrand<E>`, `TryThunkOkAppliedBrand<A>`) mirror the pattern used by `ResultBrand`/`ResultErrAppliedBrand<E>`/`ResultOkAppliedBrand<T>`, providing consistent bifunctor support.
- `TryLazyBrand<E, Config>` parameterizes over both error type and config, which is correct for a type constructor over the success value.

**Notable absence:** There is no `TrampolineBrand`. The comment on `ThunkBrand` (line 193) explicitly states: "Note: This is for `Thunk<'a, A>`, NOT for `Trampoline<A>`. `Trampoline` cannot implement HKT traits due to its `'static` requirement." This is analyzed further in Section 5.

**Missing brands for Try variants:** There is no `TryTrampolineBrand`, which is consistent since `TryTrampoline` wraps `Trampoline` and inherits the same `'static` limitation.

## 3. Functor Law Compliance for Thunk

### Identity Law: `map(id, fa) = fa`

The `Functor` implementation for `ThunkBrand` (thunk.rs line 384-417) delegates to the inherent `Thunk::map` method (line 225-230):

```rust
pub fn map<B: 'a>(self, f: impl FnOnce(A) -> B + 'a) -> Thunk<'a, B> {
    Thunk::new(move || f((self.0)()))
}
```

When `f = identity`, this becomes `Thunk::new(move || identity((self.0)()))` which simplifies to `Thunk::new(move || (self.0)())`. This is observationally equivalent to the original thunk: evaluating it produces the same value. Since `Thunk` is consumed on evaluation (taking `self` by value), the only observable operation is `evaluate()`, and `map(id, fa).evaluate() == fa.evaluate()` holds.

**Verdict: Holds.** Confirmed by QuickCheck property test at thunk.rs line 998-1000.

### Composition Law: `map(f . g, fa) = map(f, map(g, fa))`

For `map(f . g, fa)`: `Thunk::new(move || (f . g)((self.0)()))` = `Thunk::new(move || f(g((self.0)())))`.

For `map(f, map(g, fa))`: first `map(g, fa)` produces `Thunk::new(move || g((fa.0)()))`, then `map(f, ...)` produces `Thunk::new(move || f(g((fa.0)())))`.

Both evaluate to the same value. There is no effect ordering concern because `Thunk` wraps a pure computation (no observable side effects beyond the final value).

**Verdict: Holds.** Confirmed by QuickCheck property test at thunk.rs line 1004-1014.

### HKT-level vs Inherent-level

The HKT `Functor` impl (line 412-417) uses `Fn` rather than `FnOnce`:

```rust
fn map<'a, A: 'a, B: 'a>(
    func: impl Fn(A) -> B + 'a,
    fa: ...,
) -> ... {
    fa.map(func)
}
```

The `Fn` bound is more restrictive than `FnOnce` on the inherent method, but this is necessary for consistency across the HKT hierarchy (some types like `Vec` need to call the function multiple times). The laws hold for both levels.

## 4. Monad Law Compliance for Thunk

### Left Identity: `pure(a).bind(f) = f(a)`

`Thunk::pure(a)` creates `Thunk::new(move || a)`. Then `bind(f)` (thunk.rs line 194-203) produces:

```rust
Thunk::new(move || {
    let a = (self.0)();  // evaluates to `a`
    let thunk_b = f(a);   // produces f(a)
    (thunk_b.0)()          // evaluates f(a)
})
```

Evaluating this yields the same as evaluating `f(a)` directly.

**Verdict: Holds.** Confirmed by QuickCheck at thunk.rs line 1017-1023 (inherent) and trampoline.rs line 829-832 (inherent Trampoline version).

### Right Identity: `m.bind(pure) = m`

`m.bind(Thunk::pure)` produces:

```rust
Thunk::new(move || {
    let a = (m.0)();
    let thunk_b = Thunk::pure(a);  // Thunk::new(move || a)
    (thunk_b.0)()                   // evaluates to a
})
```

This evaluates to the same value as `m.evaluate()`.

**Verdict: Holds.** Confirmed by QuickCheck at thunk.rs line 1026-1030.

### Associativity: `m.bind(f).bind(g) = m.bind(|a| f(a).bind(g))`

Both sides evaluate to `g(f(m.evaluate()).evaluate()).evaluate()` after expansion. The nested closures execute in the same order.

**Verdict: Holds.** Confirmed by QuickCheck at thunk.rs line 1034-1043.

### Important Caveat: Semimonad uses `Fn`, not `FnOnce`

The HKT-level `Semimonad::bind` (thunk.rs line 572-577) takes `impl Fn(A) -> ... + 'a`, not `FnOnce`. This means the HKT bind requires a clonable/re-invocable function. The inherent `Thunk::bind` takes `FnOnce`, which is more permissive. This discrepancy is intentional and documented: "Prefer this inherent method when you do not need HKT generality" (thunk.rs line 170-173).

## 5. Why Trampoline Lacks HKT Traits

### The Fundamental Limitation

`Trampoline<A>` requires `A: 'static` (trampoline.rs line 73), because internally it is `Free<ThunkBrand, A>` which uses `Box<dyn Any>` for type erasure in its bind stack (CatList of continuations). The `Any` trait requires `'static`.

The library's Kind trait signature for functors is:
```rust
type Of<'a, A: 'a>: 'a;
```

This signature is polymorphic over lifetime `'a`, meaning a `Functor` impl must work for *any* `'a`, not just `'static`. `Trampoline` can only work when `'a = 'static`, so it cannot satisfy the universally quantified lifetime.

### Is This Fixable?

**Partially, with significant trade-offs:**

1. **A separate Kind signature `type Of<A>;`** (without lifetime) already exists in `kinds.rs` (line 41-44, generating `Kind_ad6c20556a82a1f0`). One could define a `StaticFunctor` trait using this Kind. However, this would create a parallel typeclass hierarchy that cannot interoperate with the existing one. All generic code written against `Functor` would not work with `Trampoline`.

2. **Removing the `'static` requirement** from `Trampoline` would require replacing `Box<dyn Any>` with a different mechanism for the continuation stack. This would likely sacrifice the O(1) bind guarantee or require unsafe code.

3. **The current design is well-reasoned.** Thunk fills the HKT niche (lifetime-polymorphic, composable), and Trampoline fills the stack-safety niche (guaranteed stack safety, deep recursion). The inherent methods on `Trampoline` (`map`, `bind`, `lift2`, `then`, `tail_rec_m`) provide the same API surface as the typeclasses, just without HKT generality. Users who need both HKT and stack safety can compose: use `Thunk` with `MonadRec::tail_rec_m` (which is stack-safe via a loop implementation at thunk.rs line 616-625).

**Verdict: This is a fundamental limitation, not a fixable gap.** The trade-off is intentional and well-documented (brands.rs line 192-193, thunk.rs line 54-63).

## 6. RefFunctor for Lazy: Right Level of Abstraction?

### Current Design

`Lazy` implements `RefFunctor` (lazy.rs line 880-913 for `RcLazyConfig`) rather than `Functor`. The `RefFunctor` trait (ref_functor.rs line 62-97) takes `func: impl FnOnce(&A) -> B` rather than `func: impl Fn(A) -> B`.

### Why Lazy Cannot Support Full Functor

The `Functor::map` signature requires:
```rust
fn map<'a, A: 'a, B: 'a>(func: impl Fn(A) -> B + 'a, fa: ...) -> ...
```

This takes `A` by value. But `Lazy::evaluate()` returns `&A`, not `A`. To implement `Functor`, Lazy would need to either:

1. **Clone the value**: `Lazy::new(move || func(fa.evaluate().clone()))`. This requires `A: Clone`, which is an additional bound not present in `Functor`.
2. **Consume the Lazy**: Move out of the `Rc<LazyCell<...>>`, which is impossible with shared ownership.

`RefFunctor` correctly models Lazy's semantics: the mapping function receives a reference and produces an owned value, which is then memoized in a new `Lazy` cell.

### Comparison with PureScript

PureScript's `Data.Lazy` implements full `Functor`:
```purescript
instance functorLazy :: Functor Lazy where
  map f l = defer \_ -> f (force l)
```

This works because PureScript is a garbage-collected language where `force` returns an owned value (or rather, everything is reference-counted under the hood). In Rust, the ownership model makes `Lazy::evaluate()` return `&A`, not `A`.

PureScript's Lazy also implements `Apply`, `Applicative`, `Bind`, and `Monad`:
```purescript
instance applyLazy :: Apply Lazy where
  apply f x = defer \_ -> force f (force x)
instance applicativeLazy :: Applicative Lazy where
  pure a = defer \_ -> a
instance bindLazy :: Bind Lazy where
  bind l f = defer \_ -> force $ f (force l)
```

All of these consume the inner value by-value, which is only possible in a GC'd language.

### Assessment

**RefFunctor is the correct abstraction for Lazy in Rust.** The alternative would be to require `A: Clone` everywhere, which is overly restrictive and would make `Lazy<SomethingExpensive>` impractical. The library could potentially add a `CloneFunctor` trait for types where `A: Clone` is acceptable, but `RefFunctor` serves the primary use case well.

**One potential enhancement:** A `Functor` impl gated on `A: Clone` would allow Lazy to participate in generic Functor code when the inner type is clonable. This could be done with a newtype wrapper (e.g., `CloneLazy<'a, A: Clone, Config>`) to avoid orphan/overlap issues.

## 7. The Evaluable Trait

### Design

`Evaluable` (evaluable.rs line 41-68) is defined as:
```rust
pub trait Evaluable: Functor {
    fn evaluate<'a, A: 'a>(fa: Apply!(...)) -> A;
}
```

It represents a natural transformation `F ~> Id`, extracting an owned value from a functor context. Only `ThunkBrand` implements it (thunk.rs line 628-658).

### Assessment

**Well-designed for its purpose.** The trait serves a specific role in the `Free` monad machinery: `Free::evaluate` needs to run the effect layer to extract the next step. The `Evaluable` constraint captures exactly this capability.

**Placement in the hierarchy:** The `Functor` superclass bound is appropriate since `Evaluable` requires the type to already be a functor (you need to map over the continuation inside `Free`).

**Limitations are correctly documented** (evaluable.rs line 38-40):
- `Lazy` cannot implement it because `evaluate` returns `&A`, not `A`.
- `Trampoline` has no brand, so it cannot participate.

**One observation:** `Evaluable` is essentially `Comonad::extract` restricted to the `Functor`-based Kind. In PureScript, `Data.Lazy` implements `Comonad` with `extract = force`. The library does not currently have a `Comonad` typeclass, so `Evaluable` fills a niche role. If `Comonad` were added, `Evaluable` could potentially become a superclass or be subsumed.

## 8. The Deferrable Trait: Comparison with PureScript's Lazy Class

### Rust's Deferrable

`Deferrable<'a>` (deferrable.rs line 54-78):
```rust
pub trait Deferrable<'a> {
    fn defer(f: impl FnOnce() -> Self + 'a) -> Self where Self: Sized;
}
```

Implemented by:
- `Thunk<'a, A>` (thunk.rs line 356-382)
- `Trampoline<A>` (trampoline.rs line 530-556)
- `Lazy<'a, A, RcLazyConfig>` (lazy.rs line 812-844, requires `A: Clone`)
- `Lazy<'a, A, ArcLazyConfig>` via `SendDeferrable` (lazy.rs line 847-878)
- `TryThunk`, `TryTrampoline`, `TryLazy` (their respective files)

### PureScript's Control.Lazy

```purescript
class Lazy l where
  defer :: (Unit -> l) -> l
```

Instances: `Lazy (a -> b)`, `Lazy Unit`, `Lazy (Lazy a)`.

The `fix` combinator:
```purescript
fix :: forall l. Lazy l => (l -> l) -> l
fix f = go where go = defer \_ -> f go
```

### Key Differences

1. **No `fix` on `Deferrable`.** The Rust trait does not provide `fix`, and the documentation (deferrable.rs line 26-37) explains why: lazy self-reference requires `Rc`/`Arc` + interior mutability, which is specific to `Lazy`, not all `Deferrable` types. `Thunk` is consumed on evaluation, making self-reference impossible. Instead, concrete `rc_lazy_fix` and `arc_lazy_fix` functions are provided (lazy.rs line 735-803).

2. **Lifetime parameterization.** PureScript's `Lazy` has no lifetime parameter. Rust's `Deferrable<'a>` must be parameterized to support borrowed data in closures.

3. **The `FnOnce` vs `Unit ->` distinction.** PureScript's `defer` takes `Unit -> l`, which is re-callable. Rust's takes `impl FnOnce() -> Self`, which can only be called once. This is correct for Rust: the closure may capture move-only data.

4. **The `Clone` bound on Lazy's `Deferrable` impl.** `Lazy`'s `Deferrable::defer` (lazy.rs line 839-843) requires `A: Clone`:
   ```rust
   fn defer(f: impl FnOnce() -> Self + 'a) -> Self {
       RcLazy::new(move || f().evaluate().clone())
   }
   ```
   This flattens `Lazy<Lazy<A>>` into `Lazy<A>` by evaluating the inner lazy and cloning. PureScript does the same thing (line 142-143) but without needing an explicit `Clone` bound because of GC.

### Assessment

**Good translation with pragmatic Rust adaptations.** The decision to exclude `fix` from the trait and provide it as concrete functions is the correct Rust-native choice. The `Clone` requirement on `Lazy`'s `Deferrable` impl is an unavoidable consequence of Rust's ownership model.

**One gap:** There is no `Deferrable` instance for `TryThunk` documented in the module-level imports at try_thunk.rs line 23 (it imports `Deferrable`), suggesting the impl exists but I should verify. The `TryThunk::defer` inherent method exists (try_thunk.rs line 140-142), and the `Deferrable` trait impl likely delegates to it.

## 9. Missing Typeclass Instances

### Instances That Should Arguably Exist

1. **`Foldable` for `ArcLazy` (via `LazyBrand<ArcLazyConfig>`)**: Already exists (lazy.rs line 1168). Good.

2. **`Traversable` for `Thunk`**: Explicitly documented as impossible (thunk.rs line 74-78). `FnOnce` closures cannot be cloned, and `Traversable` requires the ability to reconstruct the structure. This is a fundamental incompatibility.

3. **`Traversable` for `Lazy`**: PureScript's `Lazy` implements `Traversable`:
   ```purescript
   instance traversableLazy :: Traversable Lazy where
     traverse f l = defer <<< const <$> f (force l)
   ```
   In Rust, this would require `A: Clone` (to extract from the Lazy) and would produce a `F<Lazy<B>>` where the inner Lazy is already evaluated. This is technically possible but requires care with the `Clone` bound. **This is a potentially useful addition.**

4. **`Comonad` for `Lazy`**: PureScript implements `Extend` and `Comonad` for `Lazy`. In Rust, `Comonad` does not exist as a typeclass yet. If added, `Lazy` would be a natural instance (`extract = evaluate().clone()` for `A: Clone`).

5. **`Applicative`/`Monad` for `Lazy`**: PureScript has these. In Rust, they would require `A: Clone` everywhere. The library has chosen `RefFunctor` instead, which is a more Rust-idiomatic approach. Adding `Applicative`/`Monad` for Lazy with a `Clone` bound would be possible but would create a less ergonomic API.

6. **`Semigroup`/`Monoid` for `Thunk`**: Already implemented (confirmed by QuickCheck tests at thunk.rs line 1049-1082).

7. **`Semigroup`/`Monoid` for `Trampoline`**: Not implemented. Since `Trampoline` has no brand, these would need to be inherent methods. A `lift2` method already exists (trampoline.rs line 322-328), so `Semigroup::append` could be `self.lift2(other, A::append)`. **This is a potential addition** via inherent methods.

### Instances That Correctly Do Not Exist

- `Functor` for `LazyBrand<ArcLazyConfig>`: Would require `Send` on the mapping function, which `Functor` does not guarantee.
- `RefFunctor` for `LazyBrand<ArcLazyConfig>`: Same issue. The inherent `ref_map` method exists but requires `Send` explicitly.
- Any HKT trait for `Trampoline`: Fundamentally blocked by `'static` (see Section 5).

## Summary

The HKT/Brand system integration with lazy types is well-designed and correctly implemented. The key findings are:

- **Brand definitions are complete and correct** for all lazy types, following established patterns.
- **Kind implementations correctly map brands to concrete types** with proper lifetime bounds.
- **Functor and Monad laws are satisfied** for Thunk, with thorough QuickCheck coverage.
- **Trampoline's lack of HKT traits is a fundamental limitation**, not a fixable gap, stemming from the `'static` requirement of `Box<dyn Any>` in the Free monad's continuation stack.
- **RefFunctor is the correct abstraction for Lazy in Rust**, dictated by the ownership model where `evaluate()` returns `&A`.
- **Evaluable is well-scoped** as a natural transformation `F ~> Id` for the Free monad machinery.
- **Deferrable is a good Rust adaptation of PureScript's Lazy class**, with the `fix` combinator correctly extracted to concrete functions.
- **The main gaps are in test coverage** (TryThunk, TryLazy, Foldable laws, Applicative laws) and **potentially useful missing instances** (Traversable for Lazy, Semigroup for Trampoline).
