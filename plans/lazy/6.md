# Analysis 6: The Deferrable Trait (Type Class)

## Overview

This analysis examines the `Deferrable` trait, which is the library's type class for types that support deferred/lazy construction. It covers the trait's design, its relationship to PureScript's `Control.Lazy` class, its implementations across the lazy type hierarchy, and areas for improvement.

## 1. Deferrable Trait Design

### Current Definition

File: `/home/jessea/Documents/projects/rust-fp-lib/fp-library/src/classes/deferrable.rs`

```rust
pub trait Deferrable<'a> {
    fn defer(f: impl FnOnce() -> Self + 'a) -> Self
    where
        Self: Sized;
}
```

The trait is minimalist: a single method, no supertraits, parameterized by a lifetime `'a`.

### Key Design Decisions

**Lifetime parameter on the trait itself.** `Deferrable<'a>` makes the lifetime part of the trait bound, not just the method. This means a type can implement `Deferrable` for different lifetimes. In practice, `Thunk<'a, A>` implements `Deferrable<'a>` (flexible lifetime), while `Trampoline<A>` implements `Deferrable<'static>` (fixed to `'static`). This is the correct choice given the constraints.

**`FnOnce` (not `Fn` or `FnMut`).** The thunk only needs to be called once. This is consistent with the uncurried/zero-cost philosophy and matches PureScript's `Unit -> l` semantics (which is effectively called once).

**No supertraits.** Unlike `Functor` or `MonadRec`, `Deferrable` has zero supertraits. It does not require `Kind` or any HKT machinery. This is notable because `Deferrable` is implemented on concrete types (e.g., `Thunk<'a, A>`), not on brand types. This puts it in a different category from the HKT-based type classes.

**`Self: Sized` bound.** Required because `impl FnOnce() -> Self` needs `Self` to be sized for the return position. This is a necessary constraint and excludes unsized types.

### Free Function Wrapper

```rust
pub fn defer<'a, D: Deferrable<'a>>(f: impl FnOnce() -> D + 'a) -> D {
    D::defer(f)
}
```

The free function is provided and follows the standard pattern. Usage: `let eval: Thunk<i32> = defer(|| Thunk::new(|| 42));`. The type must be annotated or inferable, which is typical for this library's dispatch pattern.

### SendDeferrable Companion

File: `/home/jessea/Documents/projects/rust-fp-lib/fp-library/src/classes/send_deferrable.rs`

```rust
pub trait SendDeferrable<'a> {
    fn send_defer(f: impl FnOnce() -> Self + Send + Sync + 'a) -> Self
    where
        Self: Sized;
}
```

This is the thread-safe counterpart, requiring the thunk to be `Send + Sync`. It parallels the `Deferrable`/`SendDeferrable` split, similar to `CloneableFn`/`SendCloneableFn` and `RefCountedPointer`/`SendRefCountedPointer`. Implemented only by `ArcLazy` and `ArcTryLazy`.

**`SendDeferrable` is NOT a supertrait of `Deferrable`.** These are independent traits. A type can implement one without the other. This is correct because `ArcLazy` cannot implement `Deferrable` (its thunk must be `Send`), so the two are genuinely distinct capabilities.

## 2. Comparison with PureScript's Control.Lazy

### PureScript Original

File: `/home/jessea/Documents/projects/purescript-control/src/Control/Lazy.purs`

```purescript
class Lazy l where
  defer :: (Unit -> l) -> l

fix :: forall l. Lazy l => (l -> l) -> l
fix f = go
  where
    go = defer \_ -> f go
```

PureScript's `Lazy` class has:
- One method: `defer :: (Unit -> l) -> l`
- One derived combinator: `fix :: (l -> l) -> l`
- Instances for functions (`a -> b`) and `Unit`

### Structural Comparison

| Aspect | PureScript `Lazy` | Rust `Deferrable` |
|--------|-------------------|-------------------|
| Method | `defer :: (Unit -> l) -> l` | `fn defer(f: impl FnOnce() -> Self + 'a) -> Self` |
| Lifetime | N/A (GC'd language) | `'a` parameter on trait |
| `fix` combinator | Yes, in module | **Missing** |
| Instances: functions | Yes (`a -> b`) | No |
| Instances: Unit | Yes | No |
| Supertraits | None | None |
| Naming | `Lazy` | `Deferrable` |

The method signatures map directly. PureScript's `Unit -> l` corresponds to Rust's `FnOnce() -> Self`. The semantic contract is identical: take a thunk that produces a value and wrap it in deferred form.

### The Missing `fix` Combinator

PureScript defines `fix` as:

```purescript
fix :: forall l. Lazy l => (l -> l) -> l
fix f = go where go = defer \_ -> f go
```

This creates a value as the fixed point of a function, using laziness to break the apparent circularity. `go` refers to itself via a lazy thunk.

**Can `fix` be implemented in Rust?** The answer depends on which `Deferrable` implementor we are considering.

For `Lazy<'a, A, RcLazyConfig>` (which is `Clone`), a version is theoretically possible:

```rust
fn fix<'a, A: Clone + 'a>(f: impl Fn(Lazy<'a, A, RcLazyConfig>) -> Lazy<'a, A, RcLazyConfig> + 'a) -> Lazy<'a, A, RcLazyConfig> {
    // We need self-referential laziness here.
    // The PureScript version works because `go` is a lazy value
    // that, when forced, calls `f(go)`.
    // In Rust, we'd need the Lazy to hold a clone of itself.
}
```

The core problem is that PureScript's `fix` relies on the runtime's lazy evaluation (the `go` binding is itself lazy because PureScript is lazy by default in this context). In Rust, we would need to create a `Lazy` that, when forced, references a clone of itself. Because `RcLazy` is `Clone` (sharing the underlying `Rc`), this is actually workable:

```rust
pub fn fix<'a, A: Clone + 'a>(
    f: impl Fn(RcLazy<'a, A>) -> RcLazy<'a, A> + 'a,
) -> RcLazy<'a, A> {
    use std::cell::OnceCell;
    use std::rc::Rc;

    let cell: Rc<OnceCell<RcLazy<'a, A>>> = Rc::new(OnceCell::new());
    let cell_ref = cell.clone();
    let result = RcLazy::new(move || {
        let go = cell_ref.get().expect("fix: cell not initialized").clone();
        f(go).evaluate().clone()
    });
    cell.set(result.clone()).ok();
    result
}
```

This is not trivial but is feasible for `Clone + Clone`-sharing types like `RcLazy`. For `Thunk` (which is not `Clone`), `fix` is not possible because `Thunk` is consumed on evaluation.

**Recommendation:** Add `fix` as a standalone function (not a trait method) in the `deferrable` module, specialized to `RcLazy`. It could also be added for `ArcLazy` via `SendDeferrable`. This would enable tying-the-knot patterns for lazy infinite data structures.

## 3. Implementation Survey

### Types That Implement Deferrable

| Type | Lifetime | Extra Bounds | Notes |
|------|----------|-------------|-------|
| `Thunk<'a, A>` | `'a` | `A: 'a` | Flattens: `f().evaluate()` |
| `Trampoline<A>` | `'static` | `A: 'static + Send` | Stack-safe defer via `Free::wrap` |
| `Lazy<'a, A, RcLazyConfig>` | `'a` | `A: Clone + 'a` | Flattens + clones: `f().evaluate().clone()` |
| `TryThunk<'a, A, E>` | `'a` | `A: 'a, E: 'a` | Same pattern as `Thunk` |
| `TryTrampoline<A, E>` | `'static` | `A + E: 'static + Send` | Delegates to `Trampoline::defer` |
| `TryLazy<'a, A, E, RcLazyConfig>` | `'a` | `A: Clone, E: Clone` | Flattens + clones result |
| `Free<ThunkBrand, A>` | `'static` | `A: 'static` | `Free::wrap(Thunk::new(move \|\| f().0))` |

### Types That Implement SendDeferrable

| Type | Bounds |
|------|--------|
| `Lazy<'a, A, ArcLazyConfig>` | `A: Clone + Send + Sync + 'a` |
| `TryLazy<'a, A, E, ArcLazyConfig>` | `A + E: Clone + Send + Sync + 'a` |

### Types That Should But Do Not Implement Deferrable

**`Free<ThunkBrand, A>` for non-`'static` lifetimes.** Currently only `impl Deferrable<'static> for Free<ThunkBrand, A>`. This is a fundamental limitation: `Free` uses `Box<dyn Any>` for type erasure, which requires `'static`. This cannot change without redesigning `Free`.

**Functions (`Fn(A) -> B`).** PureScript has `instance lazyFn :: Lazy (a -> b) where defer f = \x -> f unit x`. In Rust, this would be:

```rust
impl<'a, A, B> Deferrable<'a> for Box<dyn Fn(A) -> B + 'a> { ... }
```

However, `Deferrable` requires `Self: Sized`, and `dyn Fn` is unsized. For the `RcFnBrand`/`ArcFnBrand` wrappers (which are `Rc<dyn Fn(A) -> B>`), an implementation would be possible. This would be useful for composing lazily-constructed function pipelines.

**`Option<A>` and other standard types.** PureScript does not give `Option` a `Lazy` instance either. `defer` requires a way to "wrap" a thunk; `Option` has no natural notion of deferred computation.

## 4. Semantic Analysis of defer Across Types

The `defer` method has subtly different semantics for each implementor:

**Thunk:** `defer` is a flattening operation. It takes `FnOnce() -> Thunk<A>` and produces `Thunk<A>`, essentially a `join` on the thunk. The produced thunk, when evaluated, calls `f()` to get the inner thunk, then evaluates that.

```rust
fn defer(f: impl FnOnce() -> Self + 'a) -> Self {
    Thunk::new(move || f().evaluate())
}
```

**Trampoline:** `defer` is a stack-safety primitive. It wraps the thunk in a `Free::wrap(Thunk::new(...))` layer, ensuring that when the trampoline is evaluated, the construction of the inner trampoline happens in the trampoline loop rather than on the call stack. This is critical for recursive functions.

```rust
pub fn defer(f: impl FnOnce() -> Trampoline<A> + 'static) -> Self {
    Trampoline(Free::wrap(Thunk::new(move || f().0)))
}
```

**Lazy (RcLazyConfig):** `defer` flattens and clones. It creates a new `Lazy` that, when evaluated, forces the inner `Lazy` and clones the result. The `Clone` bound is required because `Lazy::evaluate()` returns `&A`, not `A`.

```rust
fn defer(f: impl FnOnce() -> Self + 'a) -> Self {
    RcLazy::new(move || f().evaluate().clone())
}
```

This reveals an important asymmetry: `Lazy::defer` requires `A: Clone` while `Thunk::defer` does not. The `Clone` bound is an artifact of `Lazy`'s memoization semantics (evaluation returns a shared reference, so extracting an owned value requires cloning). This means `Deferrable` cannot provide truly uniform abstraction over the lazy types; code generic over `Deferrable` cannot use `Lazy` unless `A: Clone`.

## 5. Interaction with Other Type Classes

### Deferrable and Functor/Monad

`Deferrable` and `Functor`/`Monad` live in different worlds:
- `Functor`, `Semimonad`, etc. are implemented on **brand types** (e.g., `ThunkBrand`).
- `Deferrable` is implemented on **concrete types** (e.g., `Thunk<'a, A>`).

There is no formal relationship between them in the type system. You cannot write a generic function that says "give me something that is both a Monad and Deferrable" using brand-level constraints, because `Deferrable` is not a brand-level trait.

This is actually appropriate. `Deferrable` is about constructing a value of a specific type from a thunk. It is closer to a constructor pattern than a structure-preserving transformation. PureScript's `Lazy` class is similarly independent of the Monad hierarchy.

### Deferrable and MonadRec

`MonadRec` (implemented by `ThunkBrand`) provides `tail_rec_m`, which uses a loop for stack safety within the monadic context. `Deferrable` (on `Thunk`) provides `defer` for constructing a thunk from a computation that returns a thunk.

For `Trampoline`, `defer` is the primary recursion mechanism: stack-safe recursive functions use `Trampoline::defer(move || ...)` to suspend recursive calls. But `Trampoline` cannot implement `MonadRec` (requires `'static`, cannot participate in HKT).

The relationship is complementary:
- `ThunkBrand + MonadRec`: stack-safe via `tail_rec_m` loop, `defer` is convenience.
- `Trampoline + Deferrable`: stack-safe via `defer` + trampoline loop, `MonadRec` unavailable.

### Deferrable and Evaluable

`Evaluable` (on brand types) provides `evaluate :: F<A> -> A`. `Deferrable` (on concrete types) provides `defer :: (() -> Self) -> Self`. These are conceptual duals:
- `Evaluable`: eliminates the lazy context, forcing computation.
- `Deferrable`: introduces the lazy context, deferring computation.

They are not formally connected in the type system, but they form a natural pair. Every type that implements `Deferrable` also has an `evaluate` method (either via `Evaluable` or inherent).

## 6. The Naming Choice: "Deferrable" vs "Lazy"

### Rationale for "Deferrable"

The naming analysis in `plans/lazy-revamp/naming-consistency-analysis.md` documents the decision. The original name was `Defer`, which collided with the `defer()` method on `Thunk` and `Trampoline`. The trait was renamed to `Deferrable` (adjective form) to avoid this collision.

### Comparison with Alternatives

| Name | Pros | Cons |
|------|------|------|
| `Lazy` (PureScript) | Matches upstream; short | Collides with `Lazy<'a, A>` type name |
| `Deferrable` (current) | Avoids collision; adjective form; descriptive | Longer; slightly awkward ("is this type deferrable?") |
| `Defer` (original) | Short; verb form | Collides with `defer()` method names |
| `LazyConstruct` | Descriptive | Too verbose; not standard FP |

**Assessment:** `Deferrable` is a reasonable choice. The name `Lazy` would be ideal from a FP purity standpoint but would cause confusion with the concrete `Lazy` type. The current name avoids ambiguity and is discoverable. The method name `defer` (matching PureScript) is good.

One minor issue: the free function is also called `defer`, which matches the trait method name. This is clean and consistent.

## 7. Does the Trait Properly Abstract Over the Three Lazy Types?

### The Abstraction Gap

The three core lazy types have fundamentally different characteristics:

| Property | Thunk | Trampoline | Lazy |
|----------|-------|-----------|------|
| Memoized | No | No | Yes |
| Stack-safe | No | Yes | N/A |
| Lifetime | `'a` | `'static` | `'a` |
| Clone bound on A for defer | No | No | Yes (`A: Clone`) |
| HKT-compatible | Yes | No | Partial (`RefFunctor` only) |

`Deferrable` abstracts over the construction pattern (take a thunk, produce a deferred value) but cannot abstract over the evaluation semantics. Code generic over `D: Deferrable<'a>` can construct a value but has no standard way to extract the result (no `evaluate` in the trait).

### The Clone Bound Problem

The `Lazy` implementation requires `A: Clone`, while `Thunk` and `Trampoline` do not. This means that a function generic over `Deferrable` cannot use `Lazy` unless it also requires `Clone`. In practice, this limits the trait's usefulness as a generic abstraction.

### What Deferrable Actually Achieves

Despite these gaps, `Deferrable` serves a specific purpose: it enables polymorphic construction of deferred values. For example, a library could define:

```rust
fn create_computation<'a, D: Deferrable<'a>>(x: i32) -> D
where
    D: From<Thunk<'a, i32>>,
{
    defer(move || Thunk::new(move || x * 2).into())
}
```

But in practice, this pattern is rarely used in the codebase. The concrete constructors (`Thunk::new`, `Trampoline::defer`, `RcLazy::new`) are preferred because they provide more precise control.

## 8. Trait Bound Analysis

### Are Bounds Too Restrictive?

**`Self: Sized`:** Necessary for `impl FnOnce() -> Self` in return position. Cannot be relaxed.

**`'a` on the trait:** Appropriate. Allows different types to specify different lifetime constraints. `Trampoline` uses `'static`, `Thunk` uses `'a`.

**No `Clone` bound on the trait:** Correct. `Clone` is only needed for `Lazy`'s implementation, not for `Thunk` or `Trampoline`. Pushing `Clone` into the trait would unnecessarily restrict the other implementors.

### Are Bounds Too Permissive?

**No bound connecting `Self` to any specific structure.** The trait says nothing about what `defer` should do semantically. It could be implemented as `fn defer(f: impl FnOnce() -> Self + 'a) -> Self { f() }`, which eagerly evaluates, technically satisfying the signature but violating the intent. There is no way to enforce laziness in Rust's type system, so this is a documentation contract rather than a type-level guarantee.

This matches PureScript's `Lazy` class, which also relies on convention rather than enforcement.

## 9. Ergonomics and Free Function Wrappers

### Current State

Two free functions are provided:
- `defer` in `classes/deferrable.rs` for `Deferrable`.
- `send_defer` in `classes/send_deferrable.rs` for `SendDeferrable`.

Both are re-exported via `functions.rs` through the `generate_function_re_exports!` macro.

### Usage Ergonomics

Typical usage:
```rust
let eval: Thunk<i32> = defer(|| Thunk::new(|| 42));
let memo: RcLazy<i32> = defer(|| RcLazy::new(|| 42));
let task: Trampoline<i32> = Deferrable::defer(|| Trampoline::pure(42));
```

The type annotation is required because `defer` is generic over `D: Deferrable<'a>` and the compiler cannot infer the output type from the thunk alone. This is consistent with other free functions in the library (e.g., `pure::<ThunkBrand, _>(42)`) but slightly less ergonomic because the user must annotate the result type rather than specifying a brand.

### Missing Derived Combinators

Beyond `defer` itself, PureScript's `Control.Lazy` module provides `fix`. The Rust module provides no derived combinators. Potential additions:

1. **`fix`** (discussed above in section 2).
2. **`defer_with`** (a version that takes an argument, like Scala's `Eval.defer`).
3. **`force`** (a complementary free function for evaluation, though this overlaps with `evaluate`).

## 10. Potential for Other Types

### Could IO/Future/Streams Implement Deferrable?

**Futures (`std::future::Future`).** A future is already inherently lazy (it does nothing until polled). However, `impl Deferrable<'a> for Pin<Box<dyn Future<Output = A>>>` would make sense semantically: `defer` would create a future that, when polled, creates and polls the inner future. This matches `async { f().await }`.

**Streams.** Similar to futures, a stream could implement `Deferrable` where `defer` creates a stream that, when first polled, constructs the inner stream. This would be useful for lazy stream initialization.

**IO types (hypothetical).** If the library had an IO monad type, `Deferrable` would be a natural fit. `defer` would suspend an IO action, and `fix` would enable recursive IO definitions.

**Iterator adaptors.** `std::iter::from_fn` already serves a similar role. An iterator could implement `Deferrable` where `defer` produces an iterator that lazily constructs the inner iterator on first `next()` call.

**`Vec`, `Option`, `Result`.** These do not have natural `Deferrable` implementations because they are eager data structures, not deferred computations. PureScript also does not provide `Lazy` instances for these.

### Recommendations for Expansion

The most valuable expansion would be for `Fn`-brand wrappers (`Rc<dyn Fn(A) -> B>`), matching PureScript's `instance lazyFn`. This would require relaxing or working around the `Self: Sized` constraint, possibly via a newtype wrapper.

## Summary of Findings and Recommendations

### Strengths

1. **Clean minimal design.** One method, no supertraits, correct lifetime parameterization.
2. **Comprehensive implementation coverage.** All six deferred types plus `Free` implement it.
3. **Good naming.** `Deferrable` avoids collision with concrete `defer()` methods and the `Lazy` type name.
4. **Thread-safety split.** `SendDeferrable` correctly handles the `Send + Sync` dimension.
5. **Free function provided.** `defer()` and `send_defer()` are available and re-exported.

### Weaknesses and Gaps

1. **No `fix` combinator.** PureScript's most important derived operation from `Lazy` is missing. This limits the ability to define self-referential lazy values.
2. **No evaluation counterpart in the trait.** `Deferrable` provides construction but not elimination. Pairing it with an `evaluate` method (or requiring `Evaluable`) would make it more useful for generic programming.
3. **`Clone` asymmetry.** `Lazy`'s `Deferrable` implementation requires `A: Clone` while others do not. This weakens generic abstraction.
4. **Concrete-type vs brand-type split.** `Deferrable` operates on concrete types while the rest of the class hierarchy operates on brands. This prevents composing it with `Functor`, `Monad`, etc. in generic bounds.
5. **No `Deferrable` for function types.** PureScript's `Lazy (a -> b)` instance has no Rust counterpart.
6. **Limited documentation of laws.** The trait should document the expected law: `defer(|| x) ` is observationally equivalent to `x` (modulo laziness/memoization).

### Concrete Recommendations

1. **Add `fix` for `RcLazy` and `ArcLazy`.** Implement as standalone functions in the `deferrable` module, not as trait methods (since it requires `Clone` and self-referential structure).

2. **Document the Deferrable law.** Add to the trait doc comment: "Implementors should satisfy: for any `x: Self`, `defer(|| x)` behaves identically to `x` when evaluated."

3. **Consider a `Deferrable + Evaluable` combo trait or function.** For code that needs both construction and elimination generically, a helper that combines both could be useful, though this spans the concrete/brand type boundary.

4. **Consider `Deferrable` for `Rc<dyn Fn(A) -> B>` / `Arc<dyn Fn(A) -> B>`.** This matches PureScript's function instance and would enable lazy function pipeline construction.

5. **Keep the naming.** `Deferrable` is good. Do not rename to `Lazy` (collision) or back to `Defer` (collision with methods).
