# Analysis: `Evaluable` trait

**File:** `fp-library/src/classes/evaluable.rs`

## Summary

`Evaluable` is an HKT type class that provides a natural transformation `F ~> Id`, meaning it extracts an owned value from a functor. Its sole purpose in the codebase is to serve as the interpretation mechanism for `Free::evaluate`, where it peels off one `F` layer from the `Wrap(F<Free<F, A>>)` variant. Currently only `ThunkBrand` implements it.

## Design Assessment

### Does the design make sense?

Yes, within its narrow role. `Free::evaluate` works on type-erased `FreeInner` variants in an iterative loop. When it encounters `FreeInner::Wrap(fa)` where `fa: F::Of<'static, Free<F, TypeErasedValue>>`, it needs to unwrap one `F` layer generically. Because `F` is a brand (not a concrete type), the only way to express "unwrap F for any inner type A" is through an HKT trait. An inherent method like `Thunk::evaluate(self)` cannot be called generically over the brand `F`, so the trait is necessary for the `Free` machinery.

The `Functor` superclass bound is justified: `FreeInner` requires `F: Functor + 'static`, and `Evaluable` should not be implementable for brands that cannot participate in `Free`.

### Is the trait necessary?

**For `Free::evaluate`, yes.** The alternative would be to remove the generic `Free<F, A>` and hardcode `Free` to work only with `ThunkBrand`, making `Trampoline` (which is `Free<ThunkBrand, A>`) the only Free monad. That would eliminate the need for the trait but also eliminate the generality of `Free`.

**For general use outside `Free`, no.** Every type that has an `evaluate` method already provides it as an inherent method (`Thunk::evaluate`, `SendThunk::evaluate`, `Trampoline::evaluate`, `Lazy::evaluate`, etc.). Users never call `evaluate::<ThunkBrand, _>(thunk)` in application code; they call `thunk.evaluate()`. The free function `evaluate` exists for consistency with the library's convention of providing free function wrappers, but it adds no practical value for end users.

### Relationship to standard FP abstractions

`Evaluable` is the library's name for the categorical concept of a natural transformation `F ~> Id` (the counit of an adjunction, or equivalently, `Comonad::extract`). The documentation acknowledges this by saying it "witnesses that a functor always holds a single value that can be extracted."

In PureScript, `Free`'s `runFree` takes a natural transformation `f ~> m` at the call site rather than requiring a type class on `F`. The Rust approach of baking interpretation into `Evaluable` is a reasonable concession to Rust's type system, and the documentation in `free.rs` already explains this trade-off clearly. The `fold_free` method provides the more flexible PureScript-style interpretation for cases where a separate target monad is needed.

## Issues and Concerns

### 1. Single implementor makes the trait feel like over-engineering

Only `ThunkBrand` implements `Evaluable`. The documentation correctly explains why `Lazy` (returns `&A`, not `A`) and `Trampoline` (no brand, no HKT) cannot implement it. `SendThunkBrand` does not implement `Functor` at all, so it is also excluded. This means the trait exists solely to parameterize `Free` over its interpretation functor, a generality that is used exactly once (`Trampoline = Free<ThunkBrand, A>`).

**Assessment:** This is not necessarily a flaw. The abstraction is clean and the overhead is minimal (one trait, one impl, ~100 lines). It keeps `Free` generic so that if a new evaluable functor were added in the future, it would "just work." The cost of removing it (hardcoding `ThunkBrand` into `Free`) would be low but would also mean `Free` could never be used with a different base functor.

### 2. The `evaluate` free function is not re-exported in `functions.rs` explicitly

The `evaluate` free function is auto-generated by the `generate_function_re_exports!` macro, so it does appear in `functions::*`. However, the name `evaluate` is generic enough that it could collide with other meanings in user code. This is a minor ergonomic concern.

### 3. Documentation says "extractable" but the trait consumes the functor

The doc says "A functor containing exactly one extractable value." The word "extractable" could suggest non-destructive access, but `evaluate` takes `fa` by value (consuming it). This is accurate for `Thunk` (which is `FnOnce`-based) but worth noting for clarity. The signature makes ownership transfer clear, so this is a documentation nuance rather than a bug.

### 4. Lifetime parameter `'a` in the signature is somewhat misleading

The trait method signature is:

```rust
fn evaluate<'a, A: 'a>(
    fa: Apply!(<Self as Kind!( type Of<'a, T: 'a>: 'a; )>::Of<'a, A>)
) -> A;
```

In practice, `Free` only uses this with `'static` types (due to `Any` requiring `'static`). The `'a` parameter exists because the HKT machinery requires it, but `Evaluable` is never called with non-`'static` lifetimes via `Free`. It could theoretically be called with non-`'static` lifetimes directly (e.g., `evaluate::<ThunkBrand, _>(Thunk::new(|| &local_ref))`), which is fine since `Thunk` supports arbitrary lifetimes.

**Assessment:** The lifetime generality is correct and appropriate. It does not over-constrain the trait.

### 5. No laws are documented

Unlike `Deferrable` (which documents a transparency law), `Evaluable` documents no laws. For a natural transformation `F ~> Id`, the expected law would be naturality:

```
evaluate(map(f, fa)) == f(evaluate(fa))
```

This follows from the `Functor` superclass and the fact that `evaluate` is a natural transformation, but documenting it would improve rigor and help implementors verify correctness.

### 6. Naming: `Evaluable` vs alternatives

The name `Evaluable` is clear in the context of lazy evaluation but somewhat non-standard in FP terminology. Alternatives considered:

- `Extractable` / `Comonad::extract`: More standard FP name for `F ~> Id`, but `Evaluable` does not require the full comonadic structure (`extend`/`duplicate`). Using a comonad-related name could be misleading.
- `Runnable`: Common in effect systems, but could be confused with running side effects.
- `Forceable`: Aligns with lazy evaluation terminology ("forcing a thunk"), but less standard.

**Assessment:** `Evaluable` is a reasonable name. It aligns with the `evaluate` method on all lazy types and is consistent with the library's terminology.

## Trait Bounds Assessment

The bounds are appropriate and minimal:

- `Evaluable: Functor` is necessary because `Free` requires `F: Functor`.
- `evaluate` takes `fa` by value, which is correct for consuming a thunk.
- The `A: 'a` bound comes from the Kind machinery and is standard.
- There are no unnecessary bounds like `Clone`, `Send`, or `Debug`.

## Edge Cases

1. **Zero-sized types:** `evaluate` works fine with ZSTs since it is just calling through to `Thunk::evaluate`.
2. **Panicking thunks:** If the thunk panics, `evaluate` will propagate the panic. This is expected behavior, not an edge case to handle.
3. **Recursive types:** `evaluate` on `Free` handles deep recursion via the iterative loop, which is the whole point.

## Recommendations

1. **Add a naturality law to the documentation.** This is the only concrete improvement worth making. The law `evaluate(map(f, fa)) == f(evaluate(fa))` should be documented and ideally tested via QuickCheck.

2. **Keep the trait as-is.** Despite having a single implementor, the abstraction is clean, minimal, and serves a clear purpose in the `Free` monad. The alternative (hardcoding `ThunkBrand` into `Free`) would save ~100 lines but sacrifice generality for no meaningful gain.

3. **Consider clarifying "extractable" in the documentation.** A minor wording improvement: "A functor holding exactly one value that can be extracted by consuming the functor" would make the ownership semantics clearer.

4. **No action needed on naming, bounds, or architecture.** The current design is sound.
