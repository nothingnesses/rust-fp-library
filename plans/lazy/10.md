# Lazy Hierarchy: Alternative Designs and Redesign Evaluation

## 1. Current Design Summary

The library provides six lazy evaluation types arranged along two axes (infallible/fallible, evaluation strategy):

| Type | Memoized | Stack-safe | HKT | Lifetime | Key Trait |
|------|----------|------------|-----|----------|-----------|
| `Thunk<'a, A>` | No | Partial | Full Monad | `'a` | `Evaluable`, `Functor`, `Semimonad` |
| `Trampoline<A>` | No | Yes | None | `'static` | Inherent methods only |
| `Lazy<'a, A, Config>` | Yes | N/A | `RefFunctor` | `'a` | `Deferrable` |
| `TryThunk<'a, A, E>` | No | Partial | Full | `'a` | `Bifunctor`, `Semimonad` |
| `TryTrampoline<A, E>` | No | Yes | None | `'static` | Inherent methods only |
| `TryLazy<'a, A, E, Config>` | Yes | N/A | Partial | `'a` | `Deferrable` |

The current architecture was deliberately designed this way after extensive prior analysis (documented in `plans/lazy-revamp/plan.md` and `plans/lazy-analysis/`). The three infallible types went through a renaming cycle: what started as `Eval`/`Task`/`Memo` became `Thunk`/`Trampoline`/`Lazy` for clarity.

## 2. Could the Three Types Be Unified into Fewer?

### 2.1 Why three types exist

The three types exist because Rust forces a choice that lazy languages like Haskell or PureScript do not:

1. **HKT compatibility requires lifetime polymorphism** (`'a`), but type erasure via `Box<dyn Any>` (needed for O(1) bind in Free) requires `'static`.
2. **Memoization requires shared ownership** (`Rc`/`Arc`), which means `evaluate` returns `&A`, not `A`. Returning `&A` makes `Functor`/`Monad` impossible (they require owned values).
3. **Stack safety requires the Free monad trampoline**, which needs type erasure.

These are three genuinely orthogonal constraints. Any two of the three types share at most one constraint, so collapsing any pair creates an irreconcilable conflict.

### 2.2 Could Thunk absorb Trampoline?

No. `Thunk` wraps `Box<dyn FnOnce() -> A + 'a>`. Its `bind` implementation is:

```rust
// fp-library/src/types/thunk.rs, line 188
pub fn bind<B: 'a>(self, f: impl FnOnce(A) -> Thunk<'a, B> + 'a) -> Thunk<'a, B> {
    Thunk::new(move || {
        let a = (self.0)();
        let thunk_b = f(a);
        (thunk_b.0)()
    })
}
```

Each `bind` nests one closure inside another, growing the call stack linearly. For 10,000 binds, this overflows the stack. `Trampoline` solves this by delegating to `Free<ThunkBrand, A>` (line 72 of `trampoline.rs`), which uses a `CatList` of type-erased continuations processed in a loop. The type erasure (`Box<dyn Any>`) forces `A: 'static`, which is why `Trampoline` cannot implement `Kind` traits.

Making `Thunk` stack-safe would require embedding the `CatList` continuation machinery, which means type erasure, which means `'static`, which means losing HKT compatibility. That would turn `Thunk` into `Trampoline` and leave no HKT-compatible deferred computation type.

### 2.3 Could Thunk absorb Lazy?

No. `Thunk` consumes itself on `evaluate(self) -> A`. `Lazy` returns a reference `evaluate(&self) -> &A` and is cloneable (all clones share the same cache). These are fundamentally different semantics. Making `Thunk` memoized would require interior mutability (`OnceCell`/`LazyCell`) and shared ownership (`Rc`/`Arc`), which would mean `evaluate` returns `&A`, which would break `Functor`/`Monad` instances.

### 2.4 Verdict

The three types cannot be reduced to two without losing functionality that the library actively uses. The separation is justified by Rust's type system constraints.

## 3. The Cats `Eval` Hierarchy (Now/Later/Always)

Scala Cats provides a single `Eval[A]` type with three internal variants:

```scala
sealed abstract class Eval[+A] {
  def value: A
  def memoize: Eval[A]
}
case class Now[A](value: A) extends Eval[A]        // Eager, memoized
case class Later[A](f: () => A) extends Eval[A]     // Lazy, memoized (computed once)
case class Always[A](f: () => A) extends Eval[A]    // Lazy, not memoized (recomputed each time)
```

`Eval` in Cats is a full `Monad` because:
- JVM has garbage collection (no ownership/borrowing concerns).
- `Later` can call `f()` and store the result, returning it by value on subsequent calls.
- `flatMap` uses a trampoline internally for stack safety.

### 3.1 Why a direct port does not work in Rust

The Cats design conflates memoization with computation. In Rust:

- **`Later` returning owned `A`** requires either cloning the cached value every time (adding `Clone` bound) or consuming the `Eval` (breaking the memoization contract). PureScript's `Data.Lazy` uses `force :: Lazy a -> a`, which works because the GC handles memory; Rust cannot replicate this.
- **Stack-safe `flatMap`** requires type erasure (`Box<dyn Any>`), which requires `'static`, which prevents capturing borrowed data.
- **The unified type** would need to be both `Clone` (for memoized variant sharing) and consumed-on-evaluate (for monadic bind), which is contradictory.

### 3.2 Partial adoption: an `Eval` enum with all three variants

One could define:

```rust
enum Eval<'a, A> {
    Now(A),
    Always(Box<dyn FnOnce() -> A + 'a>),
    Later(Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>),
}
```

But `map` and `bind` would need to handle each variant differently:
- `Now` and `Always` can produce owned `A`, enabling `Functor`/`Monad`.
- `Later` produces `&A`, requiring `Clone` to participate in `Functor`/`Monad`.

This is doable but introduces a runtime overhead (variant dispatch) and an inconsistent `Clone` bound (only needed for the `Later` case). The current design avoids this by making the memoization/non-memoization choice at the type level, where the compiler can optimize each path independently.

### 3.3 Verdict

A Cats-style `Eval` is possible but would be strictly worse in Rust: it adds runtime dispatch, inconsistent trait bounds, and makes the memoization semantics ambiguous at the type level. The current three-type design is the correct Rust adaptation of this pattern.

## 4. Should Trampoline Be a Special Case of Free?

It already is. `Trampoline<A>` is defined as:

```rust
// fp-library/src/types/trampoline.rs, line 70
pub struct Trampoline<A: 'static>(Free<ThunkBrand, A>);
```

This is a newtype wrapper around `Free<ThunkBrand, A>`. The `Free` monad (in `types/free.rs`) implements the "Reflection without Remorse" technique with `CatList`-based continuations for O(1) bind and stack safety.

### 4.1 Could Trampoline be removed in favor of exposing Free directly?

Technically yes, but there are good reasons to keep the newtype:

1. **Ergonomics**: `Trampoline<A>` is clearer than `Free<ThunkBrand, A>` for users who want stack-safe recursion.
2. **API surface control**: `Trampoline` exposes a curated API (`pure`, `new`, `defer`, `bind`, `map`, `evaluate`, `tail_rec_m`). `Free` exposes lower-level operations (`wrap`, `lift_f`, `erase_type`, `boxed_erase_type`).
3. **Documentation clarity**: The `Trampoline` name is widely recognized in FP as "stack-safe recursion via heap allocation."
4. **Future extensibility**: If the library ever adds a different base functor (e.g., for effect systems), `Free<EffectBrand, A>` would be a different specialization, and having the newtype prevents confusion.

### 4.2 Should Trampoline gain HKT traits via Free?

No. The `Free` monad documentation (lines 146-165 of `free.rs`) explains this thoroughly:

> `Free` does not implement HKT traits because it requires `A: 'static` for type erasure, which conflicts with the HKT `Kind` trait requiring support for any lifetime `'a`.

This is a fundamental and unavoidable limitation of the current HKT encoding in Rust.

### 4.3 Verdict

The current relationship (Trampoline as newtype over Free) is the right design. Keep both.

## 5. Could Thunk and Lazy Be Merged (Thunk + Memoization = Lazy)?

This question asks whether a single type could serve both roles with an optional memoization flag.

### 5.1 The ownership incompatibility

`Thunk::evaluate(self) -> A` consumes the thunk. `Lazy::evaluate(&self) -> &A` borrows and caches. These are different function signatures with different ownership semantics. A unified type would need to support both, which means:

```rust
enum Unified<'a, A> {
    Thunk(Box<dyn FnOnce() -> A + 'a>),
    Lazy(Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>),
}
```

But `evaluate` cannot have two different signatures. You would need either:
- `evaluate(self) -> A` (breaks `Lazy` semantics; consuming a shared `Rc` does not make sense).
- `evaluate(&self) -> &A` (breaks `Thunk` semantics; `FnOnce` cannot be called through `&self`).
- `evaluate(&self) -> Cow<'_, A>` where `A: Clone` (works but requires `Clone` on everything and introduces runtime dispatch).

### 5.2 The `Cow` approach

```rust
impl<'a, A: Clone> Unified<'a, A> {
    fn evaluate(&self) -> Cow<'_, A> {
        match self {
            Self::Thunk(f) => Cow::Owned(f()),   // ERROR: can't call FnOnce through &self
            Self::Lazy(cell) => Cow::Borrowed(LazyCell::force(cell)),
        }
    }
}
```

This does not even compile because `FnOnce` requires `self` ownership. Switching to `Fn` (callable by reference) means the closure cannot consume captured values, which is too restrictive.

### 5.3 Verdict

`Thunk` and `Lazy` serve fundamentally incompatible evaluation contracts. Merging them would require sacrificing either consuming evaluation (needed for `Monad`) or shared memoization (needed for caching). The current separation is correct.

## 6. Alternative Memoization Strategies

### 6.1 Current approach: `LazyCell` / `LazyLock`

The current `Lazy` implementation uses `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>` for single-threaded and `Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>` for thread-safe variants. This was a deliberate choice made during the lazy revamp (documented in `plans/lazy-revamp/plan.md`, decision "Once/Cell Removal"):

> Removed `Once`, `OnceCell`, and `OnceLock` wrappers and brands. These were largely superseded by `Memo` and the move to standard library types, simplifying the API surface.

### 6.2 `OnceCell` / `OnceLock` alternative

`OnceCell<A>` and `OnceLock<A>` are the older primitives that `LazyCell`/`LazyLock` build upon. Using them directly would mean managing the thunk separately:

```rust
struct LazyManual<'a, A> {
    cell: Rc<OnceCell<A>>,
    thunk: Rc<RefCell<Option<Box<dyn FnOnce() -> A + 'a>>>>,
}
```

This is strictly more complex than `LazyCell`, which bundles cell + thunk. The standard library's `LazyCell` was stabilized in Rust 1.80 precisely to eliminate this boilerplate. There is no benefit to reverting.

### 6.3 `std::sync::Once` + raw pointer

This was the pre-1.80 approach used by crates like `once_cell`. It requires unsafe code. The current approach delegates all unsafety to the standard library, which is safer and more maintainable.

### 6.4 External crate alternatives

- `once_cell` crate: Now redundant since `OnceCell`/`OnceLock`/`LazyCell`/`LazyLock` are in std.
- `lazy_static!`: Macro-based, `'static` only, no lifetime polymorphism.

### 6.5 Verdict

The current `LazyCell`/`LazyLock` approach is the best available option for Rust. It leverages battle-tested standard library primitives, avoids unsafe code in library code, and supports both single-threaded and thread-safe variants via the `LazyConfig` trait.

## 7. Could the `'static` Constraint on Trampoline Be Relaxed?

### 7.1 Source of the constraint

The `'static` requirement comes from `Free`, which uses `Box<dyn Any>` for type erasure in its continuation list. `Any` requires `'static` because downcasting a `Box<dyn Any>` to a concrete type would be unsound if the value contained dangling references.

### 7.2 GATs (Generic Associated Types)

GATs (stabilized in Rust 1.65) allow associated types to be parameterized by lifetimes:

```rust
trait Kind {
    type Of<'a, A: 'a>: 'a;
}
```

The library already uses this pattern (via `impl_kind!`). However, GATs do not help with the `'static` constraint because the issue is not in the `Kind` trait but in `Box<dyn Any>`. There is no lifetime-polymorphic `Any` trait in Rust.

### 7.3 Custom `Any` with lifetime parameter

One could imagine:

```rust
trait AnyLifetime<'a>: 'a {
    fn type_id(&self) -> TypeId;
}
```

But `TypeId` does not distinguish lifetimes; `TypeId::of::<&'a str>() == TypeId::of::<&'b str>()` for all `'a`, `'b`. This means downcasting would be unsound. Fixing this would require changes to the Rust language itself (lifetime-aware `TypeId`), which is not on the roadmap.

### 7.4 Alternative: existential types / `impl Trait` in type position

Future Rust features like type alias `impl Trait` (TAIT) could allow:

```rust
type Continuation<'a, F> = impl FnOnce(Box<dyn Any + 'a>) -> Free<F, Box<dyn Any + 'a>> + 'a;
```

But `Box<dyn Any + 'a>` does not help because `Any` still requires `'static` for safe downcasting.

### 7.5 The real blocker: safe type erasure with lifetimes

The fundamental issue is that Rust has no safe mechanism to erase a type that contains a non-`'static` lifetime and later recover it. This is a language-level limitation, not a library design issue. Unsafe code could technically relax this (using transmute and careful lifetime tracking), but the library explicitly prioritizes safety:

> Refactored `Free` to remove all `unsafe` code, prioritizing safety and auditability over the zero-cost abstraction of `ManuallyDrop`. (plans/lazy-revamp/plan.md)

### 7.6 Verdict

The `'static` constraint on `Trampoline`/`Free` cannot be relaxed with current or foreseeable Rust features without introducing unsafe code. This is the correct trade-off.

## 8. Monomorphized vs. Polymorphic Design

### 8.1 Current design: polymorphic via `LazyConfig`

`Lazy<'a, A, Config>` is parameterized by `Config: LazyConfig`, which determines whether it uses `Rc<LazyCell>` or `Arc<LazyLock>`. This avoids duplicating the entire type for each threading model.

### 8.2 Alternative: monomorphized `RcLazy` and `ArcLazy` as separate types

```rust
struct RcLazy<'a, A>(Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>);
struct ArcLazy<'a, A>(Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>);
```

Pros:
- Simpler type signatures (no `Config` parameter).
- Potentially better compiler error messages.
- Each type can have methods specific to its threading model without trait gymnastics.

Cons:
- Code duplication across `RcLazy` and `ArcLazy`.
- Cannot write generic code over both variants.
- Would need separate brands (`RcLazyBrand`, `ArcLazyBrand`) instead of `LazyBrand<Config>`.

### 8.3 Current aliases as pragmatic middle ground

The library already provides type aliases:

```rust
// fp-library/src/types/lazy.rs, lines 625-628
pub type RcLazy<'a, A> = Lazy<'a, A, RcLazyConfig>;
pub type ArcLazy<'a, A> = Lazy<'a, A, ArcLazyConfig>;
```

These give users simple names while retaining the ability to write generic code over `Config`.

### 8.4 Verdict

The current polymorphic-with-aliases approach is the best of both worlds. No change needed.

## 9. How Other Rust FP Libraries Handle Lazy Evaluation

### 9.1 Rustica

The `rustica` library (at `/home/jessea/Documents/projects/rustica/`) takes a simpler approach:

**Thunk** (`src/datatypes/wrapper/thunk.rs`): A generic `Thunk<F, T>` where `F: Fn() -> T`. This is monomorphic over the function type (no trait object / `dyn`), so it is `Clone` when `F: Clone`, and `Fn` (not `FnOnce`), meaning the same thunk can be evaluated repeatedly. This is lighter-weight but less flexible (cannot capture move-only values via `FnOnce`).

**Memoizer** (`src/datatypes/wrapper/memoizer.rs`): A keyed memoization cache (`HashMap`-based with LRU eviction), not a lazy value. This serves a completely different use case (function memoization, not lazy evaluation).

Rustica has no `Trampoline` or `Free` monad. There is no stack-safe recursion primitive. The `Evaluate` trait in rustica is a simple `fn evaluate(&self) -> Self::Source`, evaluating by reference (not consuming).

### 9.2 Comparison

| Feature | fp-library | Rustica |
|---------|-----------|---------|
| Thunk closure type | `FnOnce` (consuming) | `Fn` (reusable) |
| Thunk type erasure | `Box<dyn FnOnce>` | Monomorphic generic `F` |
| Stack-safe recursion | `Trampoline` via `Free` | None |
| Memoized lazy value | `Lazy` via `LazyCell`/`LazyLock` | None (only keyed `Memoizer`) |
| HKT integration | Full (Brand pattern) | Minimal (`HKT` trait with `Source`/`Output`) |
| Thread safety abstraction | `LazyConfig` trait | None (only `Arc`-based `Memoizer`) |

fp-library's design is significantly more sophisticated, which is appropriate for a library whose goal is "comprehensive type classes and profunctor-based optics."

## 10. Could Async/Await or Generators Provide a Better Foundation?

### 10.1 Async/await

Rust's `async`/`await` is essentially a compiler-generated state machine for cooperative multitasking. An `async fn` that `.await`s a value is conceptually similar to a thunk that suspends and resumes. However:

- **Runtime requirement**: Async requires a runtime (tokio, async-std, etc.) to poll futures. The library targets zero-runtime pure computation.
- **`Pin` complications**: Async futures are `!Unpin` by default, adding complexity to composition.
- **No memoization**: `Future::poll` is designed for progress, not caching.
- **Different semantics**: `async` is about I/O concurrency, not lazy pure computation.

That said, for stack-safe recursion specifically, there is an interesting hack: Rust's async state machines are heap-allocated and do not grow the stack. A recursive `async fn` that `.await`s itself is stack-safe. This is occasionally used in practice (e.g., `async_recursion` crate). But it requires a runtime, introduces `Pin`/`Unpin` complexity, and cannot integrate with the library's synchronous HKT system.

### 10.2 Generators (coroutines)

Rust generators (currently unstable behind `#![feature(coroutines)]`) provide `yield` semantics:

```rust
fn fibonacci() -> impl Iterator<Item = u64> {
    std::iter::from_coroutine(|| {
        let (mut a, mut b) = (0, 1);
        loop {
            yield a;
            (a, b) = (b, a + b);
        }
    })
}
```

Generators could replace some uses of `Trampoline` for producing sequences of values. But:

- **Unstable**: Not available on stable Rust.
- **Iterator-focused**: Generators yield multiple values; `Trampoline` produces one final value.
- **No monadic composition**: Generators do not naturally support `bind`/`flatMap`.

### 10.3 Verdict

Neither async/await nor generators are suitable replacements for the current lazy hierarchy. They solve different problems (I/O concurrency and iterator generation, respectively). The trampoline pattern remains the correct approach for stack-safe pure computation in synchronous Rust.

## 11. PureScript Reference Designs

### 11.1 `Control.Lazy` (purescript-control)

PureScript's `Lazy` type class (`/home/jessea/Documents/projects/purescript-control/src/Control/Lazy.purs`) is:

```purescript
class Lazy l where
  defer :: (Unit -> l) -> l
```

This is the equivalent of the library's `Deferrable` trait:

```rust
// fp-library/src/classes/deferrable.rs, line 21
pub trait Deferrable<'a> {
    fn defer(f: impl FnOnce() -> Self + 'a) -> Self where Self: Sized;
}
```

The mapping is exact. PureScript's `fix` combinator (lazy fixed point) is also expressible via `Deferrable`.

### 11.2 `Data.Lazy` (purescript-lazy)

PureScript's `Lazy a` type (`/home/jessea/Documents/projects/purescript-lazy/src/Data/Lazy.purs`) is a foreign type implemented in JavaScript with full `Monad` instances:

```purescript
instance bindLazy :: Bind Lazy where
  bind l f = defer \_ -> force $ f (force l)
```

This works because JavaScript is garbage-collected, so `force` can return the value by value without ownership concerns. In Rust terms, this would require `A: Clone` everywhere (which PureScript does not need to state because all PureScript values are immutable and reference-counted by the runtime).

The library's `Lazy` correctly uses `RefFunctor` instead of `Functor`, acknowledging this fundamental difference. The library's `Thunk` serves the role that PureScript's `Lazy` serves for monadic composition (consuming evaluation, no memoization).

## 12. Trade-off Summary

| Design Decision | Current Choice | Alternative | Why Current Is Better |
|----------------|---------------|-------------|----------------------|
| Three types vs. one | Three separate types | Cats-style `Eval` enum | No runtime dispatch; constraints at type level; cleaner trait impls |
| Trampoline via Free | Newtype over `Free<ThunkBrand, A>` | Inline trampoline | Reusable Free monad; O(1) bind via CatList; code sharing |
| LazyCell/LazyLock | Standard library primitives | Manual OnceCell + UnsafeCell | No unsafe; battle-tested; future-proof |
| Config trait for threading | `LazyConfig` with associated types | Separate `RcLazy`/`ArcLazy` types | Code reuse; generic programming; aliases for ergonomics |
| Thunk uses FnOnce | `Box<dyn FnOnce() -> A + 'a>` | `Fn` (like Rustica) | Supports move semantics; more flexible captures |
| Lazy returns &A | `evaluate(&self) -> &A` | `evaluate(self) -> A` with Clone | No Clone bound; enables shared caching |

## 13. Concrete Redesign Proposal

After thorough analysis, the current design is well-justified and does not warrant a fundamental redesign. However, there are several targeted improvements worth considering:

### 13.1 Add `Comonad` to `Lazy` (when ready)

PureScript's `Data.Lazy` implements `Comonad` with `extract = force`. The library's `Lazy` could implement a `RefComonad` (analogous to `RefFunctor`):

```rust
pub trait RefComonad: RefFunctor {
    fn extract<'a, A: 'a>(
        wa: Apply!(<Self as Kind!( type Of<'a, T: 'a>: 'a; )>::Of<'a, A>),
    ) -> &'a A;  // This requires careful lifetime management

    fn extend<'a, A: 'a, B: 'a>(
        f: impl FnOnce(&Apply!(<Self as Kind!( type Of<'a, T: 'a>: 'a; )>::Of<'a, A>)) -> B + 'a,
        wa: Apply!(<Self as Kind!( type Of<'a, T: 'a>: 'a; )>::Of<'a, A>),
    ) -> Apply!(<Self as Kind!( type Of<'a, T: 'a>: 'a; )>::Of<'a, B>);
}
```

However, the lifetime semantics of `extract` returning `&'a A` from a `Lazy<'a, A>` are tricky because the reference is tied to the `Rc`/`Arc` inside `Lazy`, not to the `Lazy` value itself. This needs careful design; it may require `Lazy` to expose the inner `Rc`/`Arc` for lifetime anchoring.

### 13.2 Consider adding a `Thunk::memoize` convenience

While `Thunk` and `Lazy` serve different roles, a convenience method to convert a `Thunk` into a `Lazy` already exists via `From`:

```rust
// fp-library/src/types/lazy.rs, lines 535-549
impl<'a, A> From<Thunk<'a, A>> for Lazy<'a, A, RcLazyConfig> {
    fn from(eval: Thunk<'a, A>) -> Self {
        Self::new(move || eval.evaluate())
    }
}
```

An inherent `memoize` method on `Thunk` could improve discoverability:

```rust
impl<'a, A: 'a> Thunk<'a, A> {
    pub fn memoize(self) -> Lazy<'a, A, RcLazyConfig> {
        Lazy::from(self)
    }
}
```

### 13.3 Consider a `Trampoline::memoize` convenience

Similarly:

```rust
impl<A: 'static + Send> Trampoline<A> {
    pub fn memoize(self) -> Lazy<'static, A, RcLazyConfig> {
        Lazy::from(self)
    }
}
```

This already works via the existing `From` impl at line 553 of `lazy.rs`.

### 13.4 Document the Cats `Eval` correspondence explicitly

Users coming from Scala/Cats will look for `Eval`. A documentation section or type alias could help:

```rust
/// Cats `Eval.now(a)` corresponds to `Thunk::pure(a)`.
/// Cats `Eval.always(f)` corresponds to `Thunk::new(f)`.
/// Cats `Eval.later(f)` corresponds to `Lazy::new(f)`.
/// Cats `Eval.defer(f)` corresponds to `Thunk::defer(f)`.
///
/// For stack-safe monadic recursion (Cats `Eval.flatMap` with trampolining),
/// use `Trampoline` instead.
```

### 13.5 No changes needed to Free/Trampoline relationship

The current `Trampoline` as newtype over `Free<ThunkBrand, A>` is clean, correct, and well-documented. The `Free` monad's "Reflection without Remorse" implementation provides O(1) bind, which is critical for deep recursion patterns.

## 14. Conclusion

The current Lazy hierarchy is the result of iterative refinement through multiple rounds of analysis and redesign (documented extensively in `plans/lazy-revamp/` and `plans/lazy-analysis/`). The three-type design (Thunk/Trampoline/Lazy) reflects genuine, irreconcilable constraints in Rust's type system:

1. **HKT compatibility** conflicts with **type erasure** (needed for stack safety).
2. **Owned evaluation** conflicts with **shared memoization**.
3. **Lifetime polymorphism** conflicts with **`Box<dyn Any>`** (needed for heterogeneous continuation lists).

Alternative designs from other ecosystems (Cats' `Eval`, PureScript's `Lazy`) work because those languages have garbage collection and do not distinguish ownership from borrowing. The current design is the correct Rust adaptation.

The six-type system (three infallible + three fallible) is a natural consequence of the library's principle of explicit error handling. `TryThunk`, `TryTrampoline`, and `TryLazy` mirror their infallible counterparts with `Result` semantics, providing type-level documentation of failure modes.

No fundamental redesign is warranted. The suggested improvements are incremental: convenience methods, documentation for cross-ecosystem users, and future `Comonad` integration.
