# Lazy Evaluation Hierarchy: API Ergonomics, Documentation Quality, and User Experience

## 1. Naming Analysis

### Thunk

The name `Thunk` is well-chosen. It is a standard term in programming language theory for a deferred, unevaluated computation. Rust developers familiar with functional programming or compiler internals will recognize it immediately. For Rust newcomers, the doc comment on line 41 of `thunk.rs` ("A deferred computation that produces a value of type `A`") clarifies the meaning promptly.

One subtle issue: the name `Thunk` does not communicate its key differentiator from `Trampoline`, which is that it supports arbitrary lifetimes and HKT traits but is not stack-safe for deep `bind` chains. A user skimming type names alone cannot distinguish the two.

### Trampoline

`Trampoline` is also a standard term for the pattern of converting recursive calls into iterative steps. The name accurately conveys the "bouncing" evaluation loop that ensures stack safety. This is immediately recognizable to anyone who has implemented or read about tail-call optimization via trampolining.

The doc comment on line 37 of `trampoline.rs` opens with "A lazy, stack-safe computation," which is accurate and clear.

### Lazy

`Lazy` communicates memoization semantics well, aligning with Rust's own `std::cell::LazyCell` and `std::sync::LazyLock`. The type does exactly what its name promises: defers a computation and caches the result.

### Try* Variants

`TryThunk`, `TryTrampoline`, and `TryLazy` follow the `Try` prefix convention used in `TryFrom`/`TryInto` in the standard library. This is intuitive: "try" signals fallibility, so users will expect `Result`-based evaluation. The convention is consistent across all three types.

### Verdict

The naming is strong overall. The one gap is that the *relationship* between the three core types is not obvious from names alone. A user encountering `Thunk` and `Trampoline` for the first time may not understand why both exist. The documentation addresses this (see section 2), but the names themselves do not.

## 2. API Surface: Which Type to Use When

### Decision Tree in Documentation

The library provides a comparison table in multiple locations:

1. **lib.rs** (lines 114-118): A comprehensive table with columns for stack safety, memoization, lifetimes, and HKT traits.
2. **thunk.rs** (lines 56-63): A comparison table embedded in the Thunk doc comment.
3. **trampoline.rs** (lines 54-57): A brief "When to Use" section.

The table in `lib.rs` is the most complete and well-structured. It accurately represents the implementation:

| Type | Stack Safe? | Memoized? | Lifetimes? | HKT Traits |
|------|-------------|-----------|------------|------------|
| `Thunk<'a, A>` | Partial (`tail_rec_m` only) | No | `'a` | Full (Functor, Applicative, Monad) |
| `Trampoline<A>` | Yes | No | `'static` | None |
| `Lazy<'a, A>` | N/A | Yes | `'a` | `RefFunctor` |

The workflow example in `lib.rs` (lines 127-178) is excellent. It demonstrates a realistic scenario (expression evaluator) where all three types (in their `Try*` forms) work together. This is the strongest piece of user-facing documentation in the hierarchy.

### Missing Documentation

The `Try*` variants are not represented in the comparison table. A user has to extrapolate that `TryThunk` is "Thunk but with `Result`." The table should be expanded to include the fallible variants, or a note should explain their relationship. The description at `try_thunk.rs` line 53 ("This is `Thunk<'a, Result<A, E>>` with ergonomic combinators for error handling") is precise but only visible after navigating to that type's docs.

### Discoverability Gaps

1. **No centralized "lazy evaluation" module.** The six types are spread across `types/thunk.rs`, `types/trampoline.rs`, `types/lazy.rs`, and their `try_*` counterparts. There is no `types/lazy_hierarchy.rs` or umbrella module with a conceptual overview. A user browsing `types.rs` sees them listed alphabetically among 20+ other modules.

2. **`LazyConfig` is exposed but not well introduced.** A new user encountering `Lazy<'a, A, RcLazyConfig>` in method signatures may be confused by the third type parameter. The type alias `RcLazy<'a, A> = Lazy<'a, A, RcLazyConfig>` (line 689 of `lazy.rs`) and `ArcLazy<'a, A> = Lazy<'a, A, ArcLazyConfig>` (line 692) help, but the default parameter (`Config: LazyConfig = RcLazyConfig`) means that `Lazy<'_, i32>` works for the simple case. The doc examples inconsistently use `Lazy::<_, RcLazyConfig>::new(...)` vs `RcLazy::new(...)`, which could confuse users about the preferred spelling.

## 3. Method Naming Consistency

### The `evaluate()` Method

All six types use `evaluate()` as the primary forcing/running method:

| Type | Method | Signature | Returns |
|------|--------|-----------|---------|
| `Thunk` | `evaluate(self) -> A` | Consumes self | Owned value |
| `Trampoline` | `evaluate(self) -> A` | Consumes self | Owned value |
| `Lazy` | `evaluate(&self) -> &A` | Borrows self | Reference |
| `TryThunk` | `evaluate(self) -> Result<A, E>` | Consumes self | Owned Result |
| `TryTrampoline` | `evaluate(self) -> Result<A, E>` | Consumes self | Owned Result |
| `TryLazy` | `evaluate(&self) -> Result<&A, &E>` | Borrows self | Reference Result |

**Consistency:** The name `evaluate()` is used uniformly across all types, which is a significant strength. No type uses `run()`, `force()`, or `get()`.

**Semantic mismatch with `Lazy`:** The critical difference is that `Lazy::evaluate` takes `&self` and returns `&A`, while `Thunk::evaluate` and `Trampoline::evaluate` take `self` (consuming it) and return `A`. This is a correct design decision (Lazy is shared via Rc/Arc and memoized, so consuming it would be wrong), but users who switch between types may be surprised that `lazy.evaluate()` returns a reference while `thunk.evaluate()` returns an owned value. The `*` dereference is required:

```rust
// Thunk: owned
assert_eq!(thunk.evaluate(), 42);
// Lazy: reference, must deref
assert_eq!(*lazy.evaluate(), 42);
```

This is visible throughout the doc examples (e.g., `lazy.rs` line 483: `assert_eq!(*memo.evaluate(), 42)`). The consistency of `*` in Lazy examples is good, but it would be worth a note in the Lazy docs explaining why evaluate returns a reference.

### Other Method Names

- `bind`, `map`, `map_err`, `defer`, `catch`, `lift2`, `then`: These are consistent across all types that support them.
- `memoize()` and `memoize_arc()`: Present on `Thunk`, `Trampoline`, `TryThunk`, and `TryTrampoline`. This is a well-named bridge from computation types to caching types.
- `ref_map` on `Lazy`: Named to distinguish it from `Functor::map` (which takes owned values). The `RefFunctor` trait makes this clear at the trait level.

**No inconsistencies found.** Method naming is one of the strongest aspects of this API.

## 4. Constructor Patterns

### Infallible Types

| Constructor | `Thunk` | `Trampoline` | `Lazy` |
|-------------|---------|-------------|--------|
| `new(f)` | Wraps a closure | Wraps a closure | Wraps a closure |
| `pure(a)` | Wraps a value | Wraps a value | Wraps a value |
| `defer(f)` | Defers construction of a Thunk | Defers construction of a Trampoline | N/A |

This is consistent and intuitive. `new` takes a closure, `pure` takes a value, `defer` takes a closure that returns the type itself.

### Fallible Types

| Constructor | `TryThunk` | `TryTrampoline` | `TryLazy` |
|-------------|------------|-----------------|-----------|
| `new(f)` | `f: FnOnce() -> Result<A, E>` | `f: FnOnce() -> Result<A, E>` | `f: FnOnce() -> Result<A, E>` |
| `ok(a)` | Wraps a success value | Wraps a success value | Wraps a success value |
| `err(e)` | Wraps an error value | Wraps an error value | Wraps an error value |
| `defer(f)` | Defers construction | Defers construction | N/A |
| `pure(a)` | **Deprecated** | N/A | N/A |

The `ok`/`err` naming for fallible types is good, mirroring `Result::Ok`/`Result::Err`. The deprecation of `TryThunk::pure` (line 100-103 of `try_thunk.rs`) in favor of `ok` is the right call, since `pure` is confusing when the type inherently involves `Result`.

**Issue:** `TryTrampoline` and `TryLazy` never had `pure` at all, so the API was already inconsistent before the deprecation. `TryThunk::pure` was the outlier that got corrected.

**Missing:** `TryLazy` does not have a `defer` method. This is correct since `Lazy` is about memoization (not deferred construction chains), but it could surprise users who expect the pattern to extend.

### The `From` Conversions

The library provides comprehensive `From` implementations for inter-conversion:

- `Thunk -> Lazy` (via `From`)
- `Trampoline -> Lazy` (via `From`)
- `Lazy -> Thunk` (requires `Clone`, via `From`)
- `Lazy -> Trampoline` (requires `Clone`, via `From`)
- `Thunk -> Trampoline` (via `From`)
- `TryThunk -> RcTryLazy` (via `From`)
- `TryTrampoline -> RcTryLazy` (via `From`)

And convenience methods: `memoize()`, `memoize_arc()`.

This is thorough and well-designed. The `Clone` requirement on `Lazy -> Thunk/Trampoline` is correctly documented (line 305 of `thunk.rs`).

## 5. Error Messages and Compiler Guidance

### Missing Compile-Fail Tests

There are **no compile-fail tests** (`trybuild`) related to any of the lazy types. The `tests/ui/` directory contains only tests for `Send`/`Sync` bounds on function brands (`rc_fn_not_send.rs`, `new_send_not_send.rs`, `new_send_not_sync.rs`).

This is a significant gap. Common mistakes that should have helpful error messages include:

1. **Trying to send a `Thunk` across threads.** `Thunk` wraps `Box<dyn FnOnce() -> A + 'a>`, which is not `Send`. A user who tries `std::thread::spawn(move || thunk.evaluate())` will get a raw compiler error about `dyn FnOnce() -> A` not implementing `Send`, which may be hard to interpret.

2. **Trying to use `Trampoline` with borrowed data.** The `'static` bound on `Trampoline` will produce a lifetime error, but the error message will point at the `Free<ThunkBrand, A>` internals rather than saying "Trampoline requires 'static; use Thunk if you need borrowing."

3. **Using `Lazy<'a, A, ArcLazyConfig>::new` without `Send` on the closure.** The `ArcLazyConfig` variant requires `Send` on the closure, which will produce errors about unsatisfied bounds. Users may not immediately understand they need `RcLazyConfig` instead.

### Type Annotations Required

The `TryThunk`, `TryTrampoline`, and `TryLazy` types require explicit type annotations in many contexts because the error type `E` cannot be inferred. Every doc example includes type annotations like `TryThunk<i32, ()>` or `TryTrampoline<i32, String>`. This is inherent to the design (two type parameters with no default) but could be improved with type aliases for common error types, or a note in the docs about why annotations are needed.

## 6. Documentation Quality

### Strengths

1. **Every public method has doc comments.** The library uses custom procedural macros (`#[document_signature]`, `#[document_parameters]`, `#[document_returns]`, `#[document_examples]`) to enforce a structured documentation format. This is extremely thorough.

2. **Every method has a code example.** This is rare and valuable. The examples are runnable (they appear in `///` doc comments) and tested by `cargo test --doc`.

3. **Trade-off tables are embedded in type-level docs.** The `Thunk` doc comment (lines 56-63 of `thunk.rs`) includes a comparison table, making it easy to understand the type's position in the hierarchy.

4. **Algebraic properties are documented.** The `Thunk` doc comment (lines 66-70 of `thunk.rs`) states the monad laws that Thunk satisfies.

5. **Limitations are documented.** The `Thunk` doc (lines 74-78) explains why `Traversable` cannot be implemented.

### Weaknesses

1. **No module-level overview linking all six types.** The `types.rs` module doc (lines 1-14) mentions `Identity` and `Lazy` but does not provide a lazy evaluation overview. Users must find the overview in `lib.rs`.

2. **`Lazy` type-level doc is less detailed than `Thunk`.** The `Lazy` struct doc (lines 400-429 of `lazy.rs`) does not include a comparison table or guidance on when to use it vs other types.

3. **`LazyConfig` trait is documented but intimidating.** Lines 43-65 of `lazy.rs` describe it as "open for third-party implementations," which may confuse users who just want to use `RcLazy` or `ArcLazy`. The extensibility documentation is accurate but serves an advanced audience.

4. **Panic documentation on `Lazy` is good** (lines 405-412 of `lazy.rs`), directing users to `TryLazy` for panic-safe memoization.

5. **`Evaluable` trait doc** (lines 37-40 of `evaluable.rs`) correctly explains that `Lazy` cannot implement it because `evaluate` returns `&A`. This is helpful for users who wonder why `Lazy` doesn't implement the same HKT traits as `Thunk`.

## 7. README Accuracy

The README's "Lazy Evaluation & Effect System" section (lines 178-249 of `README.md`) is **accurate and well-written**. Specific findings:

1. **Table is correct.** The comparison table at lines 182-186 matches the implementation. `Thunk` is partial stack-safe, `Trampoline` is fully stack-safe, `Lazy` is N/A for stack safety.

2. **"Why Three Types" section is accurate.** Line 192 correctly states that `Thunk`'s `tail_rec_m` is stack-safe but deep `bind` chains will overflow. Line 193 correctly states that `Thunk` and `Trampoline` re-run every time.

3. **Workflow example is correct and compilable.** The expression evaluator example (lines 201-248) accurately demonstrates all three `Try*` types working together. The comment at line 213 ("Capture owned data for 'static closure") is a helpful hint about the `'static` requirement.

4. **Missing: `Try*` types are not in the table.** The README table only covers the three base types. Users must extrapolate that `TryThunk = Thunk + Result`, etc. A note or expanded table would help.

5. **Minor inaccuracy potential:** Line 193 says "re-run every time you call `.evaluate()`." For `Thunk`, this is somewhat misleading because `Thunk::evaluate` consumes `self`, so it can only be called once. Saying "re-computes when evaluated" would be more precise. (Technically, if you `.clone()` first and evaluate each clone, you get re-computation, but Thunk is not Clone.)

## 8. Discoverability

### Finding the Right Type

A user who searches "lazy" in docs will find `Lazy`, `LazyConfig`, `LazyBrand`, `RcLazy`, `ArcLazy`, `TryLazy`, and `rc_lazy_fix`/`arc_lazy_fix`. This is good.

A user who searches "deferred" or "delayed" will find nothing in type names but will find it in doc comments (e.g., `thunk.rs` line 1: "Deferred, non-memoized computation").

A user who searches "stack safe" or "recursion" will find `Trampoline` and `TryTrampoline` via doc comments.

### Finding the Right Method

The `functions.rs` re-exports (lines 36-53) include `rc_lazy_fix` and `arc_lazy_fix` but do not re-export `evaluate`, `defer`, or `memoize` as free functions. These are only available as methods on the types themselves. For `defer`, there is a free function in `deferrable.rs` (line 105), re-exported via the `generate_function_re_exports!` macro. For `evaluate`, there is a free function in `evaluable.rs` (line 96), but it works via the HKT `Evaluable` trait, meaning it only supports `ThunkBrand`, not `Trampoline` or `Lazy`.

This creates an asymmetry:
- `defer(|| Thunk::new(|| 42))` works as a free function (via `Deferrable`).
- `evaluate(thunk)` works as a free function (via `Evaluable`), but only for `Thunk`.
- `trampoline.evaluate()` must be called as a method.
- `lazy.evaluate()` must be called as a method.

This is a correct design decision (since `Trampoline` and `Lazy` do not participate in HKT), but it may surprise users.

## 9. Standard Trait Implementations

### Debug

All six types implement `Debug`:
- `Thunk`: `"Thunk(<unevaluated>)"` (line 858 of `thunk.rs`)
- `Trampoline`: `"Trampoline(<unevaluated>)"` (line 576 of `trampoline.rs`)
- `Lazy`: Confirmed at line 1365 of `lazy.rs`
- `TryThunk`: Confirmed at line 1761 of `try_thunk.rs`
- `TryLazy`: Confirmed at line 668 of `try_lazy.rs`
- `TryTrampoline`: Not explicitly searched but likely follows the pattern.

The `Debug` impls do not evaluate the computation, which is the correct behavior (evaluating on debug-print would violate laziness).

### Display

**No type implements `Display`.** This is a reasonable choice since there is no meaningful way to display an unevaluated computation, and displaying would require evaluation.

### Clone

- `Thunk`: **Not Clone.** This is correct; it wraps `Box<dyn FnOnce()>`, which cannot be cloned.
- `Trampoline`: **Not Clone.** Same reason.
- `Lazy`: **Clone.** Cloning shares the underlying memoized cell (Rc/Arc). This is correct and documented (line 437-460 of `lazy.rs`).
- `TryThunk`: **Not Clone.**
- `TryTrampoline`: **Not Clone.**
- `TryLazy`: **Clone.** Same as `Lazy`.

### Send/Sync

- `Thunk`: Not Send (wraps `Box<dyn FnOnce()>`).
- `Trampoline`: Not Send (wraps `Free<ThunkBrand, A>` which contains non-Send thunks).
- `Lazy<_, _, RcLazyConfig>` (RcLazy): Not Send (uses `Rc`).
- `Lazy<_, _, ArcLazyConfig>` (ArcLazy): Send + Sync (uses `Arc<LazyLock>`).
- `TryLazy` follows the same pattern.

The thread safety story is documented in the `ArcLazyConfig` doc (line 286 of `lazy.rs`) and in the `memoize_arc()` methods on `Thunk`/`Trampoline`.

### Missing Standard Traits

- **`PartialEq`/`Eq`:** None of the types implement equality comparison. This is correct since you would need to evaluate to compare, which is a side effect.
- **`Hash`:** Same reasoning.
- **`Default`:** Not implemented. Could be useful for `Lazy` if `A: Default`, but the absence is reasonable.
- **`From<A>`:** Not implemented for direct value wrapping. Users must use `pure(a)` or `ok(a)`. This is a conscious design choice to keep the API functional-style.

## 10. Common Pitfalls and Guardrails

### Pitfall 1: Deep `bind` chains on `Thunk` cause stack overflow

`Thunk::bind` (line 194-203 of `thunk.rs`) creates a nested closure. Deep chains will overflow the stack. The doc comment at line 167-168 warns: "Note: Each `bind` adds to the call stack. For deep recursion, use `Trampoline` instead." This is appropriate but could be more prominent (e.g., a `# Panics` or `# Stack Safety` section).

The stack safety test (`stack_safety.rs`) tests `Thunk` only via `tail_rec_m` (line 69-80), not via deep `bind` chains. There is no test demonstrating that deep `Thunk::bind` chains *do* overflow. Adding such a test (or a compile-fail test that documents the limitation) would be valuable.

### Pitfall 2: Evaluating a `Thunk` twice

Since `Thunk::evaluate` consumes `self`, calling it twice is a compile-time error. This is a good guardrail. However, users coming from Haskell may expect to evaluate a thunk multiple times. The doc comment at line 42-44 of `thunk.rs` explains this: "Since `evaluate` takes `self` by value, a `Thunk` can only be evaluated once."

### Pitfall 3: `Lazy::evaluate` returns a reference, not an owned value

Users who switch from `Thunk::evaluate() -> A` to `Lazy::evaluate() -> &A` will get type errors. The `*` dereference is visible in all Lazy examples, but there is no explicit warning about this semantic difference.

### Pitfall 4: `Trampoline` requires `'static`

The `'static` bound on `Trampoline` (line 73 of `trampoline.rs`) means closures captured by `new`, `defer`, and `bind` cannot borrow from the calling scope. Users must `clone()` or `move` owned data into closures. The README's expression evaluator example (line 213) shows this pattern: `let expr = expr.clone();`.

There is no compile-time guardrail beyond Rust's normal lifetime errors, which may be cryptic since they point to `Free<ThunkBrand, A>` internals.

### Pitfall 5: Forgetting to specify `LazyConfig`

When writing `Lazy::new(|| 42)`, the default config parameter (`RcLazyConfig`) kicks in, so this works. But `Lazy::pure(42)` is defined on the specific `impl<'a, A> Lazy<'a, A, RcLazyConfig>` block, so it requires the config to be inferred or specified. In practice, the default parameter handles most cases. The type aliases `RcLazy` and `ArcLazy` simplify this further.

### Pitfall 6: `memoize_arc()` evaluates eagerly

Both `Thunk::memoize_arc()` (line 290-295 of `thunk.rs`) and `Trampoline::memoize_arc()` (line 293-298 of `trampoline.rs`) evaluate the computation eagerly because the inner closure is not `Send`. The result is then wrapped in an `ArcLazy`. The doc comments explain this clearly ("The thunk is evaluated eagerly because its inner closure is not Send"), but users may expect `memoize_arc` to be lazy.

### Pitfall 7: Panic poisoning on `Lazy`

If the initializer closure panics, the `Lazy` instance is poisoned. The `Lazy` struct doc (lines 405-412 of `lazy.rs`) documents this and directs users to `TryLazy` for panic-safe memoization. This is a good guardrail in documentation.

### Pitfall 8: `TryThunk::pure` deprecation

`TryThunk::pure` is deprecated (line 100-103 of `try_thunk.rs`) in favor of `ok`. Users who find older examples or copy patterns from `Thunk` will get a deprecation warning. The deprecation message is clear: "Use `ok` instead for consistency with TryTrampoline and TryLazy."

## Summary of Recommendations

### High Priority

1. **Add `Try*` types to the comparison table** in `lib.rs` and the README. Users need to see the full 6-type picture at a glance.
2. **Add compile-fail tests** for common misuse patterns: sending Thunk across threads, using borrowed data with Trampoline, and using non-Send closures with ArcLazy.
3. **Clarify that `Thunk::evaluate` consumes self vs `Lazy::evaluate` borrows self** in a shared overview section. The semantic difference in `evaluate()` return types is the single most likely source of confusion.

### Medium Priority

4. **Document the `memoize_arc()` eager evaluation behavior more prominently.** Consider renaming it to `evaluate_and_share_arc()` or adding a `# Note` section.
5. **Add a warning section to Thunk about deep bind chains.** The current note at line 167-168 is easy to miss.
6. **Standardize doc examples to prefer type aliases** (`RcLazy::new(...)` over `Lazy::<_, RcLazyConfig>::new(...)`) for clarity.
7. **Correct the README claim** (line 193) that Thunk and Trampoline "re-run every time you call `.evaluate()`." Since both consume self, the phrasing should be "re-compute the result when evaluated" without implying repeated calls.

### Low Priority

8. Add `Default` implementation for `Lazy<'a, A>` where `A: Default`.
9. Consider adding a `SendDeferrable` bound on `Trampoline` closures to enable sending Trampolines across threads (currently they are !Send due to the internal Thunk).
10. Expand the `types.rs` module doc to include a brief overview of the lazy evaluation hierarchy, with links to the `lib.rs` documentation.
