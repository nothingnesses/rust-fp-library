# Deep Analysis: Trampoline, Free, and Stack-Safe Evaluation

## 1. Internal Representation

### The Enum-Based Approach

`Trampoline<A>` is a thin newtype wrapper around `Free<ThunkBrand, A>` (line 73-76 of `trampoline.rs`):

```rust
pub struct Trampoline<A: 'static>(Free<ThunkBrand, A>);
```

The actual structure lives in `Free<F, A>`, which uses a three-variant `FreeInner` enum (lines 110-138 of `free.rs`):

```rust
pub enum FreeInner<F, A> {
    Pure(A),
    Wrap(Apply!(<F as Kind!(...)>::Of<'static, Free<F, A>>)),
    Bind {
        head: Box<Free<F, TypeErasedValue>>,
        continuations: CatList<Continuation<F>>,
        _marker: PhantomData<A>,
    },
}
```

**Assessment: This is correct and well-designed.** The three-variant approach follows the "Reflection without Remorse" paper (Atze van der Ploeg and Oleg Kiselyov). In a classical trampoline, you would see `Done`, `Suspend` (or `More`), and sometimes `FlatMap`. Here:

- `Pure` corresponds to `Done`.
- `Wrap` corresponds to `Suspend`, parameterized over a functor (in this case `Thunk`).
- `Bind` corresponds to `FlatMap`, but instead of a single continuation, it stores a `CatList` of type-erased continuations for O(1) associative bind.

The key insight is that `Bind` stores a `CatList<Continuation<F>>` rather than a single boxed closure. This prevents the quadratic blowup from left-associated bind chains. A classical `FlatMap(Box<Free>, Box<dyn FnOnce(A) -> Free>)` representation degrades to O(n^2) when you have chains like `((a.bind(f)).bind(g)).bind(h)` because each evaluation must traverse from the outside in. The `CatList` approach accumulates continuations in O(1) and processes them sequentially.

### The `Option` Wrapper

`Free<F, A>` wraps `FreeInner` in an `Option`:

```rust
pub struct Free<F, A>(pub(crate) Option<FreeInner<F, A>>);
```

This serves a dual purpose:
1. It allows `take()` semantics for move-out-of-self in `bind` and `evaluate` (lines 306, 361, 429 of `free.rs`).
2. It enables an iterative `Drop` implementation (lines 490-512) that avoids stack overflow when dropping deep bind chains.

The `Option` wrapper is a pragmatic Rust-specific choice. In languages with garbage collection, this is unnecessary. Here it prevents recursive `Drop` from blowing the stack on deep chains, which is a real concern since a chain of 100,000 binds would create 100,000 nested `Bind` variants.

**Potential concern:** The `expect()` calls on `.take()` are technically infallible if the API is used correctly (each `Free` value is consumed exactly once), but they introduce a runtime panic path rather than compile-time enforcement. The comments acknowledge this as an internal invariant. An alternative would be `MaybeUninit`, but `Option` is simpler and safer in practice.

## 2. Stack Safety of `evaluate()` / `run()`

The evaluation loop lives at lines 419-467 of `free.rs`:

```rust
pub fn evaluate(self) -> A where F: Evaluable {
    let mut current: Free<F, TypeErasedValue> = self.erase_type();
    let mut continuations: CatList<Continuation<F>> = CatList::empty();

    loop {
        let inner = current.0.take().expect("...");
        match inner {
            FreeInner::Pure(val) => {
                match continuations.uncons() {
                    Some((continuation, rest)) => {
                        current = continuation(val);
                        continuations = rest;
                    }
                    None => {
                        return *val.downcast::<A>().expect("...");
                    }
                }
            }
            FreeInner::Wrap(fa) => {
                current = <F as Evaluable>::evaluate(fa);
            }
            FreeInner::Bind { head, continuations: inner_continuations, .. } => {
                current = *head;
                continuations = inner_continuations.append(continuations);
            }
        }
    }
}
```

**This is genuinely stack-safe.** The loop maintains two mutable variables (`current` and `continuations`) and never recurses. Each iteration either:

1. **Pure + continuations remaining:** applies the next continuation, producing a new `Free` to process in the next iteration.
2. **Pure + no continuations:** terminates with the final value.
3. **Wrap:** evaluates the suspended effect (a `Thunk` in the `Trampoline` case), producing a new `Free`.
4. **Bind:** flattens the nested structure by moving the `head` to `current` and appending inner continuations to the outer list.

### Edge Cases

**Deeply nested Bind-in-Bind:** The `Bind` case handles this by flattening via `CatList::append`, which is O(1). Each loop iteration peels one layer off, so arbitrarily nested `Bind` structures are handled without stack growth.

**Wrap chains:** When a `Wrap(Thunk(...))` evaluates, it calls `Thunk::evaluate`, which invokes a closure. That closure may return another `Free::Wrap(...)`. This is fine because the outer loop will process the new `Wrap` in the next iteration. However, there is a subtlety: if the `Thunk` closure itself does deep recursion *before returning*, that recursion is NOT trampolined by `Free`. The trampoline only governs the iteration of the `Free` structure. This is why `Trampoline::defer` exists: it ensures the recursive call returns a `Free::Wrap(Thunk(...))` rather than executing inline, keeping the stack depth constant.

**`erase_type` on Wrap:** When `evaluate` starts, it calls `self.erase_type()` which, for `Wrap` variants, maps over the functor to erase inner types. For `Thunk`, this is O(1) (it wraps the thunk in another thunk). But if the Free structure contains deeply nested `Wrap` variants at the top level, the initial `erase_type` could recurse. In practice this does not happen because `Wrap` contains exactly one `Free` child, and the `Bind` case stores the head as already type-erased `Free<F, TypeErasedValue>`. The `erase_type` for a `Bind` variant just reuses the existing head and continuations (line 370-379), so there is no deep recursion.

**Memory concern during evaluation:** Each continuation application (`continuation(val)`) allocates a new `Free` value. For a chain of N binds, the loop runs N iterations, each performing a `downcast` and a continuation call. The `CatList::uncons` is amortized O(1). Total evaluation cost is O(N) in time and the peak memory is bounded by the size of the largest intermediate value plus the remaining continuation list.

## 3. Why Trampoline Does NOT Implement HKT Functor/Monad

This is **intentional, not a design flaw**, and the documentation is explicit about why (lines 143-165 of `free.rs`).

### The Core Conflict

The library's `Kind` trait requires:

```rust
type Of<'a, A: 'a>: 'a;
```

This means the type constructor must accept *any* lifetime `'a`. But `Free`'s internal representation uses `Box<dyn Any>` for type erasure of continuations, and `Any` requires `'static`:

```rust
pub type TypeErasedValue = Box<dyn Any>;  // line 93 of free.rs
pub type Continuation<F> = Box<dyn FnOnce(TypeErasedValue) -> Free<F, TypeErasedValue>>;
```

You cannot downcast a `Box<dyn Any>` that contains a non-`'static` reference. This is a fundamental limitation of Rust's `Any` trait. Therefore, `Free<F, A>` requires `A: 'static`, which means it cannot satisfy `Kind::Of<'a, A: 'a>` for arbitrary `'a`.

### Why Not Use a Naive Free?

The library explicitly considered and rejected a naive `enum Free { Pure(A), Wrap(F<Free<F,A>>) }` definition that would support HKT (lines 157-165 of `free.rs`). The reasons are:

1. **Stack safety:** Evaluation of the naive `Free` would require recursion (pattern matching and calling itself), which overflows on deep chains.
2. **Performance:** `bind` on the naive `Free` is O(N) because it must traverse to the leaf `Pure` to attach the continuation. Left-associated binds become O(N^2).

This is the correct tradeoff. Stack safety and O(1) bind are essential for a production trampoline. HKT compatibility is a nice-to-have that would require either unsafe code or a different type erasure mechanism.

### Consequences

- `Trampoline` provides `map`, `bind`, `pure` as inherent methods rather than trait implementations.
- It cannot be used polymorphically where `Functor<Brand>` or `Monad<Brand>` is expected.
- There is no `TrampolineBrand` in `brands.rs` (confirmed by inspection).
- The `MonadRec` trait (line 59 of `monad_rec.rs`) is implemented by `ThunkBrand`, not by `Trampoline`.

The library effectively maintains two parallel hierarchies: the HKT hierarchy (for `Thunk`, `Option`, `Vec`, etc.) and the standalone `Trampoline`/`Free` hierarchy (for stack safety). This is documented and intentional.

### Potential Future Direction

A hypothetical `TypeErasedValue` that uses unsafe transmute-based erasure instead of `dyn Any` could lift the `'static` restriction. Libraries like `erased-serde` demonstrate this pattern. However, the unsafety burden would be significant and the benefit marginal, since `Trampoline` values are typically constructed from owned data.

## 4. The `'static` Lifetime Requirement

### Source of the Requirement

The `'static` bound comes from `Box<dyn Any>` (line 93 of `free.rs`). Every value that passes through the continuation chain must be downcasted via `Any`, which requires `'static`. The `Trampoline` struct enforces this with `A: 'static` (line 73 of `trampoline.rs`), and all closures passed to `new`, `bind`, `defer`, etc. require `+ 'static`.

### Consequences

1. **Cannot borrow into a Trampoline:** You cannot create a `Trampoline<&'a str>` for a non-static `'a`. This means borrowed data must be cloned or owned before being placed in a `Trampoline`.

2. **Closures cannot borrow from the environment:** All closures passed to `bind`, `map`, `new`, and `defer` must be `'static`, meaning they can only capture owned values or `'static` references. This is natural for most recursive computations but can require explicit `move` and cloning.

3. **Interop friction with Thunk:** `Thunk<'a, A>` supports arbitrary lifetimes, but conversion to `Trampoline` requires `'a = 'static` (line 512 of `trampoline.rs`): `impl From<Thunk<'static, A>> for Trampoline<A>`.

### Is It Too Restrictive?

**In practice, no.** The `'static` requirement is the price for stack safety via type erasure. Consider the alternatives:

- **GATs (Generic Associated Types):** Rust's GATs could theoretically enable a `Free` that is polymorphic over lifetimes without type erasure, by encoding the continuation type at the type level. However, this would require existential types or trait objects for heterogeneous continuation chains, leading back to `'static` or unsafe code.

- **Unsafe erasure:** One could use `unsafe` transmutes to erase and restore types without the `Any: 'static` constraint. This would be complex and error-prone.

- **Arena allocation:** Using an arena allocator could allow borrowed references within trampolined computations, but this would change the API surface dramatically.

The `Thunk<'a, A>` type fills the gap for computations that need lifetime polymorphism but do not need deep stack safety. The library documents this split clearly in the module docs and the `When to Use` section (lines 54-57 of `trampoline.rs`).

## 5. Relationship to Free Monad: Is Trampoline `Free Identity`?

### Classical Theory

In Haskell, `Trampoline a = Free Identity a`, where:

```haskell
data Free f a = Pure a | Free (f (Free f a))
-- Trampoline a = Free Identity a
-- which simplifies to: Pure a | Free (Identity (Free Identity a))
-- which is: Pure a | Free (Trampoline a)
-- i.e., Done a | More (Trampoline a)
```

### What This Library Does

This library uses `Free<ThunkBrand, A>` rather than `Free<IdentityBrand, A>`. The key difference:

- `Identity<Free<IdentityBrand, A>>` would just be `Free<IdentityBrand, A>` itself (identity adds no effect).
- `Thunk<Free<ThunkBrand, A>>` is `FnOnce() -> Free<ThunkBrand, A>`, which adds *laziness* as the effect.

This is actually `Free ((->) ()) a`, which in Haskell is also used as a trampoline. It is the correct choice for Rust because:

1. **Laziness is not the default in Rust.** In Haskell, `Free Identity` works because `Identity (Free Identity a)` is already lazy (by Haskell's evaluation model). In Rust, you need explicit laziness via a closure wrapper, which is what `Thunk` provides.

2. **Using `IdentityBrand` would make `Wrap` useless.** `Free::Wrap(Identity(inner_free))` would be equivalent to just `inner_free`. There would be no way to suspend computation.

So `Trampoline` is not `Free Identity` but rather `Free Thunk`, which is the correct Rust analogue of Haskell's `Free Identity` (where Haskell's implicit laziness plays the role of `Thunk`).

### Should Trampoline Be Defined as `Free Identity`?

No. `Free<IdentityBrand, A>` would not provide any suspension mechanism in Rust's eager evaluation model. The `Wrap` variant would contain `Identity<Free<IdentityBrand, A>>`, which is just `Free<IdentityBrand, A>`. Without a way to defer computation, you lose the core purpose of trampolining.

The current design is correct: `Free<ThunkBrand, A>` gives you `Thunk<Free<ThunkBrand, A>>` in the `Wrap` position, which is `FnOnce() -> Free<ThunkBrand, A>`. This is exactly the laziness needed for stack-safe recursion.

## 6. TryTrampoline: Error Handling Soundness

### Representation

`TryTrampoline<A, E>` is a newtype over `Trampoline<Result<A, E>>` (line 38-41 of `try_trampoline.rs`):

```rust
pub struct TryTrampoline<A: 'static, E: 'static>(Trampoline<Result<A, E>>);
```

### Assessment

This is the standard "error monad transformer" approach: layer `Result` inside the base monad. It is sound and follows established patterns from Haskell's `ExceptT e m a = m (Either e a)`.

**Strengths:**

- **Short-circuit semantics in bind** (lines 216-220): On `Err`, the continuation is not called and the error propagates immediately (well, as immediately as the trampoline evaluates it). This is correct.
- **Error recovery via `catch`** (lines 238-246): Symmetric to `bind`, matches on `Err` to invoke the recovery function.
- **Stack safety is inherited:** Since the inner `Trampoline` handles all stack safety, `TryTrampoline` gets it for free.

**Potential concern:** The `bind` implementation (line 216) pattern-matches the `Result` inside a closure passed to `Trampoline::bind`. When the result is `Err(e)`, it creates a new `Trampoline::pure(Err(e))`. This means error propagation through N binds creates N `Pure(Err(e))` values, each immediately unwrapped by the next continuation. The cost is O(N) for propagating through N binds after an error, which matches the semantics (each bind must check for error) and cannot be improved without short-circuit evaluation at the `Free` level.

### `tail_rec_m` Implementation

The `tail_rec_m` for `TryTrampoline` (lines 360-375) wraps the state in `Result<S, E>`, threading errors through the loop state. This is clever: instead of creating a separate `TryStep` type, it reuses `Trampoline::tail_rec_m` with `Result<S, E>` as the state. When an error occurs, it becomes `Step::Done(Err(e))`, terminating the loop. This is correct and efficient.

## 7. `catch_unwind` on TryTrampoline

### Implementation

Lines 535-539 of `try_trampoline.rs`:

```rust
pub fn catch_unwind(f: impl FnOnce() -> A + std::panic::UnwindSafe + 'static) -> Self {
    TryTrampoline::new(move || {
        std::panic::catch_unwind(f).map_err(crate::utils::panic_payload_to_string)
    })
}
```

This is constrained to `TryTrampoline<A, String>` (line 504), meaning it converts panic payloads to string error messages.

### Is This a Good Idea?

**Partially.** There are legitimate use cases and real concerns:

**Legitimate uses:**
- Catching panics from third-party code or FFI boundaries.
- Providing a last-resort error boundary for computations that might panic.
- The `UnwindSafe` bound is correctly required, which prevents catching panics in the presence of broken invariants.

**Concerns:**

1. **Panics are not errors.** Rust's convention is that panics indicate programming bugs, not recoverable errors. Catching panics and converting them to `Result::Err` can mask bugs.

2. **`UnwindSafe` is a weak guarantee.** The `UnwindSafe` trait does not prevent all problems with panic recovery. Mutexes can be poisoned, data structures can be in inconsistent states. The bound is a heuristic, not a proof of safety.

3. **String error type is lossy.** Converting a panic payload to `String` discards type information. The original panic payload could be any type, and the `panic_payload_to_string` utility can only extract `&str` and `String` payloads reliably.

4. **Not all panics are catchable.** `abort`-on-panic (configured via `panic = "abort"` in Cargo.toml) makes `catch_unwind` a no-op. The behavior becomes platform-dependent.

5. **Stack overflows are not catchable.** The documentation mentions stack overflow as a motivation (line 512-513), but `catch_unwind` does NOT catch stack overflows on most platforms. Stack overflows trigger a signal (SIGSEGV) that terminates the process, not a Rust panic. This is potentially misleading.

**Recommendation:** The feature is acceptable as an opt-in utility, but the documentation should clarify that stack overflows are not catchable and that this is intended for panic recovery, not stack overflow recovery. The `TryTrampoline` itself provides stack safety; `catch_unwind` addresses a different concern (panicking closures).

## 8. Bind/Map Chain Behavior: Memory and Performance

### Allocation Patterns

Each operation on `Trampoline` delegates to `Free`, which allocates:

**`Trampoline::new(f)`** (line 133-135):
- Allocates a `Thunk` (one boxed closure).
- Creates `Free::wrap(Thunk::new(...))`, which is one `FreeInner::Wrap` variant.

**`Trampoline::bind(f)`** (line 199-204):
- Delegates to `Free::bind`, which:
  - Boxes the continuation `f` into a `Continuation<F>` (one `Box<dyn FnOnce>`).
  - For `Pure` and `Wrap` variants: boxes the head, creates a `CatList::singleton`, and constructs a `Bind` variant. This is 2 allocations (box + catlist node).
  - For `Bind` variants: calls `CatList::snoc` which is O(1) amortized. This is 1 allocation (the boxed continuation) plus potentially one catlist node.

**`Trampoline::map(f)`** (line 224-229):
- Implemented as `self.bind(move |a| Trampoline::pure(f(a)))`, so it has the same allocation cost as `bind` plus one additional `Pure` allocation at evaluation time.

### Performance Characteristics

For a chain of N operations:

- **Construction:** O(N) time, O(N) allocations. Each `bind` is O(1).
- **Evaluation:** O(N) time. The loop processes each continuation once. Each step involves one `CatList::uncons` (amortized O(1)), one `downcast` (O(1)), and one continuation call.
- **Peak memory:** O(N) for the continuation list, plus whatever the closures capture.

**Comparison with a simple recursive approach:**
A hand-written loop would have O(1) overhead per step with no allocation. The trampoline adds approximately 2-3 heap allocations per bind (for the boxed closure, the CatList node, and the type-erased value). This is the expected cost of generic stack safety.

**The `map` inefficiency:** Since `map` is defined via `bind`, a chain of N `map` calls creates N boxed continuations. Each continuation wraps the mapped value in a `Pure`, then immediately unwraps it. A dedicated `Map` variant in `FreeInner` could optimize this by fusing consecutive maps, avoiding intermediate `Pure` allocations. However, this is a minor optimization and the current approach is simpler.

### CatList Overhead

`CatList` is defined as:

```rust
pub enum CatList<A> {
    Nil,
    Cons(A, VecDeque<CatList<A>>, usize),
}
```

The `VecDeque` provides O(1) amortized `snoc` and `uncons`. The `append` operation (used in the `Bind` case of `evaluate`) is O(1) because it pushes one `CatList` into the other's deque. This is critical for preventing quadratic behavior.

## 9. Could Trampoline Be Made Generic Over Lifetimes?

### The Fundamental Obstacle

The `'static` requirement comes from `Box<dyn Any>`. To support `Trampoline<'a, A>`, you would need a type erasure mechanism that works with non-`'static` types.

### Possible Approaches

**Approach 1: Unsafe transmute-based erasure.**
Replace `Box<dyn Any>` with `Box<dyn Any + 'a>` and use unsafe code for downcasting. Rust's `Any` trait only supports `'static` because `TypeId` (used for downcasting) is only defined for `'static` types. You could maintain the type ID manually and use `transmute` for downcasting. This is unsafe and error-prone but technically possible.

**Approach 2: Monomorphized continuations via GATs.**
Instead of type-erasing continuations, encode the continuation chain at the type level using a heterogeneous list. This eliminates the need for `Any` but makes the `Free` type grow with each `bind`, which is impractical for dynamic bind chains.

**Approach 3: Arena-based allocation.**
Allocate all values in a lifetime-bounded arena. The arena's lifetime would bound the `Free` monad's lifetime. This could work but would require threading an arena reference through all operations.

**Approach 4: Hybrid approach.**
Keep `Free` as `'static`-only for stack safety, but provide a separate `TrampolineRef<'a, A>` type that uses a simpler representation (e.g., the naive `FlatMap` variant with a single continuation) and a bounded recursion depth. This would sacrifice O(1) bind but support lifetimes.

### Practical Assessment

None of these approaches are clearly better than the current design. The library's strategy of using `Thunk<'a, A>` for lifetime-polymorphic deferred computation and `Trampoline<A>` for stack-safe computation is the right split. In practice, most stack-safe computations deal with owned data (`i32`, `String`, `Vec<T>`, etc.) where `'static` is not a meaningful restriction.

The one scenario where this hurts is when you have a deeply recursive computation that needs to reference data with a bounded lifetime. In that case, you must clone the data into the `Trampoline`, which may be expensive. But this is an inherent tension in Rust's ownership model, not specific to this library's design.

## 10. Comparison with Other Implementations

### Haskell (`Control.Monad.Free`)

Haskell's `Free` is:

```haskell
data Free f a = Pure a | Free (f (Free f a))
```

With the "Reflection without Remorse" optimization, it becomes the same three-variant structure. The key differences from this library:

- Haskell's laziness means `Free Identity` already provides trampolining.
- Haskell uses `unsafeCoerce` where this library uses `Box<dyn Any>`.
- Haskell's `Free` implements `Functor` and `Monad` via type classes.

### PureScript (`Control.Monad.Free`)

PureScript's implementation is the direct inspiration for this library (cited in `free.rs` line 9). Key differences:

- PureScript provides `resume`, `foldFree`, `hoistFree`, which this library lacks.
- PureScript's `foldFree` enables interpreting `Free` into any target monad via a natural transformation. This library's `Free` can only be evaluated via the `Evaluable` trait, which is baked into the functor.
- PureScript uses `unsafeCoerce` for type erasure.

The missing `resume` function is the most significant API gap. It would allow inspecting a `Free` computation step-by-step, enabling interpreters, debuggers, and alternative execution strategies. Adding it would require exposing the `FreeInner` variants (or an equivalent `FreeView` type) in the public API.

### Scala (Cats `Free`)

Cats' `Free` monad uses a similar structure with `Pure`, `Suspend`, and `FlatMapped`. It uses Java's type erasure (everything is `Object` at runtime) for the same purpose as `Box<dyn Any>`. The API includes `foldMap` (equivalent to PureScript's `foldFree`).

### Comparison Summary

| Feature | This Library | Haskell | PureScript | Scala Cats |
|---------|-------------|---------|------------|------------|
| O(1) bind | Yes (CatList) | Yes (reflection without remorse) | Yes | Yes |
| Stack-safe evaluate | Yes | N/A (lazy) | Yes | Yes |
| HKT Functor/Monad | No (`'static`) | Yes | Yes | Yes (via implicits) |
| `resume`/step | No | Yes | Yes | Yes (`resume`) |
| `foldFree`/interpret | No (Evaluable only) | Yes | Yes | Yes (`foldMap`) |
| Lifetime support | `'static` only | N/A (GC) | N/A (GC) | N/A (GC) |
| Stack-safe drop | Yes (iterative) | N/A (GC) | N/A (GC) | N/A (GC) |

The most notable gaps compared to other implementations are the lack of `resume` and `foldFree`. These would make `Free` useful as a general-purpose effect system, not just a trampolining mechanism. However, adding them is non-trivial due to the type erasure, and the library's documented scope positions `Free` as an internal implementation detail for `Trampoline`, not as a user-facing abstraction for building DSLs.

## Summary of Findings

### Strengths

1. **Correct representation.** The three-variant `FreeInner` with `CatList` continuations is the state-of-the-art approach, directly from the "Reflection without Remorse" paper.
2. **Genuine stack safety.** The evaluation loop is iterative, handles nested `Bind` flattening, and the `Drop` implementation prevents stack overflow on destruction.
3. **Sound type erasure.** The use of `Box<dyn Any>` with `downcast` is safe Rust with no undefined behavior.
4. **Clean layering.** `Trampoline` as a newtype over `Free<ThunkBrand, A>` is a good separation of concerns.
5. **Well-documented tradeoffs.** The `free.rs` module documentation clearly explains why HKT traits are not implemented and what the consequences are.

### Areas for Improvement

1. **Missing `resume` and `foldFree`.** These would significantly increase the utility of `Free` as a general-purpose abstraction.
2. **`map` via `bind` is slightly inefficient.** A dedicated `Map` variant or fusion pass could reduce allocations for map-heavy chains.
3. **`catch_unwind` documentation is slightly misleading.** It implies stack overflow recovery, but Rust stack overflows are not catchable via `catch_unwind`.
4. **No way to inspect or transform Free structures.** Without `hoistFree`, you cannot change the base functor. Without `resume`, you cannot implement custom interpreters.
5. **Thread safety.** `Trampoline` is `!Send` because `Thunk` uses `FnOnce` closures without `Send` bounds. The `memoize_arc` method works around this by eagerly evaluating, but a `Send`-capable trampoline (using `Send` closures) would be useful for concurrent recursive computations.
