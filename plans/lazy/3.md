## 1. Overview and Role

`Thunk<'a, A>` is the lightweight deferred computation type in the lazy hierarchy. It fills a specific niche: the HKT-compatible, lifetime-polymorphic, non-memoized lazy value. The library's lazy hierarchy consists of three types:

| Type | HKT? | Stack-safe? | Memoized? | Lifetimes? |
|------|-------|-------------|-----------|------------|
| `Thunk<'a, A>` | Yes | Partial (`tail_rec_m` only) | No | `'a` |
| `Trampoline<A>` | No | Yes (unlimited) | No | `'static` only |
| `Lazy<'a, A, Config>` | Partial (`RefFunctor`) | N/A | Yes | `'a` |

`Thunk` exists because `Trampoline` requires `'static` (due to `Box<dyn Any>` in the `Free` monad internals) and therefore cannot implement the library's HKT traits. `Thunk` can borrow from its environment and integrates fully with the brand/kind system via `ThunkBrand`.

## 2. Internal Representation

### Current Design

```rust
pub struct Thunk<'a, A>(Box<dyn FnOnce() -> A + 'a>);
```

This is a newtype around a single boxed `FnOnce` closure. The representation is maximally simple: a tuple struct with one field.

### Assessment: The Enum Question

Unlike many functional library designs that use a two-variant enum (e.g., `Value(A)` / `Deferred(Box<dyn FnOnce() -> A>)`), `Thunk` uses a single closure variant. This is a deliberate and correct choice for several reasons:

1. **`pure` wraps values in closures anyway.** `Thunk::pure(a)` calls `Thunk::new(move || a)`, so there is no semantic difference between "already computed" and "deferred." Both are closures.

2. **Avoids branching on every operation.** An enum representation would require matching on `Value` vs `Deferred` in `map`, `bind`, `evaluate`, etc., adding code complexity for zero benefit since `Thunk` is not memoized.

3. **Uniform closure representation keeps the code predictable.** The constructor `Thunk::new` is the only way to create a value, and `evaluate` is the only way to extract one.

**Potential concern:** Wrapping an already-computed value in a closure adds one indirection (the `Box` allocation plus the function call overhead). For `Thunk::pure(42)`, the closure captures `42` into a heap-allocated `Box<dyn FnOnce()>` even though the value is immediately available. However, this overhead is negligible for the use case (glue code, composition) and the uniformity benefit outweighs it.

**Verdict:** The single-variant design is optimal for `Thunk`'s intended role.

## 3. Stack Safety Analysis

The documentation says `Thunk` has "Partial" stack safety. This deserves precise characterization.

### What IS Stack-Safe: `tail_rec_m`

The `MonadRec` implementation (lines 503-548) uses an explicit loop:

```rust
fn tail_rec_m<'a, A: 'a, B: 'a>(
    f: impl Fn(A) -> Thunk<'a, Step<A, B>> + Clone + 'a,
    a: A,
) -> Thunk<'a, B> {
    Thunk::new(move || {
        let mut current = a;
        loop {
            match f(current).evaluate() {
                Step::Loop(next) => current = next,
                Step::Done(res) => break res,
            }
        }
    })
}
```

This is truly stack-safe. The loop replaces recursion, and each iteration evaluates a `Thunk<Step<A, B>>` (which is a single closure call) and either continues or breaks. No unbounded stack growth occurs. The computation is deferred: the loop only runs when `evaluate()` is called on the outer `Thunk`.

### What is NOT Stack-Safe: `bind` Chains

The inherent `bind` method (lines 184-193):

```rust
pub fn bind<B: 'a>(self, f: impl FnOnce(A) -> Thunk<'a, B> + 'a) -> Thunk<'a, B> {
    Thunk::new(move || {
        let a = (self.0)();       // evaluate self
        let thunk_b = f(a);       // apply continuation
        (thunk_b.0)()             // evaluate result
    })
}
```

Each `bind` creates a new closure that captures the previous `Thunk`. When the final `evaluate()` is called, the closures nest: the outermost closure calls the next inner closure, which calls the next, etc. For a chain of depth N:

```rust
let chain = Thunk::pure(0)
    .bind(|x| Thunk::pure(x + 1))
    .bind(|x| Thunk::pure(x + 1))
    // ... N times
```

Evaluation creates N nested stack frames. On a typical system with 8MB default stack, this overflows around N = 10,000-50,000 depending on closure capture sizes.

### What is NOT Stack-Safe: `map` Chains

Similarly, `map` (lines 215-220):

```rust
pub fn map<B: 'a>(self, f: impl FnOnce(A) -> B + 'a) -> Thunk<'a, B> {
    Thunk::new(move || f((self.0)()))
}
```

Each `map` nests one closure inside another. A chain of N `map` calls requires N stack frames at evaluation time.

### What is NOT Stack-Safe: `defer`

```rust
pub fn defer(f: impl FnOnce() -> Thunk<'a, A> + 'a) -> Self {
    Thunk::new(move || f().evaluate())
}
```

`defer` is not recursive in isolation, but chained defers (e.g., recursive functions using `defer`) will nest arbitrarily deep.

### Precise Stack Safety Characterization

- **`tail_rec_m`**: Fully stack-safe. Uses an iterative loop.
- **`bind` chains**: O(N) stack depth where N is the chain length. Overflows for large N.
- **`map` chains**: O(N) stack depth where N is the chain length.
- **`defer` chains**: O(N) stack depth if recursively composed.
- **`Semigroup::append` chains**: O(N) stack depth; each `append` creates a closure that evaluates both operands.

### Recommendation

The documentation currently says "Partial (tail_rec_m only)." This is accurate but could be more explicit. Consider documenting the approximate overflow threshold and recommending `Trampoline` for chains exceeding ~1,000 bind/map operations.

## 4. Memory Layout and Allocation

### Size of `Thunk<'a, A>`

`Thunk<'a, A>` is `Box<dyn FnOnce() -> A + 'a>`, which is a fat pointer (two `usize` values on the stack: pointer + vtable). On 64-bit systems:

- **Stack size:** 16 bytes (pointer + vtable pointer for the trait object).
- **Heap allocation:** One allocation per `Thunk::new` call; the closure is boxed.

### Allocation Behavior Per Operation

| Operation | Heap allocations |
|-----------|-----------------|
| `Thunk::new(f)` | 1 (boxes the closure) |
| `Thunk::pure(a)` | 1 (boxes the closure wrapping `a`) |
| `Thunk::map(f)` | 1 (new boxed closure capturing `self` and `f`) |
| `Thunk::bind(f)` | 1 (new boxed closure) |
| `Thunk::defer(f)` | 1 (new boxed closure) |
| `Thunk::evaluate()` | 0 (consumes the box, calls the closure) |

Every transformation allocates exactly one `Box`. A chain of N operations allocates N boxes. When `evaluate()` is called, the outermost box is consumed, which triggers the chain of inner evaluations, each consuming its box.

### Chain Allocation Pattern

For a chain like:
```rust
Thunk::pure(0).map(|x| x + 1).map(|x| x + 2).bind(|x| Thunk::pure(x * 3))
```

This creates 4 heap allocations during construction (one per operation). At `evaluate()` time, each box is consumed in sequence. The consumed boxes are freed as the stack unwinds. There is no persistent heap growth beyond the chain itself.

### Comparison with Trampoline

`Trampoline` uses `Free<ThunkBrand, A>` with `CatList` of continuations. Its `bind` is O(1) in both time and allocations (appends to a `CatList`), but `evaluate()` processes continuations iteratively. `Thunk`'s bind is also O(1) in construction time (creates one closure), but evaluation is O(N) in stack depth.

The key difference: `Thunk` chains create nested closures (onion layers); `Trampoline` chains create a flat `CatList` of continuations.

## 5. Functor/Monad Implementation and Chain Behavior

### Type Class Hierarchy

`ThunkBrand` implements:
- `Functor` (via `ThunkBrand`)
- `Pointed` (via `ThunkBrand`)
- `Lift` (via `ThunkBrand`)
- `Semiapplicative` (via `ThunkBrand`)
- `ApplyFirst`, `ApplySecond` (blanket impls via `ThunkBrand`)
- `Semimonad` (via `ThunkBrand`)
- `Applicative` (blanket: `Pointed + Semiapplicative + ApplyFirst + ApplySecond`)
- `Monad` (blanket: `Applicative + Semimonad`)
- `MonadRec` (via `ThunkBrand`)
- `Foldable` (via `ThunkBrand`)
- `Evaluable` (via `ThunkBrand`)

Additionally, `Thunk<'a, A>` directly implements:
- `Deferrable<'a>`
- `Semigroup` (when `A: Semigroup`)
- `Monoid` (when `A: Monoid`)
- `From<Lazy<'a, A, Config>>`

### FnOnce vs Fn Discrepancy

A notable design tension: the inherent `bind` method takes `impl FnOnce(A) -> Thunk<'a, B>`, but the `Semimonad` trait requires `impl Fn(A) -> ...`. This means:

- **Inherent `bind`:** More flexible; accepts any `FnOnce` closure.
- **Trait `bind` (via `Semimonad`):** More restrictive; requires `Fn` (callable multiple times).

The trait implementation delegates to the inherent method, and since `Fn: FnOnce`, this compiles. But it means users who go through the generic `bind::<ThunkBrand, _, _>` free function must provide an `Fn` closure even though `Thunk` only calls it once. This is a systemic limitation of the library's type class design, not a `Thunk`-specific issue.

### Does `bind` Create Chains? Are They Stack-Safe?

Yes, `bind` creates chains. No, they are not stack-safe. Each `bind` wraps the previous `Thunk` in a new closure. When evaluated, this creates a call chain:

```
evaluate(outer) -> call outer closure
  -> (self.0)()  -- evaluates inner Thunk
    -> ... (recursively, N levels deep)
  -> f(a)        -- applies the continuation
  -> (thunk_b.0)()  -- evaluates the result
```

For left-associated binds like `t.bind(f1).bind(f2).bind(f3)`, each level adds two stack frames (one for the wrapping closure, one for the inner `FnOnce` call). This is the fundamental limitation that `Trampoline` exists to solve.

### Applicative Implementation

The `Semiapplicative::apply` implementation (lines 454-464) delegates to `bind` and `map`:

```rust
fn apply<...>(ff: Thunk<...>, fa: Thunk<...>) -> Thunk<...> {
    ff.bind(move |f| fa.map(move |a| f(a)))
}
```

This is standard (applicative derived from monad). It adds one `bind` and one `map` to the chain. For deeply nested `apply` chains, the same stack safety concerns apply.

The `Lift::lift2` also delegates to `bind` and `map`:

```rust
fn lift2<...>(func, fa, fb) -> ... {
    fa.bind(move |a| fb.map(move |b| func(a, b)))
}
```

## 6. Closure and Boxing Behavior

### Boxing Strategy

Every `Thunk` creation boxes the closure via `Box::new(f)` in `Thunk::new`. This is unavoidable for type erasure: `Thunk<'a, A>` must store closures of varying sizes in a uniform representation, and `Box<dyn FnOnce() -> A>` is the only way to do this in Rust without `alloca` or arena allocation.

### Capture Semantics

`Thunk::new` takes `impl FnOnce() -> A + 'a`. The closure can capture by move or by reference (constrained by `'a`). The boxing moves the closure to the heap, which means:

- **Move captures:** The captured values are moved into the heap allocation.
- **Reference captures:** The references are stored in the heap allocation; the referents must outlive `'a`.

### Chain Capture Growth

Each `map`/`bind` captures the previous `Thunk` (which is itself a `Box`) plus the new function. For a chain of N operations:

- N heap allocations exist simultaneously (until `evaluate()` starts consuming them).
- Each allocation holds a `Box<dyn FnOnce()>` that captures the previous `Box` and a function.
- Total heap usage is proportional to N, with each allocation being the size of the closure environment.

This is reasonable for the intended use case (short chains, glue code). For long chains, `Trampoline` is the correct choice.

### Could `Thunk` Avoid Boxing?

Not without fundamentally changing its design. Without boxing:

1. `Thunk` would need to be generic over the closure type: `Thunk<'a, A, F: FnOnce() -> A>`. This makes it unusable in collections, trait objects, and HKT contexts since each closure has a unique type.
2. Enum representation with an `Inline` variant and a size limit (small closure optimization) would add complexity for marginal benefit, since most closures capture at least one value.

The current boxing approach is the standard Rust idiom for type-erased closures and is appropriate here.

## 7. Lifetime Handling

### The `'a` Parameter

`Thunk<'a, A>` is parameterized by lifetime `'a`, which bounds the closure: `Box<dyn FnOnce() -> A + 'a>`. This means:

- The closure can borrow data that lives at least as long as `'a`.
- The `Thunk` itself cannot outlive `'a`.
- The produced value `A` must satisfy `A: 'a` (enforced on `impl` blocks).

### HKT Integration

The `Kind` implementation:

```rust
impl_kind! {
    for ThunkBrand {
        type Of<'a, A: 'a>: 'a = Thunk<'a, A>;
    }
}
```

This maps `ThunkBrand` into the kind `type Of<'a, A: 'a>: 'a`, meaning:
- The type constructor takes a lifetime `'a` and a type `A: 'a`.
- The resulting type `Thunk<'a, A>` is bounded by `'a`.

This is correct and allows `Thunk` to participate in all the generic type class machinery.

### Borrowing Test

The test at line 788-792 confirms borrowing works:

```rust
fn test_borrowing() {
    let x = 42;
    let thunk = Thunk::new(|| &x);
    assert_eq!(thunk.evaluate(), &42);
}
```

Here `'a` is inferred as the local scope lifetime. The thunk captures `&x` (a reference to a stack variable) and produces `&i32`. This is a clean demonstration of `Thunk`'s lifetime flexibility.

### Lifetime Flexibility Assessment

The lifetime design is correct and flexible. Key properties:

1. **Can borrow from the environment:** Unlike `Trampoline`, `Thunk` can capture references.
2. **Can be `'static`:** Closures that own all their data satisfy `'static`.
3. **Composes correctly:** `map` and `bind` preserve the `'a` lifetime through their closure captures.

### One Subtle Point: `bind` Lifetime Propagation

```rust
pub fn bind<B: 'a>(self, f: impl FnOnce(A) -> Thunk<'a, B> + 'a) -> Thunk<'a, B>
```

Both the continuation `f` and the resulting `Thunk<'a, B>` share the same lifetime `'a` as the original `Thunk<'a, A>`. This is correct: the composed closure captures both `self` (bounded by `'a`) and `f` (bounded by `'a`), so the result is also bounded by `'a`. However, it means you cannot bind a shorter-lived thunk into a longer-lived one, which is the expected covariance behavior.

## 8. Force/Evaluate Semantics

### Evaluation Model

`Thunk` uses eager-on-demand (call-by-name) semantics:

- **Construction is lazy:** `Thunk::new(f)` just boxes the closure. No computation happens.
- **Evaluation is eager and consuming:** `Thunk::evaluate(self)` takes ownership, calls the closure, and returns the result. The `Thunk` is consumed.
- **No memoization:** Each `evaluate()` call re-runs the entire computation chain. Since `evaluate` takes `self` by value, a given `Thunk` can only be evaluated once (it is consumed).

### Consuming Semantics

The `evaluate` method takes `self` (not `&self`), which means:

1. A `Thunk` can only be evaluated once.
2. The `FnOnce` closure is consumed during evaluation.
3. No accidental double-evaluation is possible (enforced by the type system).

This is appropriate for `FnOnce` closures and aligns with linear usage patterns.

### Comparison: call-by-name vs call-by-need

- **Call-by-name** (Thunk): The computation is re-executed each time it is "needed." Since `evaluate(self)` consumes the thunk, it can only be called once, so the distinction is moot for a single thunk. However, if a thunk is reconstructed multiple times (e.g., in a loop), each construction and evaluation pair re-runs the computation.
- **Call-by-need** (Lazy): The computation runs at most once; subsequent accesses return the cached result. `Lazy` achieves this via `LazyCell`/`LazyLock`.

`Thunk`'s consuming semantics mean it is effectively single-use call-by-name, which is appropriate for its role as a composition/transformation building block.

### The `Evaluable` Trait

`ThunkBrand` implements `Evaluable`, which is used by `Free::evaluate` to step through `Free<ThunkBrand, A>` (i.e., `Trampoline`) computations. The implementation simply delegates to `Thunk::evaluate`:

```rust
impl Evaluable for ThunkBrand {
    fn evaluate<'a, A: 'a>(fa: Thunk<'a, A>) -> A {
        fa.evaluate()
    }
}
```

This is the mechanism by which `Trampoline` uses `Thunk` as its base functor. Each step of the trampoline loop calls `ThunkBrand::evaluate` to advance one layer.

## 9. Comparison with Haskell Thunks and PureScript Data.Lazy

### Haskell Thunks (GHC Runtime)

In Haskell, every expression is a thunk by default (lazy evaluation). GHC thunks:

- Are managed by the runtime (heap-allocated closures with an info pointer).
- Are memoized (call-by-need): once evaluated, the thunk is overwritten with the result (thunk update).
- Are transparent: no explicit `force`/`evaluate`; pattern matching forces evaluation.
- Support arbitrary nesting without stack overflow due to the RTS stack/heap interaction and garbage collection.

`Thunk<'a, A>` differs fundamentally:

- **Explicit:** Must be explicitly constructed and evaluated.
- **Not memoized:** Re-evaluates each time (though the consuming `self` prevents literal re-evaluation of the same thunk).
- **Boxed:** Each thunk is a heap allocation with a vtable.
- **Not stack-safe for deep chains:** Haskell's RTS handles deep thunk chains; Rust's `Thunk` does not.

`Thunk` is closer to Haskell's `IO ()` (a deferred action) than to Haskell's general thunk mechanism, though without the `IO` monad's guarantees about effect ordering.

### PureScript `Data.Lazy`

PureScript's `Data.Lazy` (from `purescript-lazy`):

```purescript
foreign import data Lazy :: Type -> Type
foreign import defer :: forall a. (Unit -> a) -> Lazy a
foreign import force :: forall a. Lazy a -> a
```

Key properties:
- **Memoized:** `force` computes the value once and caches it.
- **Full type class support:** Functor, Applicative, Monad, Foldable, Traversable, Comonad, Extend.
- **Cloneable:** Can be shared; all copies see the cached result.

Comparison with `Thunk`:

| Aspect | PureScript `Lazy` | Rust `Thunk` | Rust `Lazy` |
|--------|-------------------|-------------|-------------|
| Memoized | Yes | No | Yes |
| Functor/Monad | Yes (full) | Yes (full) | RefFunctor only |
| Traversable | Yes | No (FnOnce) | No |
| Comonad | Yes (`extract = force`) | No | No |
| Cloneable | Yes | No (FnOnce) | Yes (Rc/Arc) |
| Thread-safe | N/A (JS) | No (not Send) | Config-dependent |

The Rust library splits PureScript's `Lazy` into two types: `Thunk` (the monadic, non-memoized part) and `Lazy` (the memoized, shared part). This split is necessary because Rust's ownership system prevents a single type from being both memoized-and-shared and also a proper Monad (bind requires consuming the input, but memoization requires sharing).

### Missing Type Classes Compared to PureScript

PureScript's `Lazy` implements several type classes that `Thunk` does not:

1. **Traversable:** PureScript's `traverse f l = defer <<< const <$> f (force l)`. In Rust, this would require cloning the `Thunk` or its closure, which `FnOnce` prevents. This is correctly documented as an intentional limitation.

2. **Comonad (`extract = force`):** `Thunk::evaluate` is essentially `extract`. However, implementing `Comonad` would also require `Extend`, which needs `fn extend(f: impl Fn(Thunk<A>) -> B, x: Thunk<A>) -> Thunk<B>`. This requires `f` to receive the whole thunk, implying the thunk must be shareable. Since `Thunk` wraps `FnOnce`, it cannot be shared.

3. **Eq, Ord, Show:** PureScript implements these by forcing evaluation and comparing/displaying the results. Rust could implement `PartialEq`, `Eq`, `Ord`, `Debug`, `Display` for `Thunk` by evaluating. However, `evaluate` consumes the thunk, so these traits would also consume it, which violates the expectation that `eq(&self, &other)` is non-destructive. This is a fundamental impediment.

## 10. Thread Safety

### Current State

`Thunk<'a, A>` is `!Send` and `!Sync` because:

- `Box<dyn FnOnce() -> A + 'a>` is `!Send` unless the closure is `Send`.
- The `dyn` trait object does not include `Send` or `Sync` bounds.

This means `Thunk` values cannot be sent across threads.

### Could `Thunk` Be Made `Send`?

Yes, with a separate type or a generic parameter:

```rust
// Option A: A separate type
pub struct SendThunk<'a, A>(Box<dyn FnOnce() -> A + Send + 'a>);

// Option B: A marker trait parameter (similar to Lazy's Config)
pub struct Thunk<'a, A, Config>(Box<dyn FnOnce() -> A + 'a>);
// where Config controls whether Send is required
```

However, the current design keeps `Thunk` simple and single-threaded, which is appropriate for its role as glue code. For thread-safe deferred computation, `Trampoline` (which wraps `Free<ThunkBrand, A>` and requires `A: 'static + Send`) is the intended choice.

### Assessment

The `!Send` constraint is acceptable given the library's hierarchy:
- `Thunk` = single-threaded, lifetime-flexible, HKT-compatible.
- `Trampoline` = `Send`, `'static`, stack-safe.
- `Lazy<ArcLazyConfig>` = `Send + Sync`, memoized.

However, it is worth noting that `Trampoline` internally uses `Thunk` as its base functor (via `Free<ThunkBrand, A>`). Since `Trampoline` requires `A: 'static + Send`, and `Thunk` is `!Send`, the `Free` evaluation loop must be careful. Looking at the `Free::evaluate` implementation, the `Thunk` closures are created and consumed on the same thread (within the trampoline loop), so this is safe. The `Send` bound on `Trampoline<A>` applies to the result type `A`, not to the intermediate `Thunk` closures.

## 11. Edge Cases and Potential Unsoundness

### 11.1 Panic Safety

If the closure inside a `Thunk` panics during `evaluate()`, the panic propagates normally. Since `evaluate` takes `self` by value, the `Box` is consumed and there is no double-free risk. The `Thunk` is in a valid (consumed) state. This is safe.

However, in `bind`:

```rust
Thunk::new(move || {
    let a = (self.0)();       // Could panic here
    let thunk_b = f(a);       // If above panics, f is leaked
    (thunk_b.0)()
})
```

If `(self.0)()` panics, `f` is dropped as the closure unwinds. This is standard Rust drop behavior and is safe. The `Box<dyn FnOnce()>` for `f` is cleaned up by the drop glue.

### 11.2 No `Drop` Implementation Concerns

`Thunk` has no custom `Drop` impl. The default drop behavior is:
1. Drop the `Box<dyn FnOnce() -> A + 'a>`, which drops the boxed closure.
2. The closure's captures are dropped.

This is correct. There is no resource leak risk.

### 11.3 Consuming Semantics Prevent Misuse

Since `evaluate(self)` takes ownership, a `Thunk` cannot be evaluated twice. This prevents:
- Double-free of captured resources.
- Observing partially-evaluated state.
- Race conditions (in hypothetical concurrent scenarios).

### 11.4 The `From<Lazy>` Conversion

```rust
impl<'a, A, Config> From<Lazy<'a, A, Config>> for Thunk<'a, A>
where
    A: Clone + 'a,
    Config: LazyConfig,
{
    fn from(lazy: Lazy<'a, A, Config>) -> Self {
        Thunk::new(move || lazy.evaluate().clone())
    }
}
```

This requires `A: Clone` because `Lazy::evaluate` returns `&A` (a reference to the cached value), and the `Thunk` must produce an owned `A`. The `Clone` is necessary and correct.

**Subtle point:** The `Lazy` value is moved into the `Thunk`'s closure. Since `Lazy` is `Clone` (it uses `Rc`/`Arc` internally), this does not invalidate other clones of the `Lazy`. However, the thunk closure will call `lazy.evaluate().clone()` which forces evaluation of the `Lazy` if it has not been evaluated yet. This is a one-time cost; subsequent clones of the same `Lazy` will see the cached value.

### 11.5 No Unsoundness Detected

The implementation uses only safe Rust. The `Box<dyn FnOnce() -> A + 'a>` is a standard pattern. The lifetime parameter `'a` correctly bounds all closure captures. The consuming `evaluate(self)` prevents use-after-move. There are no `unsafe` blocks, raw pointer dereferences, or transmute calls.

## 12. Relationship to TryThunk

`TryThunk<'a, A, E>` is the fallible counterpart:

```rust
pub struct TryThunk<'a, A, E>(Box<dyn FnOnce() -> Result<A, E> + 'a>);
```

It mirrors `Thunk` exactly but produces `Result<A, E>` instead of `A`. The relationship is:

- `Thunk<'a, A>` is isomorphic to `TryThunk<'a, A, !>` (infallible computation).
- `TryThunk` implements `Bifunctor` (can map over both success and error types).
- Both share the same limitations: `!Send`, `!Clone`, not stack-safe for deep chains.

There is no conversion from `TryThunk` to `Thunk` (would require handling the error), but there could be a `TryThunk::from(Thunk)` that wraps in `Ok`. Currently this is not implemented.

## 13. Summary of Recommendations

### Strengths (No Changes Needed)

1. **Single-variant tuple struct design** is optimal; no enum needed.
2. **Lifetime parameterization** is correct and flexible.
3. **Consuming evaluate semantics** are safe and prevent misuse.
4. **`MonadRec::tail_rec_m` implementation** is genuinely stack-safe.
5. **Full monad hierarchy** (Functor through MonadRec) is well-implemented.
6. **`From<Lazy>` conversion** is correct with appropriate bounds.

### Documentation Improvements

1. **Stack safety documentation** should explicitly state that `bind`/`map` chains of depth >~1,000 risk stack overflow, and that `tail_rec_m` is the only stack-safe operation on `Thunk`. The current documentation says "Partial (tail_rec_m only)" in a table, which is accurate but could be clearer about the failure mode (stack overflow, not incorrect results).

2. **The `FnOnce` vs `Fn` discrepancy** between inherent `bind` and trait `Semimonad::bind` should be documented. Users accessing `bind` through the generic free function must provide `Fn` closures, which is more restrictive than the inherent method.

### Potential Enhancements

3. **Consider a `SendThunk<'a, A>` variant** (or a generic `Thunk<'a, A, Marker>` with a marker for `Send`). This would allow thread-safe thunks without requiring `'static`, filling a gap between `Thunk` (not `Send`) and `Trampoline` (requires `'static`). Priority: low, since `Trampoline` covers the `Send` use case for most practical scenarios.

4. **Consider `Debug` implementation** that shows `Thunk(...)` without evaluating. This would help with debugging:
   ```rust
   impl<'a, A> fmt::Debug for Thunk<'a, A> {
       fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
           f.write_str("Thunk(<unevaluated>)")
       }
   }
   ```

5. **Consider `TryThunk::from(Thunk)` conversion** that wraps the result in `Ok`.

6. **Add a stack safety stress test** that verifies `tail_rec_m` does not overflow for large N (e.g., 1,000,000 iterations), and a separate test that documents the bind chain limit.

### Non-Recommendations (Things That Should NOT Change)

- **Do not add memoization.** That is `Lazy`'s job.
- **Do not add an enum variant for pre-computed values.** The closure-only design is simpler and incurs negligible overhead.
- **Do not add `Clone`.** `FnOnce` closures cannot be cloned; this is fundamental to the design.
- **Do not try to make `Thunk` stack-safe for `bind` chains.** That would require the `Free` monad machinery (which needs `'static`), defeating `Thunk`'s purpose. Use `Trampoline` instead.
