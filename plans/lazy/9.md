# Agent 9: Rust-Specific Analysis of the Lazy Hierarchy

This analysis examines the Thunk, Lazy, and Trampoline types through the lens of Rust's ownership system, lifetime semantics, Send/Sync safety, and general type-system ergonomics.

## 1. Ownership Semantics

### Thunk: Linear (Move-Only) Ownership

`Thunk<'a, A>` wraps a `Box<dyn FnOnce() -> A + 'a>`. Because `FnOnce` consumes itself when called, `Thunk` is a linear type: it can be moved but not cloned or called twice.

- `evaluate(self)` correctly takes `self` by value, consuming the thunk.
- `map(self, ...)` and `bind(self, ...)` also take `self` by value, which is correct since they consume the inner closure to build a new one.
- `Thunk` does not implement `Clone`, which is the correct choice given `FnOnce` semantics.

This is sound and well-designed. The one ergonomic consequence is that `Thunk` cannot be shared or reused; you must evaluate it exactly once. This is clearly documented as a trade-off against `Lazy` (which is cloneable and memoized).

### Lazy: Shared Ownership via Rc/Arc

`Lazy<'a, A, Config>` wraps either `Rc<LazyCell<A, ...>>` or `Arc<LazyLock<A, ...>>` depending on `Config`. Because the inner cell is behind a reference-counted pointer:

- `Clone` is implemented and shares the underlying cache (all clones see the same value). This is correct.
- `evaluate(&self)` takes `&self`, returning `&A`. This is correct because LazyCell/LazyLock provide interior mutability for the one-time initialization.
- The `ref_map(self, ...)` method takes `self` by value but first calls `self.clone()`, capturing the clone in the new closure. This means the original `Lazy` is consumed at the call site even though the underlying cache is shared. This is a minor ergonomic choice; taking `&self` would also work since `Lazy` is `Clone`, but taking `self` signals that this is a chaining/transformation operation.

### Trampoline: Linear Ownership via Free Monad

`Trampoline<A>` wraps `Free<ThunkBrand, A>`. The `Free` monad uses `Option<FreeInner<F, A>>` internally (with `take()` semantics) to support linear consumption.

- `evaluate(self)`, `bind(self, ...)`, `map(self, ...)` all take `self` by value. Correct.
- `Free::bind` uses `self.0.take().expect(...)` internally, which panics if the `Option` has already been taken. This is a runtime enforcement of linear usage. The `Option` wrapper exists because `Drop` needs to iteratively unwind `Bind` chains to prevent stack overflow during drop. This is a pragmatic solution.
- The `expect` calls in `Free::bind` and `Free::evaluate` (lines 306, 429, 445 of free.rs) are documented as internal invariants, not user-facing. Since `Free` is consumed by value, the only way to hit these panics would be through an internal bug.

**Concern:** The `Free<F, A>` type wraps an `Option<FreeInner<F, A>>` solely to support iterative drop. This adds a branch on every `bind` and `evaluate` call (to check `Some`/`None`). This is a minor performance concern but necessary for correctness.

## 2. Lifetime Analysis

### Thunk: `'a` Lifetime Parameter

`Thunk<'a, A>` is parameterized over lifetime `'a`, where the inner closure is `Box<dyn FnOnce() -> A + 'a>`. This means Thunk can borrow from its environment.

- The `A: 'a` bound in `impl<'a, A: 'a>` is correct: if `A` contains references, they must outlive the thunk itself.
- The test `test_borrowing` (line 788) demonstrates borrowing from the stack: `let x = 42; let thunk = Thunk::new(|| &x);`. This works correctly.
- The HKT brand `ThunkBrand` maps to `type Of<'a, A: 'a>: 'a = Thunk<'a, A>`, which is the correct lifetime-polymorphic signature.

No unnecessary lifetime restrictions are present. The `'a` parameter is minimally constrained.

### Lazy: `'a` Lifetime Parameter

`Lazy<'a, A, Config>` also carries `'a`, and the internal types use it:
- `RcLazyConfig::Lazy<'a, A> = Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>`
- `ArcLazyConfig::Lazy<'a, A> = Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>`

This is correct. The closure can borrow from the environment with lifetime `'a`.

**Observation:** The `LazyConfig` trait has a `'static` bound on itself (`pub trait LazyConfig: 'static`). This is necessary because the Config type parameter is used as a type-level marker (like a Brand), not as a value carrier. The `'static` bound on the trait does not restrict the `'a` parameter on the associated types, which is correct.

### Trampoline: `'static` Only

`Trampoline<A>` requires `A: 'static + Send`. The `'static` bound comes from `Free`'s use of `Box<dyn Any>` for type erasure (since `Any: 'static`). This is an inherent limitation, clearly documented.

- The `From<Lazy<'static, A, Config>> for Trampoline<A>` conversion correctly requires `'static` on the Lazy's lifetime parameter.
- The `From<Trampoline<A>> for Lazy<'a, A, RcLazyConfig>` conversion requires `A: Send` because `Trampoline` requires `Send`, not because `Lazy` itself needs it. This bound propagates correctly.

**No issues found.** The `'static` restriction is inherent to the Free monad's type erasure approach and is well-documented.

## 3. Send/Sync Safety

### Thunk: Not Send, Not Sync (by Default)

`Thunk<'a, A>` contains `Box<dyn FnOnce() -> A + 'a>`. Trait objects are `Send`/`Sync` only if explicitly bounded. Since the closure is `dyn FnOnce() -> A + 'a` (without `+ Send`), `Thunk` is:
- Not `Send` (the closure might capture `Rc`, non-Send data, etc.).
- Not `Sync` (it contains a `FnOnce` which requires `&mut self` to call, but `Box<dyn FnOnce>` is consumed by value anyway).

There are no manual `Send`/`Sync` implementations. The auto-trait derivation is correct.

**Potential improvement:** Consider offering a `SendThunk<'a, A>` type alias or a separate `Send`-bounded variant, similar to how `Lazy` has `RcLazy`/`ArcLazy`. Currently, if you need a `Send`-able deferred computation, you must use `Trampoline` (which also requires `'static`). A `SendThunk<'a, A>` wrapping `Box<dyn FnOnce() -> A + Send + 'a>` would fill the gap for non-`'static`, `Send`-able deferred computations.

### Lazy: Send/Sync Depends on Config

- `Lazy<'a, A, RcLazyConfig>` wraps `Rc<LazyCell<...>>`. `Rc` is `!Send` and `!Sync`, so `RcLazy` is neither `Send` nor `Sync`. Correct.
- `Lazy<'a, A, ArcLazyConfig>` wraps `Arc<LazyLock<...>>`. `Arc` is `Send + Sync` when `T: Send + Sync`. `LazyLock` is `Send + Sync` when the value and closure are `Send`. The closure is `Box<dyn FnOnce() -> A + Send + 'a>`, so `ArcLazy` is `Send + Sync` when `A: Send + Sync`. Correct.

The `SendDeferrable` impl for `ArcLazy` correctly requires `A: Clone + Send + Sync + 'a` and `f: impl FnOnce() -> Self + Send + Sync + 'a`. No issues.

### Trampoline: Requires `A: Send`

`Trampoline<A>` requires `A: 'static + Send` on its impl block. However, the struct definition itself is `pub struct Trampoline<A: 'static>(Free<ThunkBrand, A>)`, with the `Send` bound only on the impl block, not the struct.

This means:
- You can create a `Trampoline<A>` type where `A: !Send`, but you cannot call any methods on it (since the impl block requires `Send`).
- Auto-derived `Send`/`Sync` for `Trampoline<A>` depends on `Free<ThunkBrand, A>`.

Looking deeper: `Free<ThunkBrand, A>` contains `FreeInner<ThunkBrand, A>`, which can contain `Thunk<'static, Free<ThunkBrand, A>>`, which wraps `Box<dyn FnOnce() -> ...>` (not `Send`). So `Free<ThunkBrand, A>` is `!Send` by auto-derivation, and therefore `Trampoline<A>` is `!Send`.

**This is a notable finding:** Even though `Trampoline` requires `A: Send`, the `Trampoline` value itself is `!Send`. This means you cannot send a `Trampoline` across threads; you can only construct and evaluate it on the same thread. The `Send` bound on `A` exists because `Free`'s `Box<dyn Any>` requires `downcast::<A>()` to work, and `Any` is `'static`. The `Send` bound on `A` appears to be required because `Free::bind`'s continuation closure captures `A` values, and somewhere in the chain, type erasure through `Box<dyn Any>` needs `Send + 'static`. However, upon closer inspection, `Box<dyn Any>` only requires `'static`, not `Send`. The `Send` bound on `A` in `Trampoline` may be an unnecessary over-constraint inherited from an earlier design, unless there is a specific downstream requirement.

**Recommendation:** Audit whether `A: Send` is actually required on `Trampoline`. If it was added purely for future thread-safety and `Trampoline` is `!Send` anyway (due to the internal `Thunk` closures), then the `Send` bound on `A` is an unnecessary restriction that prevents using `Trampoline` with `Rc<T>` and other `!Send` types.

## 4. Unsafe Code Audit

**No `unsafe` code exists in any of the three lazy types or the `Deferrable` trait.** The only mentions of "unsafe" in the types directory are:
- A comment in `fn_brand.rs` (line 17) about why `Costrong` cannot be safely implemented.
- A comment in `free.rs` (line 45) about PureScript's use of `unsafeCoerce`.

The `Free` monad uses `Box<dyn Any>` with `downcast()` for type erasure. The `downcast()` calls use `.expect()` (not `unsafe` transmute/pointer casts). This is safe Rust; the worst case is a panic, not undefined behavior. The internal invariant that types match is maintained by the construction pattern (type-erasing and then downcasting within the same bind chain).

**Verdict:** Sound. No unsafe code to audit.

## 5. Interior Mutability in Lazy

`Lazy` achieves interior mutability through standard library types:
- `RcLazy` uses `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>`.
- `ArcLazy` uses `Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>`.

`LazyCell` (stabilized in Rust 1.80) provides `!Sync` interior mutability. It stores either the closure or the computed value, transitioning from one to the other on first access. `LazyLock` provides `Sync` interior mutability using OS-level synchronization primitives.

Key properties:
- **Single initialization:** Both `LazyCell` and `LazyLock` guarantee the closure runs at most once. This is correct for memoization.
- **No data races:** `LazyCell` is `!Sync`, preventing concurrent access. `LazyLock` uses internal locking. Both are correct.
- **Panic safety:** If the closure panics during initialization, `LazyCell`/`LazyLock` poison the cell. Subsequent accesses will also panic. This is reasonable behavior, matching `std::sync::Once`.

**No issues found.** The design correctly delegates interior mutability to well-tested standard library primitives.

## 6. Closure Capture Semantics

### Thunk: Move Captures

`Thunk::new(f: impl FnOnce() -> A + 'a)` accepts any closure, which will typically use `move` captures:

```rust
let x = 42;
let thunk = Thunk::new(move || x);  // x moved into closure
```

Since `Thunk` uses `FnOnce`, the closure can consume captured values. This is the most flexible capture mode.

Borrowing also works:

```rust
let x = 42;
let thunk = Thunk::new(|| &x);  // x borrowed
```

The `'a` lifetime parameter correctly constrains how long borrows can live.

### Lazy: Move Captures with Boxing

`Lazy::new(f: impl FnOnce() -> A + 'a)` boxes the closure: `Box::new(f)`. This adds a heap allocation but is necessary because `LazyCell` requires a `Box<dyn FnOnce()>`.

The `ref_map` method (line 524-531) clones `self` first, then captures the clone in a new closure:

```rust
pub fn ref_map<B: 'a>(self, f: impl FnOnce(&A) -> B + 'a) -> Lazy<'a, B, RcLazyConfig> {
    let fa = self.clone();
    let init: Box<dyn FnOnce() -> B + 'a> = Box::new(move || f(fa.evaluate()));
    Lazy(RcLazyConfig::lazy_new(init))
}
```

This works because `Lazy` is cheaply cloneable (Rc/Arc clone). The original `self` is consumed but a clone is captured. This is correct but slightly surprising; a user might wonder why `self` is consumed when `evaluate` only needs `&self`. This is because `ref_map` is positioned as a chaining/fluent API method.

### Trampoline: `'static` Move Captures

All closures in `Trampoline` methods require `'static`:

```rust
pub fn new(f: impl FnOnce() -> A + 'static) -> Self
pub fn bind<B: 'static + Send>(self, f: impl FnOnce(A) -> Trampoline<B> + 'static) -> Trampoline<B>
```

This prevents borrowing from the environment. All captures must be owned. This is an inherent limitation of the `Free` monad's `Box<dyn Any>` design.

### Fn vs FnOnce Discrepancy in the HKT Trait Layer

There is a notable discrepancy between the inherent methods and the HKT trait implementations for `Thunk`:

- **Inherent `bind`** (line 184): takes `f: impl FnOnce(A) -> Thunk<'a, B> + 'a`
- **Semimonad::bind** (line 495-499): takes `func: impl Fn(A) -> ... + 'a`

The `Semimonad` trait uses `Fn` (not `FnOnce`), which is more restrictive. This means closures passed through the HKT layer must be callable multiple times, even though `Thunk` only calls them once. The same pattern holds for `Functor::map` (`Fn` in the trait, `FnOnce` in the inherent method).

This is a design-level decision: the HKT traits use `Fn` because some types (like `Vec`) need to call the mapping function multiple times. For `Thunk`, the inherent `FnOnce` methods are more ergonomic and should be preferred when not using generic HKT code.

## 7. Drop Behavior

### Thunk: Trivial Drop

`Thunk` contains `Box<dyn FnOnce() -> A + 'a>`. Dropping it drops the boxed closure, which drops any captured values. No issues. If the thunk is never evaluated, captured resources are still cleaned up on drop.

### Lazy: Reference-Counted Drop

`Lazy` wraps `Rc<LazyCell<...>>` or `Arc<LazyLock<...>>`. Dropping a `Lazy` decrements the reference count. When the last reference is dropped:
- If the cell was initialized, the computed value `A` is dropped.
- If the cell was never initialized, the closure (and its captures) is dropped.

No cycle risk exists between `Lazy` values themselves, because `Lazy` values do not reference each other in their internal structure. However, a user could create a cycle by having a `Lazy` closure that captures an `Rc` pointing to something that holds another `Lazy` sharing the same `Rc<LazyCell>`. This is the standard `Rc` cycle concern, not specific to `Lazy`.

### Trampoline/Free: Iterative Drop

`Free<F, A>` has a custom `Drop` implementation (lines 472-512 of free.rs) that iteratively unwinds `Bind` chains. Without this, a deeply nested `Bind` chain would cause stack overflow during recursive drop. The implementation:

1. Takes the inner `Option` value.
2. If it is a `Bind`, iteratively follows the `head` chain (which is always `Box<Free<F, TypeErasedValue>>`).
3. For each `Bind` in the chain, takes its inner value and continues.

This is well-designed. The continuations in the `CatList` are dropped normally (which is fine because `CatList` itself handles deep drop through its tree structure).

**Potential concern:** If a `Free` has a deeply nested `Wrap` chain (not `Bind`), the drop is still recursive through the `Thunk` closures. However, in practice, `Wrap` chains are bounded because each `Wrap` is immediately resolved during `evaluate()`, and the construction pattern (`defer` wraps a single layer at a time) does not create deep `Wrap` nesting.

## 8. Size and Alignment

### Thunk

```
Thunk<'a, A> = Box<dyn FnOnce() -> A + 'a>
```

A `Box<dyn FnOnce()>` is a fat pointer: 2 words (pointer to data + pointer to vtable) = 16 bytes on 64-bit. `Thunk` is a newtype wrapper, so `size_of::<Thunk<A>>() == 16` bytes. This is optimal for a dynamically-dispatched closure.

### Lazy

```
Lazy<'a, A, RcLazyConfig> = Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>
```

`Rc<T>` is a single pointer (8 bytes on 64-bit). So `size_of::<RcLazy<A>>() == 8`. Very compact.

The pointed-to `LazyCell<A, Box<dyn FnOnce()>>` on the heap contains a `UnsafeCell<State<A, F>>` where `State` is either the closure or the computed value. Its size depends on `A` and the closure, but this is heap-allocated so the stack footprint is minimal.

```
Lazy<'a, A, ArcLazyConfig> = Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>
```

Same stack size: 8 bytes (single pointer). `LazyLock` has additional synchronization overhead on the heap, but this is hidden from the user.

### Trampoline

```
Trampoline<A> = Free<ThunkBrand, A> = Option<FreeInner<ThunkBrand, A>>
```

`FreeInner` is an enum with three variants:
- `Pure(A)`: size of `A`
- `Wrap(Thunk<'static, Free<ThunkBrand, A>>)`: 16 bytes (the Thunk)
- `Bind { head: Box<Free<...>>, continuations: CatList<Continuation<F>>, _marker: PhantomData<A> }`: 8 (Box pointer) + size of CatList + 0 (PhantomData)

The `Option` wrapping adds the discriminant. The total size depends on `A` and the `CatList` size. For the common case where `A` is small, `Trampoline` is dominated by the enum discriminant + the largest variant. This is reasonably efficient; the main cost is in heap allocations for closures and continuations, not in the stack representation.

## 9. Interaction with Rc/Arc

### Lazy: Core Usage of Rc/Arc

The `LazyConfig` trait abstracts over the pointer type. `RcLazyConfig` uses `Rc<LazyCell<...>>` and `ArcLazyConfig` uses `Arc<LazyLock<...>>`.

- Cloning a `Lazy` clones the `Rc`/`Arc`, which is O(1) (reference count increment).
- All clones share the same memoization cache.
- No `Weak` references are used, so there is no automatic cycle-breaking. Users must be aware of potential reference cycles.

The design is clean. The `LazyConfig` trait effectively parameterizes over the pointer/cell combination without requiring the user to think about the internals.

### Thunk: No Rc/Arc

`Thunk` does not use reference counting. It is a standalone `Box<dyn FnOnce()>`. This is the correct choice for a non-memoized, single-use computation.

### Trampoline: Indirect Rc/Arc via Free

`Free` does not use `Rc`/`Arc` directly. It uses `Box` for heap allocation. The `CatList` (used for continuations) may internally use reference counting for its tree structure, but this is an implementation detail of `CatList`.

The `arc_tail_rec_m` method wraps the step function in an `Arc` to allow cloning for iteration. This is an explicit user-facing escape hatch for non-`Clone` closures.

### From Conversions

The `From` conversions between types interact with Rc/Arc correctly:
- `From<Lazy<'a, A, Config>> for Thunk<'a, A>` requires `A: Clone` because `Lazy::evaluate()` returns `&A`, so the conversion must clone the value.
- `From<Trampoline<A>> for Lazy<'a, A, RcLazyConfig>` creates a new `RcLazy` that evaluates the trampoline on first access. The `A: Send` bound propagates from `Trampoline`.
- `From<Lazy<'static, A, Config>> for Trampoline<A>` requires `A: Send + Clone + 'static`.

## 10. Ergonomics

### Thunk: Good Ergonomics

`Thunk` is straightforward. The API is small: `new`, `pure`, `defer`, `map`, `bind`, `evaluate`. The ownership model (move-only, single evaluation) is easy to understand.

One ergonomic friction point: the `Semimonad::bind` trait uses `Fn` where the inherent `bind` uses `FnOnce`. Users working generically through the HKT layer face stricter closure requirements.

### Lazy: Good Ergonomics with Config Noise

`Lazy` requires specifying the `Config` type parameter in many positions:
```rust
let lazy = Lazy::<_, RcLazyConfig>::new(|| 42);
```

The type aliases `RcLazy` and `ArcLazy` mitigate this:
```rust
let lazy = RcLazy::new(|| 42);
```

The `evaluate(&self) -> &A` return type means users always work with references. This is correct for a memoized value but can be slightly inconvenient when you need the value by-value (requiring `.clone()`). The `From<Lazy<...>> for Thunk<...>` conversion addresses this by cloning internally.

### Trampoline: Heavy but Necessary

`Trampoline` requires `A: 'static + Send`, which is a significant restriction. Users cannot:
- Use borrowed data.
- Use `Rc<T>` (which is `!Send`).

This pushes users toward owned, thread-safe data, which is sometimes more restrictive than necessary (given that `Trampoline` itself is `!Send`; see Section 3).

The `tail_rec_m` API requires `Clone` on the step function, with `arc_tail_rec_m` as a fallback. This two-method pattern is pragmatic but adds API surface.

### Missing Debug/Display

None of the three types implement `Debug` or `Display`. This makes debugging difficult; you cannot `println!("{:?}", thunk)`. This is understandable for `Thunk` (closures are not `Debug`), but `Lazy` could implement `Debug` when `A: Debug` by checking whether it has been evaluated.

**Recommendation:** Implement `Debug` for `Lazy<'a, A, Config>` where `A: Debug`, displaying either `Lazy(<unevaluated>)` or `Lazy(<value>)`. This would significantly improve debugging ergonomics. Similarly, `Trampoline` could show `Trampoline(<unevaluated>)`.

## 11. Pin Analysis

None of the three types use `Pin`. Should they?

### Thunk: No Need for Pin

`Thunk` contains a `Box<dyn FnOnce()>`. The closure is heap-allocated and not self-referential. Moving a `Thunk` moves the `Box` (a pointer), not the closure data. No pinning needed.

### Lazy: No Need for Pin

`Lazy` contains an `Rc<LazyCell<...>>` or `Arc<LazyLock<...>>`. The reference-counted pointer is a single word that can be freely moved. The pointed-to data is heap-allocated and stable. No pinning needed.

### Trampoline: No Need for Pin

`Trampoline` contains `Free<ThunkBrand, A>`, which uses `Box` and `Option` wrappers. None of the internal structures are self-referential. No pinning needed.

**If async evaluation were ever added** (e.g., `async fn evaluate(self) -> A`), pinning would become relevant because `Future`s returned by async functions are self-referential. But for the current synchronous design, `Pin` is unnecessary.

## 12. Variance Analysis

### Thunk<'a, A>: Covariant in A, Covariant in 'a

`Thunk<'a, A>` wraps `Box<dyn FnOnce() -> A + 'a>`. A function that returns `A` is covariant in `A` (if `A` is a subtype of `B`, then `FnOnce() -> A` is a subtype of `FnOnce() -> B`). The `'a` parameter appears in covariant position (the closure outlives `'a`; a longer-lived closure can be used where a shorter-lived one is expected).

This is correct. You can assign a `Thunk<'long, SpecificType>` where a `Thunk<'short, GeneralType>` is expected (assuming the appropriate subtyping relationships).

### Lazy<'a, A, Config>: Covariant in A (via Rc/Arc indirection), Covariant in 'a

`Lazy` wraps `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>` or the `Arc` equivalent. `Rc<T>` is covariant in `T`, and `LazyCell<T, F>` is covariant in `T` (it stores `T` as an output). The closure `Box<dyn FnOnce() -> A + 'a>` is covariant in `A` and covariant in `'a`.

However, `LazyCell<A, F>` stores the closure `F` internally, and `F = Box<dyn FnOnce() -> A + 'a>`. The variance of `LazyCell` in `F` is invariant (since `F` is stored in an `UnsafeCell`). But since `F` is fixed (always `Box<dyn FnOnce() -> A + 'a>`), the user-facing variance is determined by the exposed parameters `A` and `'a`, which are both covariant. Correct.

### Trampoline<A>: Invariant in A

`Trampoline<A>` wraps `Free<ThunkBrand, A>`. `Free<F, A>` can contain `FreeInner::Bind { head: Box<Free<F, TypeErasedValue>>, ... }` where `A` appears only in `PhantomData<A>`. However, the `downcast::<A>()` in `evaluate()` depends on the exact type `A`. More critically, `Free::bind` takes `A` in input position (via the closure `FnOnce(A) -> ...`) and produces `Free<F, B>` with `A` consumed.

Due to the type erasure (`Box<dyn Any>`), the actual runtime type of `A` must match exactly. The `PhantomData<A>` in the `Bind` variant does not affect variance directly, but the overall type is effectively invariant in `A` because `A` is used in both covariant (output of evaluation) and contravariant (input to continuations) positions through the type-erased machinery.

This invariance is correct for a monadic type that both produces and consumes values of type `A`.

### Config in Lazy: Phantom, Invariant

The `Config` parameter in `Lazy<'a, A, Config>` is used only at the type level (to select associated types). Since `Config` appears in `Config::Lazy<'a, A>` (as a trait), it is effectively invariant. This is fine since `Config` is always a concrete type (`RcLazyConfig` or `ArcLazyConfig`), not a lifetime-parameterized or generic type.

## Summary of Findings

### Sound and Well-Designed

1. Ownership semantics are correct across all three types.
2. No unsafe code exists; type erasure uses safe `downcast()`.
3. Interior mutability in `Lazy` correctly delegates to `LazyCell`/`LazyLock`.
4. Drop behavior is carefully handled, especially the iterative drop in `Free`.
5. Variance is correct for all types.
6. Lifetime parameters are minimally constrained and correctly used.
