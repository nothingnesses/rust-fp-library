# Performance Characteristics of the Lazy Evaluation Hierarchy

## 1. Box Allocation in Thunk

### Current Pattern

Every `Thunk` construction allocates a `Box<dyn FnOnce() -> A + 'a>` (line 90 of `thunk.rs`):

```rust
pub struct Thunk<'a, A>(Box<dyn FnOnce() -> A + 'a>);
```

The `new` constructor (line 114) boxes the closure immediately:

```rust
pub fn new(f: impl FnOnce() -> A + 'a) -> Self {
    Thunk(Box::new(f))
}
```

### Cost per Operation

- **`map`** (line 225-229): Creates one new `Thunk`, meaning one `Box` allocation. The closure captures `self.0` (the old boxed closure) and `f`, so the new box stores a closure that holds a box. This is two pointer indirections at evaluation time.
- **`bind`** (line 194-203): Creates one new `Thunk`, same pattern. The closure captures `self.0` and `f`, evaluates the first closure, passes the result to `f`, then evaluates the result. Two `Box` allocations live simultaneously during evaluation (the outer thunk and the inner thunk returned by `f`).
- **`pure`** (line 137-141): Allocates one `Box` even for an already-known value. The closure `move || a` captures `a` by value and returns it. This is a non-zero cost; a direct `enum` variant like `Thunk::Pure(A)` would avoid it.
- **Chain of N maps**: Produces N nested `Box` allocations. At evaluation time, all N closures are called in sequence. The closures are consumed (`FnOnce`), so each `Box` is freed as it runs.

### Can the Box Be Avoided?

Not easily, given the current design goals:

1. **`dyn FnOnce`** requires type erasure because each closure has a unique anonymous type. Without the `Box`, you would need a `Thunk<F>` parameterized by the concrete closure type, which makes `Thunk` non-uniform (every chain of `map`/`bind` produces a different type).
2. **Enum-based approach**: A `Thunk` could use an enum `{ Pure(A), Deferred(Box<dyn FnOnce() -> A>) }` to avoid boxing pure values. This would save one allocation for `pure` but not for `map`/`bind`.
3. **Small-closure optimization**: In theory, a `SmallBox` or inline storage (like `smallvec` for closures) could store small closures inline. In practice, this is fragile: closure sizes depend on captures and are not easily bounded at compile time.

### Verdict

The `Box` allocation per `map`/`bind` is an inherent cost of type erasure. For short chains (1-5 operations), this is negligible. For long chains (hundreds of operations), each allocation touches the global allocator. The real risk is not the allocation cost per se, but the recursive evaluation pattern (see Section 8).

## 2. Trampoline and Free Monad Allocation Patterns

### Architecture

`Trampoline<A>` is a newtype over `Free<ThunkBrand, A>` (line 73-76 of `trampoline.rs`):

```rust
pub struct Trampoline<A: 'static>(Free<ThunkBrand, A>);
```

`Free<F, A>` uses the "Reflection without Remorse" pattern (see `free.rs`). Its internal representation has three variants (`FreeInner`, lines 110-138 of `free.rs`):

- `Pure(A)`: A completed value.
- `Wrap(F<Free<F, A>>)`: A suspended computation (for `ThunkBrand`, this is `Thunk<'static, Free<ThunkBrand, A>>`).
- `Bind { head: Box<Free<F, TypeErasedValue>>, continuations: CatList<Continuation<F>>, _marker }`: A computation followed by a queue of continuations.

### Allocation Per Operation

- **`Trampoline::pure(a)`**: Creates `Free::pure(a)`, which is `Free(Some(FreeInner::Pure(a)))`. Zero heap allocations (ignoring the `Option` wrapper).
- **`Trampoline::new(f)`** (line 133-135): Creates a `Thunk` (one `Box`) wrapping a closure that returns `Free::pure(f())`, then wraps that in `Free::wrap`. This is 1 `Box` allocation for the thunk closure.
- **`Trampoline::bind(f)`** (line 199-204): Delegates to `Free::bind`, which:
  - **Always** boxes the continuation `f` as a `Continuation<F>` (line 295 of `free.rs`): `Box<dyn FnOnce(TypeErasedValue) -> Free<F, TypeErasedValue>>`. That is 1 `Box` allocation.
  - For the `Pure` case: Also creates a `Free::pure(a).erase_type()` which boxes `a` as `Box<dyn Any>` (line 364 of `free.rs`), then boxes the entire `head` as `Box<Free<...>>` (line 312). That is 2-3 more allocations.
  - For the `Bind` case: Only appends to the `CatList` (O(1), no new `Box` for the head). The continuation itself is still boxed, so 1 allocation.
- **`Trampoline::defer(f)`** (line 170-172): Creates a `Thunk::new(move || f().0)` and wraps it. 1 `Box` for the thunk.

### Heap Pressure for Deep Chains

For a chain of N `bind` operations on a `Trampoline`:

- **Construction**: O(N) allocations total. Each `bind` appends one boxed continuation to the `CatList`. The `CatList` uses `VecDeque` internally (see `cat_list.rs` line 90-95), so appends are amortized O(1) and don't allocate per element (they batch in the deque's buffer).
- **Evaluation** (`Free::evaluate`, lines 419-467 of `free.rs`): The trampoline loop processes the `CatList` iteratively. Each continuation is consumed once. The `downcast` calls (`Box<dyn Any>` to concrete type) add a vtable check but no allocation. The main costs are:
  - Unboxing each continuation (already allocated).
  - Boxing intermediate results as `TypeErasedValue` (one `Box<dyn Any>` per step).
  - Evaluating `Thunk` closures for `Wrap` variants.

**Summary**: For N binds, expect approximately 2N heap allocations (N continuations + N type-erased values). This is competitive with other trampolined free monad implementations but significantly more expensive than a simple loop.

### Type Erasure Overhead

The `Box<dyn Any>` pattern in `Free::bind` (lines 294-301 of `free.rs`) introduces:

1. A `Box` allocation to erase the type.
2. A `downcast` (pointer comparison of `TypeId`) to recover it.
3. A `Box` deallocation after downcast.

The `downcast` itself is very cheap (single `TypeId` comparison), but the boxing/unboxing cycle is an unavoidable cost of the "Reflection without Remorse" approach.

## 3. Lazy Memoization Overhead

### RcLazy (Single-Threaded)

The structure is `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>` (line 178 of `lazy.rs`).

Allocation breakdown for `Lazy::new(f)`:

1. `Box::new(f)`: Boxes the initializer closure. One allocation.
2. `Rc::new(LazyCell::new(...))`: Allocates the `Rc` control block + `LazyCell`. One allocation.

Total: **2 heap allocations** per `Lazy::new`.

Evaluation cost (`evaluate`, line 485-487):

- First call: `LazyCell::force` runs the closure, stores the result in-place. The boxed closure is consumed and deallocated. Cost: one function call + one deallocation.
- Subsequent calls: `LazyCell::force` returns the cached value immediately. Cost: one pointer dereference through `Rc`, then a branch (checking if initialized). This is effectively free.

### ArcLazy (Thread-Safe)

The structure is `Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>` (line 292).

Same allocation count as `RcLazy` (2 allocations). The evaluation difference:

- First call: `LazyLock::force` uses an internal `Once` synchronization primitive. On first call, this acquires a lock (typically an atomic compare-and-swap), runs the closure, stores the result, and releases. Cost: one atomic CAS + one function call.
- Subsequent calls: `LazyLock::force` performs a single atomic load to check the initialization flag. On x86, this compiles to a plain `mov` instruction (acquire semantics are free on x86). Cost: effectively one branch.

### Overhead Assessment

For single evaluations, the overhead of `Lazy` compared to just calling the closure directly is:

- 2 extra heap allocations (the `Box` for the closure and the `Rc`/`Arc` wrapper).
- 1 pointer indirection through `Rc`/`Arc`.
- For `ArcLazy`: 1 atomic load per subsequent access (negligible on x86).

This overhead is justified when the computation is expensive and accessed multiple times. For cheap computations accessed once, `Lazy` is strictly worse than `Thunk` (more allocations, same work done).

### `pure` Allocates Unnecessarily

`Lazy::pure(a)` (line 532-534) still boxes a closure `move || a` and wraps in `Rc<LazyCell>`. An optimization for already-known values could skip the closure entirely, storing the value directly. However, `LazyCell`'s API does not expose a "pre-initialized" constructor in stable Rust, making this difficult without unsafe code or a custom cell type.

## 4. Unnecessary Clones and Allocations

### Lazy `ref_map` Clones the Lazy

`Lazy::ref_map` (lines 554-561) calls `self.clone()` to capture the `Lazy` in a new closure:

```rust
pub fn ref_map<B: 'a>(self, f: impl FnOnce(&A) -> B + 'a) -> Lazy<'a, B, RcLazyConfig> {
    let fa = self.clone();
    let init: Box<dyn FnOnce() -> B + 'a> = Box::new(move || f(fa.evaluate()));
    Lazy(RcLazyConfig::lazy_new(init))
}
```

This takes `self` by value but also clones it. The `self` is then dropped at the end of the method. Since `self` is moved in, the clone is necessary for the closure to capture it, but `self` itself is wasted. A more efficient pattern would move `self` directly into the closure:

```rust
pub fn ref_map<B: 'a>(self, f: impl FnOnce(&A) -> B + 'a) -> Lazy<'a, B, RcLazyConfig> {
    let init: Box<dyn FnOnce() -> B + 'a> = Box::new(move || f(self.evaluate()));
    Lazy(RcLazyConfig::lazy_new(init))
}
```

However, `self.evaluate()` returns `&A` with lifetime tied to `self`, and moving `self` into the closure means the reference would borrow from the closure's captured data, which is fine since `FnOnce` consumes the closure. This should work and would save one `Rc::clone` (which is cheap but nonzero: an atomic increment for `Arc`, a non-atomic increment for `Rc`).

### Trampoline `map` Desugars Through `bind`

`Trampoline::map` (lines 224-229 of `trampoline.rs`) is implemented as:

```rust
pub fn map<B: 'static>(self, f: impl FnOnce(A) -> B + 'static) -> Trampoline<B> {
    self.bind(move |a| Trampoline::pure(f(a)))
}
```

This means every `map` pays the full cost of `bind` (boxing a continuation, type erasure, etc.) plus creates an intermediate `Trampoline::pure`. A dedicated `map` on `Free` could avoid the type erasure roundtrip by applying `f` directly when the inner value is `Pure`. This is a known optimization opportunity.

### `Free::erase_type` Recurses Into Wrap

`Free::erase_type` (lines 358-380 of `free.rs`) for the `Wrap` variant calls `F::map(|inner| inner.erase_type(), fa)`. For `ThunkBrand`, this creates a new `Thunk` that wraps the old one, adding another layer of indirection. Each type erasure of a `Wrap` adds one `Thunk` allocation.

### `From<Lazy> for Trampoline` Clones

The `From<Lazy<'static, A, Config>> for Trampoline<A>` impl (lines 490-509 of `trampoline.rs`) uses `lazy.evaluate().clone()`:

```rust
fn from(lazy: Lazy<'static, A, Config>) -> Self {
    Trampoline::new(move || lazy.evaluate().clone())
}
```

This creates a new `Trampoline` that, when evaluated, forces the `Lazy` and clones the result. This is necessary because `Lazy::evaluate` returns `&A`, not `A`. The clone is unavoidable given the API, but callers should be aware that converting `Lazy<String>` to `Trampoline<String>` copies the cached string on each evaluation.

## 5. Zero-Cost Abstraction Assessment

### What "Zero-Cost" Means

The README (lines 151-168) claims that core operations like `map` and `bind` use `impl Fn` for "zero-cost abstractions" with no heap allocation and static dispatch. It then explicitly carves out exceptions (lines 170-176) for "Functions as Data" and "Lazy Evaluation."

### Thunk: Not Zero-Cost

Every `Thunk` operation allocates a `Box<dyn FnOnce>`. This is dynamic dispatch (through the `dyn` vtable) and heap allocation. Compared to hand-written eager code:

```rust
// Hand-written eager equivalent of Thunk::pure(10).map(|x| x * 2).evaluate()
let result = (|x| x * 2)(10);  // Zero allocations, fully inlined
```

The `Thunk` version allocates 2 boxes (one for `pure`, one for `map`) and performs 2 indirect calls. This is fundamentally not zero-cost.

### Trampoline: Not Zero-Cost

`Trampoline` adds even more overhead: `Box<dyn Any>` type erasure, `CatList` management, and the trampoline loop itself. Compared to a hand-written loop:

```rust
// Hand-written equivalent of a sum via Trampoline
let mut acc = 0;
for i in 0..1000 { acc += i; }
```

The `Trampoline` version would allocate ~2000 boxes for the bind chain. The trade-off is justified only when stack safety is required (deep recursion that would otherwise overflow).

### Lazy: Justified Overhead

`Lazy` is not zero-cost, but its overhead (2 allocations + 1 pointer indirection) is the minimum possible for shared memoization with type erasure. The standard library's `LazyCell`/`LazyLock` are as efficient as possible for this pattern. The overhead is amortized over repeated accesses.

### Accurate Characterization

The README's characterization is honest: it claims zero-cost for `map`, `bind`, `fold` on standard types (`Option`, `Vec`, `Result`) where `impl Fn` static dispatch applies, and explicitly notes the exceptions for lazy types. The lazy types are not zero-cost and should not be described as such.

## 6. `dyn Fn` vs `impl Fn`: Performance Implications

### Thunk Uses `Box<dyn FnOnce>`

`Thunk`'s internal representation uses dynamic dispatch:

```rust
pub struct Thunk<'a, A>(Box<dyn FnOnce() -> A + 'a>);
```

This means:
- Every call to the closure goes through a vtable indirection.
- The compiler cannot inline the closure body across the `dyn` boundary.
- The closure's captures are heap-allocated (in the `Box`).

### Lazy Uses `Box<dyn FnOnce>` Inside LazyCell

`RcLazyConfig::Lazy` is `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>` (line 178 of `lazy.rs`). The initializer is stored as a boxed trait object. Same implications as `Thunk`: no inlining, vtable dispatch.

### Impact

For operations like `Thunk::map(|x| x + 1, thunk)`, the compiler sees:
1. A `Box<dyn FnOnce>` for the outer thunk.
2. Inside, another `Box<dyn FnOnce>` for the inner thunk.
3. The `|x| x + 1` closure is erased behind the outer box.

The optimizer cannot see through `dyn` boundaries. It cannot determine that the entire chain is equivalent to `value + 1` and optimize accordingly. This is the fundamental cost of deferred evaluation with type erasure.

### Comparison with `impl Fn` Paths

When using `map::<OptionBrand, _, _>(|x| x + 1, Some(5))`, the compiler monomorphizes the function, inlines the closure, and can optimize the entire expression to `Some(6)`. There are zero vtable lookups and zero heap allocations.

### Alternative: Enum-Based Thunk

An alternative design uses an enum to represent common operations:

```rust
enum Thunk<'a, A> {
    Pure(A),
    Map(Box<dyn FnOnce(???) -> A + 'a>, Box<Thunk<'a, ???>>),
    Bind(Box<dyn FnOnce(???) -> Thunk<'a, A> + 'a>, Box<Thunk<'a, ???>>),
}
```

This does not actually help with inlining (the closures are still `dyn`), but it could avoid one `Box` allocation for `Pure` values and enable pattern-matching optimizations in the evaluator.

## 8. Stack Usage: Thunk Recursive Evaluation vs Trampoline Iterative Loop

### Thunk: Recursive Evaluation

`Thunk::evaluate` (line 248-250) simply calls the stored closure:

```rust
pub fn evaluate(self) -> A {
    (self.0)()
}
```

For a chain of N maps, evaluation unfolds as nested closure calls:

```
evaluate outer_thunk
  -> calls outer_closure
    -> evaluates inner_thunk (calls inner_closure)
      -> evaluates deeper_thunk ...
```

Each closure call adds a stack frame. For a chain of N maps, this is N stack frames deep. The default Rust thread stack is 8 MiB; each frame is typically 32-128 bytes, so chains beyond ~50,000-100,000 operations risk stack overflow.

`Thunk::bind` (lines 194-203) has the same pattern:

```rust
Thunk::new(move || {
    let a = (self.0)();
    let thunk_b = f(a);
    (thunk_b.0)()
})
```

Each bind adds 2 stack frames (one for the outer closure, one for the inner thunk evaluation).

### Thunk's `tail_rec_m`: Stack-Safe

`ThunkBrand`'s `MonadRec::tail_rec_m` (lines 610-625 of `thunk.rs`) uses an explicit loop:

```rust
Thunk::new(move || {
    let mut current = a;
    loop {
        match f(current).evaluate() {
            Step::Loop(next) => current = next,
            Step::Done(res) => break res,
        }
    }
})
```

This is stack-safe because the loop replaces recursion. However, this only applies to computations expressed through `tail_rec_m`; ad-hoc `bind` chains are still recursive.

### Trampoline: Iterative Loop

`Free::evaluate` (lines 419-467 of `free.rs`) uses an explicit `loop`:

```rust
loop {
    match inner {
        FreeInner::Pure(val) => { /* apply continuation or return */ }
        FreeInner::Wrap(fa) => { current = <F as Evaluable>::evaluate(fa); }
        FreeInner::Bind { head, continuations, .. } => {
            current = *head;
            continuations = inner_continuations.append(continuations);
        }
    }
}
```

This is fully stack-safe regardless of chain depth. The `CatList` of continuations is processed iteratively, and `Wrap` variants are evaluated by the `Evaluable` trait for `ThunkBrand`, which calls the `Thunk` closure directly (one stack frame, not recursive).

### Stack Safety Comparison

| Type | `map` chain of N | `bind` chain of N | `tail_rec_m` depth N |
|------|-------------------|--------------------|-----------------------|
| Thunk | N stack frames | 2N stack frames | O(1) stack frames |
| Trampoline | O(1) stack | O(1) stack | O(1) stack |

For chains deeper than ~10,000 operations, `Thunk` will overflow the stack while `Trampoline` will not.

## 9. Arena Allocation and Small-Buffer Optimization

### Arena Allocation

For `Trampoline` chains that create many short-lived `Box<dyn Any>` and `Box<dyn FnOnce>` values, a bump allocator (arena) could significantly reduce allocation overhead:

1. **Observation**: During `Free::evaluate`, each step allocates a `TypeErasedValue` (`Box<dyn Any>`) that is immediately consumed by the next continuation. These are sequential, short-lived allocations.
2. **Opportunity**: A thread-local bump allocator could serve these allocations from a pre-allocated buffer. Allocation becomes a pointer bump (a single addition), and deallocation is a no-op (the arena is reset after evaluation completes).
3. **Challenge**: Rust's `Box` uses the global allocator. Using a custom allocator requires either nightly's `allocator_api` feature or a custom smart pointer type. This is invasive.

### Small-Buffer Optimization for Thunk

Many `Thunk` closures are small (capturing 1-2 values, typically 8-24 bytes). A `SmallThunk` type could store closures inline up to a threshold (e.g., 64 bytes) and fall back to `Box` for larger closures:

```rust
// Hypothetical
enum ThunkStorage<'a, A> {
    Inline([u8; 64], /* vtable pointer */),
    Boxed(Box<dyn FnOnce() -> A + 'a>),
}
```

This would avoid heap allocation for the majority of closures. However:
- Implementing this safely is non-trivial (requires `MaybeUninit`, manual vtable management, or `unsafe`).
- The `smallbox` crate provides this pattern but adds a dependency.
- The benefit is most pronounced in tight loops creating many thunks.

### CatList Already Uses VecDeque

The `CatList` implementation (line 90-95 of `cat_list.rs`) uses `VecDeque<Vec<A>>` internally, which amortizes allocation across multiple appends. This is already a good choice; further optimization (e.g., arena-backed deque) would provide diminishing returns.

### Recommendation

Arena allocation for `Free::evaluate` is the highest-impact optimization opportunity. It would eliminate the per-step `Box<dyn Any>` allocation/deallocation cycle during trampoline evaluation. This should be measured first to confirm the allocation overhead is significant in practice.

## 10. Inlining Opportunities

### Current `#[inline]` Usage

- **`Trampoline`**: `#[inline]` is applied to `pure`, `new`, `defer`, `bind`, `map`, and `lift2` (lines 92, 118, 148, 186, 214 of `trampoline.rs`). Good coverage.
- **`Thunk`**: **No `#[inline]` annotations at all.** None of `new`, `pure`, `map`, `bind`, `evaluate`, `defer`, or any HKT trait implementations are marked `#[inline]`.
- **`Lazy`**: **No `#[inline]` annotations at all.** Neither `new`, `pure`, `evaluate`, nor `ref_map` are marked.
- **`Free`**: Only `Free::pure` has `#[inline]` (line 190 of `free.rs`). `Free::wrap`, `Free::bind`, `Free::evaluate`, `Free::erase_type` are not marked.

### Impact

For `Thunk` and `Lazy`, the lack of `#[inline]` means the compiler may not inline these methods across crate boundaries. Since the library is consumed as a dependency, cross-crate inlining requires either `#[inline]` hints or LTO (Link-Time Optimization).

Critical methods that should be `#[inline]`:

- `Thunk::new`, `Thunk::pure`, `Thunk::evaluate`: These are trivial wrappers (1-2 lines). The compiler almost certainly inlines them within the crate, but not across crate boundaries without the annotation.
- `Thunk::map`, `Thunk::bind`: These are small but create closures. Inlining allows the compiler to see the closure structure, potentially optimizing the capture.
- `Lazy::new`, `Lazy::evaluate`, `Lazy::pure`: Same reasoning as `Thunk`.
- `Free::bind`: This is the performance-critical path. While the function body is non-trivial, `#[inline]` would allow the compiler to specialize for the common `Pure` and `Bind` match arms.
- `Free::evaluate`: The evaluation loop is too large to benefit from `#[inline]`, but `#[inline(never)]` might actually help by preventing code bloat at call sites while keeping the hot loop compact.

### Recommendation

Add `#[inline]` to all trivial wrapper methods on `Thunk` and `Lazy`. For `Free::bind`, consider `#[inline]` to enable cross-crate specialization. For `Free::evaluate`, leave without annotation or add `#[inline(never)]` to avoid code bloat.
