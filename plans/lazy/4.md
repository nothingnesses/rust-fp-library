# Deep Analysis: Lazy/RcLazy/ArcLazy Type Implementations

## 1. The Three Lazy Variants: Is the Split Justified?

There are not truly three separate types. There is one generic type `Lazy<'a, A, Config: LazyConfig>` (line 424 of `lazy.rs`) with two concrete configurations:

- `RcLazy<'a, A>` = `Lazy<'a, A, RcLazyConfig>` (line 689)
- `ArcLazy<'a, A>` = `Lazy<'a, A, ArcLazyConfig>` (line 692)

**Justification.** The split is well-justified. `Rc<LazyCell>` is not `Send + Sync`, so it cannot cross thread boundaries. `Arc<LazyLock>` is thread-safe but imposes `Send` bounds on the initializer closure. Making these two configurations of a single generic type is cleaner than two completely separate structs, and it avoids code duplication for the shared `evaluate()` method and `Clone` impl.

**Naming.** The naming is clear. `RcLazy`/`ArcLazy` follows the established library convention (`RcFnBrand`/`ArcFnBrand`). The `LazyConfig` trait (line 65) is well-named as a strategy/configuration trait. The type aliases at lines 689 and 692 are concise and predictable.

**The `LazyConfig` trait approach.** This is a good design choice. The trait (lines 65-170) bundles together four associated types (`Lazy`, `TryLazy`, `Thunk`, `TryThunk`) and four methods, providing a single extension point. The documentation at lines 55-64 explicitly calls out that the trait is open for third-party implementations (e.g., `parking_lot`-based locks, async-aware cells), which is a forward-looking design.

**Minor concern.** The `LazyConfig` trait requires `'static` (line 65: `pub trait LazyConfig: 'static`). This is necessary because brand types like `LazyBrand<Config>` are used as type-level markers, but it means custom configs cannot carry lifetime-parameterized data. This is acceptable since configs are purely type-level strategy selectors.

## 2. Interior Mutability Approach: Correctness and Soundness

### RcLazyConfig (lines 177-284)

- Uses `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>`.
- `LazyCell` (from `std::cell`, stabilized in Rust 1.80) provides single-threaded, initialize-once semantics.
- `Rc` provides shared ownership so clones share the same cell.

**Soundness.** `LazyCell` is a well-audited standard library type. It uses internal `UnsafeCell` with a state flag but exposes a safe API. The `Rc` wrapper ensures single-threaded access. This is sound.

### ArcLazyConfig (lines 291-398)

- Uses `Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>`.
- `LazyLock` (from `std::sync`, stabilized in Rust 1.80) provides thread-safe, initialize-once semantics with internal blocking.
- `Arc` provides thread-safe shared ownership.

**Soundness.** `LazyLock` uses internal synchronization (a `Once`-like mechanism) to guarantee exactly one initialization even under contention. Blocking threads wait until initialization completes. The `Send` bound on the closure at line 293 is correct; without it, the closure could capture non-`Send` data that would then be executed on a different thread.

**No `OnceCell`/`OnceLock` for core storage.** The design correctly uses `LazyCell`/`LazyLock` rather than raw `OnceCell`/`OnceLock`. The `Once*` cell variants are used only in the `fix` functions (lines 738 and 791) for the bootstrap cell that breaks the self-referential cycle. This is the right choice: `LazyCell`/`LazyLock` encapsulate the "store the initializer, run it once, then return the value" pattern more cleanly than manually managing a `OnceCell` plus a separate `Option<FnOnce>`.

### Concern: Box allocation for the initializer

Both configs wrap the initializer in a `Box<dyn FnOnce() -> A + 'a>` (lines 178, 292). This means every `Lazy::new` allocates twice: once for the boxed closure, once for the `Rc`/`Arc`. The `LazyConfig::lazy_new` signature at line 95 takes `Box<Self::Thunk<'a, A>>`, mandating this boxing.

This is a necessary trade-off. `LazyCell` requires `FnOnce()` as a concrete type parameter, and since the library needs to accept arbitrary closures of different sizes, type erasure via `Box<dyn FnOnce>` is the only practical approach. The alternative (making `Lazy` generic over the closure type `F`) would make the type non-object-safe and prevent cloning (since `Clone` for `Rc<LazyCell<A, F>>` would need `F: Clone`, but `FnOnce` closures are typically not `Clone`).

## 3. Memoization Semantics

### Evaluate-once guarantee

**RcLazy.** `LazyCell::force` (called at line 254) guarantees exactly one execution of the initializer. The `LazyCell` transitions from "uninitialized with closure" to "initialized with value" atomically (in the single-threaded sense). After initialization, `force` returns the cached reference. This is correct.

**ArcLazy.** `LazyLock::force` (called at line 368) provides the same guarantee with thread safety. If multiple threads call `force` concurrently, exactly one thread runs the initializer and all others block until it completes. This is correct.

### Thread safety of ArcLazy

The `ArcLazy` implementation at lines 291-398 and 602-692 is thread-safe because:

1. `Arc` provides `Send + Sync` when `T: Send + Sync`.
2. `LazyLock<A, F>` is `Sync` when `F: Send` (the closure may be sent to the initializing thread) and the output `A` is `Sync`.
3. The `Send` bound on the closure type at line 293 (`dyn FnOnce() -> A + Send + 'a`) ensures the closure can safely be invoked on whichever thread triggers initialization first.

**The test at lines 1449-1478** verifies this: 10 threads concurrently evaluate an `ArcLazy`, and the atomic counter confirms exactly one initialization occurs.

**Subtle point.** `ArcLazy` does not require `A: Send + Sync` at construction time (line 622: `A: 'a` only). The `Send + Sync` bounds appear only where needed, such as in `ref_map` (line 682: `A: Send + Sync`) and `pure` (line 646: `A: Send`). This is a deliberate and correct design; it allows constructing an `ArcLazy<NonSendType>` in theory, though it would not be `Send` itself. The bounds propagate naturally through `Arc`'s conditional `Send`/`Sync` implementations.

## 4. Why Only RefFunctor and Not Full Functor?

The key constraint is that `Lazy::evaluate()` returns `&A`, not `A` (line 485). This means a `Functor::map` implementation with signature `fn map(f: impl Fn(A) -> B, fa: Lazy<A>) -> Lazy<B>` would need to move `A` out of the lazy cell, which is not possible since the cell owns the value and shares it via reference counting.

`RefFunctor::ref_map` (line 93 of `ref_functor.rs`) accepts `FnOnce(&A) -> B`, which works with the reference returned by `evaluate()`. The implementation at lines 554-561 of `lazy.rs` creates a new `Lazy<B>` whose initializer forces the original `Lazy<A>` and applies `f` to the resulting reference.

**Is this the right trade-off?** Yes, for several reasons:

1. **Correctness.** A `Functor` impl would require `A: Clone` to extract the value, which violates the `Functor` contract (functors should work for all types). `RefFunctor` avoids this.
2. **Consistency with Haskell/PureScript.** In PureScript, `Lazy` has only a `Lazy` class (analogous to `Deferrable`), not `Functor`. Mapping is done through explicit forcing and re-wrapping.
3. **Semantics.** `Lazy` is fundamentally a caching wrapper, not a computation context. The `ref_map` operation creates a *new* `Lazy` cell with its own cache; it does not modify the original. This is semantically cleaner than pretending it is a functor.

**Note on ArcLazy.** `ArcLazy` does not get a `RefFunctor` trait impl (the comment at lines 660-663 explains why: `RefFunctor::ref_map` does not require `Send` on the mapping function, but `ArcLazy::new` does). Instead, `ArcLazy` provides an inherent `ref_map` method (lines 677-685). This is a pragmatic compromise. An alternative would be a `SendRefFunctor` trait, mirroring the `Deferrable`/`SendDeferrable` split.

## 5. Foldable Implementation for ArcLazy

Both `LazyBrand<RcLazyConfig>` (lines 1043-1166) and `LazyBrand<ArcLazyConfig>` (lines 1168-1291) implement `Foldable`.

**Does this make sense semantically?** Yes. `Lazy` is a container of exactly one element (once forced). Foldable for a single-element container is straightforward: `fold_right(f, z, lazy) = f(lazy.evaluate().clone(), z)`. This is analogous to `Identity` being foldable.

**The `Clone` requirement.** All three methods (`fold_right`, `fold_left`, `fold_map`) call `fa.evaluate().clone()` (e.g., line 1085). This is necessary because `evaluate()` returns `&A`, but the fold functions expect owned `A` values. The `Foldable` trait already requires `A: Clone` (line 102 of `foldable.rs`), so this is consistent.

**Duplication.** The `Foldable` impls for `RcLazy` and `ArcLazy` are identical in structure. This is because the `Foldable` trait is implemented on the brand (`LazyBrand<Config>`), not on the type directly, and Rust does not allow a blanket impl over all `Config: LazyConfig` if it could conflict with other trait impls. The duplication is unfortunate but unavoidable given the HKT encoding.

## 6. Memory Management

### When is the thunk dropped after evaluation?

For `LazyCell<A, F>`, after `force()` runs `F` and stores `A`, the closure `F` is dropped. This is guaranteed by the standard library's `LazyCell` implementation. The same applies to `LazyLock`.

So the lifecycle is:
1. `Lazy::new(f)` creates `Rc<LazyCell<A, Box<dyn FnOnce() -> A>>>`. The box holds the closure.
2. First `evaluate()` calls `LazyCell::force()`, which invokes the closure, stores the result, and drops the closure (the `Box<dyn FnOnce>` is consumed).
3. Subsequent `evaluate()` calls return a reference to the cached value.
4. When all `Rc`/`Arc` references are dropped, the `LazyCell`/`LazyLock` and its cached value are deallocated.

**No leaks.** The closure is dropped after evaluation, so any captured values are freed. The cached result lives as long as at least one `Rc`/`Arc` clone exists. This is correct reference-counted behavior.

### Potential concern: `ref_map` chains

When `ref_map` is called (line 554-561), it creates a new `Lazy<B>` that captures a clone of the original `Lazy<A>`. This means the original's `Rc`/`Arc` refcount is incremented, keeping the original's cached value alive as long as the mapped `Lazy<B>` exists. In a long chain of `ref_map` calls, all intermediate `Lazy` values remain alive. This is inherent to the lazy memoization model; each cell independently caches its result and holds a reference to its dependency.

This is not a leak in the traditional sense; it is the expected behavior for a graph of memoized computations. However, users building deep `ref_map` chains should be aware that all intermediate results remain in memory.

### Concern: `pure` allocates a closure unnecessarily

`RcLazy::pure(a)` at line 533 wraps the value in `Box::new(move || a)`, allocating a closure that just returns the value. An alternative would be to directly create a pre-forced `LazyCell`, but `LazyCell` does not expose a "from value" constructor in the standard library; it always requires a closure. This is a minor inefficiency but is dictated by the standard library API.

## 7. Deferrable: Does `defer` Compose Correctly with Memoization?

### RcLazy's Deferrable (lines 812-844)

```rust
fn defer(f: impl FnOnce() -> Self + 'a) -> Self {
    RcLazy::new(move || f().evaluate().clone())
}
```

This creates a single `RcLazy` whose initializer:
1. Calls `f()` to produce an inner `RcLazy<A>`.
2. Forces the inner `RcLazy` with `evaluate()`.
3. Clones the result.

**Correctness.** This correctly flattens `Lazy<Lazy<A>>` into `Lazy<A>`. The transparency law (`defer(|| x)` is observationally equivalent to `x`) holds because `defer(|| lazy).evaluate()` produces the same value as `lazy.evaluate()`, albeit with a fresh cache cell.

**The `Clone` bound.** `Deferrable` for `RcLazy` requires `A: Clone` (line 814), which is necessary because `evaluate()` returns `&A` and we need to move the value into the outer cell. This is a semantic restriction; not all types can participate in `defer` for `Lazy`.

### ArcLazy's SendDeferrable (lines 847-878)

```rust
fn send_defer(f: impl FnOnce() -> Self + Send + Sync + 'a) -> Self {
    ArcLazy::new(move || f().evaluate().clone())
}
```

The same pattern, with additional `Send + Sync` bounds. This is correct.

### Composition with memoization

When `defer` is used, the inner `Lazy` is evaluated and its result is cloned into a new cell. The inner cell's memoization is still effective: if the inner `Lazy` was already evaluated, the cached value is returned. If not, it is computed once. The outer cell then caches its own copy of the cloned value. This means two separate cache entries exist, which is slightly wasteful but semantically correct.

## 8. TryLazy: Error Handling and Error Caching

### Structure

`TryLazy<'a, A, E, Config>` (line 44 of `try_lazy.rs`) wraps `Config::TryLazy<'a, A, E>`, which for `RcLazyConfig` is `Rc<LazyCell<Result<A, E>, Box<dyn FnOnce() -> Result<A, E> + 'a>>>`.

### Error caching

The `evaluate` method (line 106) calls `Config::try_evaluate`, which for `RcLazyConfig` (lines 279-283) does:
```rust
fn try_evaluate(...) -> Result<&'b A, &'b E> {
    LazyCell::force(lazy).as_ref()
}
```

This forces the cell, which stores a `Result<A, E>`. Then `.as_ref()` converts `&Result<A, E>` to `Result<&A, &E>`.

**Can errors be cached incorrectly?** No. The `Result<A, E>` is stored exactly as the initializer returned it. If the initializer returns `Err(e)`, that error is cached permanently. Subsequent calls return `Err(&e)`. This is the intended semantics: the computation ran, it failed, and the failure is permanent.

**Is this always desirable?** This depends on the use case. For transient errors (network timeouts, lock contention), caching the error might be undesirable. However, `TryLazy` is designed for "run at most once" semantics, not for retry semantics. Users who need retry should use a different abstraction. The documentation at lines 29-32 is clear about this: "If it succeeds, the value is cached. If it fails, the error is cached."

### Panic handling

`TryLazy` provides `catch_unwind` (lines 402-426 of `try_lazy.rs`) for `TryLazy<'a, A, String, RcLazyConfig>`. This wraps the closure in `std::panic::catch_unwind` and converts the panic payload to a `String` error. This is a useful escape hatch, especially since `Lazy` itself will poison the `LazyCell`/`LazyLock` on panic (as documented at lines 405-412 of `lazy.rs`).

**The `String` specialization.** `catch_unwind` is only available when `E = String`, which is limiting. A more general approach would parameterize over an `E: From<String>` or accept a panic-to-error conversion function. However, the current approach is simple and covers the most common case.

### map/map_err cloning behavior

`TryLazy::map` (lines 200-207) and `map_err` (lines 231-238) both create new `TryLazy` cells. In `map`:
```rust
RcTryLazy::new(move || self.evaluate().map(|a| f(a)).map_err(|e| e.clone()))
```

The error must be cloned because the new cell needs its own copy. Similarly, in `map_err`, the success value must be cloned. These `Clone` bounds (`E: Clone` for `map`, `A: Clone` for `map_err`) are explicit in the method signatures, which is correct.

## 9. Pointer Parameterization: Rc/Arc Split

The Rc/Arc split is handled through the `LazyConfig` trait rather than through the library's `Pointer`/`RefCountedPointer` hierarchy. This is a deliberate divergence from how `FnBrand<P>` handles pointer parameterization.

**Why the divergence?** The `Pointer`/`RefCountedPointer` traits abstract over `Rc`/`Arc` as containers (`Rc<T>` vs `Arc<T>`), but `Lazy` needs to abstract over *lazy cell types* (`LazyCell` vs `LazyLock`) in addition to the pointer. These are fundamentally different types with different APIs (one is `!Sync`, the other is `Sync`). A single `RefCountedPointer` parameter would not capture this distinction.

**Is this well-handled?** Yes. The `LazyConfig` trait cleanly encapsulates both the pointer choice and the lazy cell choice as a single strategy parameter. The `RcLazyConfig` and `ArcLazyConfig` implementations are self-contained and correct.

**Missed opportunity.** The `LazyConfig` trait could in principle be connected to the `RefCountedPointer` hierarchy via an associated type (e.g., `type Ptr: RefCountedPointer`), which would allow generic code to extract the pointer brand from a `LazyConfig`. Currently there is no such connection, meaning code that is generic over `LazyConfig` cannot also be generic over the pointer type. This is a minor gap but could matter for future abstractions that compose lazy evaluation with pointer-parameterized function types.

## 10. Comparison with `once_cell::Lazy` and `std::sync::LazyLock`

### `std::cell::LazyCell` / `std::sync::LazyLock` (Rust 1.80+)

These are the underlying primitives that `Lazy` wraps. On their own, they are:
- **Not shared.** `LazyCell` and `LazyLock` are single-owner types. You cannot clone them.
- **Not lifetime-polymorphic.** The closure type parameter must be known at construction time.

### What `Lazy` adds

1. **Shared memoization via reference counting.** `Lazy` wraps `LazyCell`/`LazyLock` in `Rc`/`Arc`, enabling clone-based sharing. All clones see the same cached value. This is the primary value-add.

2. **Lifetime polymorphism.** Through type erasure (`Box<dyn FnOnce() -> A + 'a>`), `Lazy` accepts closures with arbitrary lifetimes, enabling it to capture borrowed data. Raw `LazyCell`/`LazyLock` require knowing the concrete closure type, which makes lifetime-polymorphic usage awkward.

3. **HKT integration.** `Lazy` participates in the library's type class hierarchy via `LazyBrand<Config>`, enabling it to be used with `RefFunctor`, `Foldable`, `Deferrable`, `Semigroup`, and `Monoid`. This is the main reason for the type's existence within the library.

4. **Unified API.** `RcLazy` and `ArcLazy` share the same `evaluate()` method and most of the same API surface. Standard library `LazyCell` and `LazyLock` have different method names (`LazyCell::force` vs `LazyLock::force`, though they happen to be the same) and different trait implementations.

5. **Fixed-point combinators.** `rc_lazy_fix` (lines 735-750) and `arc_lazy_fix` (lines 788-803) enable tying the recursive knot, analogous to PureScript's `fix :: Lazy l => (l -> l) -> l`. This is not available with raw `LazyCell`/`LazyLock`.

6. **Fallible variant.** `TryLazy` extends the model to `Result`-producing computations, with error caching. There is no standard library equivalent; `LazyCell`/`LazyLock` only support infallible initialization (panics poison the cell).

### What `once_cell::Lazy` provided (now mostly in std)

The `once_cell` crate's `Lazy` and `sync::Lazy` were the precursors to `std::cell::LazyCell` and `std::sync::LazyLock`. They offered the same initialize-once semantics. The library's `Lazy` type builds on the now-standard versions and adds the shared-ownership, HKT, and fallible layers described above.

## Summary of Findings

| Aspect | Assessment |
|--------|-----------|
| Rc/Arc split via `LazyConfig` | Well-designed, justified, extensible. |
| Interior mutability | Correct; delegates to well-audited `std` types. |
| Evaluate-once guarantee | Correct for both single-threaded and concurrent use. |
| Thread safety of ArcLazy | Sound; `Send` bounds are correctly placed. |
| RefFunctor instead of Functor | Correct trade-off given reference-returning `evaluate()`. |
| Foldable for Lazy | Semantically sound (single-element container). |
| Memory management | No leaks; closures dropped after first evaluation. `ref_map` chains retain intermediates as expected. |
| Deferrable composition | Correct; requires `A: Clone` which is documented. |
| TryLazy error caching | Correct and intentional; errors are permanent. |
| Pointer parameterization | Clean but disconnected from the `RefCountedPointer` hierarchy. |
| Value over raw `LazyCell`/`LazyLock` | Substantial: shared ownership, lifetime polymorphism, HKT, `fix`, `TryLazy`. |

### Potential Improvements

1. **Reduce ArcLazy code duplication.** Consider a `SendRefFunctor` trait to give `ArcLazy` trait-level `ref_map`, mirroring the `Deferrable`/`SendDeferrable` pattern already used.
2. **Generalize `catch_unwind`.** Accept a conversion function `Box<dyn Any> -> E` instead of hardcoding `E = String`.
3. **Connect `LazyConfig` to `RefCountedPointer`.** Add an associated type `type Ptr: RefCountedPointer` to enable generic code that composes lazy evaluation with pointer-parameterized abstractions.
4. **Consider `Eq` and `Ord`.** `PartialEq` and `PartialOrd` are implemented (lines 1301-1357), but `Eq` and `Ord` are not. If the inner type supports them, `Lazy` could too.
5. **Document the double-allocation cost.** The `Box<dyn FnOnce>` + `Rc`/`Arc` double allocation is a necessary trade-off, but it is not documented for performance-sensitive users.
