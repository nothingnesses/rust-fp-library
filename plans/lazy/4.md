# Deep Analysis: Trampoline

## 1. Internal Representation

`Trampoline<A>` is a thin newtype wrapper around `Free<ThunkBrand, A>`:

```rust
pub struct Trampoline<A: 'static>(Free<ThunkBrand, A>);
```

All the interesting structure lives in `Free<F, A>`, which has three variants via `FreeInner`:

```rust
pub enum FreeInner<F, A> {
    Pure(A),
    Wrap(Apply!(<F as Kind!(...)>::Of<'static, Free<F, A>>)),
    Bind {
        head: Box<Free<F, TypeErasedValue>>,
        continuations: CatList<Continuation<F>>,
        _marker: PhantomData<A>,
    },
}
```

For `F = ThunkBrand`, the `Wrap` variant holds a `Thunk<'static, Free<ThunkBrand, A>>`, i.e., a boxed closure that, when called, produces the next `Free` step. This is the classic trampoline encoding: instead of recursing, you return a thunk that the driver loop calls iteratively.

The `Bind` variant stores a type-erased head computation plus a `CatList` of type-erased continuations (`Box<dyn FnOnce(TypeErasedValue) -> Free<F, TypeErasedValue>>`). This is the "Reflection without Remorse" optimization.

**Key observation**: `TypeErasedValue = Box<dyn Any>` is the mechanism that forces `'static`. The `Any` trait requires `'static` to guarantee memory safety (no dangling references after type erasure). Every continuation in the CatList operates on `Box<dyn Any>` and downcasts at each step.

### Comparison to a Naive Free Monad

A naive `Free` would be:

```rust
enum NaiveFree<F, A> {
    Pure(A),
    Wrap(F<NaiveFree<F, A>>),
}
```

With `bind` defined as:

```rust
fn bind(self, f) -> NaiveFree<F, B> {
    match self {
        Pure(a) => f(a),
        Wrap(fa) => Wrap(map(|inner| inner.bind(f), fa)),
    }
}
```

This naive definition would allow lifetimes (no `dyn Any` needed), but left-associated binds would be O(n^2) because each `evaluate` call traverses the entire chain of `Wrap(map(...))` layers. The Reflection without Remorse approach transforms this to O(n) total by accumulating continuations in a CatList.


## 2. Stack Safety Guarantees

### The Evaluation Loop

The evaluation loop in `Free::evaluate` (line 419-467 of `free.rs`) is a `loop {}` with no recursion:

```rust
pub fn evaluate(self) -> A where F: Evaluable {
    let mut current: Free<F, TypeErasedValue> = self.erase_type();
    let mut continuations: CatList<Continuation<F>> = CatList::empty();

    loop {
        let inner = current.0.take().expect("...");
        match inner {
            FreeInner::Pure(val) => {
                match continuations.uncons() {
                    Some((continuation, rest)) => {
                        current = continuation(val);
                        continuations = rest;
                    }
                    None => {
                        return *val.downcast::<A>().expect("...");
                    }
                }
            }
            FreeInner::Wrap(fa) => {
                current = <F as Evaluable>::evaluate(fa);
            }
            FreeInner::Bind { head, continuations: inner_conts, .. } => {
                current = *head;
                continuations = inner_conts.append(continuations);
            }
        }
    }
}
```

**Is this truly stack-safe?** Yes, with one nuance:

1. The `loop` itself never recurses; it uses mutable state (`current`, `continuations`) and iterates.
2. The `Bind` case merges inner continuations with outer ones via `CatList::append` (O(1)) and replaces `current` with the head. This is the key reassociation step.
3. The `Wrap` case calls `Evaluable::evaluate(fa)`, which for `ThunkBrand` calls the wrapped `FnOnce` closure. This closure returns a `Free` value, which becomes the new `current`.
4. The `Pure` case applies the next continuation, which may itself return a `Bind` or `Wrap`, keeping the loop going.

**The nuance**: The `Wrap` step calls `Thunk::evaluate`, which calls the closure. If a user writes `Trampoline::defer(|| ...)`, the closure inside the `Thunk` produces a `Free` value, which is returned to the loop. This is fine; the closure call is a single stack frame that returns immediately. But if a user writes `Trampoline::new(|| expensive_recursive_function())`, the closure itself might recurse on the Rust call stack. The trampoline only guarantees stack safety for computations expressed through its combinators (`bind`, `defer`, `tail_rec_m`), not for arbitrary closures.

**Drop safety**: The `Drop` implementation for `Free` (line 490-512) iteratively walks `Bind` chains to avoid stack overflow during destruction. This is critical because a deep `Bind` chain would otherwise cause recursive `Drop` calls. The test `test_free_drop_safety` verifies this works for 100,000-deep chains.

### Stack safety test coverage

The integration test file `fp-library/tests/stack_safety.rs` tests three scenarios:
- Deep `tail_rec_m` (1,000,000 iterations).
- Deep `bind` chain (100,000 iterations, built in a loop).
- Deep `defer` chain (100,000 iterations, recursive function).

All pass without stack overflow.


## 3. The `'static` Lifetime Requirement

### Why `'static` is Required

The `'static` bound comes from `Box<dyn Any>` used in the "Reflection without Remorse" technique. Specifically:

1. `Continuation<F> = Box<dyn FnOnce(TypeErasedValue) -> Free<F, TypeErasedValue>>`.
2. `TypeErasedValue = Box<dyn Any>`.
3. `Any: 'static` in Rust's standard library.

The CatList stores heterogeneously-typed continuations (each `bind` may have a different intermediate type). To store them in a single list, types must be erased to `dyn Any`, which requires `'static`.

### Is It Too Restrictive?

**Yes, in specific scenarios.** The `'static` requirement means:
- Trampoline cannot capture borrows from the enclosing scope.
- All values flowing through the computation must be owned.
- Types like `&str` cannot be used; `String` must be used instead.

This prevents patterns like:

```rust
fn process(data: &[u8]) -> Trampoline<usize> {
    // ERROR: &[u8] is not 'static
    Trampoline::new(move || data.len())
}
```

**Mitigation in this library**: `Thunk<'a, A>` exists as a complement that supports borrowing and HKT traits, at the cost of stack safety. The library documentation makes this trade-off explicit with a comparison table in the `Thunk` docs.

### The `Send` Bound

Notably, `Trampoline<A>` requires `A: 'static + Send` on all its methods, even though `Free` itself does not require `Send`. This is an additional restriction layered on top of Free's `'static` requirement.

The `Send` bound is likely added for forward compatibility with concurrent use cases (e.g., `TryTrampoline` being used across thread boundaries), but it excludes types containing `Rc`, `Cell`, or other non-`Send` types. For a single-threaded trampoline, `Send` is unnecessarily restrictive. A user who wants to trampoline over `Rc<RefCell<...>>` data would be blocked.

**Suggestion**: Consider splitting into `Trampoline<A: 'static + Send>` (current) and a `LocalTrampoline<A: 'static>` without the `Send` bound, or removing `Send` from `Trampoline` and adding a `SendTrampoline` alias/wrapper. Since `Free` itself does not impose `Send`, the underlying machinery already supports this.


## 4. Trampoline as Free Monad over Identity/Thunk

### The Classical Connection

In the functional programming literature, `Trampoline` is typically defined as `Free Identity`, i.e., the free monad over the identity functor. The idea is:

- `Pure(a)` = the computation is done with value `a`.
- `Suspend(Identity(Free Identity A))` = one more step of computation, wrapping the continuation in `Identity` (which is a no-op).

This library instead defines `Trampoline` as `Free<ThunkBrand, A>`, which is the free monad over the `Thunk` functor. The `Wrap` variant holds a `Thunk<'static, Free<ThunkBrand, A>>`, meaning each suspension is a deferred computation (a closure).

**Is this connection made explicit?** Partially. The module documentation for `trampoline.rs` states:

> Built on the `Free` monad with O(1) `bind` operations.

And the `Free` module documentation references PureScript's `Control.Monad.Free`. However, the classical "Trampoline = Free Identity" connection is not explicitly discussed. The choice of `Thunk` over `Identity` as the base functor is an intentional departure, because `Thunk` adds the ability to defer computation (laziness), whereas `Identity` would provide no deferral mechanism.

**Why Thunk, not Identity?** In a strict language like Rust (and PureScript, which is also strict), using `Identity` as the base functor would mean `Wrap(Identity(free))` forces the inner `free` eagerly. This defeats the purpose of trampolining, because the recursive call would still happen on the stack. Using `Thunk` (i.e., `() -> Free<...>`) defers the recursive step until the driver loop asks for it. This is why PureScript also uses a function-based suspension rather than raw `Identity`.

**Suggestion**: The module-level documentation for `trampoline.rs` and `free.rs` would benefit from a brief note explaining this relationship. Something like: "Trampoline is the free monad over Thunk, analogous to the classical 'Trampoline = Free Identity' in lazy languages; in strict languages, the thunk layer is necessary to defer recursive steps."


## 5. bind/flat_map: Left-Associated Bind Handling

### The O(1) bind Implementation

`Trampoline::bind` delegates to `Free::bind`:

```rust
pub fn bind<B: 'static + Send>(
    self,
    f: impl FnOnce(A) -> Trampoline<B> + 'static,
) -> Trampoline<B> {
    Trampoline(self.0.bind(move |a| f(a).0))
}
```

`Free::bind` (line 290-340 of `free.rs`) handles three cases:

1. **`Pure(a)`**: Creates a new `Bind` node with the pure value (type-erased) as head and the new continuation as a singleton `CatList`.
2. **`Wrap(fa)`**: Wraps the `Wrap` into a `Bind` node similarly.
3. **`Bind { head, continuations }`**: Appends (`snoc`) the new continuation to the existing CatList. This is the crucial O(1) case.

For left-associated binds like:

```rust
pure(x).bind(f1).bind(f2).bind(f3)...bind(fn)
```

Each `.bind()` call hits case 1 (first call) then case 3 (all subsequent calls), each doing O(1) `CatList::snoc`. Total construction time is O(n).

During evaluation, the loop processes the CatList by calling `uncons()`, which is O(1) amortized (the CatList uses a `VecDeque` of sublists internally). Total evaluation time is O(n).

**Right-associated binds** like `pure(x).bind(|a| f1(a).bind(|b| f2(b).bind(|c| ...)))` create a deeper `Free` structure but the evaluation loop handles them identically; the `Bind` case merges inner and outer continuation lists.

### Comparison to Thunk::bind

`Thunk::bind` (line 184-193 of `thunk.rs`) simply nests closures:

```rust
pub fn bind<B: 'a>(self, f: impl FnOnce(A) -> Thunk<'a, B> + 'a) -> Thunk<'a, B> {
    Thunk::new(move || {
        let a = (self.0)();
        let thunk_b = f(a);
        (thunk_b.0)()
    })
}
```

For n binds, evaluation creates n nested function calls on the stack. This is O(n) time but uses O(n) stack space, so it will overflow for deep chains.


## 6. Memory Behavior

### Allocation Pattern

Each operation on `Trampoline` allocates:

- **`Trampoline::new(f)`**: One `Box<dyn FnOnce>` (for the Thunk), plus the `Free` structure (Pure + Wrap).
- **`Trampoline::bind(f)`**: One `Box<dyn FnOnce(TypeErasedValue) -> Free<...>>` for the type-erased continuation, plus potentially a `CatList::Cons` node.
- **`Trampoline::pure(a)`**: One `Free` (Pure variant), minimal allocation.
- **`Trampoline::defer(f)`**: One `Box<dyn FnOnce>` (for the Thunk), same as `new`.

### Closure Accumulation

During construction, a chain of n `bind` calls accumulates n closures in the CatList. These closures are all stored as `Box<dyn FnOnce(TypeErasedValue) -> Free<F, TypeErasedValue>>`, so each one is a heap allocation (the closure itself) plus its captured environment.

During evaluation, each continuation is consumed exactly once (it's `FnOnce`) and dropped immediately after use. So the total live memory during evaluation is bounded by the number of pending continuations, not the total number ever created.

For `tail_rec_m`, each iteration creates one `defer` (one Thunk allocation) and one `bind` (one continuation allocation). However, the previous step's allocations are freed before or during the next step's processing, so memory usage stays roughly constant per step.

### Box<dyn Any> Overhead

Every intermediate value flowing through the continuation pipeline is boxed into `Box<dyn Any>`. This means:
- An `i32` passing through `bind` gets heap-allocated as `Box<i32>` (8 bytes for the i32, plus allocation overhead).
- Each `downcast::<T>()` call checks a `TypeId` at runtime, adding a small per-step overhead.
- For large values, the boxing/unboxing overhead is negligible relative to the value size. For small values (primitives), it is proportionally significant.

### Comparison to Direct Recursion

A directly recursive function uses only stack frames. `Trampoline` replaces stack frames with heap-allocated closures and `Box<dyn Any>` values. This trades stack overflow risk for heap allocation overhead. For deep recursions (>1000 levels), this is a worthwhile trade.


## 7. Comparison with Other Ecosystems

### Cats/Scalaz (Scala)

Scala's `cats.free.Free` and `scalaz.Free` follow a similar structure:
- `Pure[A]`, `Suspend[F[A]]`, `FlatMap[S, A]` (the bind node).
- Left-associated bind optimization using a "stack" of continuations.
- `foldMap` for natural transformation-based interpretation.
- `Trampoline` is `Free[Function0, A]` in Scalaz, exactly analogous to `Free<ThunkBrand, A>`.

**Differences from this library**:
- Scala's `Free` supports `resume` (inspect the next step) and `foldMap` (generic interpretation). This library's `Free` does not, because interpretation is baked into the `Evaluable` trait.
- Scala can use `unsafeCoerce`-like tricks (via JVM type erasure) more easily; Rust's `Box<dyn Any>` + `downcast` achieves the same effect with runtime type checks.

### PureScript

PureScript's `Control.Monad.Free` from `purescript-free`:
- Uses the same "Reflection without Remorse" technique with `CatList`.
- `resume :: Free f a -> Either (f (Free f a)) a` allows stepping through the computation.
- `runFree :: (f (Free f a) -> Free f a) -> Free f a -> a` provides the trampoline loop.
- `foldFree :: MonadRec m => (f ~> m) -> Free f a -> m a` allows interpreting into any `MonadRec`.
- PureScript's `Trampoline` is defined as `Free (Lazy Unit)` or `Free Function0`.

**What this library lacks compared to PureScript**:
1. `resume`: Structural inspection of the computation. Not possible with the current type-erased `CatList` approach. Would require a different encoding.
2. `hoistFree`: Natural transformation of the base functor. Could be added but would need to map over the CatList's closures.
3. `foldFree` / generic interpretation: The `Evaluable` trait bakes in the interpretation strategy. To support multiple interpretations, a `run_free` function taking a natural transformation `F ~> G` would be needed.

### Haskell

Haskell's `Control.Monad.Free` benefits from laziness, so the classic definition works:
```haskell
data Free f a = Pure a | Free (f (Free f a))
```

No special `CatList` optimization is needed for stack safety (Haskell's lazy evaluation handles it), though `Control.Monad.Free.Church` uses Church encoding for performance. The "Reflection without Remorse" paper applies to Haskell too, but it's less critical there because laziness prevents the O(n^2) issue from manifesting in the same way.


## 8. Why Trampoline Lacks HKT Traits (and Should It Have Them?)

### Current Status

The library documentation explicitly states that `Trampoline` does not implement HKT traits. From `brands.rs` (line 192-193):

> Note: This is for `Thunk<'a, A>`, NOT for `Trampoline<A>`.
> `Trampoline` cannot implement HKT traits due to its `'static` requirement.

There is no `TrampolineBrand` in `brands.rs`.

### Why Not?

The HKT machinery in this library uses `Kind` traits with the signature:

```rust
type Of<'a, A: 'a>: 'a;
```

This requires the type constructor to accept any lifetime `'a`. Since `Trampoline<A>` requires `A: 'static`, it cannot satisfy `Of<'a, A: 'a>` for arbitrary `'a`; it can only work when `'a = 'static`. Implementing the trait would be unsound or require a restricted kind that only works for `'static`.

### Should It Have Them?

**Arguments for**:
- Users could use `map`, `bind`, `pure` through the HKT free functions, enabling generic programming over monads.
- `Trampoline` satisfies the monad laws; it just has a restricted domain.
- A `TrampolineBrand` could be added with `type Of<'a, A: 'a>: 'a = Trampoline<A>` if `A: 'static` is assumed, but this would be vacuous for non-static lifetimes.

**Arguments against (and why the current design is correct)**:
- The `Kind` trait's signature promises that `Of<'a, A>` works for any `'a`. A `TrampolineBrand` that only works at `'static` would violate this contract at the type level.
- Generic code that works with `Functor` brands would be written to accept any `'a`, but `TrampolineBrand` would panic or fail to compile when instantiated at non-static lifetimes.
- `Thunk` already fills the HKT-compatible niche for deferred computation.

**Possible middle ground**: A `Kind` variant with the signature `type Of<A: 'static>` (no lifetime parameter) could support types like `Trampoline`. However, this would require a separate trait hierarchy (`StaticFunctor`, `StaticMonad`, etc.) or a unified approach that abstracts over the lifetime parameter. This is a significant design decision that would affect the entire library.


## 9. Performance Characteristics

### Overhead Breakdown (per step of evaluation)

| Operation | Cost |
|-----------|------|
| `CatList::uncons` | O(1) amortized |
| Continuation call | One virtual dispatch + `Box<dyn Any>` downcast |
| `Box<dyn Any>` downcast | TypeId comparison (cheap but not free) |
| `Thunk` evaluation (Wrap case) | One closure call |
| `CatList::append` (Bind case) | O(1) |

### Benchmarked Expectations

For a chain of n operations:
- **Construction time**: O(n), one heap allocation per `bind`/`defer`.
- **Evaluation time**: O(n), one iteration per step.
- **Memory**: O(n) during construction (all closures live), O(1) steady-state during evaluation (closures are consumed and freed).
- **Overhead vs direct iteration**: Approximately 2-5x slower due to boxing, type erasure, virtual dispatch, and allocation. The library's benchmarks (in `benches/`) should confirm this.

### When Trampoline Is Not Worth It

- For shallow recursion (< ~100 levels), the overhead exceeds the benefit. Use direct recursion or `Thunk`.
- For tight loops over primitives, the `Box<dyn Any>` overhead per iteration is significant. Consider using `tail_rec_m` with `Thunk` (which uses a direct loop) instead.
- For pure accumulation patterns (e.g., summing a list), a `while` loop or iterator is strictly better.

### When Trampoline Is Essential

- Recursive algorithms with unbounded depth (tree traversals, parser combinators, graph algorithms).
- Monadic pipelines with thousands of `bind` steps.
- Any situation where a stack overflow is a production risk.


## 10. Composability with Thunk and Lazy

### Current Conversion Paths

The library provides these conversions:

```
Lazy<'static, A> --From--> Trampoline<A>      (evaluates Lazy, wraps in Trampoline::new)
Lazy<'a, A>      --From--> Thunk<'a, A>        (evaluates Lazy, wraps in Thunk::new)
Thunk<'a, A>     (no direct conversion to/from Trampoline)
Trampoline<A>    (no direct conversion to Thunk)
```

### Missing Conversions

**Thunk -> Trampoline**: Not provided. Would be straightforward:

```rust
impl<A: 'static + Send> From<Thunk<'static, A>> for Trampoline<A> {
    fn from(thunk: Thunk<'static, A>) -> Self {
        Trampoline::new(move || thunk.evaluate())
    }
}
```

This is limited to `'static` thunks but would enable a natural migration path from Thunk to Trampoline when stack safety becomes a concern.

**Trampoline -> Thunk**: Also possible:

```rust
impl<A: 'static> From<Trampoline<A>> for Thunk<'static, A> {
    fn from(trampoline: Trampoline<A>) -> Self {
        Thunk::new(move || trampoline.evaluate())
    }
}
```

This would allow using Trampoline computations in HKT-generic code that expects a `Thunk`.

### Composing Trampoline with Lazy

The current `From<Lazy<'static, A>> for Trampoline<A>` implementation:

```rust
fn from(lazy: Lazy<'static, A, Config>) -> Self {
    Trampoline::new(move || lazy.evaluate().clone())
}
```

Note the `.clone()` call. `Lazy::evaluate` returns a `&A` (shared reference to the cached value), so conversion to `Trampoline` requires cloning. This is a necessary consequence of Lazy's memoization-by-sharing design.

### The Deferrable Trait

All three types implement `Deferrable`:
- `Thunk<'a, A>: Deferrable<'a>`.
- `Trampoline<A>: Deferrable<'static>`.
- `Free<ThunkBrand, A>: Deferrable<'static>`.

This provides a unified interface for deferred construction, but does not help with interconversion. The `Deferrable` trait is more about constructing values than composing them.


## Summary of Suggestions

1. **Document the Free-over-Thunk connection**: Add a note in both `trampoline.rs` and `free.rs` explaining that `Trampoline = Free<ThunkBrand, A>` is analogous to the classical `Trampoline = Free Identity`, and why `Thunk` is used instead of `Identity` in a strict language.

2. **Consider removing the `Send` bound from `Trampoline`**: The `Send` bound on all `Trampoline` methods is more restrictive than `Free` itself requires. This could be relaxed to broaden usability (e.g., trampolining over `Rc`-based structures), with a `Send`-bounded variant available separately.

3. **Add `From<Thunk<'static, A>> for Trampoline<A>`**: A natural conversion that lets users migrate from Thunk to Trampoline.

4. **Add `From<Trampoline<A>> for Thunk<'static, A>`**: Enables using Trampoline results in HKT-generic code.

5. **Consider `resume` for Free**: If there is ever a need for DSL/interpreter patterns (decoupling definition from interpretation), adding `resume :: Free f a -> Step (f (Free f a)) a` would bring the library closer to PureScript/Haskell capabilities. This would require exposing the `FreeInner` structure or providing a safe accessor.

6. **Consider a `Kind` variant for `'static`-only types**: A `StaticKind` or similar mechanism would allow `Trampoline`, `Free`, and other `'static`-bound types to participate in the HKT hierarchy without violating the lifetime-polymorphic contract of the existing `Kind` traits.

7. **Performance documentation**: The existing benchmark infrastructure should include Trampoline-specific benchmarks comparing: (a) Trampoline vs direct recursion at various depths, (b) Trampoline vs Thunk for bind chains at the stack limit boundary, (c) overhead per step for `tail_rec_m`.
