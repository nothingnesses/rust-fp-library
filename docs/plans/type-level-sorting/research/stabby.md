### What this codebase does

Stabby provides ABI stability for FFI by declaring a type-level representation of struct/enum layouts and computing stable type identities. The `IStable` trait (`istable.rs:30`) requires implementations to supply an associated constant `const ID: u64`, generated by the `gen_id` function (`report.rs:172`). This function hashes a `TypeReport` (the type's declared shape) and architecture metadata (pointer size, endianness) via SHA256, reducing to a u64. Types are marked ABI-stable when their `IStable` impl is unsafe and correct; the macro-generated implementation runs at compile-time to embed these IDs in binaries. Stabby's type-level machinery (`typenum2/unsigned.rs`) uses a hand-rolled type-level arithmetic system (not extern typenum) to represent sizes and alignments as types, enabling compile-time memory layout proofs.

### Type-level sorting capability

Stabby's hash lives as a **runtime constant u64** (`const ID: u64`), not as a type-level quantity. The hash is computed at compile-time via `const fn gen_id(report: &TypeReport) -> u64` (`report.rs:172-176`) using a const SHA256 implementation (`sha2_const_stable` crate), but the result is opaque to the type system. Two types with different IDs yield distinct `u64` values; however, there is **no type-level comparison machinery** to order these at compile-time. The hash cannot drive trait dispatch because it is not a type parameter or const generic. Stabby's type-level arithmetic (size/alignment) is separate from hashing and does not expose type identity as an orderable type-level value.

### Approach used (or enabled)

Stabby implements a **hybrid of approaches 3 and 1**: approach 3 (hash-based identity) in the runtime sense, plus Peano-style type-level arithmetic (`typenum2::unsigned::UInt` chains, `build.rs:38-66`) for size/alignment. Approach 3 hashing alone cannot sort coproducts because the hash is a u64 constant, not a type that participates in trait resolution. Stabby does not use const generics (`const PARAM: u64`) to lift the hash into type-level ordering. The type-level system is entirely for layout proofs, not identity-driven dispatch.

### Stable or nightly

Stabby targets **stable Rust** for its core (`stabby-abi/Cargo.toml:15-63`). The crate uses `const fn` throughout (`const fn gen_id`, `const fn hash_report`, `const fn fields`) which is stable. The `sha2_const_stable` dependency provides const SHA256, also stable. The `stabby-macros` proc-macro can generate const ID assignments in trait impls. No feature gates or nightly CFG dependencies are required for hashing itself; compilation is stable-compatible with MSRV likely Rust 1.61+ (const trait-associated consts).

### Ergonomics and compile-time profile

Users opt types into hashing via the `#[stabby]` macro or manual `unsafe impl IStable`. The macro generates `const REPORT: &'static TypeReport` (struct/enum metadata) and `const ID: u64 = crate::report::gen_id(Self::REPORT)` (`stable_impls/mod.rs` examples). Compile-time cost is dominated by SHA256 hashing of the type report string (module + name + field list); the hash is computed once per type at compile-time and embedded in the binary as a constant. No specialization or proc-macro expansion complexity arises. The hash computation is deterministic: identical type metadata always yields the same ID.

### Applicability to coproduct row canonicalisation

**Verdict: not applicable.** Stabby's hash cannot directly sort a coproduct row at the type level. To use it, a user would need to (1) declare each effect type as a `#[stabby]` struct to generate an ID, (2) extract those runtime u64 IDs, and (3) sort the row at runtime, recomputing the coproduct structure. There is no trait or const-generic mechanism to ask "which type has the smaller hash?" and drive type-level dispatch accordingly. Attempting approach 3 (hash-based sorting) would require lifting each ID into a const generic parameter (e.g., `Coproduct<A, B, {A::ID}, {B::ID}>`) and implementing trait dispatch on the ordering of those const u64s, which is not possible on stable Rust (const generic ordering in trait bounds requires nightly const trait comparison). Integrating stabby would add ~10 lines per effect (one `#[stabby]` impl) but would not reduce the manual effort of specifying correct type-level order; users would still need a parallel const-generic mechanism (e.g., approach 8 const generics with custom traits) to sort at compile-time.
