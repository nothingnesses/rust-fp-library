# Analysis 7: Lifetime Design and Ergonomics

## 1. Why Thunk and Lazy Use `'a` but Trampoline Requires `'static`

The lifetime split across the hierarchy is not arbitrary; it follows directly from the internal representation of each type.

**Thunk** stores a `Box<dyn FnOnce() -> A + 'a>` (thunk.rs, line 89). The `'a` parameter on `Thunk<'a, A>` is the lifetime of the boxed closure. Because the closure is type-erased behind `dyn FnOnce`, the compiler needs an explicit lifetime to know how long the closure (and any references it captures) will remain valid. This is the same `'a` that appears in the Kind trait: `type Of<'a, A: 'a>: 'a = Thunk<'a, A>` (thunk.rs, line 319).

**Lazy** stores `Config::Lazy<'a, A>`, which resolves to `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>` for `RcLazyConfig` (lazy.rs, line 178) or `Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>` for `ArcLazyConfig` (lazy.rs, line 292). The `'a` again bounds the initializer closure.

**Trampoline** wraps `Free<ThunkBrand, A>` (trampoline.rs, line 74). The `Free` monad uses `Box<dyn Any>` for type erasure of intermediate values in its continuation stack (free.rs, line 93). Since `Any` in Rust requires `'static` (it is defined as `dyn Any + 'static`), every value flowing through the `Free` monad must be `'static`. This is why `Trampoline<A>` requires `A: 'static` (trampoline.rs, line 72) and every closure passed to `bind`, `map`, `new`, and `defer` must be `'static` (trampoline.rs, lines 132, 169, 198-200, 223-226).

The `'static` requirement is a direct consequence of choosing `Box<dyn Any>` for type erasure inside `Free`. The `Any` trait's `'static` bound exists to prevent dangling references when downcasting; without it, you could store a `&'a str`, erase it to `Any`, and then downcast it after the reference has expired.

## 2. Are the Lifetime Constraints Too Restrictive or Too Permissive?

### Trampoline: Appropriately Restrictive, but with Costs

The `'static` bound on Trampoline is **necessary given the current implementation** (using `Box<dyn Any>` inside `Free`). However, it is more restrictive than the semantic minimum. A trampoline fundamentally only needs to defer and chain computations; it does not inherently require `'static` data. The constraint arises from the implementation strategy (type erasure via `Any`), not from the concept itself.

This means users cannot use Trampoline with borrowed data:

```rust
fn process(data: &[u8]) -> Trampoline<usize> {
    // ERROR: `data` does not live long enough
    Trampoline::new(move || data.len())
}
```

### Thunk: Appropriately Flexible

Thunk's `'a` parameter correctly tracks the lifetime of captured references. The constraint `A: 'a` on `Thunk<'a, A>` ensures the produced value does not outlive the closure that creates it. This is sound and appropriately permissive.

### Lazy: Appropriately Flexible

Lazy mirrors Thunk's lifetime design. The `'a` parameter on `Lazy<'a, A, Config>` bounds the initializer closure. The `A: 'a` bound (lazy.rs, line 429) ensures values stored in the cache do not contain dangling references. This is correct.

## 3. Lifetime Interaction with the HKT/Brand System

The Kind trait used by most types in this library is `Kind_cdc7cd43dac7585f` with signature `type Of<'a, A: 'a>: 'a` (kinds.rs, line 58). This means any brand implementing this trait must produce types that are parameterized by a lifetime and bounded by it.

**Thunk and Lazy integrate cleanly with HKT:**
- `ThunkBrand` implements `Kind` with `type Of<'a, A: 'a>: 'a = Thunk<'a, A>` (thunk.rs, line 319).
- `LazyBrand<Config>` implements `Kind` with `type Of<'a, A: 'a>: 'a = Lazy<'a, A, Config>` (lazy.rs, line 807).

**Trampoline cannot implement HKT traits.** The brands.rs file explicitly documents this: "This is for `Thunk<'a, A>`, NOT for `Trampoline<A>`. `Trampoline` cannot implement HKT traits due to its `'static` requirement." (brands.rs, lines 192-193). The problem is that `Kind::Of<'a, A: 'a>` requires the brand to produce a type for any lifetime `'a`, but `Trampoline<A>` only works with `A: 'static`. You cannot implement `type Of<'a, A: 'a>: 'a = Trampoline<A>` because `Trampoline<A>` does not accept non-static `A`.

This creates a fundamental gap: Trampoline provides stack safety but is excluded from generic HKT programming. Users who need both stack safety and HKT integration must use `Thunk` with its `MonadRec::tail_rec_m` implementation (thunk.rs, lines 581-596), which uses a loop internally for limited stack safety but is not fully stack-safe for arbitrary nesting.

## 4. Is the `'static` Requirement on Trampoline Truly Necessary?

**Under the current architecture, yes.** The `Free` monad's continuation stack (`CatList<Continuation<F>>`) uses `Box<dyn FnOnce(TypeErasedValue) -> Free<F, TypeErasedValue>>` where `TypeErasedValue = Box<dyn Any>` (free.rs, lines 93, 100). Since `dyn Any` requires `'static`, all intermediate and final values must be `'static`.

**Could it be relaxed?** In theory, a different implementation strategy could avoid `'static`:

1. **Parameterize `Free` with a lifetime.** Instead of erasing to `Box<dyn Any>`, the continuation stack could use a lifetime-parameterized `Free<'a, F, A>` with `Box<dyn FnOnce(Box<dyn Any + 'a>) -> Free<'a, F, Box<dyn Any + 'a>> + 'a>`. However, `Any` itself is `'static`-bounded in Rust's standard library. There is no `Any + 'a` in stable Rust.

2. **Use unsafe transmutes.** One could use `unsafe` to relax the `'static` bound, similar to what some async runtimes do. This would sacrifice safety guarantees.

3. **Avoid type erasure entirely.** Replace `CatList` of type-erased continuations with a different structure (e.g., nested closures or a GAT-based approach). This would require a fundamentally different `Free` implementation.

The most practical path would be option 3, but it would be a significant rewrite and might sacrifice the O(1) bind guarantee that `CatList` provides. The current design is a deliberate trade-off: O(1) bind and unlimited stack safety, at the cost of `'static`.

## 5. Ergonomic Impact of the Lifetime Parameter

### Thunk: Moderate Friction

In practice, Thunk's `'a` parameter is often elided or inferred. Simple usage is ergonomic:

```rust
let thunk = Thunk::new(|| 42);  // 'a inferred as the enclosing scope
```

The friction appears when Thunk values need to be stored in structs or returned from functions, where the lifetime must be explicitly named. The HKT trait signatures also become verbose, as documented in the lifetime-ablation-experiment.md (lines 9-13): signatures like `fn map<'a, Brand: Functor, A: 'a, B: 'a, Func>(...)` create "Lifetime Soup."

### Lazy: Moderate Friction (Same as Thunk)

Lazy has the same ergonomic profile as Thunk regarding lifetimes. The additional `Config` type parameter adds further verbosity, mitigated by the type aliases `RcLazy<'a, A>` and `ArcLazy<'a, A>` (lazy.rs, lines 689, 692).

### Trampoline: Lower Friction, Higher Walls

Trampoline has no lifetime parameter, making simple usage very clean:

```rust
let task = Trampoline::new(|| 42);
task.bind(|x| Trampoline::pure(x * 2));
```

But the `'static` requirement creates hard walls. Users cannot capture borrows at all, which forces either cloning or wrapping in `Rc`/`Arc`. This is a different kind of ergonomic cost: instead of verbose signatures, users face puzzling compiler errors about values "not living long enough."

### The Ablation Experiment

The lifetime-ablation-experiment.md documents an attempt to remove `'a` from the Kind trait entirely. It produced 17 E0310 errors in types using trait objects (thunk, lazy, free, fn brands). The conclusion (line 128): "The 'Lifetime Soup' is the price of admission for safe, zero-cost functional abstractions in Rust." The experiment confirms that for closure-wrapping types, the lifetime parameter is non-negotiable.

## 6. Closure Capture Patterns and Lifetime Interaction

### `Box<dyn FnOnce() -> A + 'a>` in Thunk

Thunk uses `FnOnce` (thunk.rs, line 89), which is consumed on evaluation. This means a Thunk can only be evaluated once (the inherent `evaluate` takes `self`). The `'a` on the trait object correctly bounds any captured references.

The HKT-level `Semimonad::bind` requires `Fn` (not `FnOnce`), because some brands (like `Vec`) call the function multiple times (thunk.rs, lines 171-173). This mismatch means the HKT bind is more restrictive than necessary for Thunk; the inherent `bind` method accepting `FnOnce` is preferred when HKT generality is not needed.

### `Box<dyn FnOnce() -> A + 'a>` in Lazy (via LazyCell)

Lazy's initializer is also `FnOnce`, boxed into the `LazyCell`. The key difference is that `LazyCell` consumes the closure internally on first access and then stores the result. The `'a` on the box bounds the closure's captures, which is valid because the closure is consumed before the `Lazy` value could outlive `'a`.

### `Box<dyn FnOnce() -> Free<F, TypeErasedValue>>` in Free/Trampoline

The `Thunk` instances stored inside `Free` (via `Trampoline::new`, line 133: `Thunk::new(move || Free::pure(f()))`) use `Thunk<'static, ...>` because all values in the `Free` pipeline must be `'static`. The `FnOnce` closures here cannot capture borrowed data at all.

### Interaction with `Fn` (multi-use closures)

`Trampoline::tail_rec_m` requires `Fn` (not `FnOnce`) with `Clone` (trampoline.rs, line 378), because the recursive step function is called multiple times. This further constrains what closures can be used. The `arc_tail_rec_m` variant (trampoline.rs, line 447) wraps the closure in `Arc` to sidestep the `Clone` requirement.

## 7. Variance Issues

### Thunk is Covariant in `'a`

`Thunk<'a, A>` contains `Box<dyn FnOnce() -> A + 'a>`. A `Box<dyn Trait + 'long>` is a subtype of `Box<dyn Trait + 'short>` when `'long: 'short`. Therefore, `Thunk<'long, A>` can be used where `Thunk<'short, A>` is expected (assuming `A` is covariant too). This is correct and expected behavior.

### Lazy is Covariant in `'a`

`Lazy<'a, A, RcLazyConfig>` stores `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>`. The lifetime `'a` appears only in the closure's trait object bound, in covariant position (inside `Box` and `Rc`). So `Lazy` is covariant in `'a`, which is correct.

### Trampoline Has No Lifetime Variance

`Trampoline<A>` has no lifetime parameter, so variance is not applicable.

### No Contravariance Issues

None of these types use `'a` in contravariant position (e.g., as a function parameter type). The lifetime only bounds the closure's captures, which is always covariant. This is sound.

## 8. Soundness Issues

### No Identified Soundness Issues

The lifetime handling appears sound:

1. **Thunk:** The `Box<dyn FnOnce() -> A + 'a>` correctly prevents use-after-free. The `'a` bound ensures the thunk cannot outlive its captured references. The `evaluate(self)` method consumes the thunk, so there is no aliasing concern.

2. **Lazy:** The `Rc<LazyCell<...>>` wrapping is sound because `LazyCell` handles interior mutability safely (it uses `UnsafeCell` internally but exposes a safe API). The `'a` bound on the initializer prevents dangling references. Cloning a `Lazy` shares the `Rc`/`Arc`, which is safe because the memoized value's lifetime is bounded by `'a`.

3. **Trampoline/Free:** The `'static` bound is overly conservative but sound. `Box<dyn Any>` ensures all values are owned and cannot dangle. The `CatList` of continuations uses `Box<dyn FnOnce(...)>` which is also owned.

4. **Cross-type conversions:** The `From` implementations between types correctly enforce lifetime constraints:
   - `From<Thunk<'static, A>> for Trampoline<A>` (trampoline.rs, line 483): requires `'static` Thunk.
   - `From<Lazy<'static, A, Config>> for Trampoline<A>` (trampoline.rs, line 465): requires `'static` Lazy with `A: Clone`.
   - `From<Trampoline<A>> for Thunk<'static, A>` (thunk.rs, line 300): produces `'static` Thunk, requires `A: Send`.
   - `From<Thunk<'a, A>> for Lazy<'a, A, RcLazyConfig>` (lazy.rs, line 565): preserves `'a`.

One minor observation: `From<Trampoline<A>> for Thunk<'static, A>` requires `A: Send` (thunk.rs, line 300), which seems unnecessary since both Trampoline and Thunk are `!Send` types. The `Send` bound here may be overly restrictive, preventing conversion of `Trampoline<Rc<T>>` to `Thunk<'static, Rc<T>>` even though neither type would be sent across threads. This is not a soundness issue but an ergonomic one.

## 9. Composability Between Types

### Thunk Inside Trampoline: Only with `'static`

You can put a `Thunk<'static, A>` inside a `Trampoline<A>` via `From` (trampoline.rs, line 483-497):

```rust
let thunk = Thunk::pure(42);
let task = Trampoline::from(thunk);  // OK: Thunk<'static, i32>
```

But a `Thunk<'a, A>` where `'a` is not `'static` cannot be converted:

```rust
let x = 42;
let thunk = Thunk::new(|| x);  // Thunk<'_, i32>
// Cannot convert to Trampoline because 'a is not 'static
```

### Trampoline Inside Thunk: Requires `A: Send`

`From<Trampoline<A>> for Thunk<'static, A>` requires `A: 'static + Send` (thunk.rs, line 300). This is more restrictive than necessary, as noted in section 8.

### Lazy to/from Thunk: Seamless within the Same Lifetime

`From<Thunk<'a, A>> for Lazy<'a, A, RcLazyConfig>` (lazy.rs, line 565) and `From<Lazy<'a, A, Config>> for Thunk<'a, A>` where `A: Clone` (thunk.rs, line 278) both preserve the `'a` lifetime. This composability is clean.

### Lazy to/from Trampoline: Only at `'static`

`From<Trampoline<A>> for Lazy<'a, A, RcLazyConfig>` where `A: 'static` (lazy.rs, line 583) works because the `'a` on the output Lazy can be any lifetime (it is covariant), and the Trampoline owns its data statically. The reverse, `From<Lazy<'static, A, Config>> for Trampoline<A>` (trampoline.rs, line 465), requires `'static` Lazy and `A: Clone`.

### Cross-type Composability: The Try Variants

The Try variants mirror the same patterns:
- `TryLazy` can be created from `TryThunk` within the same `'a` (try_lazy.rs, line 184-201).
- `TryLazy` can be created from `TryTrampoline` only with `A: Send, E: Send` (try_lazy.rs, line 208-229).
- `TryTrampoline` requires `A: 'static, E: 'static` (try_trampoline.rs, line 38), same as `Trampoline`.

### Summary of Composability Constraints

| From | To | Constraints |
|------|----|-------------|
| `Thunk<'a, A>` | `Lazy<'a, A>` | `A: 'a` (seamless) |
| `Lazy<'a, A>` | `Thunk<'a, A>` | `A: Clone + 'a` (seamless) |
| `Thunk<'static, A>` | `Trampoline<A>` | `A: 'static` |
| `Trampoline<A>` | `Thunk<'static, A>` | `A: 'static + Send` (Send is surprising) |
| `Lazy<'static, A>` | `Trampoline<A>` | `A: 'static + Clone` |
| `Trampoline<A>` | `Lazy<'a, A>` | `A: 'static` |
| `Thunk<'a, A>` | `Trampoline<A>` | Impossible unless `'a = 'static` |

The fundamental barrier is clear: the `'a` vs `'static` divide creates a one-way gate. Values can move from the `'static` world into the `'a` world (by subsumption), but not the reverse. This means Trampoline is isolated from the broader HKT ecosystem unless all data is `'static`.

## Key Findings

1. **The lifetime split is architecturally motivated.** `Thunk` and `Lazy` use `'a` because their closures are stored as trait objects needing explicit lifetime bounds. `Trampoline` requires `'static` because its `Free` monad uses `Box<dyn Any>` which mandates `'static`.

2. **The `'static` requirement on Trampoline is an implementation artifact, not a semantic necessity.** Relaxing it would require replacing `dyn Any` with a different type erasure strategy, which is non-trivial but theoretically possible.

3. **The lifetime parameter is essential for HKT integration.** The ablation experiment (lifetime-ablation-experiment.md) proved that removing `'a` from Kind breaks all closure-wrapping types with 17 E0310 errors.

4. **Trampoline's exclusion from HKT is the most significant design consequence.** Users who need both stack safety and generic HKT programming are forced to use `Thunk::tail_rec_m`, which provides only loop-based tail recursion, not full stack safety for arbitrary nesting.

5. **No soundness issues were identified.** All lifetime bounds are correct and conservative. The variance is covariant where expected.

6. **One unnecessarily restrictive bound exists.** `From<Trampoline<A>> for Thunk<'static, A>` requires `A: Send` (thunk.rs, line 300), which prevents converting `Trampoline<Rc<T>>` to `Thunk` even though neither type is `Send`.

7. **Composability is clean within the `'a` world** (Thunk and Lazy interconvert freely) **but gated at the `'static` boundary** (Trampoline conversions require `'static` and sometimes additional bounds like `Clone` or `Send`).
