# Analysis 4: Memoization Implementation (Lazy/TryLazy)

## 1. Internal Representation

### Lazy

The `Lazy` type (`fp-library/src/types/lazy.rs`, line 424) is a newtype wrapper:

```rust
pub struct Lazy<'a, A, Config: LazyConfig = RcLazyConfig>(
    pub(crate) Config::Lazy<'a, A>,
)
```

The actual storage is determined by the `LazyConfig` trait (line 65), which abstracts over two configurations:

**RcLazyConfig** (line 175, impl at line 177):
- `Lazy<'a, A>` = `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>`
- `TryLazy<'a, A, E>` = `Rc<LazyCell<Result<A, E>, Box<dyn FnOnce() -> Result<A, E> + 'a>>>`

**ArcLazyConfig** (line 289, impl at line 291):
- `Lazy<'a, A>` = `Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>`
- `TryLazy<'a, A, E>` = `Arc<LazyLock<Result<A, E>, Box<dyn FnOnce() -> Result<A, E> + Send + 'a>>>`

The design layers three concerns:
1. **Initialization-once semantics**: delegated to `std::cell::LazyCell` (single-threaded) or `std::sync::LazyLock` (thread-safe).
2. **Shared ownership**: `Rc` or `Arc` wrapping enables clone-to-share-cache semantics.
3. **Closure boxing**: `Box<dyn FnOnce() -> A + 'a>` for type erasure of the initializer.

### TryLazy

`TryLazy` (`fp-library/src/types/try_lazy.rs`, line 44) follows an identical pattern but stores a `Result<A, E>` in the cell:

```rust
pub struct TryLazy<'a, A, E, Config: LazyConfig = RcLazyConfig>(
    pub(crate) Config::TryLazy<'a, A, E>,
)
```

Evaluation (line 106) returns `Result<&A, &E>` by calling `LazyCell::force(lazy).as_ref()` (line 282 of lazy.rs for the Rc variant), which forces the cell and then splits the reference to the stored `Result`.

## 2. Thread Safety: Rc vs Arc Separation

The separation is **correct and well-enforced**:

- **RcLazyConfig** uses `Rc<LazyCell<...>>`. `Rc` is `!Send + !Sync`, and `LazyCell` is `!Sync`. Neither can cross thread boundaries. The closure type is `dyn FnOnce() -> A + 'a` (no `Send` bound).

- **ArcLazyConfig** uses `Arc<LazyLock<...>>`. `Arc` is `Send + Sync` when its contents are, and `LazyLock` provides internal synchronization via a blocking mutex. The closure type is `dyn FnOnce() -> A + Send + 'a`, correctly requiring `Send`.

The `ArcLazy::new` constructor (line 622) requires `f: impl FnOnce() -> A + Send + 'a`, while `RcLazy::new` (line 511) only requires `f: impl FnOnce() -> A + 'a`. The `Send` bound is consistently threaded through the Arc path.

One detail worth noting: `ArcLazyConfig::Lazy` wraps `Arc<LazyLock<A, ...>>`, but `LazyLock` itself already provides thread-safe initialization. The `Arc` is needed purely for shared ownership (clone semantics), not for synchronization. This is correct.

**Verification in tests**: `test_arc_memo_thread_safety` (line 1324) spawns 10 threads sharing an `ArcLazy` and asserts the computation counter is exactly 1, confirming thread-safe evaluate-at-most-once.

## 3. Evaluate-At-Most-Once Guarantee

### Infallible (Lazy)

The guarantee is **fully sound** and delegated to std:

- `LazyCell::force` (used at line 254) initializes the cell on first call using the stored `FnOnce` closure, then returns a reference to the cached value on all subsequent calls. The closure is consumed (moved out) on first use, making re-invocation impossible.
- `LazyLock::force` (used at line 368) does the same with internal mutex synchronization; concurrent callers block until the first completes.

### Fallible (TryLazy)

TryLazy stores `Result<A, E>` in the cell (not just `A`), so both success and failure are memoized. `try_evaluate` (line 279-283) calls `LazyCell::force(lazy).as_ref()`, which:
1. Forces the `LazyCell<Result<A, E>, ...>`, computing the `Result` at most once.
2. Calls `.as_ref()` to convert `&Result<A, E>` to `Result<&A, &E>`.

This means errors are sticky: once a `TryLazy` evaluates to `Err`, it always returns that same `Err`. This is confirmed by `test_try_memo_caching_err` (try_lazy.rs line 616-629).

### Panic Behavior

The `Lazy` doc (line 405-412) explicitly documents that if the initializer panics, the cell is "poisoned":
- For `LazyCell`: the cell remains uninitialized; subsequent `force` calls will panic because the closure has been consumed but no value was stored.
- For `LazyLock`: the lock is poisoned; subsequent `force` calls panic.

`TryLazy::catch_unwind` (try_lazy.rs line 364) provides a safe alternative by wrapping the closure in `std::panic::catch_unwind`.

## 4. Interior Mutability and Soundness

The interior mutability chain is:

- **RcLazy**: `Rc` (shared pointer, no interior mutability) -> `LazyCell` (interior mutability via `UnsafeCell` internally).
- **ArcLazy**: `Arc` (shared pointer, no interior mutability) -> `LazyLock` (interior mutability via blocking mutex).

`LazyCell` and `LazyLock` are standard library types that have been audited for soundness. They guarantee:
- No data races (LazyLock uses mutex; LazyCell is `!Sync`).
- No double initialization (the `FnOnce` is consumed).
- References returned by `force` are valid for the lifetime of the cell.

**No custom unsafe code** is used anywhere in the memoization implementation. The `pub(crate)` visibility on the inner field (line 426) prevents external code from directly accessing the cell, though crate-internal code could theoretically bypass the abstraction.

**One potential concern**: The `Lazy` struct derives `Clone` manually (line 437-460) by cloning the inner `Rc`/`Arc`. This is correct because `Rc::clone`/`Arc::clone` only increments the reference count, sharing the same `LazyCell`/`LazyLock`. Multiple clones force the same cell, which is the intended semantics.

## 5. RefFunctor Implementation

The `RefFunctor` trait (ref_functor.rs, line 62) is designed for types where `evaluate` returns `&A` rather than `A`:

```rust
pub trait RefFunctor {
    fn ref_map<'a, A: 'a, B: 'a>(
        func: impl FnOnce(&A) -> B + 'a,
        fa: ...,
    ) -> ...;
}
```

**Why RefFunctor instead of Functor**: `Lazy::evaluate` returns `&A` (line 485), not an owned `A`. A regular `Functor::map` would need to produce a `Lazy<'a, B>` from a function `A -> B` and a `Lazy<'a, A>`. But since `evaluate` only gives `&A`, you would need `A: Clone` to extract an owned `A`. `RefFunctor` sidesteps this by having the mapping function accept `&A -> B`, which works naturally with the reference-returning API.

**Implementation for RcLazyConfig** (line 880-913):

```rust
impl RefFunctor for LazyBrand<RcLazyConfig> {
    fn ref_map<'a, A: 'a, B: 'a>(
        f: impl FnOnce(&A) -> B + 'a,
        fa: Lazy<'a, A, RcLazyConfig>,
    ) -> Lazy<'a, B, RcLazyConfig> {
        fa.ref_map(f)
    }
}
```

This delegates to the inherent `ref_map` method (line 554-561):

```rust
pub fn ref_map<B: 'a>(self, f: impl FnOnce(&A) -> B + 'a) -> Lazy<'a, B, RcLazyConfig> {
    let fa = self.clone();
    let init: Box<dyn FnOnce() -> B + 'a> = Box::new(move || f(fa.evaluate()));
    Lazy(RcLazyConfig::lazy_new(init))
}
```

**Correctness analysis**: The `self.clone()` creates a second `Rc` handle to the same `LazyCell`. The returned `Lazy<B>` captures this clone. When the new `Lazy<B>` is forced, it forces the original `Lazy<A>` (which may already be cached), applies `f` to the reference, and returns `B`. The result `B` is itself memoized in the new `Lazy<B>`.

**No RefFunctor for ArcLazyConfig** (documented at line 660-663): The `RefFunctor` trait does not require `Send` on the mapping function, but `ArcLazy::new` requires `Send`. The library correctly provides only an inherent `ref_map` method on `ArcLazy` (line 677-685) with explicit `Send` bound, rather than an unsound blanket `RefFunctor` impl.

**Law compliance**: The identity law (`ref_map(|x| x.clone(), fa)` evaluates equally) and composition law (`ref_map(|x| f(&g(x)), fa)` equals `ref_map(f, ref_map(g, fa))`) hold because:
- Each `ref_map` creates a new independent `Lazy` that, when forced, forces its input and applies its function.
- Memoization is per-`Lazy` instance, so composed chains evaluate correctly.

## 6. Comparison with Standard Library and once_cell

### std::cell::LazyCell (Rust 1.80+)

The library uses `LazyCell` directly as the inner cell for `RcLazyConfig` (line 178). The wrapping `Rc` adds shared ownership. This is the canonical pattern; the library does not reinvent the cell.

### std::sync::LazyLock (Rust 1.80+)

Similarly, `LazyLock` is used directly for `ArcLazyConfig` (line 292). The wrapping `Arc` adds shared ownership.

### once_cell crate

The `once_cell` crate's `OnceCell`/`Lazy` and `sync::OnceCell`/`sync::Lazy` served as the precursors to the std types. Since this library targets Rust 1.80+, it correctly uses the std types instead, avoiding the external dependency.

The library does use `OnceCell` and `OnceLock` directly in the `rc_lazy_fix`/`arc_lazy_fix` functions (lines 738, 791) for the fixed-point combinator, where the cell needs to be populated after construction (a use case that `LazyCell`/`LazyLock` cannot serve because they take the initializer at construction time).

### Key difference from std

The std types (`LazyCell`, `LazyLock`) do not support shared ownership by themselves. The library's contribution is:
1. Wrapping in `Rc`/`Arc` for clone-to-share-cache semantics.
2. The `LazyConfig` trait for abstracting over Rc/Arc choice.
3. The `TryLazy` variant for fallible computations.
4. The fixed-point combinators.

## 7. Whether Lazy Should Implement More Type Classes

### Currently implemented

| Type class | RcLazy | ArcLazy |
|---|---|---|
| RefFunctor | Yes (trait) | No (inherent only) |
| Foldable | Yes | No |
| Semigroup | Yes | Yes |
| Monoid | Yes | Yes |
| Deferrable | Yes | Yes (SendDeferrable) |
| PartialEq | Yes | Yes |
| PartialOrd | Yes | Yes |

### What PureScript's Data.Lazy implements

PureScript's `Lazy` (purescript-lazy) implements a rich set of instances (lines 41-143):
- Functor, Apply, Applicative, Bind, Monad
- Foldable, FoldableWithIndex, Foldable1
- Traversable, TraversableWithIndex, Traversable1
- FunctorWithIndex
- Extend, Comonad
- Semigroup, Monoid, Semiring, Ring, EuclideanRing
- Eq, Ord, Bounded
- Show
- Invariant (via imapF)
- Control.Lazy (tie-the-knot)

### What Rust's Lazy cannot (easily) implement and why

**Functor**: Would require `fn map(f: impl FnOnce(A) -> B, fa: Lazy<A>) -> Lazy<B>`. Since `evaluate()` returns `&A`, the function would need `A: Clone` or the signature would need `FnOnce(&A) -> B` (which is `RefFunctor`). Adding a `Clone` bound to make `Functor` work is possible but constrains the type parameter. The library chose `RefFunctor` to avoid this.

**Applicative/Monad**: `pure` exists as an inherent method (line 532). `bind` would need `fn bind(fa: Lazy<A>, f: impl FnOnce(A) -> Lazy<B>) -> Lazy<B>`, again requiring `A: Clone` to extract a value from `&A`. This is technically implementable with `Clone` bounds but clashes with the library's uncurried, zero-cost design goals.

**Traversable**: PureScript's `traverse f l = defer <<< const <$> f (force l)` (line 109) works because `force` returns an owned `a`. In Rust, `evaluate` returns `&A`, so traversal would need cloning.

**Comonad**: `extract = force` in PureScript (line 137). In Rust, `evaluate` returns `&A`, not `A`. Comonad's `extract` typically returns an owned value. This would require `A: Clone`.

**Assessment**: The library's choice to keep Lazy's type class implementations minimal is **well-reasoned**. The fundamental tension is that Lazy's API returns `&A` (zero-copy), while most type classes expect owned values. PureScript does not have this distinction; `force` returns an owned `a` (in a GC'd runtime, this is just pointer copying). Adding `Clone` bounds to enable Functor/Monad would work but would limit generality and add hidden allocations, conflicting with the zero-cost philosophy.

The library could consider:
- **Eq, Ord** (already done, correctly forces both sides).
- A `Comonad`-like trait that returns `&A` instead of `A` (a `RefComonad`), analogous to `RefFunctor`.
- More `Foldable` variants (ArcLazy Foldable is missing; likely just an oversight).

## 8. Pointer Abstraction (RcLazy vs ArcLazy)

The `LazyConfig` trait (line 65) is the abstraction mechanism, bundling:
- `type Lazy<'a, A>` (the cell type)
- `type TryLazy<'a, A, E>` (the fallible cell type)
- `type Thunk<'a, A>` (the initializer type, `dyn FnOnce`)
- `type TryThunk<'a, A, E>` (the fallible initializer type)
- Methods: `lazy_new`, `try_lazy_new`, `evaluate`, `try_evaluate`

**Strengths**:
- Clean separation: code that is generic over `Config: LazyConfig` works with both Rc and Arc variants.
- Extensibility: third-party configs (e.g., `parking_lot`-based) can be plugged in.
- Type aliases `RcLazy`/`ArcLazy` (lines 689, 692) and `RcTryLazy`/`ArcTryLazy` (try_lazy.rs lines 540, 543) provide ergonomic shorthand.

**Design tension**: The `LazyConfig` trait lives in `lazy.rs` rather than in `classes/`, which means it is not a first-class type class in the library's hierarchy. It does not follow the `Brand` pattern. This is reasonable because `LazyConfig` is a configuration trait, not a type class that types implement to gain algebraic structure.

**Relationship to the Pointer hierarchy** (`Pointer` -> `RefCountedPointer` -> `SendRefCountedPointer`): The `LazyConfig` trait does not directly use the `Pointer` hierarchy. `RcLazyConfig` hard-codes `Rc` and `ArcLazyConfig` hard-codes `Arc`, rather than being parameterized by a `RefCountedPointer` brand. This is arguably appropriate because `LazyCell` and `LazyLock` are not parameterized over pointer types; the cell type and the pointer type are tightly coupled (LazyCell is !Sync, so it must be Rc; LazyLock is Sync, so it pairs with Arc).

**Potential improvement**: If `LazyConfig` were parameterized over a `Pointer` brand, it would be more general. However, the tight coupling between cell type and pointer type means there is no practical combinatorial explosion; only two valid combinations exist.

## 9. Comparison with PureScript's Data.Lazy

PureScript's `Lazy` (purescript-lazy/src/Data/Lazy.purs) is a `foreign import data` type (line 30), meaning its implementation is provided by the runtime (JavaScript). Key differences:

### Runtime model
- PureScript: `defer` and `force` are FFI primitives. The JS implementation uses a thunk object with a `value` field and a `forced` flag. Forcing is assignment-based memoization. GC handles lifetime management.
- Rust: `Lazy` uses `LazyCell`/`LazyLock` for initialization-once, `Rc`/`Arc` for shared ownership. Lifetimes are statically tracked.

### Owned vs borrowed values
- PureScript: `force :: Lazy a -> a` returns an owned value (cheap in GC'd language).
- Rust: `evaluate(&self) -> &A` returns a reference. This is the fundamental design difference that cascades through all type class implementations.

### Type class coverage
- PureScript: Full Functor, Applicative, Monad, Foldable, Traversable, Comonad, plus algebraic instances (Semiring, Ring, etc.). All are trivially correct because `force` returns an owned value and `defer` creates a new lazy thunk.
- Rust: RefFunctor, Foldable (Rc only), Semigroup, Monoid, PartialEq, PartialOrd. The reference-returning API constrains what is implementable without `Clone`.

### Fixed-point combinator
- PureScript: `lazyLazy :: CL.Lazy (Lazy a)` (line 142-143) provides `defer f = defer \_ -> force (f unit)`, which is the `Control.Lazy` class's `defer` method.
- Rust: `rc_lazy_fix` (line 735-750) and `arc_lazy_fix` (line 788-803) use `OnceCell`/`OnceLock` to break the self-referential cycle. This is more complex because Rust cannot create cyclic references directly. The implementation is sound: the cell is populated before the closure can be invoked (the comment at line 741-743 explains the invariant).

### Thread safety
- PureScript: Not relevant; single-threaded JS runtime.
- Rust: The Rc/Arc split provides explicit opt-in to thread safety, which is a strict improvement.

### Error handling
- PureScript: No `TryLazy` equivalent; errors would be handled via `Lazy (Either e a)`.
- Rust: `TryLazy` provides first-class fallible memoization with `Result<&A, &E>` return type, plus `catch_unwind` for panic capture. This is a meaningful addition.

### Memoization guarantee
Both implementations guarantee evaluate-at-most-once semantics. PureScript relies on the JS runtime's mutation. Rust relies on `LazyCell`/`LazyLock` which consume the initializer closure.

## Summary of Findings

1. **Internal representation is sound**: The library correctly layers `Rc`/`Arc` for shared ownership over `LazyCell`/`LazyLock` for initialization-once semantics, with `Box<dyn FnOnce>` for type-erased initializers.

2. **Thread safety separation is correct**: `RcLazyConfig` uses non-Send types; `ArcLazyConfig` uses Send+Sync types with appropriate bounds on closures.

3. **Evaluate-at-most-once is guaranteed**: Delegated to standard library primitives. No custom unsafe code. Panic poisoning is documented.

4. **No soundness issues identified**: All interior mutability is handled by audited std types. The `pub(crate)` visibility on the inner field is the only potential leakage point.

5. **RefFunctor is correctly implemented**: The design choice to use `&A -> B` instead of `A -> B` is well-motivated by the reference-returning API. The decision not to implement `RefFunctor` for `ArcLazyConfig` (due to `Send` bound mismatch) is correct.

6. **Standard library alignment is excellent**: Direct use of `LazyCell`/`LazyLock` (Rust 1.80+) avoids reinventing the wheel.

7. **Type class coverage is deliberately minimal**: The reference-returning API prevents straightforward Functor/Monad/Comonad implementations. This is a reasonable trade-off for zero-cost semantics.

8. **The pointer abstraction is pragmatic**: `LazyConfig` couples the cell type with the pointer type, which matches the reality that only two valid combinations exist.

9. **Compared to PureScript**: Rust's version is more constrained (by ownership/borrowing) but gains thread safety, explicit error handling (`TryLazy`), and zero-cost guarantees in exchange.
