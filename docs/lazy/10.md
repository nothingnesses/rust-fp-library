# Analysis 10: API Ergonomics and Practical Usability

## 1. Discoverability: Can a User Figure Out Which Type to Use?

The library provides excellent guidance for type selection. The comparison table in `lib.rs` (lines 114-118) and `README.md` (lines 182-186) gives a clear at-a-glance summary of Thunk vs Trampoline vs Lazy across five dimensions (stack safety, memoization, lifetimes, HKT traits). The "Why of Three Types" section in both `lib.rs` (lines 120-125) and `README.md` (lines 188-193) explains the design rationale.

The workflow example in `lib.rs` (lines 127-178) showing an expression evaluator that uses TryTrampoline, TryLazy, and TryThunk together is a strong pedagogical choice, demonstrating how the types compose in a realistic scenario.

**Concern**: The six types (Thunk, Trampoline, Lazy, TryThunk, TryTrampoline, TryLazy) multiplied by two config variants (Rc/Arc) create a 12-cell matrix. A user new to the library faces a non-trivial decision tree. The documentation handles the infallible trio well, but the Try variants are less prominently documented. There is no dedicated comparison table for when to use TryThunk vs TryTrampoline vs TryLazy.

**Suggestion**: A single "choosing your lazy type" decision flowchart (need memoization? -> Lazy/TryLazy; need stack safety? -> Trampoline/TryTrampoline; need HKT? -> Thunk/TryThunk; fallible? -> Try variant) would improve discoverability significantly.

## 2. Constructor Name Intuitiveness

### Consistent constructors across types

| Constructor | Thunk | Trampoline | Lazy | TryThunk | TryTrampoline | TryLazy |
|-------------|-------|------------|------|----------|---------------|---------|
| `new(f)` | line 113 | line 132 | line 511 (Rc), 622 (Arc) | line 90 | line 99 | line 136 (Rc), 404 (Arc) |
| `pure(a)` | line 136 | line 100 | line 532 (Rc), 644 (Arc) | line 109 | N/A | N/A |
| `defer(f)` | line 161 | line 169 | N/A (via Deferrable) | line 130 | line 142 | N/A (via Deferrable) |
| `ok(a)` | N/A | N/A | N/A | line 151 | line 61 | line 155 (Rc), 423 (Arc) |
| `err(e)` | N/A | N/A | N/A | line 172 | line 80 | line 174 (Rc), 445 (Arc) |

**Strengths**:
- `new` consistently means "wrap a closure" across all types.
- `pure` consistently means "wrap an already-computed value" for non-Try types.
- `ok`/`err` on Try types mirror `Result::Ok`/`Result::Err`, which is highly idiomatic Rust.
- `defer` consistently means "defer construction of the wrapper itself."

**Concerns**:
- `TryThunk::pure` and `TryThunk::ok` are documented as aliases of each other (try_thunk.rs, line 134: "Alias for `pure`"), but their coexistence is potentially confusing. A user might wonder if they differ subtly. The `TryTrampoline` type only has `ok` (no `pure`), which is inconsistent with `TryThunk`. If `ok` is the idiomatic name for Try types, `TryThunk::pure` should be deprecated or removed to reduce the decision surface.
- `TryLazy` lacks `pure` entirely, only offering `ok`, which is the right call for a fallible type.

**Verdict**: Constructor names are mostly intuitive and idiomatic. The `pure`/`ok` duplication on `TryThunk` is the main rough edge.

## 3. Method Surface Area

### Thunk (thunk.rs)

Inherent methods: `new`, `pure`, `defer`, `bind`, `map`, `evaluate`, `memoize`.

This is a tight, well-scoped API. The seven inherent methods cover creation, transformation, execution, and migration to Lazy. The type also implements extensive HKT traits (Functor, Pointed, Semiapplicative, Semimonad, Lift, ApplyFirst, ApplySecond, Foldable, MonadRec, Evaluable, Deferrable), which surface additional free functions (`map`, `bind`, `pure`, `fold_map`, `tail_rec_m`, `defer`, `evaluate`) via the `functions` module. This dual API (inherent + trait-based) is well-motivated: inherent methods take `FnOnce` for flexibility (line 170-173), while trait methods take `Fn` for HKT generality.

### Trampoline (trampoline.rs)

Inherent methods: `pure`, `new`, `defer`, `bind`, `map`, `evaluate`, `memoize`, `lift2`, `then`, `tail_rec_m`, `arc_tail_rec_m`.

At 11 inherent methods, Trampoline has a slightly larger surface. The addition of `lift2`, `then`, `tail_rec_m`, and `arc_tail_rec_m` is justified because Trampoline does not implement HKT traits, so these operations must be provided directly. The `arc_tail_rec_m` variant (line 447) addresses a real ergonomic pain point: closures that capture non-Clone state.

### Lazy (lazy.rs)

Inherent methods: `evaluate`, `new`, `pure`, `ref_map`.

At 4 methods (per config variant), Lazy is intentionally minimal. It is a caching wrapper, not a computation type, and the small surface reflects that.

**Notable**: `ref_map` (line 554 for Rc, line 677 for Arc) takes `&A -> B` rather than `A -> B`, which is necessary because Lazy holds a shared reference to cached data. This is a subtle but important distinction that might surprise users expecting a regular `map`.

### TryThunk (try_thunk.rs)

Inherent methods: `new`, `pure`, `defer`, `ok`, `err`, `bind`, `map`, `map_err`, `catch`, `evaluate`, `catch_unwind`.

At 11 methods, this is rich but justified. The error-handling combinators (`map_err`, `catch`, `catch_unwind`) are essential for ergonomic fallible computation. `catch_unwind` (line 327) is particularly clever, converting panics to `Err(String)`.

### TryTrampoline (try_trampoline.rs)

Inherent methods: `ok`, `err`, `new`, `defer`, `map`, `map_err`, `bind`, `catch`, `lift2`, `then`, `tail_rec_m`, `arc_tail_rec_m`, `evaluate`, `memoize`.

At 14 methods, this is the largest surface. Like Trampoline, it compensates for the lack of HKT traits by providing all operations as inherent methods.

### TryLazy (try_lazy.rs)

Inherent methods per config: `new`, `ok`, `err`, `evaluate`, `catch_unwind` (Rc only).

Minimal and appropriate for a caching type.

**Overall Assessment**: The surface areas are well-calibrated. Computation types (Thunk, Trampoline) have more methods than caching types (Lazy). Try variants reasonably expand the API with error-handling combinators. No type feels bloated or anemic.

## 4. Composition Between Types

### From conversions

The library provides a comprehensive `From` conversion network:

- `Thunk -> Lazy` (lazy.rs line 565): `From<Thunk<'a, A>> for Lazy<'a, A, RcLazyConfig>`
- `Trampoline -> Lazy` (lazy.rs line 583): `From<Trampoline<A>> for Lazy<'a, A, RcLazyConfig>`
- `Lazy -> Thunk` (thunk.rs line 278): requires `A: Clone`
- `Trampoline -> Thunk` (thunk.rs line 300): requires `A: 'static + Send`
- `Thunk -> Trampoline` (trampoline.rs line 483): requires `'static`
- `Lazy -> Trampoline` (trampoline.rs line 465): requires `A: Clone + 'static`

And for Try variants:
- `TryThunk -> TryLazy` (try_lazy.rs line 184)
- `TryTrampoline -> TryLazy` (try_lazy.rs line 208): requires `A: Send, E: Send`
- `Lazy -> TryLazy` (try_lazy.rs lines 236, 262): both Rc and Arc variants
- `TryLazy -> TryThunk` (try_thunk.rs line 376): requires `A: Clone, E: Clone`
- `Lazy -> TryThunk` (try_thunk.rs line 348): requires `A: Clone`
- `Result -> TryLazy` (try_lazy.rs lines 288, 312)

### Convenience methods

Both `Thunk` and `Trampoline` provide a `.memoize()` method (thunk.rs line 268, trampoline.rs line 271) that returns `Lazy<'_, A, RcLazyConfig>`. This is the most common conversion and the dedicated method makes the workflow natural:

```rust
let lazy = Thunk::new(|| expensive()).memoize();
let lazy = Trampoline::new(|| expensive()).memoize();
```

**Concern**: `.memoize()` always returns `RcLazy`, never `ArcLazy`. A user wanting thread-safe memoization must use `Lazy::<_, ArcLazyConfig>::from(trampoline)` instead. There is no `memoize_arc()` or similar convenience. Given that the `From<Trampoline> for Lazy<ArcLazyConfig>` impl does not exist (only Rc is provided), the user would need to manually construct: `ArcLazy::new(|| trampoline.evaluate())`. This is a notable gap.

**Concern**: The `Trampoline -> Thunk` conversion (thunk.rs line 300) requires `A: Send`, which is surprising. Trampoline itself does not require Send, so forcing Send on the conversion creates a restriction that is not obviously motivated by the target type (Thunk is not Send either).

### Workflow smoothness

The expression evaluator example in `lib.rs` (lines 133-178) demonstrates the intended TryTrampoline -> TryLazy -> TryThunk pipeline well. However, the code shows a manual `.evaluate()` call at the boundary:

```rust
let result = RcTryLazy::new(move || eval(&expr).evaluate());
```

This pattern, wrapping `evaluate()` in a new lazy cell, could be simplified with a direct `.memoize()` on TryTrampoline. TryTrampoline does have `.memoize()`, but it is only shown for the infallible case. The workflow would be cleaner if the example used `eval(&expr).memoize()` directly.

## 5. Footguns

### Footgun 1: Thunk re-evaluation is invisible

`Thunk::evaluate` consumes `self` (thunk.rs line 247: `pub fn evaluate(self) -> A`), so a Thunk can only be evaluated once. This is actually *safe* because the compiler prevents accidental re-evaluation. However, if a user clones a Thunk (which is not possible since Thunk wraps `Box<dyn FnOnce>` and does not implement `Clone`), they might expect memoization. The non-Clone nature is the correct design, but it means Thunk cannot be shared at all, which forces the user to `.memoize()` early if sharing is needed.

### Footgun 2: Trampoline re-evaluation is silent

`Trampoline::evaluate` also consumes `self` (trampoline.rs line 248), but `Trampoline` is not `Clone` either, so this is equally safe. The documentation is explicit about the lack of memoization (line 60: "Trampoline does NOT memoize. Each call to `run` re-evaluates."), though the method signature already prevents double-evaluation at the type level.

### Footgun 3: Lazy::evaluate returns a reference, not an owned value

`Lazy::evaluate` returns `&A` (lazy.rs line 485), not `A`. This can surprise users coming from `Thunk::evaluate() -> A`. The dereference is required: `*lazy.evaluate()` for Copy types or `lazy.evaluate().clone()` for Clone types. This is documented in examples (e.g., trampoline.rs line 269: `assert_eq!(*lazy.evaluate(), 42)`), but the asymmetry between `Thunk::evaluate() -> A` and `Lazy::evaluate() -> &A` could trip up users.

### Footgun 4: LazyConfig type parameter in construction

Creating a Lazy requires specifying the config: `Lazy::<_, RcLazyConfig>::new(|| 42)`. The type aliases `RcLazy` and `ArcLazy` (lazy.rs lines 689, 692) mitigate this: `RcLazy::new(|| 42)`. However, the examples in the documentation inconsistently use both forms. Some examples use `Lazy::<_, RcLazyConfig>::new(...)` (lazy.rs line 508), while others use `RcLazy::new(...)` (in test code). The verbose form is a significant ergonomic tax.

### Footgun 5: Lazy panics propagate on subsequent accesses

`Lazy`'s documentation explicitly notes this (lazy.rs lines 405-412): if the initializer panics, the underlying `LazyCell`/`LazyLock` is poisoned and all subsequent access also panics. The suggestion to use `TryLazy::catch_unwind` is good, but users might not discover this until runtime.

### Footgun 6: TryLazy::evaluate returns `Result<&A, &E>`, not `Result<A, E>`

Like Lazy, TryLazy returns references (try_lazy.rs line 106). Converting back to owned values requires `.cloned().map_err(Clone::clone)`, which is verbose. The `Deferrable` implementation for TryLazy (try_lazy.rs line 489) shows this pattern, confirming it is a recurring friction point.

## 6. Comparison to Similar Rust Crates

### vs `once_cell` / `std::sync::OnceLock`

`Lazy` is conceptually similar to `once_cell::sync::Lazy` (now `std::sync::LazyLock` in std). The key differences:
- `fp-library`'s Lazy is reference-counted (Rc/Arc wrapping), so cloning shares the cache. `LazyLock` is not Clone.
- `fp-library`'s Lazy supports lifetime polymorphism (`'a`), while `LazyLock` requires `'static`.
- `fp-library` provides a `RefFunctor` implementation, enabling `.ref_map()` for transforming cached values.

The shared-cache semantics are a genuine advantage for FP workflows where multiple consumers need the same deferred value.

### vs `futures::lazy`

`futures::lazy` creates a future from a closure. The key difference is that `fp-library`'s types are synchronous; they do not integrate with async runtimes. `Trampoline` is closer to a synchronous `Future` in spirit (deferred computation with explicit evaluation), but without the polling model.

### vs Haskell's lazy semantics

Haskell's laziness is pervasive and implicit. `fp-library` makes laziness explicit and opt-in, which is the right approach for Rust. The three-type decomposition (Thunk/Trampoline/Lazy) is more granular than Haskell's unified approach but provides better control over performance characteristics. The `Deferrable` trait (deferrable.rs) is analogous to Haskell's `Control.Lazy` class.

### vs `im` / `rpds` (persistent data structures)

These crates use laziness internally for structural sharing. `fp-library`'s Lazy types are lower-level building blocks that could underpin such structures.

## 7. Type Aliases: Helpful or Confusing?

The aliases are:
- `RcLazy<'a, A>` = `Lazy<'a, A, RcLazyConfig>` (lazy.rs line 689)
- `ArcLazy<'a, A>` = `Lazy<'a, A, ArcLazyConfig>` (lazy.rs line 692)
- `RcTryLazy<'a, A, E>` = `TryLazy<'a, A, E, RcLazyConfig>` (try_lazy.rs line 540)
- `ArcTryLazy<'a, A, E>` = `TryLazy<'a, A, E, ArcLazyConfig>` (try_lazy.rs line 543)

**Verdict**: These aliases are clearly helpful. The fully-qualified `Lazy<'a, A, RcLazyConfig>` is verbose and the Config parameter is noise in most usage. The aliases follow the Rc/Arc naming convention established by the standard library (`Rc` vs `Arc`). They are re-exported from `types.rs` (lines 61-67 and 78-82), making them discoverable.

**However**: The base types `RcLazyConfig` and `ArcLazyConfig` are also re-exported (types.rs lines 62, 63), which is necessary for advanced usage (e.g., writing generic code over `LazyConfig`) but adds noise to the `types::*` namespace. A user doing `use fp_library::types::*` gets 10+ lazy-related names: `Lazy`, `LazyConfig`, `RcLazyConfig`, `ArcLazyConfig`, `RcLazy`, `ArcLazy`, `TryLazy`, `RcTryLazy`, `ArcTryLazy`, `Thunk`, `Trampoline`, `TryThunk`, `TryTrampoline`. This is a lot, though each name earns its place.

## 8. Module Organization and Re-exports

### Module structure

The lazy types are spread across six files in `fp-library/src/types/`:
- `thunk.rs`
- `trampoline.rs`
- `lazy.rs`
- `try_thunk.rs`
- `try_trampoline.rs`
- `try_lazy.rs`

All are re-exported from `types.rs` (lines 48-85) and thus available via `use fp_library::types::*`.

**Strength**: The one-type-per-file organization is consistent with the rest of the codebase and makes it easy to find the implementation of any specific type.

**Concern**: There is no `lazy` submodule that groups all six files together. A user browsing the docs or source sees `thunk`, `trampoline`, `lazy`, `try_thunk`, etc., interspersed with unrelated types like `additive`, `cat_list`, `conjunctive`. A `types::lazy` submodule re-exporting all six would improve discoverability in the API documentation.

### Free function re-exports

The `functions.rs` module (lines 25-35) auto-generates re-exports from class modules. The lazy-relevant free functions (`defer`, `evaluate`) come from `deferrable.rs` and `evaluable.rs` respectively. The `rc_lazy_fix` and `arc_lazy_fix` functions are explicitly re-exported (functions.rs lines 37-40).

**Strength**: `defer` and `evaluate` are available as free functions, enabling generic programming:
```rust
let thunk: Thunk<i32> = defer(|| Thunk::new(|| 42));
```

**Concern**: There is no free function for `memoize`. The pattern `thunk.memoize()` is an inherent method, not a trait method, so it cannot be used generically. A `Memoizable` trait or free `memoize` function would complete the generic API.

## 9. Simplification Opportunities

### 9a. Unify `pure`/`ok` on Try types

`TryThunk` has both `pure(a)` and `ok(a)` (try_thunk.rs lines 109, 151), where `ok` is documented as an alias. Pick one. Since the Try types produce `Result`, `ok` is the natural name, and `pure` should be deprecated.

### 9b. Add `memoize_arc()` convenience

Both Thunk and Trampoline's `.memoize()` hardcode `RcLazyConfig`. Adding `.memoize_arc()` (or a generic `.memoize_with::<Config>()`) would close the gap for thread-safe memoization.

### 9c. Consider a `LazyExt` trait

A trait unifying the `.evaluate()` method across Thunk, Trampoline, and Lazy would enable generic code:
```rust
trait Evaluate {
    type Output;
    fn evaluate(self) -> Self::Output;
}
```
However, this conflicts with `Lazy::evaluate(&self) -> &A` vs `Thunk::evaluate(self) -> A`. The by-ref vs by-value distinction is fundamental, so a unified trait may not be feasible without compromising the design.

### 9d. Default Config for Lazy

`Lazy<'a, A, Config: LazyConfig = RcLazyConfig>` (lazy.rs line 424) already has a default config of `RcLazyConfig`. This means `Lazy::<_, _>::new(...)` should work without specifying the config. This default is good but underutilized in the examples.

### 9e. Reduce TryLazy construction verbosity

Currently: `TryLazy::<_, _, RcLazyConfig>::new(|| Ok::<i32, ()>(42))` (try_lazy.rs line 133). This requires both the config parameter and explicit `Ok` type annotation. Using `RcTryLazy::new(|| Ok(42))` is much cleaner but still requires explicit type annotation on the Ok variant when it cannot be inferred. This is a Rust limitation, not a library issue.

## 10. Performance Characteristics

### Documentation quality

Performance characteristics are well-documented:

- **Thunk**: `O(1)` creation, `O(1)` evaluation per step, but `O(n)` stack depth for bind chains (thunk.rs lines 166-173 warn about stack growth).
- **Trampoline**: Explicitly states "O(1) bind" (trampoline.rs line 50) and "Stack safe: Will not overflow regardless of recursion depth" (line 49). Complexity annotations on constructors (e.g., line 83: "O(1) creation, O(1) evaluation").
- **Lazy**: Evaluation is implicit O(1) after first access. The shared semantics (via Rc/Arc) mean cloning is O(1).

### Intuitive performance model

The mental model is straightforward:
- Thunk = closure + Box (one heap allocation per construction).
- Trampoline = Free monad + CatList (amortized O(1) bind via concatenation-friendly list).
- Lazy = Rc<LazyCell> or Arc<LazyLock> (one allocation, one initialization).

**Concern**: The cost of `From` conversions is not documented. Converting `Lazy -> Thunk` (thunk.rs line 294: `Thunk::new(move || lazy.evaluate().clone())`) clones the cached value, which may be expensive for large types. Converting `Trampoline -> Lazy` (lazy.rs line 597: `Self::new(move || task.evaluate())`) defers the full trampoline evaluation, which is fine. Users should be aware that conversions involving `.clone()` have data-dependent cost.

**Concern**: The `LazyConfig` trait (lazy.rs line 65) uses `Box<dyn FnOnce()>` for thunk storage, which means every Lazy creation involves a heap allocation for the closure itself, in addition to the Rc/Arc allocation. This double-allocation is inherent to the design (the LazyCell needs an owned closure) but is not called out explicitly.

## Summary of Findings

**Strengths**:
1. Consistent naming (`new`, `pure`, `defer`, `evaluate`) across all types.
2. Comprehensive `From` conversion network enabling smooth composition.
3. Well-scoped method surfaces, with computation types having more methods than caching types.
4. Excellent documentation with comparison tables, workflow examples, and complexity annotations.
5. Type aliases (`RcLazy`, `ArcLazy`, etc.) substantially reduce verbosity.
6. Ownership-based evaluation (`self` vs `&self`) prevents accidental re-evaluation footguns.

**Weaknesses**:
1. `TryThunk::pure`/`TryThunk::ok` duplication adds confusion.
2. `.memoize()` only produces `RcLazy`; no convenience for `ArcLazy`.
3. `Lazy::evaluate() -> &A` vs `Thunk::evaluate() -> A` asymmetry can surprise users.
4. No grouping submodule for lazy types in `types.rs`; they are interspersed with unrelated types.
5. `From<Trampoline> for Thunk` requires `Send`, which is non-obvious.
6. Verbose `TryLazy` construction with explicit config and Result type annotations.
7. Missing "choosing your type" decision flowchart for the full 6-type matrix.
8. `From` conversion costs (especially those involving `.clone()`) are undocumented.
