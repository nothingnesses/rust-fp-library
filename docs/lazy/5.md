# Analysis 5: Try Variant Design and Consistency

## 1. Separate Try* Types vs Result Inside Regular Types

The library takes the approach of creating dedicated `TryThunk`, `TryTrampoline`, and `TryLazy` types that wrap their base counterparts with `Result` inside:

- `TryThunk<'a, A, E>` wraps `Thunk<'a, Result<A, E>>` (try_thunk.rs, line 63-66)
- `TryTrampoline<A, E>` wraps `Trampoline<Result<A, E>>` (try_trampoline.rs, line 38-41)
- `TryLazy<'a, A, E, Config>` wraps `Config::TryLazy<'a, A, E>` (try_lazy.rs, line 44-50)

**Assessment:** This is the right approach. The alternative, using `Thunk<'a, Result<A, E>>` directly, would force users to manually unwrap `Result` at every chaining step. The Try* types provide short-circuiting `bind` semantics where an `Err` propagates without executing the continuation. This mirrors the EitherT monad transformer pattern from Haskell, which is well-established.

The newtype approach also enables ergonomic methods like `map_err`, `catch`, `ok`, and `err` that would be awkward on the raw `Thunk<Result<...>>` type. Additionally, it allows separate HKT branding: `TryThunkErrAppliedBrand<E>` gives a proper `Functor`/`Monad` over the success type with error short-circuiting.

However, `TryLazy` diverges from the other two in its internal representation. Rather than wrapping `Lazy<'a, Result<A, E>, Config>`, it uses `Config::TryLazy<'a, A, E>`, a separate associated type on `LazyConfig`. This is necessary because `TryLazy::evaluate` returns `Result<&A, &E>` (references to the inner values), not `&Result<A, E>`. This is the correct design for memoization since it allows borrowing the cached Ok or Err value independently. But it means `TryLazy` is structurally different from the other two Try* types.

## 2. Code Duplication Between Base and Try* Variants

There is significant structural duplication across the Try* types compared to their base types.

### catch_unwind Duplication

The `catch_unwind` method is identically implemented in both `TryThunk` (try_thunk.rs, lines 327-339) and `TryLazy` (try_lazy.rs, lines 364-376). The panic-to-string conversion logic is copied verbatim:

```rust
std::panic::catch_unwind(f).map_err(|e| {
    if let Some(s) = e.downcast_ref::<&str>() {
        s.to_string()
    } else if let Some(s) = e.downcast_ref::<String>() {
        s.clone()
    } else {
        "Unknown panic".to_string()
    }
})
```

**Notably missing:** `TryTrampoline` has no `catch_unwind` method at all. This is an inconsistency.

**Recommendation:** Extract the panic-to-string conversion into a shared helper function and add `catch_unwind` to `TryTrampoline`.

### Constructor Pattern Duplication

All three types repeat the same constructor patterns (`new`, `ok`, `err`), but this is acceptable since each has distinct type signatures and constraints.

### From Impl Duplication

Each Try* type implements `From` conversions from the other types and from `Result`. The patterns are consistent but the implementations necessarily differ due to type constraints (e.g., `TryTrampoline` requires `'static`, `TryLazy` must clone through references).

## 3. Error Handling Consistency

### Short-circuiting Semantics

`TryThunk::bind` (try_thunk.rs, lines 195-203) and `TryTrampoline::bind` (try_trampoline.rs, lines 212-220) both implement identical short-circuiting logic:

```rust
// Pattern in both:
match result {
    Ok(a) => f(a).0,
    Err(e) => BaseType::pure(Err(e)),
}
```

This is consistent and correct.

### TryLazy Has No bind/map/map_err

`TryLazy` provides only `evaluate` (try_lazy.rs, line 106), `ok`, `err`, and `catch_unwind`. It has no `bind`, `map`, `map_err`, or `catch` methods. This is a significant gap compared to `TryThunk` and `TryTrampoline`, which both provide the full set.

This omission is somewhat justified by `TryLazy`'s memoization semantics: mapping over a memoized value would need to either create a new `TryLazy` (losing the shared cache) or return a different type. But it still hurts API consistency and composability.

### catch Method

`TryThunk` has `catch` (try_thunk.rs, line 268) and `TryTrampoline` has `catch` (try_trampoline.rs, line 238). Both follow the same pattern: on `Err(e)`, apply the recovery function; on `Ok(a)`, pass through. `TryLazy` has no `catch`.

### evaluate Return Types Differ

- `TryThunk::evaluate(self) -> Result<A, E>` (consumes self, returns owned; try_thunk.rs, line 291)
- `TryTrampoline::evaluate(self) -> Result<A, E>` (consumes self, returns owned; try_trampoline.rs, line 450)
- `TryLazy::evaluate(&self) -> Result<&A, &E>` (borrows self, returns references; try_lazy.rs, line 106)

This difference is inherent to memoization (you cannot give away the cached value), but it creates friction when composing the three types together. Converting from `TryLazy` to the others requires `Clone` bounds because of this (see try_thunk.rs line 394: `memo.evaluate().cloned().map_err(Clone::clone)`).

## 4. Composition Between Try* Types (Workflow Pattern)

The README (lines 194-246) shows the intended workflow:

1. `TryTrampoline` for stack-safe fallible recursion.
2. `TryLazy` (via `RcTryLazy::new(...)`) to memoize the result.
3. `TryThunk` to create lightweight deferred views.

### From Conversions Enable the Pipeline

The conversion graph is comprehensive:

**TryThunk** accepts `From`:
- `TryLazy<'a, A, E, Config>` (line 376, requires `A: Clone, E: Clone`)
- `Thunk<'a, A>` (line 403, wraps in `Ok`)
- `Lazy<'a, A, Config>` (line 348, wraps in `Ok`)
- `Result<A, E>` (line 425)

**TryTrampoline** accepts `From`:
- `TryThunk<'static, A, E>` (line 532)
- `Trampoline<A>` (line 456, wraps in `Ok`)
- `Lazy<'static, A, Config>` (line 482, wraps in `Ok`)
- `TryLazy<'static, A, E, Config>` (line 509, requires `A: Clone, E: Clone`)
- `Result<A, E>` (line 554)

**TryLazy** accepts `From`:
- `TryThunk<'a, A, E>` (line 184)
- `TryTrampoline<A, E>` (line 208, requires `A: Send, E: Send`)
- `Lazy<'a, A, RcLazyConfig>` (line 262, requires `A: Clone`)
- `Lazy<'a, A, ArcLazyConfig>` (line 236, requires `A: Clone + Send + Sync, E: Send + Sync`)
- `Result<A, E>` for both Rc (line 288) and Arc (line 312) variants

### Gap: TryTrampoline to TryLazy Requires Send

`From<TryTrampoline<A, E>> for TryLazy<'a, A, E, RcLazyConfig>` (try_lazy.rs, line 208) requires `A: Send, E: Send`. This is surprising for an `RcLazyConfig` (single-threaded) variant. The `Send` bound likely comes from `TryTrampoline` internally, but it restricts the composition unnecessarily. It means you cannot memoize a `TryTrampoline<Rc<T>, E>` (a `!Send` type) in a single-threaded `RcTryLazy`.

### Gap: No TryThunk to TryTrampoline Directly Via Inherent Method

While `From<TryThunk<'static, A, E>>` exists for `TryTrampoline`, there is no `memoize` method on any Try* type. Both `Thunk` (thunk.rs, line 268) and `Trampoline` (trampoline.rs, line 271) have `memoize()` methods that return `Lazy`. The Try* variants lack this convenience.

### Inconsistencies

1. **`pure` vs `ok`:** `TryThunk` has both `pure` and `ok` (line 151: "`ok` is alias for `pure`"). `TryTrampoline` and `TryLazy` have only `ok`. This is inconsistent. `TryThunk` having `pure` makes sense for HKT compatibility (it implements `Pointed`), but `TryTrampoline` does not implement `Pointed`, so `ok` alone is fine. The inconsistency is minor.

2. **`lift2` and `then`:** `TryTrampoline` provides these as inherent methods (lines 272, 303). `TryThunk` has them only through HKT trait implementations (`Lift for TryThunkErrAppliedBrand<E>`). Users of `TryThunk` who want `lift2` must use the free function form with brand type parameters, which is much less ergonomic.

3. **`defer`:** Present on `TryThunk` and `TryTrampoline` but missing from `TryLazy`. For `TryLazy`, `Deferrable::defer` is implemented as a trait (try_lazy.rs, line 486), so it is available via the trait, just not as an inherent method.

4. **`catch_unwind`:** Present on `TryThunk` and `TryLazy` but missing from `TryTrampoline`. Since `TryTrampoline` is specifically for stack-safe deep recursion (where panics from stack overflow are a real concern), this is a notable gap.

## 6. Missing From/Into Conversions and Traits

### Missing memoize Methods

Neither `TryThunk` nor `TryTrampoline` provides a `memoize()` convenience method to convert to `TryLazy`, analogous to `Thunk::memoize()` and `Trampoline::memoize()`. Users must write `TryLazy::from(try_thunk)` instead of `try_thunk.memoize()`.

### Missing HKT Implementations for TryTrampoline and TryLazy

- **TryTrampoline** has zero HKT trait implementations. No `impl_kind!`, no `Functor`, no `Pointed`, no `Semimonad`, no `Foldable`. This is a significant gap. While the `'static` constraint complicates HKT integration (similar to `Trampoline`), the base `Trampoline` at least has `Deferrable`. `TryTrampoline` has `Deferrable` (line 578) but nothing else.

- **TryLazy** has `impl_kind!` for `TryLazyBrand<E, Config>` (try_lazy.rs, line 493) and `Deferrable`/`SendDeferrable`, but no `Functor`, `Foldable`, or other type class implementations. This limits its utility in generic HKT code.

- **TryThunk** has the richest HKT support: `Functor`, `Pointed`, `Lift`, `Semiapplicative`, `Semimonad`, `MonadRec`, `Foldable`, `Bifunctor`, `Bifoldable`, `Semigroup`, `Monoid`, plus three brand types (`TryThunkBrand`, `TryThunkErrAppliedBrand<E>`, `TryThunkOkAppliedBrand<A>`).

### Missing From Conversions

- No `From<TryThunk<'a, A, E>> for TryTrampoline<A, E>` when `A` is not `'static`. The existing conversion (try_trampoline.rs, line 532) requires `'static`, which is correct for Trampoline's constraints, but means there is no way to upgrade a short-lived `TryThunk` to a `TryTrampoline`.

- No `From<TryTrampoline<A, E>> for TryThunk<'static, A, E>`. Users can go TryThunk -> TryTrampoline but not the reverse, even though `TryTrampoline::evaluate` returns an owned `Result`. Adding this would complete the bidirectional conversion.

## 7. Missing Functionality in Try* Variants

### Compared to Base Types

| Feature | Base Has | Try* Lacks |
|---------|----------|-----------|
| `memoize()` | `Thunk`, `Trampoline` | All Try* variants |
| `catch_unwind` | N/A (not applicable to infallible) | `TryTrampoline` |
| Inherent `lift2` | `Trampoline` (line 297) | `TryThunk` (only HKT-level) |
| Inherent `then` | `Trampoline` (line 326) | `TryThunk` (missing entirely) |
| `RefFunctor` | `Lazy` | `TryLazy` |
| Full HKT suite | `Thunk` (Functor through MonadRec) | `TryTrampoline`, `TryLazy` |

### Unique to Try* (Correctly)

- `catch` (error recovery) is present on both `TryThunk` and `TryTrampoline`. This is a valuable addition.
- `map_err` is present on both `TryThunk` and `TryTrampoline`.
- `ok`/`err` constructors are present on all three.
- `catch_unwind` on `TryThunk` and `TryLazy`.
- `Bifunctor`/`Bifoldable` on `TryThunk` (via `TryThunkBrand`).

## 8. Error Type Parameter Handling

### Consistent Patterns

The error type `E` is consistently:
- A second type parameter after `A` in all three types.
- Required to be `'static` only in `TryTrampoline` (line 38: `E: 'static`), matching `Trampoline`'s `A: 'static` requirement.
- Allowed to be any lifetime `'a` in `TryThunk` and `TryLazy`.

### Brand Encoding

The HKT brand encoding for `TryThunk` handles the error type in three ways:
- `TryThunkBrand` (2-kinded, for Bifunctor): `Of<'a, E, A> = TryThunk<'a, A, E>` (note the swap of E and A; try_thunk.rs, line 960).
- `TryThunkErrAppliedBrand<E>` (1-kinded, error fixed): `Of<'a, A> = TryThunk<'a, A, E>` (line 482).
- `TryThunkOkAppliedBrand<A>` (1-kinded, success fixed): `Of<'a, E> = TryThunk<'a, A, E>` (line 1200).

For `TryLazy`, `TryLazyBrand<E, Config>` (try_lazy.rs, line 493-498) fixes the error type and config, making it a functor over the success type.

`TryTrampoline` has no brand at all, which means it cannot participate in any generic HKT code.

### Clone Requirements on Error Type

When converting between types that return references (`TryLazy`) and those that return owned values (`TryThunk`, `TryTrampoline`), `E: Clone` is required. This is handled correctly:
- `From<TryLazy<...>> for TryThunk<...>` requires `E: Clone` (try_thunk.rs, line 379).
- `From<TryLazy<...>> for TryTrampoline<...>` requires `E: Clone` (try_trampoline.rs, line 512).
- `TryLazy::Deferrable::defer` requires `E: Clone` (try_lazy.rs, line 489).

This is correct but worth noting as a usability cost: error types that are not `Clone` cannot flow through certain conversions.

## Summary of Recommendations

1. **Add `memoize()` to `TryThunk` and `TryTrampoline`** returning `TryLazy`, for parity with base types.
2. **Add `catch_unwind` to `TryTrampoline`** for consistency and because stack overflow panics are a real concern in its use case.
3. **Add inherent `lift2` and `then` methods to `TryThunk`** for ergonomic parity with `TryTrampoline`.
4. **Extract the panic-to-string logic** from duplicated `catch_unwind` implementations into a shared utility.
5. **Add `From<TryTrampoline<A, E>> for TryThunk<'static, A, E>`** to complete bidirectional conversion.
6. **Investigate the `Send` bound on `From<TryTrampoline> for RcTryLazy`** (try_lazy.rs, line 208). If `TryTrampoline` does not inherently require `Send`, this bound may be unnecessarily restrictive for single-threaded use.
7. **Consider adding `map`/`map_err` to `TryLazy`** that return new `TryLazy` instances (creating a new lazy cell that transforms the result of the original). This would improve composability at the cost of one extra indirection.
8. **Consider adding a brand for `TryTrampoline`** so it can participate in HKT-generic code. Even a minimal `TryTrampolineBrand<E>` with `Functor` and `Pointed` would be valuable.
