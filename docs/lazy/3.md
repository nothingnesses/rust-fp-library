# Analysis 3: Stack Safety Analysis

## Overview

This document analyzes the stack safety properties of the Lazy hierarchy in the fp-library crate, covering `Trampoline`, `Thunk`, `TryTrampoline`, `TryThunk`, and the underlying `Free` monad. The analysis examines how stack safety is achieved (or not), the correctness of `tail_rec_m` implementations, edge cases, and comparison with PureScript's approach.

---

## 1. How Trampoline Achieves Stack Safety

### The Free Monad Foundation

`Trampoline<A>` is a newtype wrapper around `Free<ThunkBrand, A>` (line 72-75, `trampoline.rs`). The stack safety guarantee comes entirely from the `Free` monad's evaluate loop.

### The Trampoline Loop (`Free::evaluate`, lines 419-467, `free.rs`)

The core mechanism is an iterative loop that processes the `Free` computation without recursion:

```
loop {
    match inner {
        Pure(val) => { apply next continuation or return }
        Wrap(fa)  => { evaluate the functor layer to get next Free }
        Bind { head, continuations } => { merge continuations, continue with head }
    }
}
```

Key properties:

1. **Pure(val)** (line 432): If there are remaining continuations, pop the next one from the `CatList` and apply it to get a new `Free` to process. If no continuations remain, downcast and return the final value.

2. **Wrap(fa)** (line 450-452): Calls `<F as Evaluable>::evaluate(fa)` which, for `ThunkBrand`, simply invokes the thunk's closure (line 625-629, `thunk.rs`). This produces the next `Free` value to process. Crucially, this is a single function call that returns a `Free`; it does not recurse.

3. **Bind { head, continuations }** (line 455-464): Merges inner continuations with outer continuations via `CatList::append` (O(1)), then continues processing the head. This is the "Reflection without Remorse" optimization that prevents quadratic blowup.

The loop never recurses. Each iteration processes one layer of the `Free` structure. The `CatList` of continuations lives on the heap, so bind chains of arbitrary depth are handled without stack growth.

### O(1) Bind via CatList

`Free::bind` (lines 290-340, `free.rs`) appends to the `CatList` in O(1) via `snoc`. For the `Bind` case (line 330-338), the new continuation is appended to the existing list. This prevents the left-associated bind degradation that plagues naive free monad implementations.

### Stack-Safe Drop

The `Free::drop` implementation (lines 472-512, `free.rs`) iteratively walks the `Bind` chain rather than relying on recursive drop:

```rust
while let Some(FreeInner::Bind { mut head, .. }) = current {
    current = head.0.take();
}
```

This prevents stack overflow when dropping deeply nested `Free` computations. The test `test_free_drop_safety` (line 610-618) verifies this for 100,000 levels.

## 2. Thunk Stack Safety (or Lack Thereof)

### Where Thunk is NOT Stack Safe

**`bind` chains** (lines 193-202, `thunk.rs`): Thunk's bind is implemented as nested closure composition:

```rust
pub fn bind<B: 'a>(self, f: impl FnOnce(A) -> Thunk<'a, B> + 'a) -> Thunk<'a, B> {
    Thunk::new(move || {
        let a = (self.0)();
        let thunk_b = f(a);
        (thunk_b.0)()
    })
}
```

Each `bind` creates a new closure that captures the previous thunk. When `evaluate` is called, the entire chain unwinds through nested closures on the call stack. For N binds, this requires O(N) stack frames. This WILL overflow for sufficiently deep chains (roughly 10,000-50,000 depending on frame size and stack limit).

**`defer`** (lines 161-163, `thunk.rs`): Similarly, `defer` is just:

```rust
pub fn defer(f: impl FnOnce() -> Thunk<'a, A> + 'a) -> Self {
    Thunk::new(move || f().evaluate())
}
```

Deeply nested `defer` chains also grow the stack linearly.

**`map`** (lines 224-229, `thunk.rs`): Same pattern; each `map` nests another closure.

### Where Thunk IS Stack Safe

**`tail_rec_m`** (lines 581-596, `thunk.rs`): The `MonadRec` implementation for `ThunkBrand` uses an imperative loop:

```rust
Thunk::new(move || {
    let mut current = a;
    loop {
        match f(current).evaluate() {
            Step::Loop(next) => current = next,
            Step::Done(res) => break res,
        }
    }
})
```

This is genuinely stack-safe. Each iteration of the loop evaluates the step function and either continues looping or returns. No recursion, no stack growth. The loop runs in constant stack space.

### Documentation Accuracy

The documentation is **accurate and clear** on this point:

- The module-level doc (line 3, `thunk.rs`): "Builds computation chains without stack safety guarantees."
- The struct doc table (lines 56-62): Shows "Stack-safe: Partial (tail_rec_m only)" for Thunk vs "Yes (unlimited)" for Trampoline.
- The `bind` doc (line 167-168): "Note: Each bind adds to the call stack. For deep recursion, use Trampoline instead."
- The `MonadRec` trait doc (lines 47-49, `monad_rec.rs`): "Thunk's tail_rec_m implementation uses a loop and is stack-safe. However, Thunk's bind chains are NOT stack-safe."

This is well-documented throughout. No gaps found.

## 3. `tail_rec_m` Implementations: Correctness Analysis

### Trampoline::tail_rec_m (lines 377-398, `trampoline.rs`)

```rust
fn go<A, B, F>(f: F, a: A) -> Trampoline<B>
where F: Fn(A) -> Trampoline<Step<A, B>> + Clone + 'static {
    let f_clone = f.clone();
    Trampoline::defer(move || {
        f(a).bind(move |step| match step {
            Step::Loop(next) => go(f_clone.clone(), next),
            Step::Done(b) => Trampoline::pure(b),
        })
    })
}
```

**Stack safety**: The `Trampoline::defer` wraps the recursive call in a `Free::wrap(Thunk::new(...))`. The actual recursive call to `go` happens inside the thunk closure, which is only invoked during evaluation. At evaluation time, the `Free` trampoline loop processes each `defer`/`bind` layer iteratively. So even though `go` appears recursive, each call only builds a `Free` structure; it does not evaluate it. The evaluation loop in `Free::evaluate` handles the actual execution iteratively.

However, there is a subtlety: the `go` function itself IS called recursively during evaluation, because each `Trampoline::defer` creates a `Thunk::new(move || ...)`, and when the trampoline loop evaluates that thunk, it calls `go(f_clone, next)`. This builds a new `Trampoline::defer(...)` which returns a `Free::wrap(...)`. The trampoline loop then processes this `Wrap` by calling the thunk again...

Wait, let me trace this more carefully. `go(f, a)` returns:
1. `Trampoline::defer(closure)` which is `Trampoline(Free::wrap(Thunk::new(closure)))`
2. When the trampoline evaluates the `Wrap`, it calls `<ThunkBrand as Evaluable>::evaluate(thunk)` which executes the thunk closure.
3. The closure calls `f(a)` to get a `Trampoline<Step<A, B>>`, then `.bind(...)` on it.
4. The `bind` closure, when eventually applied to the step, calls `go(f_clone, next)` which returns immediately with a new `Trampoline::defer(...)`.
5. The returned value is a new `Free` structure, not a recursion into `go`'s body.

So `go` is called once per iteration but returns immediately (building a `Free` node). The stack depth of `go` itself is O(1) per step. The concern would be if `go` were called recursively before returning, but it is not: the recursive call to `go` inside the `bind` closure only executes when the trampoline loop reaches that continuation, at which point the previous `go` call has already returned.

**Correctness**: The implementation correctly models tail-recursive monadic computation. Each `Step::Loop` produces a new `defer`/`bind` that the trampoline loop processes. Each `Step::Done` terminates via `Trampoline::pure`. The semantics match the expected `tailRecM` from PureScript/Haskell.

**Clone concern**: `f` is cloned at each step (`f_clone = f.clone()`, then `f_clone.clone()` inside the bind). This is necessary because each recursive `go` call needs its own copy. For closures capturing large state, this could be expensive. The `arc_tail_rec_m` variant (lines 447-458) addresses this by wrapping in `Arc`.

### ThunkBrand::tail_rec_m (lines 581-596, `thunk.rs`)

```rust
Thunk::new(move || {
    let mut current = a;
    loop {
        match f(current).evaluate() {
            Step::Loop(next) => current = next,
            Step::Done(res) => break res,
        }
    }
})
```

**Correctness**: This is straightforward and correct. It is a simple loop that repeatedly applies `f` to the state until `Done` is reached. Stack-safe by construction.

**Potential issue**: `f(current).evaluate()` creates a `Thunk<Step<A, B>>` and immediately evaluates it. If `f` returns a deeply composed thunk (e.g., `f(x) = Thunk::pure(Step::Loop(...)).bind(...).bind(...)...`), that inner evaluation could still overflow the stack. In practice, `f` typically returns `Thunk::pure(...)`, so this is not an issue for normal usage. But the guarantee is only as strong as the step function's depth.

### TryThunkErrAppliedBrand::tail_rec_m (lines 734-751, `try_thunk.rs`)

```rust
TryThunk::new(move || {
    let mut current = a;
    loop {
        match f(current).evaluate() {
            Ok(Step::Loop(next)) => current = next,
            Ok(Step::Done(res)) => break Ok(res),
            Err(e) => break Err(e),
        }
    }
})
```

**Correctness**: Correctly handles the three cases: continue looping, return success, or short-circuit on error. Stack-safe by the same reasoning as Thunk's `tail_rec_m`.

### TryTrampoline::tail_rec_m (lines 360-375, `try_trampoline.rs`)

```rust
TryTrampoline(Trampoline::tail_rec_m(
    move |state: Result<S, E>| match state {
        Err(e) => Trampoline::pure(Step::Done(Err(e))),
        Ok(s) => f(s).0.map(|result| match result {
            Ok(Step::Loop(next)) => Step::Loop(Ok(next)),
            Ok(Step::Done(a)) => Step::Done(Ok(a)),
            Err(e) => Step::Done(Err(e)),
        }),
    },
    Ok(initial),
))
```

**Correctness**: This lifts the fallible recursion into Trampoline by threading `Result<S, E>` as the state type. The mapping correctly translates between `TryTrampoline<Step<S, A>, E>` and `Step<Result<S, E>, Result<A, E>>`. Error short-circuits by producing `Step::Done(Err(e))`. This is correct.

**Stack safety**: Delegates to `Trampoline::tail_rec_m`, so inherits its full stack safety.

## 4. Bind/Flat_map Chain Stack Safety Summary

| Type | `bind` chains | `map` chains | `defer` chains | `tail_rec_m` |
|------|--------------|-------------|---------------|-------------|
| `Trampoline` | Stack-safe (Free trampoline loop) | Stack-safe (implemented via `bind`) | Stack-safe (Free `Wrap` nodes) | Stack-safe (defer + bind) |
| `Thunk` | NOT stack-safe (nested closures) | NOT stack-safe (nested closures) | NOT stack-safe (nested closures) | Stack-safe (imperative loop) |
| `TryTrampoline` | Stack-safe (delegates to Trampoline) | Stack-safe (delegates to Trampoline) | Stack-safe (delegates to Trampoline) | Stack-safe (delegates to Trampoline) |
| `TryThunk` | NOT stack-safe (delegates to Thunk) | NOT stack-safe (delegates to Thunk) | NOT stack-safe (delegates to Thunk) | Stack-safe (imperative loop) |

## 5. Evaluate Method Correctness

### Free::evaluate (lines 419-467, `free.rs`)

The implementation is correct and handles all three `FreeInner` variants:

- **Pure**: Either applies the next continuation or returns the final value via `downcast`.
- **Wrap**: Evaluates the functor layer via the `Evaluable` trait, producing the next `Free` to process.
- **Bind**: Merges continuation lists and continues with the head computation.

The `Option<FreeInner<F, A>>` wrapper (line 171) with `.take()` ensures each node is consumed exactly once, preventing use-after-move issues. The `.expect()` calls on `take()` and `downcast()` are internal invariant assertions, not user-facing error paths; they indicate bugs if triggered.

**Potential concern**: The `erase_type` call at line 423 (`let mut current = self.erase_type()`) processes the top-level Free node. For a `Wrap` variant, this calls `F::map(|inner| inner.erase_type(), fa)` (line 367), which for `ThunkBrand` creates a new `Thunk` that will call `erase_type` on the inner `Free` when evaluated. This is a single closure layer, not recursive, so it is safe.

### Trampoline::evaluate (line 248-250, `trampoline.rs`)

Simply delegates to `self.0.evaluate()`, which is `Free::evaluate`. Correct.

### Thunk::evaluate (lines 247-249, `thunk.rs`)

Simply calls the boxed closure: `(self.0)()`. Correct and minimal. The stack safety depends entirely on what the closure does.

### TryTrampoline::evaluate (lines 450-452, `try_trampoline.rs`)

Delegates to `self.0.evaluate()` where `self.0` is `Trampoline<Result<A, E>>`. Correct.

### TryThunk::evaluate (lines 291-293, `try_thunk.rs`)

Delegates to `self.0.evaluate()` where `self.0` is `Thunk<'a, Result<A, E>>`. Correct.

## 6. Edge Cases

### Deeply Nested Defers

For Trampoline, deeply nested defers are stack-safe. The test `test_deep_defer_chain` in `stack_safety.rs` (lines 49-60) verifies 100,000 nested defers with maps. Each defer becomes a `Free::Wrap(Thunk::new(...))`, and the trampoline loop iteratively processes each layer.

For Thunk, `defer` is `Thunk::new(move || f().evaluate())`. Deeply nested defers would create a closure chain where each closure calls the next, overflowing the stack. There is no test for this because it is a known limitation.

### Mutual Recursion

Mutual recursion (e.g., `f` calls `g` which calls `f`) is stack-safe in Trampoline if both functions return `Trampoline::defer(...)` or use `bind`. The Free trampoline loop will iteratively process the alternating `Wrap`/`Bind` nodes. No explicit test exists for this pattern, but the mechanism supports it by construction.

For Thunk, mutual recursion is NOT stack-safe for the same reasons bind chains are not.

### Error Paths in Try* Variants

**TryTrampoline::bind** (lines 212-220, `try_trampoline.rs`):
```rust
TryTrampoline(self.0.bind(|result| match result {
    Ok(a) => f(a).0,
    Err(e) => Trampoline::pure(Err(e)),
}))
```
On error, returns `Trampoline::pure(Err(e))`, which is a `Free::Pure`. The trampoline loop will immediately handle this as a pure value and apply the next continuation (or return). This is stack-safe and correct. Error propagation does not build up additional Free layers.

**TryTrampoline::catch** (lines 238-246, `try_trampoline.rs`):
```rust
self.0.bind(|result| match result {
    Ok(a) => Trampoline::pure(Ok(a)),
    Err(e) => f(e).0,
})
```
Recovery delegates to the user's function `f`, which produces a new `TryTrampoline`. This is safe as long as `f` does not itself deeply recurse without trampolining.

**TryThunk::bind** (lines 195-203, `try_thunk.rs`):
On error, returns `Thunk::pure(Err(e))`, which is evaluated immediately. This does not add stack depth beyond the bind chain itself.

**TryThunk::catch_unwind** (lines 327-339, `try_thunk.rs`):
Uses `std::panic::catch_unwind` to capture panics. Correctly converts `&str` and `String` panic payloads. Falls back to "Unknown panic" for other types. This is correct but only works with `String` error type.

### Type Erasure Downcast Failures

The `Free` monad uses `Box<dyn Any>` for type erasure. Downcast failures (lines 298, 443-445 in `free.rs`) trigger `expect` panics with "Type mismatch" messages. These should never occur in correct usage because the type is maintained by internal invariant. However, if there were a bug in the continuation chain (e.g., mismatched types in the `CatList`), this would panic rather than produce incorrect results. This is the correct behavior for an internal invariant violation.

## 7. Free Monad Encoding Correctness

The `Free` monad in this library (defined in `free.rs`) follows the "Reflection without Remorse" paper. The encoding uses three variants:

1. `Pure(A)` for completed computations.
2. `Wrap(F<Free<F, A>>)` for suspended computations in functor `F`.
3. `Bind { head, continuations }` for computation + continuation queue.

This matches the standard free monad encoding with the CatList optimization. The key differences from PureScript's implementation (as noted in the module doc, lines 6-62):

- **No `resume`**: Cannot inspect computation step-by-step.
- **No `hoistFree`**: Cannot transform the base functor.
- **No `foldFree`/`runFree`**: Cannot interpret into an arbitrary target monad. The interpretation is fixed by the `Evaluable` trait.

These limitations are by design: the library prioritizes stack-safe execution over generic interpretation. The `Evaluable` trait bakes interpretation into the functor, which is sufficient for the trampolining use case.

The `lift_f` implementation (lines 265-268) correctly uses `F::map(Free::pure, fa)` followed by `Free::wrap`, matching the standard definition `liftF fa = wrap (map pure fa)`.

The bind implementation (lines 290-340) correctly handles all three variants:
- `Pure`: Creates a `Bind` with a singleton continuation.
- `Wrap`: Wraps in a `Bind` with a singleton continuation.
- `Bind`: Appends (snocs) the new continuation to the existing `CatList`.

This is correct and matches the "Reflection without Remorse" algorithm.

## 8. Comparison with PureScript

### PureScript's Data.Lazy (purescript-lazy)

PureScript's `Lazy` type is implemented as a foreign type with JavaScript runtime memoization:
- `defer :: (Unit -> a) -> Lazy a` creates a lazy value.
- `force :: Lazy a -> a` evaluates and caches.
- The `Bind` instance (line 129): `bind l f = defer \_ -> force $ f (force l)` is NOT stack-safe for deep bind chains, similar to Rust's `Thunk`.

The Rust `Lazy` type corresponds most closely to PureScript's `Data.Lazy`, with the same memoization semantics.

### PureScript's Control.Lazy (purescript-control)

The `Lazy` typeclass (line 10): `class Lazy l where defer :: (Unit -> l) -> l` is a capability for deferred construction, matching Rust's `Deferrable` trait.

The `fix` combinator (lines 22-25):
```purescript
fix f = go where go = defer \_ -> f go
```
This relies on JavaScript's lazy evaluation for the self-reference. There is no direct equivalent in the Rust library due to ownership constraints; Rust cannot easily create self-referential lazy values without `Rc`/`Arc` and interior mutability.

### Key Differences

1. **Stack safety strategy**: PureScript uses `purescript-free` and `purescript-tailrec` for stack-safe recursion, with `MonadRec` as the primary mechanism. The Rust library provides both `Trampoline` (stack-safe for everything) and `Thunk` (stack-safe only via `tail_rec_m`), making the tradeoffs explicit.

2. **Memoization**: PureScript's `Lazy` always memoizes (via JS runtime). Rust separates memoization (`Lazy`) from deferred computation (`Thunk`/`Trampoline`), giving users explicit control.

3. **Bind stack safety**: PureScript's `Lazy` bind (`defer \_ -> force $ f (force l)`) is not inherently stack-safe, same as Rust's `Thunk`. PureScript users would use `Free` or `Aff` for stack-safe monadic chains. Rust users would use `Trampoline`.

4. **HKT compatibility**: PureScript's `Lazy` implements `Functor`, `Monad`, etc. naturally. In Rust, `Thunk` implements HKT traits but `Trampoline`/`Free` cannot due to the `'static` requirement from `Box<dyn Any>`. This is a fundamental tradeoff documented at lines 143-165 in `free.rs`.

5. **Lifetime support**: PureScript has no lifetime concept. Rust's `Thunk<'a, A>` supports borrowing, while `Trampoline<A>` requires `'static`. This is the primary motivation for keeping both types.

## 9. Summary of Findings

### Correct and Well-Designed

1. The `Free` monad trampoline loop is correctly implemented with O(1) bind and guaranteed stack safety.
2. The `Free::drop` implementation prevents stack overflow on deep structures.
3. All four `tail_rec_m` implementations are correct and stack-safe.
4. Error propagation in `TryTrampoline` and `TryThunk` is correct and does not break stack safety guarantees.
5. Documentation accurately describes stack safety properties throughout.

### Potential Concerns

1. **Trampoline::tail_rec_m cloning overhead**: Each iteration clones `f`. For closures capturing large state, this could be a performance concern. The `arc_tail_rec_m` variant mitigates this but adds `Arc` overhead. This is a reasonable tradeoff, not a bug.

2. **No mutual recursion test**: While the mechanism supports it, there are no explicit tests for mutually recursive trampolined functions. Consider adding one.

3. **No stack overflow test for Thunk::bind**: The test suite verifies Trampoline's stack safety extensively (in `stack_safety.rs`) but does not have a negative test demonstrating that Thunk bind chains DO overflow. Such a test would need careful construction (e.g., `#[should_panic]` or conditional on stack size), but its absence means the documented limitation is not regression-tested.

4. **Free::evaluate expects on downcast**: The `.expect("Type mismatch in Free::bind")` at line 298 and `.expect("Type mismatch in Free::evaluate final downcast")` at line 444-445 will panic on internal bugs. These are correct internal invariant checks, but they produce unclear error messages for users who might encounter them in release builds. Consider using `debug_assert` or providing more context.

5. **TryTrampoline `From<TryLazy>` eagerly evaluates**: At line 527, `From<TryLazy<...>> for TryTrampoline` calls `memo.evaluate()` eagerly during conversion, wrapping the result in `Trampoline::pure(...)`. This means the lazy value is forced at conversion time, not deferred. This is correct (the result is memoized anyway) but may surprise users expecting deferred evaluation. The same pattern appears in `From<Lazy> for TryTrampoline` at line 500.
