# Analysis 1: Overall Design Evaluation of the Lazy Hierarchy

## 1. Summary of Current Architecture

The library provides six types organized as a 3x2 matrix:

|              | Infallible       | Fallible (`Result`)    |
|:-------------|:-----------------|:-----------------------|
| **Thunk**    | `Thunk<'a, A>`   | `TryThunk<'a, A, E>`  |
| **Trampoline** | `Trampoline<A>` | `TryTrampoline<A, E>` |
| **Lazy**     | `Lazy<'a, A, Config>` | `TryLazy<'a, A, E, Config>` |

Each infallible type has a `Try*` counterpart that is a thin newtype around the base type wrapping a `Result`. Each of `Lazy` and `TryLazy` is further parameterized by a `LazyConfig` trait that selects between `RcLazyConfig` (single-threaded, `Rc<LazyCell>`) and `ArcLazyConfig` (thread-safe, `Arc<LazyLock>`).

Supporting type classes:
- `Deferrable<'a>` (deferrable.rs): `defer(|| x)` is observationally equivalent to `x`.
- `SendDeferrable<'a>` (send_deferrable.rs): Thread-safe variant requiring `Send + Sync` on the thunk.
- `Evaluable` (evaluable.rs): Functor whose effects can be evaluated; used by `Free::evaluate`.

## 2. Evaluation of the Three-Type Split

### 2.1 The Fundamental Tension

Rust forces the decomposition that Haskell and PureScript can avoid because of three orthogonal concerns that cannot be unified under Rust's ownership and lifetime rules:

1. **Lifetime polymorphism vs. type erasure.** `Thunk<'a, A>` supports borrowing with arbitrary lifetimes. `Trampoline<A>` requires `'static` because it is built on `Free<ThunkBrand, A>`, which uses `Box<dyn Any>` internally for type erasure in the CatList-based bind stack.

2. **Stack safety vs. zero overhead.** `Thunk` is a simple `Box<dyn FnOnce() -> A>` (thunk.rs, line 89). Its `bind` eagerly composes closures on the call stack (lines 197-202: `(self.0)()` then `(thunk_b.0)()`). `Trampoline` is `Free<ThunkBrand, A>` (trampoline.rs, line 74) with O(1) bind via CatList, guaranteeing stack safety.

3. **Computation vs. memoization.** `Thunk` and `Trampoline` re-execute on each `evaluate()` call. `Lazy` wraps an `Rc<LazyCell>` or `Arc<LazyLock>` (lazy.rs, lines 178, 292) to compute at most once.

These three axes are genuinely independent in Rust. You cannot get stack-safe `Free`-based bind without `'static` (because `Box<dyn Any>` requires it). You cannot get memoization without shared ownership (because the cache must be shared across clones). You cannot get HKT trait implementations without lifetime polymorphism (because `Kind::Of<'a, A>` must be parameterized by `'a`).

**Verdict: The three-type decomposition is well-motivated and structurally necessary.** Combining any two of these types would either sacrifice a key property or require unsafe code to work around lifetime constraints.

### 2.2 The Try* Variants

The three `Try*` types are straightforward newtypes:
- `TryThunk<'a, A, E>` wraps `Thunk<'a, Result<A, E>>` (try_thunk.rs, line 65).
- `TryTrampoline<A, E>` wraps `Trampoline<Result<A, E>>` (try_trampoline.rs, line 41).
- `TryLazy<'a, A, E, Config>` wraps `Config::TryLazy<'a, A, E>` (try_lazy.rs, line 47).

These exist purely for ergonomics: `bind` short-circuits on `Err`, `map` operates on the success channel, `map_err` operates on the error channel. Without them, users would need to manually destructure `Result` at every step.

**Verdict: The Try* variants are justified.** They provide the same value as `EitherT` in Haskell or `MonadError` in Scala. The alternative (requiring users to use `Thunk<'a, Result<A, E>>` directly) would be significantly less ergonomic. The boilerplate cost is modest since each `Try*` type delegates to its base type.

## 3. Comparison with FP Literature

### 3.1 Haskell

Haskell's thunks are built into the runtime: every value is a pointer to either a computed value or an unevaluated closure. There is no user-visible distinction between "lazy" and "memoized" because the runtime handles both. The GHC runtime also provides stack-safe evaluation via "update frames."

This library necessarily decomposes what Haskell's runtime provides as a single mechanism into three explicit types. This is the correct approach for Rust, which has no lazy runtime.

### 3.2 PureScript

PureScript's `Control.Lazy` (purescript-control/src/Control/Lazy.purs) defines a type class:
```purescript
class Lazy l where
  defer :: (Unit -> l) -> l
```
This maps directly to the library's `Deferrable<'a>` trait (deferrable.rs, lines 41-64). PureScript's `Data.Lazy` (purescript-lazy/src/Data/Lazy.purs) is a memoized lazy type (FFI-backed) with `force`/`defer`, implementing full `Functor`, `Applicative`, `Monad`, `Comonad`, `Foldable`, `Traversable`. This corresponds closely to the library's `Lazy` type, though the Rust version only implements `RefFunctor` (not full `Functor`) because memoization returns `&A` rather than `A`.

PureScript does not separate Thunk from Trampoline because JavaScript is not stack-constrained in the same way (PureScript compiles to tail-call-optimized JS). The library's split is driven by Rust-specific constraints.

### 3.3 Scala's Cats `Eval`

Scala's Cats library provides a single `Eval[A]` ADT with three variants:
- `Now(a)`: Already computed.
- `Later(f)`: Memoized, computed at most once (like `Lazy`).
- `Always(f)`: Non-memoized, recomputed each time (like `Thunk`).

`Eval` also has a `FlatMap` constructor for stack-safe `flatMap` chains (like `Trampoline`). This unification into a single type is possible because:
1. Scala has no lifetime system, so there is no `'a` vs `'static` distinction.
2. The JVM garbage collector handles shared ownership, so no `Rc`/`Arc` distinction.
3. `Eval` erases the distinction at the type level, paying for unused features.

The library could theoretically define an `Eval`-like enum, but it would need to be parameterized by lifetime and config, and the `FlatMap` variant would force `'static` on all variants (since the bind stack requires type erasure). This would be strictly worse than the current design.

**Verdict: The Cats `Eval` approach is inappropriate for Rust.** The three-type split is the correct Rust-native decomposition of `Eval`'s three concerns.

### 3.4 Cats Effect `IO`

`IO[A]` combines laziness, stack safety, memoization (via `IO.memoize`), error handling, async execution, and resource management. It is far more than a lazy evaluation type; it is an effect system. The library's types occupy a lower level in the abstraction stack, providing the building blocks from which something like `IO` could be constructed. This is the right scope for a pure FP library.

## 4. Trade-off Analysis

### 4.1 Stack Safety vs. Lifetimes

The `Thunk` vs `Trampoline` split trades stack safety against lifetime flexibility:

| Property | Thunk | Trampoline |
|:---------|:------|:-----------|
| Lifetimes | `'a` (arbitrary) | `'static` only |
| Stack safety (bind) | No (grows stack) | Yes (O(1) via CatList) |
| Stack safety (tail_rec_m) | Yes | Yes |
| HKT traits | Full (Functor, Monad, etc.) | None |
| Internal representation | `Box<dyn FnOnce() -> A + 'a>` | `Free<ThunkBrand, A>` |

This is well-chosen. The `'static` requirement on `Trampoline` is unavoidable given that `Free` uses `Box<dyn Any>` for type erasure in its continuation stack. The documentation makes this trade-off clear (thunk.rs, lines 54-62; README lines 192-193).

One nuance worth noting: `Trampoline` does not implement any HKT traits (brands.rs, lines 192-193 comment: "Trampoline cannot implement HKT traits due to its 'static requirement"). This means you cannot write generic code that is polymorphic over Thunk and Trampoline. This is an inherent limitation, not a design flaw.

### 4.2 Memoization as a Separate Concern

Making `Lazy` a separate type rather than an option on `Thunk`/`Trampoline` is the right call:

1. `Lazy` returns `&A` from `evaluate()` (lazy.rs, line 485), while `Thunk` returns `A` by value (thunk.rs, line 247). These are fundamentally different evaluation semantics.
2. `Lazy` requires shared ownership (`Clone` via `Rc`/`Arc`), while `Thunk` is move-only (consumed on evaluate).
3. `Lazy` can only implement `RefFunctor` (mapping `&A -> B`), not full `Functor` (mapping `A -> B`), because it holds a reference to the cached value.

The `memoize()` methods on both `Thunk` (line 268) and `Trampoline` (line 271) provide a clean path from computation to cache: `thunk.memoize()` produces `Lazy<'a, A, RcLazyConfig>`.

### 4.3 The LazyConfig Trait

The `LazyConfig` trait (lazy.rs, lines 65-170) is a clever mechanism for abstracting over `Rc<LazyCell>` vs `Arc<LazyLock>`. It bundles four associated types and four methods, keeping the `Lazy` and `TryLazy` types generic over the pointer/synchronization strategy.

Strengths:
- Users can implement custom configs (e.g., `parking_lot`-based, async-aware).
- The `RcLazyConfig`/`ArcLazyConfig` split avoids doubling the number of types.
- Type aliases (`RcLazy`, `ArcLazy`, `RcTryLazy`, `ArcTryLazy`) provide ergonomic shorthand.

Weaknesses:
- The config also bundles `TryLazy` and `TryThunk` associated types (lines 70-76), which means the config trait knows about fallible evaluation. This couples concerns that could theoretically be separated.
- The trait has a `'static` bound (line 65: `pub trait LazyConfig: 'static`), which constrains how configs can be defined.

### 4.4 Conversion Network

The types form a rich conversion graph via `From` implementations:

```
Thunk <---> Lazy (via From, memoize)
Thunk <---> Trampoline (via From)
Trampoline ---> Lazy (via From, memoize)
Lazy ---> Thunk (requires Clone)
Lazy ---> Trampoline (requires Clone + 'static)
```

And similarly for the `Try*` variants. This is well-designed: it lets users start with one type and convert to another as requirements change, without rebuilding their computation from scratch.

## 5. Naming Analysis

### 5.1 Type Names

- **Thunk**: Standard FP terminology for a suspended computation. Good.
- **Trampoline**: Standard terminology for stack-safe recursive evaluation via heap-allocated continuations. Good.
- **Lazy**: Slightly ambiguous; in Haskell, "lazy" implies both deferral and memoization, while here it means memoization only. However, this matches Rust's `std::cell::LazyCell` and `std::sync::LazyLock` naming. Acceptable.
- **TryThunk/TryTrampoline/TryLazy**: Consistent with Rust's `Try*` naming convention (e.g., `TryFrom`, `TryInto`). Good.

### 5.2 Method Names

- **`evaluate()`**: Used consistently across all six types. On `Thunk` and `Trampoline`, it consumes `self` and returns `A`. On `Lazy`, it borrows `&self` and returns `&A`. The semantic difference (consume vs borrow) is reflected in the signature but not the name. This could surprise users, but it is consistent with PureScript's `force` being used for both memoized and non-memoized types.
- **`memoize()`**: Clear bridge from computation to cached value. Present on both `Thunk` and `Trampoline`. Good.
- **`defer()`**: Both the inherent method and the `Deferrable` trait method. Consistent with PureScript's `Control.Lazy.defer`. Good.
- **`pure()`**: Standard FP terminology. Present on all types. Good.

### 5.3 Brand Names

- `ThunkBrand` serves double duty: it is the brand for `Thunk<'a, A>` AND the functor parameter for `Free<ThunkBrand, A>` (which is `Trampoline`). This is technically correct (Trampoline IS `Free` over Thunk), but could confuse users who see `ThunkBrand` and expect it to refer only to `Thunk`. The comment at brands.rs line 193 ("Note: This is for `Thunk<'a, A>`, NOT for `Trampoline<A>`") mitigates this.
- `LazyBrand<Config>` is parameterized by config, which is consistent.
- There is no `TrampolineBrand` because `Trampoline` cannot participate in HKT. This is correct and well-documented.

### 5.4 Naming Concern: "evaluate" Overloading

The name `evaluate` is used with three different signatures:
1. `Thunk::evaluate(self) -> A` (consumes, returns owned value)
2. `Trampoline::evaluate(self) -> A` (consumes, returns owned value)
3. `Lazy::evaluate(&self) -> &A` (borrows, returns reference)
4. `TryLazy::evaluate(&self) -> Result<&A, &E>` (borrows, returns reference)

The first two are consistent. The third and fourth differ in ownership semantics. Since `Lazy` is a cache, returning `&A` is correct, but users coming from `Thunk`/`Trampoline` might expect ownership. This is documented but could benefit from a more distinctive name (e.g., `force` for `Lazy`, matching PureScript's `force :: Lazy a -> a`). However, renaming at this point would break the consistency of having a single `evaluate` operation across all lazy types.

## 6. Structural Observations

### 6.1 Thunk Cannot Implement Traversable

The documentation (thunk.rs, lines 73-77) correctly identifies that `Thunk` cannot implement `Traversable` because `FnOnce` closures are not cloneable. This is an inherent limitation of wrapping `Box<dyn FnOnce>`.

### 6.2 Lazy Only Implements RefFunctor

`Lazy` implements `RefFunctor` but not `Functor` (lazy.rs, lines 880-913). This is correct: since `evaluate()` returns `&A`, mapping must go through `&A -> B` rather than `A -> B`. The PureScript `Data.Lazy` implements full `Functor` because `force` returns an owned value, but that requires re-evaluating or cloning, which the Rust version avoids for efficiency.

### 6.3 Trampoline Has No Brand

`Trampoline` has no brand and cannot participate in HKT-polymorphic code. This is documented (brands.rs, lines 192-193). The implication is that generic FP code (e.g., `traverse`, `sequence`) cannot use `Trampoline` as a target functor. Users must use `Thunk` for HKT contexts and convert to `Trampoline` when stack safety is needed.

### 6.4 Evaluable Trait

The `Evaluable` trait (evaluable.rs) is used only by `Free::evaluate`. It requires `Functor`, which means only `ThunkBrand` implements it (since it is the only brand whose `evaluate` extracts the inner value). This is narrow but correct; `Evaluable` is an internal mechanism, not a user-facing abstraction.

## 7. Potential Improvements

### 7.1 Consider a Stack-Safe Thunk with Lifetimes

The biggest gap is that there is no type offering both stack safety and lifetime polymorphism. `Thunk` has lifetimes but no stack safety for `bind`; `Trampoline` has stack safety but requires `'static`. A hypothetical `StackSafeThunk<'a, A>` would need a `Free` monad that does not use `Box<dyn Any>` for its continuation stack, perhaps using a typed heterogeneous stack instead. This is a hard problem, but worth investigating for future versions.

### 7.2 Decouple TryLazy from LazyConfig

The `LazyConfig` trait bundles both infallible and fallible lazy cell types. Separating these into `LazyConfig` and `TryLazyConfig` would reduce coupling and allow custom configs that only support one variant. This is a minor point.

### 7.3 Document the Conversion Network

The `From` implementations scattered across files create a conversion graph that is not documented in one place. A diagram or table in the module-level docs showing all available conversions would help users discover the right path.

## 8. Conclusion

The Thunk/Trampoline/Lazy decomposition is well-motivated by Rust's ownership and lifetime constraints. It correctly separates three genuinely independent concerns (lifetime flexibility, stack safety, memoization) that languages with GC or lazy runtimes can unify. The Try* variants provide ergonomic error handling without adding architectural complexity. The `LazyConfig` trait elegantly handles the Rc/Arc split. The naming is mostly consistent with both FP tradition and Rust conventions.

The design compares favorably to Scala's `Eval` (which conflates concerns that Rust must separate) and is the natural Rust-native decomposition of what Haskell and PureScript provide via runtime support. The main gap is the absence of a type combining stack safety with lifetime polymorphism, which is a hard problem that may not have a clean solution in Rust's type system.

Overall, this is a sound, well-reasoned design that makes Rust-appropriate trade-offs explicit rather than hiding them behind a single polymorphic type.
