# Analysis 8: Pointer Abstraction and Thread Safety

## 1. How Thunk and Lazy Use Rc/Arc Internally

### Thunk: No Pointer Abstraction Needed

`Thunk` (in `fp-library/src/types/thunk.rs`) uses `Box<dyn FnOnce() -> A + 'a>` internally (line 89). It does not use `Rc` or `Arc` at all. This is appropriate because:

- `Thunk` is consumed on evaluation (`evaluate` takes `self` by value, line 247).
- `Thunk` is not cloneable; it wraps `FnOnce`, which can only be called once.
- There is no shared memoization to coordinate.

`Thunk` does not need pointer parameterization. Its role is lightweight deferred computation with lifetime flexibility, and `Box` is the correct choice for unique ownership of a closure.

### Lazy: Pointer Abstraction via LazyConfig

`Lazy` (in `fp-library/src/types/lazy.rs`) achieves pointer abstraction through the `LazyConfig` trait (lines 65-170) rather than direct parameterization over a pointer brand. The `LazyConfig` trait bundles the pointer choice, lazy cell type, and thunk type into a single configuration:

- `RcLazyConfig` (line 175-177): Uses `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>` (line 178).
- `ArcLazyConfig` (line 289-291): Uses `Arc<LazyLock<A, Box<dyn FnOnce() -> A + Send + 'a>>>` (line 292).

The `Lazy` struct itself is parameterized over `Config: LazyConfig` (line 424), with the config's associated types determining the concrete pointer and cell types. This is a well-designed approach because:

1. It prevents invalid pointer/cell combinations at compile time (e.g., `Rc` with `LazyLock` or `Arc` with `LazyCell`).
2. It keeps the type signature clean: `Lazy<'a, A, RcLazyConfig>` rather than `Lazy<'a, A, RcBrand, OnceCellBrand>`.
3. It is extensible; third-party configs can be defined.

### TryLazy: Same Pattern as Lazy

`TryLazy` (in `fp-library/src/types/try_lazy.rs`) follows the identical `LazyConfig` pattern (line 44). Its inner type is `Config::TryLazy<'a, A, E>` (line 46), which resolves to `Rc<LazyCell<Result<A, E>, ...>>` or `Arc<LazyLock<Result<A, E>, ...>>`.

## 2. The RcLazy/ArcLazy and RcTryLazy/ArcTryLazy Split

The split is well-designed. The type aliases are defined as:

- `RcLazy<'a, A>` = `Lazy<'a, A, RcLazyConfig>` (not shown in the read portion but implied by usage patterns).
- `ArcLazy<'a, A>` = `Lazy<'a, A, ArcLazyConfig>`.
- `RcTryLazy<'a, A, E>` = `TryLazy<'a, A, E, RcLazyConfig>` (try_lazy.rs line 540).
- `ArcTryLazy<'a, A, E>` = `TryLazy<'a, A, E, ArcLazyConfig>` (try_lazy.rs line 543).

### Strengths of This Design

1. **Single generic type with config-based specialization.** There is one `Lazy` struct and one `TryLazy` struct, not four separate structs. This avoids code duplication while providing convenient aliases.

2. **Config bundles prevent invalid combinations.** The `LazyConfig` trait ensures that `Rc` is always paired with `LazyCell` and `Arc` with `LazyLock`. Users cannot accidentally create a `Lazy` with mismatched pointer and synchronization primitives.

3. **Separate constructors enforce correct bounds.** The `impl` blocks for `Lazy<'a, A, RcLazyConfig>` (line 492) and `Lazy<'a, A, ArcLazyConfig>` (line 603) have different `new` signatures. The Arc variant requires `Send` on the closure (line 622: `f: impl FnOnce() -> A + Send + 'a`), while the Rc variant does not (line 511: `f: impl FnOnce() -> A + 'a`). This enforces thread safety at the API boundary.

4. **Conversion impls are config-aware.** For example, `From<TryThunk>` is only implemented for `RcTryLazy` (try_lazy.rs line 184), while `From<Lazy<ArcLazyConfig>>` to `TryLazy<ArcLazyConfig>` requires `A: Clone + Send + Sync` (try_lazy.rs lines 236-255).

### One Minor Concern

The `TryLazy::new` for `ArcLazyConfig` (try_lazy.rs line 404) requires `Send` on the closure but not `Sync`. This is consistent with `LazyLock`'s requirements (it only needs `Send` for the initializer since it synchronizes access internally). However, the `SendDeferrable` impl (try_lazy.rs line 532) requires `Send + Sync` on the closure, which is a stronger bound. This asymmetry is correct but could be confusing; it reflects that `Send + Sync` is needed when the `TryLazy` itself must be `Send + Sync` (for cross-thread sharing), while mere construction only needs `Send`.

## 3. Trampoline's Use of Box vs Rc/Arc

`Trampoline` (in `fp-library/src/types/trampoline.rs`) wraps `Free<ThunkBrand, A>` (line 74), where `Free` uses `Box`-based continuations internally (via `CatList`). `Thunk` itself uses `Box<dyn FnOnce() -> A + 'a>` (thunk.rs line 89).

This is the correct design for several reasons:

1. **Trampolines are consumed, not shared.** `Trampoline::evaluate` takes `self` by value (line 248). There is no shared memoization, so reference counting adds overhead without benefit.

2. **Box enables unique ownership of continuations.** The `CatList`-based bind stack requires moving closures through the evaluation loop. `Box` provides the needed ownership semantics without the overhead of atomic reference counting.

3. **`'static` requirement eliminates lifetime-polymorphic sharing.** Since `Trampoline` requires `A: 'static` (line 72), it cannot hold borrowed references. This makes `Rc`/`Arc` even less useful, as there is no shared state to coordinate lifetimes for.

4. **No Clone needed.** `Trampoline` is not `Clone`. Each computation is a unique chain of deferred steps. Reference counting would add unnecessary cloning capability.

### TryTrampoline Follows the Same Pattern

`TryTrampoline` (try_trampoline.rs line 38-41) simply wraps `Trampoline<Result<A, E>>`. It inherits all of `Trampoline`'s pointer characteristics and the `'static` requirement on both `A` and `E`.

### One Observation: `arc_tail_rec_m`

Interestingly, `Trampoline::arc_tail_rec_m` (trampoline.rs lines 447-458) uses `Arc` to share a non-`Clone` closure across recursive iterations. This is the one place where `Arc` appears in the trampoline hierarchy, and it is used correctly: as a mechanism to share an `Fn` closure (not `FnOnce`) across multiple invocations of the recursion step. This is not a pointer abstraction concern; it is a practical workaround for closures that capture non-Clone state.

## 4. Thread Safety: Which Types Are Send/Sync and Should They Be?

### Current Thread Safety Status

| Type | Send | Sync | Reason |
|------|------|------|--------|
| `Thunk<'a, A>` | No | No | Contains `Box<dyn FnOnce() -> A + 'a>`, which is `!Send` unless the closure is `Send`. No `Send` bound on the struct. |
| `TryThunk<'a, A, E>` | No | No | Wraps `Thunk<'a, Result<A, E>>`, inheriting its `!Send` status. |
| `RcLazy<'a, A>` | No | No | Contains `Rc<LazyCell<...>>`. `Rc` is `!Send` and `!Sync`. |
| `ArcLazy<'a, A>` | Conditional | Conditional | Contains `Arc<LazyLock<...>>`. `Send + Sync` when the closure and value are `Send`. |
| `RcTryLazy<'a, A, E>` | No | No | Same reasoning as `RcLazy`. |
| `ArcTryLazy<'a, A, E>` | Conditional | Conditional | Same reasoning as `ArcLazy`. |
| `Trampoline<A>` | No | No | Wraps `Free<ThunkBrand, A>`, which contains `Box<dyn FnOnce>` closures. Not `Send` even when `A: Send`. |
| `TryTrampoline<A, E>` | No | No | Wraps `Trampoline<Result<A, E>>`. |

### Assessment

**Thunk and TryThunk:** Correctly `!Send`. They use `Box<dyn FnOnce()>` without `Send` bounds. Since they support arbitrary lifetimes (`'a`), making them `Send` would require `Send` bounds on the closure, which would break the lightweight, unconstrained API. Users who need thread safety should use `ArcLazy` or build thread-safe wrappers explicitly.

**RcLazy and RcTryLazy:** Correctly `!Send` and `!Sync`. They use `Rc`, which is inherently single-threaded. This is by design.

**ArcLazy and ArcTryLazy:** Correctly conditionally `Send + Sync`. The `Arc<LazyLock<...>>` combination is `Send + Sync` when the value type is `Send + Sync`. The `ArcLazyConfig::new` method (lazy.rs line 622) requires `Send` on the closure, which ensures the initializer can be safely moved across threads.

**Trampoline:** This is `!Send` even when `A: Send`, because it contains `Box<dyn FnOnce() -> Free<ThunkBrand, A>>` closures that lack `Send` bounds. The tests at trampoline.rs lines 829-892 explicitly verify that `Trampoline` works with `Rc<T>` (a `!Send` type). This is correct: `Trampoline` is designed for single-threaded stack-safe recursion. For concurrent use cases, users should evaluate the trampoline first, then share the result via `Arc`.

**Should any of these be `Send`?** The current design is sound. If a `SendTrampoline` were desired, it would need `Send` bounds on all closures in the `Free` monad chain, which would be a separate type. The library already provides `ArcLazy` for thread-safe memoization, so users can compose `Trampoline` + `ArcLazy` for thread-safe lazy evaluation of stack-safe computations.

## 5. The FnBrand<P> Pattern and Its Relationship to Lazy Types

`FnBrand<P>` (in `fp-library/src/types/fn_brand.rs`) parameterizes function wrappers over a pointer brand `P: UnsizedCoercible`. The `Kind` implementation (fn_brand.rs lines 48-52) maps:

```
FnBrand<P>::Of<'a, A, B> = P::CloneableOf<'a, dyn 'a + Fn(A) -> B>
```

This means `RcFnBrand::Of<'a, A, B>` = `Rc<dyn Fn(A) -> B + 'a>` and `ArcFnBrand::Of<'a, A, B>` = `Arc<dyn Fn(A) -> B + 'a>`.

### Relationship to Lazy Types

The `LazyConfig` trait does NOT use `FnBrand` for its thunk storage. Instead, it stores `Box<dyn FnOnce() -> A>` (for `RcLazyConfig`) or `Box<dyn FnOnce() -> A + Send>` (for `ArcLazyConfig`). This is because:

1. **`LazyCell` and `LazyLock` accept `FnOnce`, not `Fn`.** The standard library's lazy types consume the initializer on first evaluation. `FnBrand` wraps `Fn` (multi-call), not `FnOnce` (single-call).

2. **No cloneability needed for the initializer.** The thunk runs at most once. Wrapping it in `Rc`/`Arc` for cloneability would be wasteful.

3. **`Box` is sufficient for unique ownership.** The `LazyCell`/`LazyLock` owns the initializer exclusively until evaluation.

This is a deliberate and correct divergence from the `FnBrand` pattern. `FnBrand` is for shareable, multi-call functions (profunctor arrows, optics, etc.). `Lazy`'s thunk is a one-shot computation.

### Where FnBrand Matters for Lazy

`FnBrand` is relevant when `Lazy` values participate in HKT operations. For instance, `Lazy` implements `RefFunctor` (via the `ref_map` method, lazy.rs line 554), which creates a new `Lazy` from the mapped value. The mapping closure is stored as a new `Box<dyn FnOnce>`, not as a `FnBrand`-wrapped function.

The `Deferrable` implementation for `RcLazy` (if present) and `SendDeferrable` for `ArcTryLazy` (try_lazy.rs line 505) create new `Lazy` values from closures, again using `Box<dyn FnOnce>` via the `new` constructor.

## 6. Whether Pointer Choice Affects Performance

### Rc vs Arc Overhead

The pointer choice has measurable performance implications:

| Operation | Rc | Arc |
|-----------|-----|-----|
| Clone (ref count increment) | ~1-3ns (non-atomic) | ~5-15ns (atomic fetch_add) |
| Drop (ref count decrement) | ~1-3ns (non-atomic) | ~5-15ns (atomic fetch_sub) |
| Deref | Same | Same |
| Allocation | Same | Same |

For `Lazy`, each `clone()` and `drop()` hits the reference count. In hot paths with many clones (e.g., sharing a lazy value across a data structure), the Arc overhead can be 3-5x more expensive than Rc for the cloning operation alone.

### LazyCell vs LazyLock Overhead

| Operation | LazyCell (RcLazy) | LazyLock (ArcLazy) |
|-----------|-------------------|---------------------|
| First evaluation | No synchronization overhead | Mutex/atomic synchronization |
| Subsequent evaluations | Direct memory read | Atomic load (acquire fence) |

`LazyLock` has higher overhead on first evaluation due to synchronization, and a small but non-zero overhead on subsequent reads due to atomic ordering guarantees.

### Box in Thunk/Trampoline

`Thunk` and `Trampoline` use `Box` for their closures, which has zero reference counting overhead. The only cost is heap allocation, which is a one-time cost at construction. This is the lightest-weight choice for non-shared, non-cloneable computations.

### Practical Implications

- **Single-threaded code should always prefer Rc variants.** The `RcLazy`/`RcTryLazy` types avoid all atomic operations.
- **Multi-threaded code must use Arc variants.** The `ArcLazy`/`ArcTryLazy` types provide the necessary `Send + Sync` guarantees.
- **Trampoline and Thunk are pointer-agnostic.** They use `Box` and avoid the Rc/Arc overhead entirely.

## 7. Consistency with Other Library Types

### How Other Types Handle Rc vs Arc

The library's approach to pointer abstraction is consistent across its type families:

| Type Family | Pattern | Pointer Abstraction |
|-------------|---------|---------------------|
| `FnBrand<P>` | Generic over `P: UnsizedCoercible` | Full; `RcFnBrand`/`ArcFnBrand` aliases |
| `Lazy<Config>` | Generic over `Config: LazyConfig` | Full; `RcLazyConfig`/`ArcLazyConfig` configs |
| `TryLazy<Config>` | Generic over `Config: LazyConfig` | Full; same configs as `Lazy` |
| `Thunk` | Uses `Box` | Not parameterized (correct; no sharing) |
| `TryThunk` | Wraps `Thunk` | Not parameterized (correct; no sharing) |
| `Trampoline` | Uses `Box` via `Free` | Not parameterized (correct; no sharing) |
| `TryTrampoline` | Wraps `Trampoline` | Not parameterized (correct; no sharing) |

The pattern is: types that need shared memoization (Lazy, TryLazy) are parameterized over pointer/config. Types that are consumed on evaluation (Thunk, TryThunk, Trampoline, TryTrampoline) use `Box` without parameterization. This is a clean separation.

### Optics (Noted Inconsistency)

The CLAUDE.md notes that "many optics currently hard-code Rc usage" and should be refactored to use `FnBrand<P>`. This is a known inconsistency that the pointer abstraction was designed to address. The lazy types have already been refactored to use the new pattern; the optics have not yet.

### Pointer Trait Hierarchy

The trait hierarchy (`Pointer` -> `RefCountedPointer` -> `SendRefCountedPointer`) is used consistently:

- `rc_ptr.rs`: `RcBrand` implements `Pointer` + `RefCountedPointer` + `UnsizedCoercible` (lines 35, 63, 168). Does NOT implement `SendRefCountedPointer` or `SendUnsizedCoercible`.
- `arc_ptr.rs`: `ArcBrand` implements all five traits (lines 37, 65, 172, 200, 230).
- `fn_brand.rs`: `FnBrand<P>` requires `P: UnsizedCoercible` for basic operations, `P: SendUnsizedCoercible` for `SendCloneableFn` (line 436).

This is consistent and well-layered.

## 8. Support for Future Refactoring to Full Pointer Abstraction

### What is Already Pointer-Abstract

1. **Lazy and TryLazy** are fully pointer-abstract via `LazyConfig`. Adding a new pointer type requires implementing a new `LazyConfig` struct.

2. **FnBrand** is fully pointer-abstract via the `UnsizedCoercible` trait hierarchy. Third-party pointer brands can get `FnBrand` support by implementing `UnsizedCoercible`.

3. **The `RefCountedPointer` trait** includes a `TakeCellOf` associated type (ref_counted_pointer.rs line 105) that pairs the appropriate interior mutability primitive with each pointer brand (`RefCell` for Rc, `Mutex` for Arc). This supports the thunk-clearing pattern used in earlier design iterations.

### What Would Need Work for Full Abstraction

1. **Thunk and TryThunk** use `Box<dyn FnOnce>` directly. If a future use case required a shareable thunk (e.g., for persistent data structures), a new type would be needed. However, this is unlikely; shareable thunks are what `Lazy` provides.

2. **Trampoline and TryTrampoline** are hardcoded to `Box` via `Free<ThunkBrand, A>`. If a thread-safe trampoline were desired, it would require a `SendThunkBrand` or similar, plus `Send` bounds throughout the `Free` monad machinery. This would be a substantial change but the architecture does not preclude it.

3. **Optics** still hardcode `Rc` in many places. The `FnBrand<P>` pattern is ready to be applied here, but the refactoring has not been done.

4. **The `LazyConfig` trait** is extensible. A `ParkingLotLazyConfig` or `TokioLazyConfig` could be implemented by third parties without changes to the core library.

### Architectural Readiness

The current design is well-positioned for future pointer abstraction:

- The `LazyConfig` trait is open for extension (line 65: it is a public trait, not sealed).
- The pointer hierarchy (`Pointer` -> `RefCountedPointer` -> `SendRefCountedPointer`) has a clean extension point for new pointer types.
- The `UnsizedCoercible`/`SendUnsizedCoercible` traits allow new pointer brands to participate in `FnBrand` without modifying library code.
- The `TakeCellOf` associated type on `RefCountedPointer` abstracts over interior mutability, making it possible to use the same patterns with different synchronization primitives.

### One Limitation: LazyConfig is Not Pointer-Brand-Parameterized

The `LazyConfig` trait does not expose a `PtrBrand` associated type in its current implementation. The original plan (in `plans/pointer-abstraction/steps/03-lazy-refactor/README.md` lines 23-33) included `type PtrBrand: RefCountedPointer` as part of `LazyConfig`, but the final implementation instead inlines the pointer and cell types directly into the associated types (e.g., `type Lazy<'a, A: 'a> = Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>`). This means that generic code cannot easily extract the pointer brand from a `LazyConfig` to use it for other purposes (e.g., creating matching `FnBrand` wrappers). However, since `LazyConfig` is a sealed set in practice (only `RcLazyConfig` and `ArcLazyConfig` are provided), this is not a practical limitation.

## Summary of Findings

1. **Thunk and Trampoline correctly use Box.** They are consumed on evaluation, not shared. Pointer parameterization would add complexity without benefit.

2. **Lazy and TryLazy correctly use the LazyConfig pattern.** The Rc/Arc split is achieved through configuration structs that bundle pointer, cell, and thunk types. This prevents invalid combinations and keeps the API clean.

3. **The RcLazy/ArcLazy split is well-designed.** It provides compile-time thread safety enforcement through separate `new` constructors with appropriate bounds.

4. **Thread safety is correctly implemented.** Rc-based types are `!Send`, Arc-based types are conditionally `Send + Sync`, and non-shared types (Thunk, Trampoline) are `!Send` by default.

5. **The FnBrand<P> pattern is correctly not used for Lazy thunks.** Lazy's one-shot initializer needs `FnOnce` in `Box`, not `Fn` in `Rc`/`Arc`.

6. **Pointer choice has real performance implications.** Rc is 3-5x cheaper for clone/drop operations than Arc due to atomic operations. The library correctly defaults to Rc and provides Arc as an opt-in.

7. **The design supports future extension.** New pointer brands, lazy configs, and synchronization primitives can be added without modifying the core library.

8. **The main gap is in optics,** where `Rc` is still hardcoded. The pointer abstraction infrastructure is ready; the optics just need to be refactored to use `FnBrand<P>`.
