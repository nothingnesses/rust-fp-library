# Analysis 2: HKT Integration and Type Class Correctness

## 1. Brand/Kind Definitions

### 1.1 ThunkBrand

Defined at `brands.rs:194-195`:

```rust
pub struct ThunkBrand;
```

Kind mapping at `thunk.rs:317-321`:

```rust
impl_kind! {
    for ThunkBrand {
        type Of<'a, A: 'a>: 'a = Thunk<'a, A>;
    }
}
```

This uses the `Kind_cdc7cd43dac7585f` trait (signature `type Of<'a, A: 'a>: 'a;`), which is the standard kind signature used by `Functor`, `Pointed`, `Semimonad`, and all other core type classes. The mapping is correct: `ThunkBrand` applied to lifetime `'a` and type `A` yields `Thunk<'a, A>`. The `'a` bound on both `A` and the output type faithfully reflects `Thunk`'s internal `Box<dyn FnOnce() -> A + 'a>`.

**Verdict:** Correct and consistent.

### 1.2 LazyBrand

Defined at `brands.rs:91-92`:

```rust
pub struct LazyBrand<Config>(PhantomData<Config>);
```

Kind mapping at `lazy.rs:805-809`:

```rust
impl_kind! {
    impl<Config: LazyConfig> for LazyBrand<Config> {
        type Of<'a, A: 'a>: 'a = Lazy<'a, A, Config>;
    }
}
```

The `Config` parameter is baked into the brand, so `LazyBrand<RcLazyConfig>` and `LazyBrand<ArcLazyConfig>` are distinct brands. This is the right approach, because the two configurations have different `Clone` and `Send` properties. The kind mapping follows the same `Kind_cdc7cd43dac7585f` pattern, which is consistent.

**Verdict:** Correct. The parameterization of the brand by `Config` is a clean design that avoids needing separate `RcLazyBrand` and `ArcLazyBrand` types.

### 1.3 TryThunkErrAppliedBrand / TryThunkOkAppliedBrand / TryThunkBrand

Defined at `brands.rs:201-211`:

```rust
pub struct TryThunkBrand;
pub struct TryThunkErrAppliedBrand<E>(PhantomData<E>);
pub struct TryThunkOkAppliedBrand<A>(PhantomData<A>);
```

Kind mappings at `try_thunk.rs:481-486` and `try_thunk.rs:1199-1204`:

```rust
impl_kind! {
    impl<E: 'static> for TryThunkErrAppliedBrand<E> {
        type Of<'a, A: 'a>: 'a = TryThunk<'a, A, E>;
    }
}

impl_kind! {
    impl<A: 'static> for TryThunkOkAppliedBrand<A> {
        type Of<'a, E: 'a>: 'a = TryThunk<'a, A, E>;
    }
}
```

`TryThunkBrand` is used only for the `Bifunctor` and `Bifoldable` impls (lines 952-1013), where a two-type-parameter kind is needed. The partially-applied brands each fix one type parameter and become standard single-parameter kinds suitable for `Functor`/`Monad` etc.

**Note on 'static bound:** Both partially-applied brands require `E: 'static` or `A: 'static` respectively. This is stricter than `Thunk` itself, which supports arbitrary lifetimes. The `'static` bound appears necessary for the `impl_kind!` macro's requirements for the fixed parameter, but it does restrict the HKT interface to static error/value types even though the underlying `TryThunk<'a, A, E>` could support non-static ones. This is an acceptable trade-off; the inherent methods remain fully flexible.

**Verdict:** Correct. The three-brand approach (bifunctor + two partial applications) mirrors the `ResultBrand` / `ResultErrAppliedBrand<E>` / `ResultOkAppliedBrand<T>` pattern used elsewhere in the library.

### 1.4 TryLazyBrand

Defined at `brands.rs:198-199`:

```rust
pub struct TryLazyBrand<E, Config>(PhantomData<(E, Config)>);
```

Kind mapping at `try_lazy.rs:493-498`:

```rust
impl_kind! {
    impl<E: 'static, Config: LazyConfig> for TryLazyBrand<E, Config> {
        type Of<'a, A: 'a>: 'a = TryLazy<'a, A, E, Config>;
    }
}
```

This correctly parameterizes the brand by both error type and configuration, leaving the success type `A` as the polymorphic parameter. The `E: 'static` bound is again required by the HKT machinery.

**Verdict:** Correct and consistent with the overall pattern.

### 1.5 Trampoline: No Brand

Trampoline has no dedicated brand. The comment at `brands.rs:191-193` explicitly states:

```
/// Note: This is for `Thunk<'a, A>`, NOT for `Trampoline<A>`.
/// `Trampoline` cannot implement HKT traits due to its `'static` requirement.
```

This is accurate. Trampoline's internal type is `Free<ThunkBrand, A>` where `A: 'static` (line `trampoline.rs:72-75`). The HKT kind signature `type Of<'a, A: 'a>: 'a` allows the lifetime to vary, but Trampoline would always force `'a = 'static`, making the polymorphism vacuous. More importantly, the `Semimonad::bind` signature takes `impl Fn(A) -> ... + 'a`, but Trampoline's bind requires `FnOnce` with `'static` bounds on both `A` and the closure, making it structurally incompatible with the HKT bind signature.

**Verdict:** The decision to exclude Trampoline from the HKT system is well-justified. See Section 3 for detailed analysis.

## 2. Type Class Implementations

### 2.1 Thunk (ThunkBrand)

Thunk implements the following HKT type classes:

| Type Class | Location | Notes |
|---|---|---|
| `Functor` | `thunk.rs:355-389` | Delegates to inherent `map` |
| `Pointed` | `thunk.rs:391-419` | Delegates to inherent `pure` |
| `Lift` | `thunk.rs:421-465` | Binary lift via `bind` + `map` |
| `ApplyFirst` | `thunk.rs:467` | Marker trait |
| `ApplySecond` | `thunk.rs:468` | Marker trait |
| `Semiapplicative` | `thunk.rs:470-513` | `apply` via `bind` + `map` |
| `Semimonad` | `thunk.rs:515-549` | Delegates to inherent `bind` |
| `MonadRec` | `thunk.rs:551-597` | Stack-based loop (not stack-safe!) |
| `Evaluable` | `thunk.rs:599-630` | Delegates to inherent `evaluate` |
| `Foldable` | `thunk.rs:632-670+` | Single-element fold |
| `Deferrable` | `thunk.rs:327-353` | Non-HKT, but integrated |

Additionally, `Thunk` implements `Semigroup` and `Monoid` (not shown in excerpts but indicated by the imports at line 25-26).

**Key observation: `Semimonad::bind` takes `impl Fn` vs inherent `bind` takes `FnOnce`.** The HKT-level `Semimonad::bind` at line 543-548 accepts `impl Fn(A) -> ... + 'a`, while the inherent `bind` at line 193-202 accepts `impl FnOnce(A) -> ... + 'a`. This divergence is documented in the code (lines 170-173) and is necessary because `Semimonad::bind` must support types like `Vec` that call the function multiple times. For `Thunk`, which holds exactly one value, `Fn` is strictly more restrictive than `FnOnce`, so this is sound but slightly less ergonomic at the HKT level.

**MonadRec implementation (lines 581-597):** The tail_rec_m for `ThunkBrand` uses a stack-based loop inside a `Thunk::new`, which is NOT stack-safe for deep recursion. This is correctly documented in the type's doc comment at line 59. The implementation is technically correct (it satisfies the `MonadRec` laws), but users must be aware of the stack limitation.

**Missing: Traversable.** This is explicitly documented at lines 73-77 as impossible due to `FnOnce` semantics.

### 2.2 Lazy (LazyBrand<RcLazyConfig>)

Lazy implements a much more restricted set of HKT type classes:

| Type Class | Location | Notes |
|---|---|---|
| `RefFunctor` | `lazy.rs:880-913` | Maps by reference, only for `RcLazyConfig` |
| `Foldable` | `lazy.rs:1043-1086+` | Single-element fold, only for `RcLazyConfig` |
| `Deferrable` | `lazy.rs:812-844` | Only for `RcLazyConfig`, requires `A: Clone` |
| `SendDeferrable` | `lazy.rs:847-878` | Only for `ArcLazyConfig`, requires `A: Clone + Send + Sync` |
| `Semigroup` | `lazy.rs:921-985` | For both Rc and Arc variants |
| `Monoid` | `lazy.rs:993-1039` | For both Rc and Arc variants |

**Why no Functor?** Lazy's `evaluate()` returns `&A`, not `A`. A standard `Functor::map` signature requires transforming `F<A>` to `F<B>`, but Lazy cannot move the `A` out; it can only provide a reference. This is why `RefFunctor` exists as a separate trait. The `ref_map` operation creates a new `Lazy` that reads from the old one by reference.

**Why no Pointed, Semimonad, Semiapplicative?** These would all require owned values, which Lazy cannot provide without `Clone`. The library chose not to add `Clone`-gated `Pointed`/`Semimonad` impls, which is a reasonable design choice. It keeps the HKT interface honest: if you want monadic composition, use `Thunk`.

**ArcLazyConfig asymmetry:** `RefFunctor` and `Foldable` are only implemented for `LazyBrand<RcLazyConfig>`, not `LazyBrand<ArcLazyConfig>`. The `ArcLazy::ref_map` method exists as an inherent method (lines 677-686) but not as a `RefFunctor` impl. The reason is documented at lines 660-663: the `RefFunctor` trait does not require `Send` on the mapping function, but `ArcLazy::new` requires `Send`. Adding `Send` would change the trait's signature and break other implementors.

**Verdict:** The restricted type class set is well-motivated. The asymmetry between Rc and Arc configs is a real limitation but is documented.

### 2.4 TryLazy (TryLazyBrand<E, Config>)

TryLazy has minimal HKT integration. It has a `Kind` mapping but no Functor, Pointed, or other standard type class implementations beyond the kind mapping itself. It implements `Deferrable` (line 458) and `SendDeferrable` (line 505). This parallels Lazy's restricted HKT surface, for the same reasons: `evaluate()` returns `Result<&A, &E>`, not an owned value.

### 2.5 Trampoline and TryTrampoline

Neither has any HKT type class implementations. Trampoline provides inherent methods: `pure`, `bind`, `map`, `lift2`, `then`, `tail_rec_m`, `evaluate`, `memoize` (lines 100-303). TryTrampoline mirrors these with error handling (lines 45-375+). Both implement `Deferrable` (Trampoline at line 501, TryTrampoline at line 460+).

## 3. Whether Trampoline's Lack of HKT Traits is a Real Limitation

### 3.1 Technical Barriers

The `'static` requirement on `Trampoline<A>` (line 72) is fundamental. The Free monad uses `Box<dyn Any>` for type erasure in its CatList-based bind stack, which requires `'static`. This means:

1. **Kind signature mismatch.** The standard `Kind_cdc7cd43dac7585f` signature is `type Of<'a, A: 'a>: 'a`, but for Trampoline, `'a` would always be `'static`. A brand could be defined with this constraint, but it would be uninhabitable at non-static lifetimes, breaking generic code that assumes lifetime polymorphism.

2. **Bind signature mismatch.** `Semimonad::bind` takes `impl Fn(A) -> ... + 'a`, but Trampoline's bind requires `FnOnce` (line 198-203). For Thunk, the `Fn`/`FnOnce` gap is acceptable because `Thunk` is a single-valued container. But forcing `Fn` on Trampoline would be more restrictive since Trampoline's raison d'etre is chaining potentially expensive one-shot computations.

3. **'static closures.** Even if we used `Fn`, every closure passed to Trampoline's bind must be `'static` (line 200), which is more restrictive than the `'a` in the HKT signature.

### 3.2 Practical Impact

The lack of HKT integration means:

- Trampoline cannot be used with `m_do!` macro or generic monadic functions like `bind::<Brand, _, _>`.
- Generic code parameterized by a monad brand cannot be instantiated with Trampoline.
- The conversion path Thunk <-> Trampoline exists via `From` impls (thunk.rs:300-315, trampoline.rs:483-498), providing an escape hatch.

### 3.3 Assessment

This is an **acceptable and well-designed limitation**. The library provides a clean separation:

- **Thunk** for HKT-integrated, lifetime-flexible, non-stack-safe deferred computation.
- **Trampoline** for stack-safe, `'static`-only computation without HKT integration.

Users who need both stack safety and HKT generality can compose them: use Trampoline internally for stack safety, then wrap in a Thunk or convert via `From` for HKT consumption. The `From` impls bridge the gap. The alternative of forcing Trampoline into the HKT system with a `'static`-only brand would create a fragile abstraction that breaks at arbitrary lifetime boundaries.

## 4. Lifetime Interactions with the HKT System

### 4.1 Thunk

The lifetime `'a` in `Thunk<'a, A>` flows directly through the HKT system via the kind signature `type Of<'a, A: 'a>: 'a`. This means:

- `Thunk` can borrow from its environment: `Thunk<'a, &'a str>` is valid.
- Generic functions like `map::<ThunkBrand, _, _>` correctly propagate lifetimes.
- The `Box<dyn FnOnce() -> A + 'a>` internal representation (line 89) is properly bounded.

### 4.2 Lazy

Same lifetime structure as Thunk. The `Lazy<'a, A, Config>` type propagates `'a` through the kind mapping. The internal `Config::Lazy<'a, A>` associated type (e.g., `Rc<LazyCell<A, Box<dyn FnOnce() -> A + 'a>>>` for `RcLazyConfig`, line 178) correctly uses the lifetime.

### 4.3 Trampoline and Lazy Interaction

The `From<Trampoline<A>> for Lazy<'a, A, RcLazyConfig>` impl (lazy.rs:583-599) has `A: 'static`, which makes the resulting `Lazy` effectively `'static` even though `'a` could be shorter. This is sound because `'static: 'a` for all `'a`.

The `From<Lazy<'static, A, Config>> for Trampoline<A>` impl (trampoline.rs:465-480) requires `'static` on the Lazy, `A: Clone`, and moves the Lazy into the closure. The `Clone` bound is needed because `Lazy::evaluate()` returns `&A` but `Trampoline::new` needs an owned `A`.

### 4.4 TryThunk Lifetime

`TryThunk<'a, A, E>` inherits lifetime handling from its inner `Thunk<'a, Result<A, E>>`. The HKT brand `TryThunkErrAppliedBrand<E>` requires `E: 'static` (line 482), which means the HKT interface cannot represent `TryThunk` with a borrowed error type. The inherent methods have no such restriction.

## 5. Type Class Law Compliance

### 5.1 Thunk: Functor Laws

**Identity:** `map(id, Thunk::pure(a)).evaluate() == a`. Since `map` at line 224-229 creates `Thunk::new(move || f((self.0)()))`, with `f = id`, this evaluates to `(self.0)()` which is `a`. Holds.

**Composition:** `map(f . g, t).evaluate() == map(f, map(g, t)).evaluate()`. Both sides evaluate the inner thunk exactly once and apply the functions. Since there are no side effects in the evaluation order (both evaluate left-to-right), this holds.

The QuickCheck tests at thunk.rs (not fully shown but referenced in imports) and trampoline.rs:782-795 verify these properties.

### 5.2 Thunk: Monad Laws

**Left identity:** `Thunk::pure(a).bind(f).evaluate() == f(a).evaluate()`. The bind at line 193-202 evaluates `self` to get `a`, then calls `f(a)` and evaluates the result. For `pure(a)`, `self.0` is `move || a`, so evaluation yields `a`, and the result is `f(a).evaluate()`. Holds.

**Right identity:** `t.bind(Thunk::pure).evaluate() == t.evaluate()`. The bind evaluates `t` to get `a`, then `Thunk::pure(a).evaluate()` returns `a`. Holds.

**Associativity:** `t.bind(f).bind(g).evaluate() == t.bind(|a| f(a).bind(g)).evaluate()`. Both sides evaluate `t` to `a`, then `f(a)` to `b`, then `g(b)`. The evaluation order is identical. Holds.

QuickCheck tests for Trampoline's monad laws are at trampoline.rs:799-820.

### 5.3 Thunk: Semimonad (HKT) vs Inherent bind

The HKT `Semimonad::bind` at line 543-548 accepts `impl Fn`, which is a subset of `FnOnce`. Since Thunk only calls the function once, any `Fn` closure works correctly. The law compliance is identical to the inherent version.

### 5.4 Lazy: RefFunctor Laws

**Identity:** `ref_map(|x| x.clone(), lazy).evaluate() == lazy.evaluate()`. The `ref_map` at line 554-561 creates a new Lazy that calls `f(fa.evaluate())`. With `f = |x| x.clone()`, the result is a clone of the original value. For types where `clone` produces an equal value, this holds.

**Composition:** `ref_map(|x| f(&g(x)), lazy) == ref_map(f, ref_map(g, lazy))`. Both create lazily-chained computations. The left side creates one level of indirection; the right side creates two. Due to memoization, the evaluation produces the same result. Holds.

### 5.5 Lazy: Deferrable Law

**Transparency:** `defer(|| x).evaluate() == x.evaluate()`. The `Deferrable` impl at line 839-843 creates `RcLazy::new(move || f().evaluate().clone())`. When `f` returns `x`, evaluating the deferred value gives `x.evaluate().clone()`. This is observationally equivalent to `x.evaluate()` for `Clone` types, which is required by the trait bound. Holds, assuming `Clone` is well-behaved.

### 5.6 Trampoline: Inherent Method Laws

Though not HKT-integrated, the QuickCheck tests at trampoline.rs:782-820 verify:
- Functor identity (line 783-785)
- Functor composition (lines 788-795)
- Monad left identity (lines 800-804)
- Monad right identity (lines 807-810)
- Monad associativity (lines 813-820)

All use `wrapping_add`/`wrapping_mul` to avoid overflow, which is a good testing practice.

### 5.7 TryThunk Laws

The monad laws for `TryThunkErrAppliedBrand<E>` hold because the bind short-circuits on `Err` (try_thunk.rs:199-202). This is the standard `Result` monad behavior:
- Left identity: `pure(a).bind(f) = TryThunk(Thunk::pure(Ok(a))).bind(f)`, which evaluates to `f(a)`. Holds.
- Right identity: `m.bind(pure) = m.bind(|a| TryThunk::pure(a))`, which maps `Ok(a)` to `Ok(a)` and `Err(e)` to `Err(e)`. Holds.
- Associativity: follows from `Result`'s associativity + Thunk's associativity.

## 6. Summary of Findings

### Correct and Well-Designed

1. **Brand definitions** are consistent across the hierarchy. Each type has the appropriate brand parameterization.
2. **Kind mappings** all use the `type Of<'a, A: 'a>: 'a` signature, ensuring compatibility with the core type class hierarchy.
3. **Thunk's type class coverage** is comprehensive: Functor, Pointed, Semimonad (Monad), Semiapplicative (Applicative), MonadRec, Evaluable, Foldable.
4. **Trampoline's exclusion from HKT** is technically necessary and well-documented.
5. **Type class laws** are satisfied, with QuickCheck property tests providing evidence.
6. **Conversion impls** (`From`) provide bridges between all lazy types, enabling composition.

### Asymmetries and Limitations

1. **ArcLazy lacks RefFunctor and Foldable at the HKT level** (only inherent methods exist). This is documented but means generic code parameterized by `LazyBrand<Config>` cannot use `ref_map` or `fold_right` when `Config = ArcLazyConfig`.
2. **TryLazy has no type class impls beyond Kind.** It only has `Deferrable`/`SendDeferrable`. No RefFunctor, no Foldable. This limits its utility in generic HKT code.
3. **TryTrampoline has no HKT integration at all**, not even a brand. Like Trampoline, this is justified by the `'static` requirement.
4. **The `'static` bound on partially-applied brands** (`TryThunkErrAppliedBrand<E>` requires `E: 'static`) restricts HKT usage to static error types, even though the underlying type supports non-static ones.

### Potential Improvements

1. **Add Foldable for `LazyBrand<ArcLazyConfig>`:** The `fold_right` operation only needs `Clone` on `A`, which could be bounded in the impl. The existing `RcLazyConfig` implementation (lazy.rs:1043-1086) would translate directly.
2. **Add RefFunctor for TryLazy:** `TryLazy` could support a `RefFunctor`-like operation (mapping over the cached success value by reference), though the error case complicates the signature.
3. **Document the `Fn` vs `FnOnce` divergence more prominently:** The difference between Thunk's inherent `bind` (`FnOnce`) and its HKT `Semimonad::bind` (`Fn`) is documented at thunk.rs:170-173, but this is a subtle point that deserves mention in the module-level docs as well.
