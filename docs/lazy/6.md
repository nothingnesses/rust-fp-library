# Analysis 6: Deferrable and Evaluable Trait Design

## 1. Overview of the Traits

The library defines three traits related to lazy evaluation:

- **`Deferrable<'a>`** (`fp-library/src/classes/deferrable.rs`, line 41): wraps a computation that produces a value into a deferred representation.
- **`Evaluable`** (`fp-library/src/classes/evaluable.rs`, line 33): extracts the inner value from a functor by running its effects.
- **`SendDeferrable<'a>`** (`fp-library/src/classes/send_deferrable.rs`, line 23): thread-safe variant of `Deferrable`.

## 2. Comparison with PureScript's `Control.Lazy`

PureScript's `class Lazy l where defer :: (Unit -> l) -> l` (line 10 of `Control/Lazy.purs`) is simple and clean. It also provides `fix :: Lazy l => (l -> l) -> l` (line 22) as a derived combinator for fixed-point recursion.

### Does `Deferrable` correctly capture this concept?

**Yes, mostly.** The Rust `Deferrable<'a>` trait maps closely:

```
PureScript:  defer :: (Unit -> l) -> l
Rust:        fn defer(f: impl FnOnce() -> Self + 'a) -> Self
```

The translation is faithful. `FnOnce() -> Self` is the Rust equivalent of `Unit -> l`, and the `'a` lifetime parameter is a necessary Rust addition. The transparency law (line 24) is correct and matches PureScript's semantics.

### Missing: `fix` as a generic free function

PureScript defines `fix` generically over any `Lazy l`. The Rust library only has concrete versions: `rc_lazy_fix` (line 735 of `lazy.rs`) and `arc_lazy_fix` (line 788 of `lazy.rs`), re-exported from `functions.rs` (lines 38-39). No `fix` exists for `Thunk` or `Trampoline`, and no generic `fix` works over arbitrary `Deferrable`.

**This is justified.** A generic `fix` requires self-referential values. In PureScript, `go = defer \_ -> f go` works because `go` is a lazy binding. In Rust, the `Lazy` implementations use `OnceCell`/`OnceLock` with `Rc`/`Arc` sharing (lines 738-749, 791-802 of `lazy.rs`). `Thunk` cannot support `fix` because it is consumed on evaluation (takes `self` by value at line 247 of `thunk.rs`), making self-reference impossible without `Clone`.

## 3. Analysis of `Evaluable`

`Evaluable` (line 33 of `evaluable.rs`) extends `Functor` and provides `fn evaluate<'a, A: 'a>(fa: ...) -> A`. It serves a specific purpose: used by `Free::evaluate` to execute effects in a free monad.

**The `Functor` superclass bound is appropriate** because `Free::evaluate` needs to `map` over the effect layer to thread continuations.

**Evaluable is not a standard FP type class.** The closest concept is a natural transformation `F ~> Id` (an "extract" operation). It essentially witnesses that `F` is a "container of exactly one extractable value." Only `ThunkBrand` implements it (line 599 of `thunk.rs`). `Trampoline` has an inherent `evaluate` but is not HKT-polymorphic. `Lazy::evaluate` returns `&A`, so it cannot implement `Evaluable`.

With only one implementor, the trait is narrow, but it is necessary because `Free<F, A>` must be generic over its effect functor.

## 4. Relationship Between Deferrable and Evaluable

These two traits are complementary but operate at different abstraction levels:

- `Deferrable` is implemented on concrete types (`Thunk<'a, A>`, `Trampoline<A>`, `Lazy<'a, A, RcLazyConfig>`).
- `Evaluable` is implemented on HKT brands (`ThunkBrand`).

This asymmetry means you cannot generically write code that defers and then evaluates using both traits together. This is a consequence of Rust's type system limitations with HKT; `Deferrable` cannot easily be defined on brands because `Self` in `defer` needs to be a concrete type. The design is pragmatically sound.

## 5. SendDeferrable Design

`SendDeferrable<'a>` (line 23 of `send_deferrable.rs`) adds `Send + Sync` bounds to the closure. The bound placement is correct: `Send + Sync` is on the closure, not on `Self`.

Only `Lazy<'a, A, ArcLazyConfig>` implements it (line 847 of `lazy.rs`), with `A: Clone + Send + Sync + 'a`.

**`SendDeferrable` should NOT be a supertrait of `Deferrable`** because not every `Deferrable` type can accept `Send + Sync` closures. The reverse (requiring `Deferrable` as a supertrait of `SendDeferrable`) is also inappropriate because `ArcLazy` intentionally does not implement `Deferrable`; the two configurations (`RcLazyConfig` and `ArcLazyConfig`) are kept separate.

The `Sync` bound on the `FnOnce` closure is unusual but necessary because `LazyLock` (used by `ArcLazyConfig`) requires `Send + Sync` on its initialization function.

## 6. Trait Bound Correctness and Minimality

All bounds are minimal and correct:
- **Deferrable:** `FnOnce() + 'a`, `Self: Sized` are all necessary.
- **Evaluable:** `Functor` superclass is needed by `Free::evaluate`; `A: 'a` is standard HKT machinery.
- **SendDeferrable:** `Send + Sync + 'a` on the closure matches `LazyLock` requirements.
- **Lazy's `Deferrable` impl** (line 812): `A: Clone + 'a` is needed because `Lazy::evaluate` returns `&A` and the `defer` impl does `f().evaluate().clone()`.
- **Lazy's `SendDeferrable` impl** (line 847): `A: Clone + Send + Sync + 'a` is needed for thread-safe storage.

## 7. Free Function Design

- `defer` (line 92): Clean, idiomatic dispatch to `D::defer(f)`.
- `send_defer` (line 72): Same pattern with `Send + Sync` propagation.
- `evaluate` (line 88): Requires turbofish `evaluate::<ThunkBrand, _>(thunk)`, which is consistent with the library's HKT conventions.
- `rc_lazy_fix` and `arc_lazy_fix` (lines 735, 788 of `lazy.rs`): Well-implemented using `OnceCell`/`OnceLock` for self-referential lazy values.

## 8. Should There Be a Combined Defer+Evaluate Trait?

**No.** The return type differences (`Thunk::evaluate` returns `A` by value; `Lazy::evaluate` returns `&A` by reference) make unification awkward. The traits operate at different abstraction levels. The current design where each type has inherent `evaluate` and optionally implements `Deferrable`/`Evaluable` is pragmatic and idiomatic.

## 9. Summary

| Aspect | Assessment |
|--------|-----------|
| `Deferrable` fidelity to PureScript `Lazy` | Faithful; law is correct. |
| Missing generic `fix` | Justified by Rust's ownership model. |
| `Evaluable` design | Sound for `Free`; `Functor` superclass is correct and minimal. |
| `Evaluable` scope | Narrow (only `ThunkBrand`); acceptable. |
| `SendDeferrable` design | Correct; bounds are necessary. |
| Trait bounds | Minimal and correct across all three traits. |
| Free functions | Well-designed; follow library conventions. |
| Need for combined trait | Not recommended. |

## 10. Potential Improvements

1. **Document `Evaluable`'s semantic contract more explicitly.** State that it witnesses "a functor containing exactly one extractable value."
2. **Consider adding `Deferrable` impl for `ArcLazy`.** Currently `ArcLazy` only implements `SendDeferrable`, so users with non-Send closures cannot `defer` into `ArcLazy`.
3. **Document why `fix` is not generic** in `deferrable.rs`, explaining that PureScript's `fix` requires lazy self-reference only achievable with reference-counted types in Rust.
4. **Fix doc typo in `evaluable.rs`, line 72.** Says "The functor instance to evaluable" instead of "The functor instance to evaluate."
